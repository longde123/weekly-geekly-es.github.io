昨天和今天的DevOops

今天我们将讨论当前，未来和DevOops 2019会议计划。DevOps每天都在变化。 您还记得2004年吗？ 我们有一位发言人，当时没有这样的术语，他曾与云一起工作。 Amazon Web Services于2006年推出。 同时在某个地方，对DevOps的第一个引用开始出现。 从那以后，整个生活已经过去了。

最近，他们再次回忆起“回归我的2007”。 想象一下这个世界，没有亚马逊上无处不在的虚拟机，没有GitHub（仅在2008年出现），没有舒适而安全的码头工人。 记住设置iptables，GNU / Linux软件包管理器，手动重建内核模块，不眠之夜。 还想去那里吗？ 如果不是这些年的飓风进展，那将是您的2019年。



技术和公司已经改变。 让我举几个例子。 例如，我们都看到了微软的出色转型-从2001年鲍尔默（Ballmer）的“ Linux is Cancer ”到广泛过渡到开源，在2018年节省了GitHub，并计划在2019年夏季引入 Linux内核，作为Windows交付的一部分。 随着所有这些运动，工程师对接收到的信息的偏好发生了变化。


2016年，谷歌出版了《 站点可靠性工程 》一书。 一方面，无论是现在还是现在，这本书都不是每个人的指南-毕竟，“ 您不是Google ”，您没有Borg，并且可能没有这样的任务。 实际上，它最初是聪明的Google PR的产品，但确实在全球范围内产生了影响。 很少有人没有听说过她。 2018年8月，她的俄语翻译与 《 网站可靠性工作手册》一起继续流行 。

会议随着形势的发展而不断变化。 在2017年出现的DevOops会议计划中反映了当时DevOps解决方案专家所占据的主要问题。 2017年的归档副本不会让您撒谎：第一项是容器，业务流程和虚拟化，包括Docker和AWS。 Docker，Docker，无处不在的Docker。 我们吸引了无休止的讨论人员，该会议由AWS Last Week的编辑Corey Quinn主持开幕。

在2018年，很明显Docker已经受够了一切。 他成为标准，他开始从各个地方来看我们。 AWS上周的新帖子开始每天出现超过60次。 围绕这些显而易见的事情召开会议不再有意义。 2018年的主题演讲由约翰·威利斯（John Willis）进行，他不仅是Docker生态系统开发总监，而且还是DevOps的始祖之一，DevOps手册和《凤凰计划》的作者之一。 令人高兴的是，John开始谈论的不是调优，而是谈论DevOps作为组织文化的实现-这个话题一直被人们遗忘，并被新技术的聪明玩具所分散。

2018年的第二个主题是Kubernetes。 如何使用它，如何实现它，是否值得做。 这个主题贯穿整个程序的红线，Kubernetes不在标题中，如果没有，则不行，它出现在幻灯片上。

您好，2019年。Kubernetes与Docker一样，已经成为标准。 热战结束了，早期介绍的受害者从眼前消失了，只有他留在战场上。 着眼于新国王，所有新项目都是以一种或另一种方式完成的。

随之而来的问题是：今年的DevOops会议应该说些什么？ 这是计划委员会当前正在研究的一个开放性问题。

会议程序可以两种方式呈现。 首先，您可以立即呈现一个现成的报告表格，然后说-看它有多酷。 这会产生令人赞叹的效果，导致快速购买门票，但并不总是完全响应访客的请求。

例如，最近有一个朋友在VKontakte上给我写信，要求我告诉程序内容。 “但是在会议还剩下六个月之后，您为什么需要这个？” 事实证明，在她的公司里，习惯上事先写一首关于领导力的诗，主题是“我为什么要去参加会议”。 而且由于公司规模很大，所有事情都需要提前计划，所以您需要写六个月的书。 她绝对不适合“程序将在开始前一个月”形式的选项。

这似乎是一个特例，但是从许多此类特例中可以形成整体图景。 还有另一种方法：您可以上传小片段的更新，而不是程序的最终快照。 这有点敏捷和精益。 如果您还记得，这里有一个价值流映射的概念，尽管它并不完全适用于使用Habré报告的公告，但也有类似的东西。 例如，如果我们扔出太多文本，您将根本没有时间阅读它，并且您需要以某种方式记住您阅读的内容，直到下次。 报告的说明会不时进行调整和更新，发言人在处理报告时会更改主题的名称，这种信息流不容易理解，甚至更是如此-将其转变为“我为什么需要这次会议”。 换句话说，在程序充满时分块发布是一种祝福。

现在我们已经在网站上发布了前几位肯定会在2019 DevOops上发表的演讲者，很快就会出现报告主题。 为了不丢失任何内容，您可以阅读我们在Habré上的博客或订阅邮件列表（为此，您需要转到会议网站并单击那里的“订阅”按钮）。 如果您突然想自己报告，您仍然有机会 。

然而，这些报告将是关于什么的呢？ 查看会议说明如何更改。 Kubernetes仍然处于第一位，但不是作为独立学科，而是作为Cloud Native运动的一部分，仅次于Helm，Istio和服务网格。 请注意，“可观察性”一词排在第三位（例如，在Mike Julian撰写的《 实用监控 》一书中，该词尚未使用，一年过去了-现在）。 报告将大致朝这个方向。 当然，有关Docker和Kubernetes的旧主题也将跳过，但在更高层次上。

我对此也有想法。 例如，对我而言，DevOps一直主要是一种方法论和文化，而不是一组调整。 您在客户的服务器上推出了企业Java Web应用程序，但没有成功推出，出现了问题，您需要立即调整方向并修复所有问题。 而且最好不要将产品推向产品梦a。 那是关于这次会议的内容-如何不与他人和发行者做噩梦。 第二个问题是将要使用的工具，我们很酷，可以解决任何问题。 我本人希望获得更多有关文化和做事方式的报告-很好，即使在最初的发言人名单中也有这样的人（例如，安东·魏斯，巴鲁克·萨多古尔斯基，罗曼·沙波斯尼克）。
