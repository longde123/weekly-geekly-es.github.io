Cómo escribir código fácil de describir

Hola Habr!


¿Con qué frecuencia tuvo que usted o sus colegas no podían describir su propio código con un par de frases?


Les traigo a su atención una traducción del artículo "Cómo escribir código fácilmente descripble" de Cedd Burge , en el que comparte su consejo sobre cómo evitar tales situaciones.



Cuando el código es difícil de describir con palabras, la mayoría de las personas tienen que usar la imaginación para hacerlo. Por esta razón, se desperdicia energía mental y aumenta el riesgo de errores en la interpretación del código. Diferentes personas aún percibirán las palabras a su manera, lo que generará confusión al discutir el código.


Como regla general, tales discusiones se convierten en terreno fértil para los errores que surgen debido a malentendidos, y la corrección de estos errores a menudo conduce a la aparición de nuevos errores por las mismas razones. Al final, obtenemos un código incomprensible con el que nadie quiere trabajar.


Ejemplo de código indescriptible


Puede pensar que el código es solo un lenguaje escrito. Si el código parece simple, entonces se puede leer, describir y comprender fácilmente. Sin embargo, este no es siempre el caso.


La decisión general a continuación determina si un año es bisiesto. 


(divisibleBy(4) and not divisibleBy(100)) or divisibleBy(400)

Este es un código simple. Llama a funciones 3 veces, tiene 3 operadores (y, o, no) y dos niveles de anidamiento.


Pero creo que si tiene un segundo para describir este algoritmo usando palabras, tendrá dificultades.


¿Quizás "un año es bisiesto si es divisible por 4 y no divisible por 100, o divisible por 400"?


El problema es que, en palabras, a diferencia del código, no hay corchetes. Por lo tanto, es difícil describir adecuadamente la condición en palabras y si la palabra "es divisible por 400" se refiere a "divisible por 4" o "no divisible por 400". Para solucionar este problema, puede especificar corchetes con la mano o hacer pausas más largas entre las condiciones, pero la probabilidad de un error seguirá siendo grande.


Refactorizando el código descrito


Primero podemos describir las condiciones en palabras y solo luego reducirlas, haciéndolas tan claras y concisas como sea posible. Comencemos así:


"400 años es un caso único. Si un año es divisible por 400, entonces este es un año bisiesto. 100 años también es un caso único. Si un año es divisible por 100, entonces no es un año bisiesto a menos que sea divisible por 400, entonces la prioridad para el caso especial es "400 años". Si no hay casos especiales, este año es bisiesto, siempre que se divida entre 4 ".


Parece claro, pero no conciso, por lo que acortaremos un poco el texto:
“Si el año se divide por 400, entonces es un año bisiesto. Si se divide entre 100, este es un año normal, pero cuando se divide entre 4, este es un año bisiesto ".


Si convertimos estas palabras en código, obtenemos algo como lo siguiente: 


 if divisbleBy(400): return LeapYear elif divisbleBy(100) return NormalYear elif divisbleBy(4): return LeapYear else: return NormalYear

Conclusiones


El código difícil de entender es común para casi todos los programadores. Nos ayudaremos a nosotros mismos y a nuestros colegas si escribimos código que se describe fácilmente en palabras.
Y lo más importante, escribir código de esta manera es más fácil, ya que no hay un esfuerzo mental vano. El "truco" es que el algoritmo se describe primero con las palabras que luego se usarán para escribir el código.


En muchas organizaciones, el código ya ha comenzado a describirse en palabras en pruebas de aceptación o historias de usuarios, lo que puede mejorar la productividad.
