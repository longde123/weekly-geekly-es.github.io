Cadangan cloud-to-cloud: apa itu dan mengapa diperlukan


Masuk dan keluar, keluar ...

Aplikasi yang berjalan di cloud hanya dilindungi sampai batas tertentu. Untuk sepenuhnya melindungi data yang dihasilkan oleh aplikasi cloud, Anda harus mencadangkan ke cloud lain (cloud-to-cloud backup).

Menurut Forrester, biaya layanan cloud publik akan meningkat pada tahun 2020 menjadi 236 miliar dolar AS. Tren ini diperkuat oleh peningkatan jumlah aplikasi yang dihosting di awan.

Mengkonsumsi layanan cloud terkadang sangat mudah sehingga pelanggan dan tim TI dipandu oleh prinsip "berfungsi - jangan sentuh" â€‹â€‹dan senang bergantung pada masalah perlindungan data dan cadangan ke penyedia.

Jadi mengapa kita melihat perlunya salinan cloud-to-cloud?

Risiko Keamanan Informasi Cloud


Memindahkan aplikasi, beban kerja, atau infrastruktur TI ke cloud memiliki manfaat dan risiko tambahan. Bermigrasi ke cloud berarti mentransfer sejumlah besar kemampuan penyimpanan dan perlindungan data ke pihak ketiga.

Layanan cloud akan atau harus memiliki beberapa pusat data dan beberapa gudang data yang berlebihan untuk memastikan kesinambungan bisnis dan kemampuan pemulihan data. Ini juga harus memastikan keamanan data di tingkat perusahaan.

Tetapi layanan cloud juga dapat macet. Mereka relatif jarang, tetapi CIO yang telah kehilangan pentingnya backup cloud membahayakan keamanan informasi di organisasi mereka.

Untuk perusahaan yang menggunakan cloud, pertanyaannya bukanlah apakah layanan cloud akan gagal, tetapi bagaimana dan seberapa sukses bisnis akan mengatasi masalah jika terjadi bencana ini. Meskipun layanan cloud dapat memberikan tingkat ketahanan yang tinggi, ini tidak akan cukup untuk memenuhi kebutuhan cadangan organisasi mana pun.

Tingkat layanan berawan vs di tempat


Layanan cloud melakukan yang terbaik untuk menjaga layanan tetap berjalan. Tentu saja, CIO perlu memeriksa rincian perjanjian tingkat layanan (SLA) . Beberapa cloud publik sama sekali tidak menjamin ketersediaan atau waktu pemulihan, tetapi hanya menawarkan tindakan "upaya terbaik".

Ketika datang ke data itu sendiri, perusahaan berada pada risiko yang lebih besar. Perangkat Lunak sebagai Penyedia Layanan (SaaS) biasanya bertanggung jawab atas ketersediaan infrastruktur, tetapi kehilangan data semata-mata menjadi tanggung jawab klien.

Ini dapat mengarah pada fakta bahwa dalam masa-masa sulit pelanggan akan dibiarkan berhadapan muka dengan pemulihan data yang kompleks, mahal dan memakan waktu setelah kegagalan.

Selain itu, penyedia layanan cloud tidak bertanggung jawab atas penghapusan data yang tidak disengaja. Kesalahan manusia - mulai dari penulisan ulang satu bidang secara tidak sengaja di basis data klien hingga menghapus seluruh kumpulan data - adalah masalah klien. Menurut Backupify, penyedia cadangan berbasis cloud, satu dari tiga perusahaan kehilangan data saat menggunakan SaaS. Kesalahan manusia adalah penyebab paling umum.

Berdasarkan persyaratan kontrak, penyedia cloud juga dapat menghapus data pelanggan yang berlangganan layanan telah berakhir. Misalnya, Microsoft menghapus semua data pengguna 30 hari setelah penghentian berlangganannya. Jika bisnis tidak memiliki rencana yang dapat diandalkan untuk mengumpulkan file karyawan ketika mereka meninggalkan organisasi, data vital dapat hilang selamanya.

Opsi pencadangan cloud


Dalam skenario skala kecil, pengguna dapat menyalin file, misalnya, dari Office 365 dan G Suite, ke volume lokal atau, jika diizinkan oleh aturan keamanan, ke drive eksternal. Tapi ini adalah proses manual yang bisa diandalkan dan sulit untuk diukur.

Untuk file besar dan aplikasi yang lebih besar, ini jarang praktis. Bisnis yang menggunakan infrastruktur sebagai layanan (IaaS) atau cloud SaaS dapat menggunakan antarmuka sistem aplikasi prosedural (API) atau perangkat lunak pihak ketiga untuk membuat cadangan ke server lokal, ke penyimpanan yang terhubung jaringan (NAS), atau ke pusat data mereka sendiri.

Namun, membuat cadangan data dari layanan cloud ke penyimpanan lokal adalah langkah mundur. Alih-alih menggunakan cloud, itu memaksa perusahaan untuk mempertahankan atau memperluas infrastruktur lokal mereka, meningkatkan biaya dan membatasi fleksibilitas.
Perusahaan yang mencadangkan aplikasi SaaS akan yakin bahwa mereka memiliki salinan data mereka, tetapi mereka tidak akan dapat meluncurkan lingkungan SaaS di dalam perusahaan. Ini membatasi kegunaan cadangan lokal. Dalam kasus terbaik, bisnis akan menghadapi pemulihan panjang atau transisi ke platform baru.

Mencadangkan data layanan cloud ke cloud mungkin merupakan opsi terbaik.

Menurut Gartner, saat ini hanya satu dari 10 perusahaan yang membuat cadangan data mereka dengan penyedia IaaS. Para peneliti berharap angka ini menjadi dua kali lipat pada tahun 2020, karena perusahaan menyadari pentingnya cadangan, dan semakin banyak penyedia menawarkan layanan cloud yang serupa.

Manfaat Pencadangan Cloud-to-Cloud


Cadangan cloud-to-cloud menjanjikan beberapa keuntungan bagi bisnis dibandingkan dengan cadangan lokal dan penawaran SaaS, termasuk biaya infrastruktur yang lebih rendah, pencadangan dan pemulihan yang lebih cepat, dan fleksibilitas yang lebih besar.

Seperti kekuatan komputasi infrastruktur cloud itu sendiri, cadangan cloud tersedia di mana saja. Organisasi juga dapat menggunakan cadangan untuk penambangan data dan analitik tanpa risiko sumber data.

Awan jatuh


Sayangnya, bagi para pembuat keputusan TI, pasar cadangan cloud-to-cloud terfragmentasi dan relatif belum matang. Mengingat berbagai penawaran komputasi awan, layanan cadangan sangat bervariasi dalam kemampuan.

Gartner mencatat bahwa penyedia layanan cadangan telah meningkatkan dukungan cloud selama dua tahun terakhir, membuatnya lebih mudah untuk melindungi data IaaS dan mencadangkan aplikasi SaaS. Area paling berkembang dalam beberapa tahun terakhir - mendukung infrastruktur TI sebagai layanan - salinan cloud dari semua mesin virtual.

Untuk mencadangkan cloud pribadi, gateway cloud adalah opsi. Cloud pribadi menggunakan API layanan cloud untuk menyediakan cadangan dan pemulihan data. Pemasok juga menyediakan cadangan virtual dan aplikasi deduplikasi yang dapat berjalan di cloud.

Pasar bergerak dari menyediakan â€œtempat pembuangan basis dataâ€ sederhana ke penyimpanan cloud - ke layanan yang lebih canggih dan dapat dikelola yang disediakan berdasarkan pembayaran saat Anda bepergian.

Jika rencana Anda gagal, rencanakan kegagalan Anda sendiri


Namun, ada jebakan. Mencadangkan aplikasi SaaS secara khusus tetap sulit.

Misalnya, cadangan Salesforce.com dari data mereka sendiri bukan salinan tepat dari data internal aplikasi. Ini disebut cadangan "satu arah" yang dapat melewati metadata, dan informasinya mungkin kurang "kaya" daripada kumpulan data asli. Mereka dapat dikembalikan kembali ke aplikasi SaaS, tetapi data pertama-tama harus diformat ulang dan dimuat ulang sebagai set data baru.

Dalam kasus banyak aplikasi SaaS, jika layanan asli tidak tersedia, pelanggan tidak akan dapat menggunakan data yang dipulihkan.

Cadangan aplikasi yang di-host oleh model IaaS jarang benar-benar platform independen. "Ada beberapa perbedaan antara format data pada platform utama untuk membuat pemulihan menjadi tugas yang solid," memperingatkan analis Forrester, Naveen Chhabra.

Pembatasan cadangan cloud berlaku untuk aplikasi pribadi untuk meningkatkan kinerja. Beberapa aplikasi di Office 365, khususnya, sulit untuk dicadangkan. Misalnya, Sharepoint hanya didukung oleh beberapa penyedia layanan cadangan.

Adapun cadangan aplikasi perusahaan, mereka harus bekerja baik pada infrastruktur lokal, dan di awan publik atau pribadi. Tim TI harus dapat memulihkan data mereka dari penyedia apa pun yang mendukung aplikasi, pada mesin virtual yang berjalan di cloud, atau di pusat data lokal. Organisasi harus melakukan pemeriksaan cadangan sendiri dan mengujinya.
