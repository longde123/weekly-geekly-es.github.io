La empresa Zfort Group ha decidido no renovar la suscripción corporativa a Habré.
Pero hay buenas noticias:


Nos gustaría anunciar el lanzamiento del sitio actualizado zfort.com.ua , hablar brevemente sobre algunas de las características técnicas de la creación del sitio y también informar sobre la decisión de transferir la publicación de resúmenes de Habr al blog del nuevo sitio. En el blog de nuestro nuevo sitio, continuaremos publicando resúmenes, artículos y anunciando reuniones profesionales. Puede suscribirse a todas las actualizaciones y publicaciones para mantenerse informado y no perderse nada.


Una versión corta de los resúmenes también permanecerá en Habré, pero no se publicará en el blog corporativo del Grupo Zfort, sino en la cuenta de alexzfort .


Uno de los objetivos que nos fijamos es actualizar nuestro sitio dirigido a un público local, para que sea rápido y fácil. Complemente el sitio con secciones para mostrar las últimas noticias de la compañía, anuncios de eventos en curso y publicar resúmenes / artículos en poco tiempo con la máxima flexibilidad y la capacidad de expandirse.


Y ahora más sobre los detalles de la creación de un sitio actualizado zfort.com.ua


Inicialmente, se consideraron tres áreas principales de desarrollo del sitio actualizado:


  1. Crea un sitio basado en WordPress;
  2. Desarrolle un sitio desde cero en la pila LAMP basado en nuestra propia experiencia;
  3. Implemente un sitio basado en el enfoque no convencional de JAMstack. Este es un enfoque alternativo que ofrece muchas herramientas para elegir para crear sitios estáticos, pero con contenido agregado y administrado dinámicamente. Esta opción implicaba el desarrollo de un sitio basado en el uso de servicios existentes, combinado en un ecosistema único para administrar y publicar contenido.

Después de sopesar las posibles opciones, se decidió ir por el tercer camino y abandonar el lado del servidor personalizado en Symfony o WordPress en favor de las tecnologías sin servidor, lo que permitiría obtener las siguientes ventajas:


  • reducir el tiempo de desarrollo, prueba, soporte adicional y corrección de errores;
  • reducir el tiempo de implementación de entornos, configuración y soporte de servidor;
  • Obtenga una infraestructura de proyecto nativa de la nube, en lugar de tener una solución local / autohospedada;
  • construir todo el proyecto en forma de un ecosistema de microservicios interactivos, en lugar de la arquitectura monolítica clásica;
  • Obtenga un sistema de administración de contenido flexible, sea capaz de estructurar contenido, portar contenido a otro sistema, etc.

La opción elegida para el desarrollo del sitio zfort.com.ua implicaba el máximo posible (con algunas limitaciones) evitando el servidor web tradicional y, en cambio, construyendo todo el sistema basado en tecnología Serverless, que involucra varios servicios interactivos.
Además, durante la planificación y el desarrollo, perseguimos el objetivo: obtener una solución en la que la interfaz se desate al máximo desde el back-end del sitio.


Los principales componentes subyacentes al sitio:


  1. CMS basado en API sin cabeza (SaaS) para administrar el contenido del sitio;
  2. Generador de sitio estático;
  3. Servicio de manejo de formularios o funcionalidad;
  4. Repositorio de código fuente y sistema de control de versiones;
  5. Plataforma de implementación y alojamiento: una plataforma capaz de escuchar las notificaciones de webhooks, comenzar a construir e implementar sitios para la producción;
  6. Plataforma de entrega de correo electrónico.

De acuerdo con cada uno de los componentes, se decidió utilizar:


  1. Contentful.com - como CMS basado en API sin cabeza
  2. GatsbyJS: como marco para generar páginas de sitio HTML estáticas;
  3. Funcionalidad personalizada PHP (Symfony) como manejador de formulario de sitio;
  4. GitLab como repositorio del código fuente del proyecto y el sistema de control de versiones;
  5. Rackspace: como proveedor de alojamiento;
  6. SendGrid: como un sistema para enviar correos electrónicos, así como un sistema para almacenar y administrar plantillas de correo electrónico.

Además, la gestión de las vacantes que se muestran en el sitio se lleva a cabo directamente en el sistema Zoho Recruit.


El servicio que se eligió como alternativa al desarrollo personalizado de la parte administrativa de la administración de contenido para zfort.com.ua es ontentful.com . Este producto es una plataforma de gestión de contenido con su panel de administración. Algo así como el CMS en la nube, pero sin una interfaz de usuario. Solo hay un panel de administración.




La obtención de contenido para mostrar en las páginas del sitio se lleva a cabo a través de la API Contentful. En nuestro caso, utilizamos el complemento gatsby-source-contentful para obtener los tipos de contenido, entradas y activos en Gatsby desde espacios Contentful.
Además de la infraestructura de administración de contenido, Contentful brinda la capacidad de cambiar el tamaño, recortar y comprimir imágenes a través de la API de imágenes.


Ventajas del enfoque elegido:


  • Rendimiento y velocidad de visualización del sitio en el navegador. Como resultado de la generación de la página de tiempo de ejecución, el HTML se almacena en caché y se muestra al usuario casi instantáneamente. Además, es posible transferir completamente el sitio a CDN, lo que, a su vez, además de aumentar la velocidad del trabajo, le permite escalarlo con las herramientas de CDN;


  • Arquitectura distribuida que le permite reemplazar cualquiera de los componentes del sitio en cualquier momento con un esfuerzo relativamente mínimo. Todos los servicios están separados entre sí, donde cada servicio es una aplicación con su propia lógica. La lógica no se mezcla, los equipos de desarrollo de cada módulo pueden trabajar casi independientemente uno del otro;


  • Seguridad De hecho, un sitio es una colección de archivos estáticos pregenerados. En general, todas las funciones dinámicas del sitio se asignan a servicios independientes de terceros, cuya interacción ocurre a través de la API y solo en el momento de generar o actualizar el contenido. Como resultado, no hay complementos donde pueda encontrar la vulnerabilidad. Este enfoque también mitiga el riesgo de ser pirateado mediante inyección SQL;


  • Portabilidad fenomenal . Los requisitos de alojamiento para el sitio generado son mínimos. Un sitio estático se puede alojar en cualquiera de los sitios de alojamiento que permiten almacenar archivos estáticos;


  • Velocidad de desarrollo y falta de necesidad de soporte de código constante. Todos los archivos de configuración necesarios para generar páginas del sitio, así como las plantillas para generar páginas HTML, se almacenan en GitLab. Basado en la oportunidad de reutilizar soluciones preparadas como algunos microservicios que interactúan entre sí. Como se mencionó anteriormente, como parte administrativa para llenar el contenido, utilizamos Contentful, el trabajo con vacantes se lleva a cabo en el sistema Zoho Recruit.



Sin embargo, hay algunas dificultades con este enfoque:


  • La complejidad de sincronizar y actualizar el contenido del sitio desde sistemas de terceros. El problema se resuelve con la configuración correcta de los webhooks, así como con el implementador y su manejo cuidadoso;


  • Puede haber problemas para conectar bibliotecas JS de terceros cuando el lado del servidor genera páginas HTML en un sitio. El problema está resuelto, aunque con algunos matices;


  • La falta de un servidor tradicional presenta sus propias dificultades e impone restricciones en los casos en que, por ejemplo, es necesario implementar una búsqueda. Sin embargo, el problema de búsqueda también se puede resolver utilizando servicios de terceros como Algolia, etc.



El siguiente diagrama muestra esquemáticamente la arquitectura del sitio zfort.com.ua , así como la interacción de los componentes principales.



GatsbyJS trabaja con datos utilizando el protocolo GraphQL: cuando compilamos, obtenemos datos de la API de servicios de terceros y creamos una base local de nodos GraphQL.


Además, las consultas GraphQL se escriben en los componentes para obtener los datos necesarios en un lugar en particular y el HTML se construye sobre la base de plantillas jsx.


Para trabajar con Zoho Recruit, se escribió un complemento Gatsby personalizado, que recibe datos por API y agrega los datos necesarios a la base de datos GraphQL.


En general, si es necesario, puede negarse a usar el sistema de administración de contenido y administrar el contenido del sitio cambiando los archivos .md ubicados directamente en la carpeta del sitio en el alojamiento de archivos estáticos. Todos los datos se agregan preliminarmente a GraphQL y, como resultado, puede buscar, ordenar y la mayoría de las otras operaciones de agregación y búsqueda más simples.


Para procesar las solicitudes de los formularios, se creó un servicio personalizado separado usando Symfony 4. Para poder ver los datos recibidos de varios formularios del sitio, se creó una aplicación usando Symfony 4 (SonataAdminBundle).
