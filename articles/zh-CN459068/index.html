Windows Server 2008 R2-国王死了，国王万岁



哈Ha！ 即将于14/01/2020到期，并且该日期标志着对Windows Server 2008 R2的扩展支持的结束，并且，如果您仍在使用Windows Server 2008 R2，则应考虑迁移到更新的平台。

削减部分是对旧的和新的Windows Server平台消耗的资源的小比较-RDSH的作用。 首先，我对有机会继续使用运行VMware Vsphere 5.5的HP G6 / G7服务器感兴趣。

有关Windows Server生命周期的信息，请访问此搜索产品生命周期链接。

我将有两个测试台：


  1. HP DL120 G7,1 * CPU英特尔至强E3-1240
  2. HP ML350 G6，2 * CPU英特尔至强E5620

这些服务器将在VMware ESXi 5.5（VMware-ESXi-5.5.0-Update1-1746018-HP-5.75.4-Dec2014的映像）下运行。 我还尝试使用VMware-ESXi-5.5.0-Update3-3116895-HP-550.9.4.26-Nov2015映像，但这不会影响结果。

但是，VMware-ESXi-5.5.0-Update2-2403361-HP-550.9.2.40.2-Sep2015的映像给出了失真的结果，由于某些原因，服务器显示的统计信息与实际的统计信息相差两倍。 使用此图像，服务器显示的最大值为处理器负载的50％，显然是某种错误。 起初，当我获得第一个结果时感到很高兴，但后来我意识到出了点问题...

还应该提到的是，对VMware Vsphere 5.5的支持也不是永恒的，并且该平台也不再适用，因此您应该考虑进行更新。 该VMware Lifecycle产品列表中提供了有关VMware Lifecycle的信息。



并且不要忘记《 VMware兼容性指南》 。 如果我们谈论的是HP DL120 G7服务器，则兼容性矩阵中永远不会出现该服务器，但是ESXi 5.5 U3被正式允许安装在HP ML350 G6服务器上。

来宾操作系统


《 VMware兼容性指南》中也提供了有关ESXi与来宾操作系统的兼容性的信息 。

ESXi 5.5 U3允许您将Windows Server 2016安装为来宾OS，即使版本5.5也足够。 但是要安装Windows Server 2019，您将需要ESXi 6+版本，但这不会阻止我测试我的计划。

我将拥有以下测试VM：WS2008R2SP1，WS2012R2，WS2016和WS2019。

VMware Tools 5.5.0-10.2.5.8068406的代理已安装在所有VM上。 已安装Microsoft于2019年5月发布的所有更新，甚至是2019年6月的某些更新，此后禁用了更新服务。

对于我的测试，我执行了最小调整，我希望所有功能都可以使用最标准的参数进行工作。 为此，我将使用本地组策略，最后我会写一些有关此的更多信息。

  1. 禁用“桌面组合”
  2. 允许重定向音频和视频
  3. 为WS2008R2SP1和WS2008R2SP1客户端启用了RemoteFX
  4. 禁用字体平滑
  5. 将调色板的位深度限制为32
  6. 将监视器的数量限制为1

为了使RemoteFX编解码器在WS2008R2SP1上开始工作，必须安装RDSH角色；对于较新的版本，可以省略此角色。 也许在WS2008R2SP1上使用RemoteFX而不安装RDSH，但是日志中没有有关此编解码器激活的事件。

测试1


作为测试，我将在YouTube上使用分辨率为720p的视频， 这是Windows Server 2019的远程桌面服务中的新增功能-BRK2242 。

在每台服务器上，我将在Chrome（版本75.0.3770.80（64位））中打开一个rdp会话和一个测试视频。

在此测试中，将使用运行HP ThinOS 4.4 OS和Freerdp-1.1hp10d-all-4.4-x86-SQ软件包的HP t510瘦客户端作为客户端。 本机连接到具有全高清分辨率的监视器，并且瘦客户端配置文件已重设为出厂设置。

我从主机拍摄了图表，所有第一个测试都将在HP DL120G7服务器上执行。


对第一个测试结果的评论：

在此测试中，ws2008消耗的处理器资源减少了三倍，但同时的流量却比测试中的其他参与者多了三倍。

在当前流量的情况下，可以在一个千兆位中放置66个会话，但是切换到新版本的服务器可能会将可以处理此处理器消耗的用户数量减少三倍。

视觉上，此演示文稿中的图片在所有版本的Windows Server上都是可以接受的。 当然，如果您运行的是动态性更高的服务器，则较新版本的服务器会显示出更好的画面。 我想这是由于帧数的原因，更多的帧=流畅的播放=更多的CPU负载。

除了使用Firefox（版本67.0.2（64位））外，第二项测试的脚本保持不变。


对第二项测试结果的评论：

与ws2019搭配使用的Firefox表现出良好的效果，与ws2008相比，消耗更多的处理器并显着减少流量。

此测试将使用Chrome，但客户端将已经是具有Windows 10 1903的笔记本电脑，该笔记本电脑已连接到具有FullHD分辨率的显示器。


对第三项测试结果的评论：

在使用ws2008的情况下，消耗的流量较少，其他更改并不重要...

我还注意到，在视频背景下显示时，Chrome进度条会消耗1000 MHz以上的频率，但是在Firefox中，我没有注意到这种效果，绘制进度条不会影响消耗的资源量。

测试4


装有Windows 10和Firefox的笔记本电脑


对第四项测试结果的评论：

在这种服务器-客户端-浏览器组合中，ws2019 + Firefox消耗的资源少于Chrome的资源，但是与Test 2的结果相比，它开始消耗的处理器明显更多。

在ws2012和ws2016的情况下，执行的所有测试的结果要均匀得多，没有ws2019这样的变化。

对第五项测试结果的评论：

出乎意料的是，2019年服务器的结果比ws2012和ws2016更差。 Server 2008可以打开8个会话，而Server 2012/2016可以打开4个会话，第5个将负载提高到100％。 Server 2019最多可以使用4个会话。

Firefox对我来说不是优先浏览器，因此我仅在ws2019中运行它来验证收到的数字。

测试6


在此测试中，具有两个E5620处理器的HP ML350G6服务器将发挥作用，我将仅测试2008和2019年服务器。 测试是一样的，开始最大会话数，仅使用Chrome。


对第六项测试结果的评论：

要在服务器2008上运行10个会话，需要60％的处理器资源，而要在2019服务器上仅启动5个会话，则需要90％以上。

最后一部分-调优


我将所有测试虚拟机返回到HP DL120G7服务器，并使用一台具有Windows 10和其本机屏幕（分辨率为1366 * 768）的笔记本电脑作为客户端。

不幸的是，我不会成功地尝试减少处理器消耗，但是也不能说也没有结果。 以下是可用于调整的所有可用本地组策略设置的列表：


在Windows Server 2008R2中，“使用RemoteFX时优化视觉体验”选项使您可以控制图像质量和帧数，这将带来明显的效果。 减少帧数可减少处理器和网络上的负载。 对于Windows Server 2019，这些选项不起作用。


Windows Server 2019中的图片质量可以通过“配置RemoteFX自适应图形的图像质量”参数来控制，但是没有更改任何数量的帧，或者我找不到这样的参数。

我尝试了各种参数组合，但是仅在少数情况下，我可以在结果中得到明显的改变。 最有趣的是，您可以注意到流量的变化，但是与此同时，处理器的负载变化很小。

-H.264 / AVC 444的优先级提高了处理器的负载，但与此同时，流量从2倍显着下降到1兆位。
对于非RemoteFX vGPU方案，此策略设置会优先考虑H.264 / AVC 444图形模式。 当您在RDP服务器上使用此设置时，服务器将使用H.264 / AVC 444作为RDP 10连接中的编解码器，客户端和服务器都可以使用H.264 / AVC 444。

-禁用RDP压缩并不能减少处理器的负载，但同时流量从2倍增加到7兆位。

-为图像质量启用“丢失”参数会导致流量消耗不足，而不会更改处理器的负载。 在什么情况下，添加到我的选项尚不清楚。
如果启用此策略设置并将质量设置为“无损”，则RemoteFX Adaptive Graphics将使用无损编码。 在这种模式下，图形数据的颜色完整性不会受到影响。 但是，此设置导致网络带宽消耗显着增加。 我们建议您仅针对非常特殊的情况进行设置。




总结


如果早些时候，对于10个活动用户的操作，8台核服务器就足够了，仍然有可用空间，现在这样的服务器仅够5个用户使用。

我认为是时候准备更换HP G6和G7代服务器了。 对于安装了大于100 GB RAM的服务器，我感到很遗憾，我认为它们会继续提供服务。

我不排除旧处理器缺乏硬件说明，因此我计划在不久的将来购买一台现代化的服务器进行测试。 但是说实话，我的预测是悲观的，我认为仅靠蛮力就能获得预期的结果，处理器核的数量会增加，这将导致所需许可证数量和成本的增加。
