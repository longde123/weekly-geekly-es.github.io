Pourquoi (aujourd'hui) retourner 444 n'est pas toujours utile

Le serveur Web Nginx possède un merveilleux code de réponse 444 qui "ferme" la connexion sans envoyer de données. Cette fonctionnalité est très utile pour filtrer le trafic parasite - si nous sommes sûrs que le client n'est pas valide selon certains critères, il n'est pas nécessaire de le notifier, par exemple, avec la réponse 403e. Il est plus efficace d'arrêter simplement le transfert de données, ce qui, souvent, peut réduire considérablement la charge sur le serveur.

Les recommandations pour l'utilisation de ces réponses se trouvent partout dans les instructions sur le blocage des clics sur les liens des sites populaires et le spam de référence, la protection contre les DDoS, etc.

Et, en général, au fil des ans, ces conseils pourraient être utilisés presque sans regarder, mais ... les navigateurs modernes ne s'arrêtent pas et nous présentent périodiquement de nouvelles surprises.

Programme obligatoire
Serveur: 

$ uname -orm FreeBSD 11.1-STABLE amd64 

 $ nginx -v nginx version: nginx/1.15.0

Client: 

 >ver Microsoft Windows [Version 10.0.15063]

Google Chrome 67.0.3396.99 (version officielle), (64 bits)
Firefox Quantum 61.0 (64 bits)

Je n'ai pris pour expérience que les navigateurs Chrome et Firefox. Non pas parce que les autres sont sans péché, juste le comportement décrit ci-dessous était assez stable dans les deux.

Donc, nous avons un site Web et nous voulons restreindre l'accès à l'emplacement sans aucune condition et sans transmettre de données: 

 server { ... location = /code/444 { return 444; } ... }

En attente:

  • le navigateur enverra une demande;
  • le serveur reçoit une demande, ferme la connexion et écrit une ligne avec le code 444 dans le journal;
  • ne reçoit pas de données, le navigateur affichera une erreur et cessera de communiquer avec le serveur.

Cap de spoiler


Ainsi, en attendant de recevoir une seule demande du client, en fait, nous recevons 3 à 20 demandes. En plus des appels qui nous sont complètement inutiles, il y a un risque d'alimenter un tel journal, par exemple, vers un script anti-DDoS et de bloquer cette ip. Et cela semblerait la moitié du problème, parce que nous lui avons déjà «donné» 444, c'est-à-dire que nous ne sommes pas désolés pour lui, mais cela peut se produire dans l'autre sens - nous montrerons au client ce que nous n'avions pas prévu du tout.

Modifiez la configuration: 

 map $http_referer $code_if { "~*https://habr.com/post/415565/" 1; "http://tison.ru/ref" 1; default 0; } ... server { ... location = /code/444 { return 444; } ... location = /codeif/444 { if ( $code_if = 1 ) { return 444; } add_header "Content-Type" "text/html; charset=UTF-8" always; return 200 "Expected code 444"; } ... }

Ici, nous avons interdit la livraison de contenu à tous les clients avec un référent "
https://habr.com/post/415565/ "lors du passage à la page tison.ru/codeif/444
