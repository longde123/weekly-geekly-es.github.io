Docker de aprendizagem Parte 6: Trabalhando com dados

Na edição de hoje da série Docker, falaremos sobre como trabalhar com dados. Em particular, volumes Docker. Nesses materiais, comparamos constantemente os mecanismos de software do Docker com várias analogias comestíveis. Não vamos nos afastar dessa tradição aqui. Os dados do Docker devem ser condimentados. Existem muitos tipos de especiarias no mundo e, no Docker, existem muitas maneiras de trabalhar com dados.

Os dados do Docker podem ser armazenados temporariamente ou permanentemente. Vamos começar com os dados de tempo.

Armazenamento temporário de dados


Há duas maneiras de organizar o trabalho com dados temporários nos contêineres do Docker.

Por padrão, os arquivos criados por um aplicativo em execução em um contêiner são salvos em uma camada de contêiner habilitada para gravação. Para que esse mecanismo funcione, você não precisa configurar nada especificamente. Acontece barato e com raiva. Um aplicativo só precisa salvar dados e continuar a fazer suas próprias coisas. No entanto, depois que o contêiner deixar de existir, os dados armazenados de maneira simples desaparecerão.

Para armazenar arquivos temporários no Docker, você pode usar outra solução adequada para os casos em que é necessário um nível mais alto de desempenho comparado ao que é possível obter usando o mecanismo padrão para armazenamento temporário de dados. Se você não precisar que seus dados sejam armazenados por mais tempo que o contêiner, existe possível conectar tmpfs ao contêiner, um armazenamento temporário de informações que usa a RAM do host. Isso irá acelerar o desempenho da gravação e leitura de dados.

Muitas vezes acontece que os dados precisam ser armazenados mesmo depois que o contêiner deixa de existir. Para fazer isso, precisamos de mecanismos úteis para armazenamento permanente de dados.

Armazenamento permanente de dados


Existem duas maneiras de aumentar a vida útil dos dados que a vida útil do contêiner. Uma maneira é usar a tecnologia de montagem de ligação. Com essa abordagem, você pode montar, por exemplo, uma pasta realmente existente no contêiner. Os processos localizados fora do Docker poderão trabalhar com os dados armazenados nessa pasta. É assim que a tecnologia tmpfs mount and bind mount se parece .


Monte tmpfs e monte bind

As desvantagens do uso da tecnologia de montagem de ligação são que seu uso complica o backup de dados, a migração de dados e o compartilhamento de dados entre vários contêineres. É muito melhor usar volumes do Docker para armazenamento de dados persistente.

Tom Docker


Um volume é um sistema de arquivos que reside em uma máquina host fora dos contêineres. A criação e o gerenciamento de volumes são feitos pelo Docker. Aqui estão as propriedades básicas dos volumes do Docker:

  • Eles representam um meio para o armazenamento permanente de informações.
  • Eles são independentes e separados dos contêineres.
  • Eles podem ser compartilhados entre diferentes contêineres.
  • Eles permitem que você organize a leitura e a gravação eficazes dos dados.
  • Os volumes podem ser hospedados no provedor de nuvem remota.
  • Eles podem ser criptografados.
  • Eles podem receber nomes.
  • O contêiner pode providenciar para que o volume seja preenchido com dados antecipadamente.
  • Eles são convenientes para teste.

Como você pode ver, os volumes do Docker têm ótimos recursos. Vamos falar sobre como criá-los.

Criação de Volume


É possível criar volumes usando o Docker ou por meio de solicitações de API.

Aqui está a instrução no Dockerfile que permite criar um volume quando o contêiner é iniciado. 

VOLUME /my_volume

Ao usar uma instrução Docker semelhante, depois de criar o contêiner, ele criará um volume contendo dados que já existem no local especificado. Observe que, se você criar um volume usando o Dockerfile, isso não dispensa a necessidade de especificar um ponto de montagem para o volume.

Você também pode criar volumes no Dockerfile usando o formato JSON.

Além disso, é possível criar volumes usando a linha de comandos enquanto o contêiner estiver em execução.

Trabalhando com volumes da linha de comandos


▍Criar um volume


Você pode criar um volume independente com o seguinte comando: 

 docker volume create —-name my_volume

InformationDobrar informações de volume


Para visualizar a lista de volumes do Docker, use o seguinte comando: 

 docker volume ls

Você pode explorar um volume específico como este: 

 docker volume inspect my_volume

EmRemover Volume


Você pode excluir um volume como este: 

 docker volume rm my_volume

Para excluir todos os volumes que não são usados ​​por contêineres, você pode recorrer ao seguinte comando: 

 docker volume prune

Antes de excluir volumes, o Docker solicitará a confirmação desta operação.

Se o volume estiver associado a um contêiner, esse volume não poderá ser excluído até que o contêiner correspondente seja excluído. Nesse caso, mesmo que o contêiner seja removido, o Docker nem sempre entende isso. Se isso acontecer, você pode usar o seguinte comando: 

 docker system prune

Ele foi projetado para limpar os recursos do Docker. Após executar este comando, você poderá excluir volumes cujo status foi determinado anteriormente incorretamente.

Sinalizadores --mount e --volume


Para trabalhar com volumes, quando você chama o docker , geralmente é necessário usar sinalizadores. Por exemplo, para criar um volume durante a criação de um contêiner, você pode usar a seguinte construção: 

 docker container run --mount source=my_volume, target=/container/path/for/volume my_image

Nos tempos antigos (até 2017), a bandeira --volume era --volume . Inicialmente, esse sinalizador (também pode ser usado de forma abreviada e, em seguida, parece -v ) foi usado para contêineres independentes, e o sinalizador --mount foi --mount no ambiente do Docker Swarm. No entanto, a partir do Docker 17.06, o sinalizador --mount pode ser usado em qualquer cenário.

Deve-se notar que, ao usar o sinalizador --mount , a quantidade de dados adicionais que devem ser especificados no comando aumenta, mas, por várias razões, é melhor usar esse sinalizador em vez de --volume . O sinalizador --mount é o único mecanismo que permite trabalhar com serviços ou especificar configurações do driver de volume. Além disso, trabalhar com esse sinalizador é mais fácil.

Nos exemplos existentes de comandos destinados a trabalhar com dados no Docker, você pode encontrar muitos exemplos do uso do sinalizador -v . Ao tentar adaptar esses comandos, lembre-se de que os --mount e --volume usam formatos de parâmetros diferentes. Ou seja, você não pode simplesmente substituir -v por --mount e obter um comando de trabalho.

A principal diferença entre --mount e --volume é que, ao usar o sinalizador --volume todos os parâmetros são coletados juntos em um campo e, ao usar --mount parâmetros são separados.

Ao trabalhar com --mount parâmetros são representados como pares de chave-valor, ou seja, se parece com key=value . Esses pares são separados por vírgulas. Aqui estão as opções --mount comumente usadas:

  • type - type montagem. O valor para a chave correspondente pode ser bind , volume ou tmpfs . Estamos falando de volumes aqui, isto é - estamos interessados ​​no valor do volume .
  • source - monte a fonte. Para volumes nomeados, este é o nome do volume. Para volumes sem nome, essa chave não está especificada. Pode ser reduzido para src .
  • destination - o caminho no qual o arquivo ou pasta está montado no contêiner. Essa chave pode ser reduzida para dst ou target .
  • readonly - monta um volume somente leitura . O uso dessa chave é opcional; nenhum valor é atribuído a ela.

Aqui está um exemplo do uso de --mount com muitas opções: 

 docker run --mount type=volume,source=volume_name,destination=/path/in/container,readonly my_image

Sumário


Aqui estão alguns comandos úteis que você pode usar ao trabalhar com volumes do Docker:

  • docker volume create
  • docker volume ls
  • docker volume inspect
  • docker volume rm
  • docker volume prune

Aqui está uma lista de opções comumente usadas para --mount , aplicáveis ​​a um comando como docker run --mount my_options my_image :

  • type=volume
  • source=volume_name
  • destination=/path/in/container
  • readonly
