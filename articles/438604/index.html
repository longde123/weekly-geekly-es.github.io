Bitrix en un montón de Nginx + PHP-FPM, configurando el CNC, así como un caché compuesto con comentarios a través de nginx. Configuración modificada

Propósito: Proporcionar la configuración del servidor virtual Nginx para la operación de Bitrix-cms en el paquete Nginx + PHP-FPM. Lo cual, entre otras cosas, también es adecuado para el paquete Nginx + Apache2, con modificaciones menores.

Público objetivo: administradores del servidor, administradores avanzados del sitio, programadores.

Hay suficientes artículos sobre este tema, pero si los mira como no oficiales, generalmente contienen errores, y los oficiales están llenos de errores que no es aconsejable usar en Nginx. Espero que después de publicar esta configuración, el paquete Nginx + PHP-FPM se vuelva más serio.

Aquí está la configuración oficial para que el caché compuesto funcione a través de nginx.

Mostraré la implementación del retorno de la memoria caché compuesta del archivo. En general, el retorno con memcached se realiza por analogía. En la configuración de retorno de la memoria caché de archivos, conté 11 si, de lo cual me deshací al rehacerlos en el mapa.

Comenzaré con una versión simplificada del CNC para aquellos que solo necesitan un montón de Nginx + PHP-FPM sin devolver el caché compuesto a través de Nginx. Se entiende que la sección del servidor ya está configurada, con nombres de dominio y envío a php-fpm. 

location / { try_files $uri $uri/ /bitrix/urlrewrite.php$is_args$args; }

No es sorprendente cuando mira esos lienzos que se me ocurrieron, esto es suficiente para que Bitrix funcione correctamente. Si necesita una redirección de index.php e index.html a sin, entonces todavía necesita agregar esta línea: 

 if ($request_uri ~ ^(.*)/index.(html|php)) { return 301 $1/$is_args$args; }

Desafortunadamente, no hay un reemplazo digno si. Pero esta línea funciona sin crear problemas.

Ejemplo de configuración minimalista 
 server { listen 80; server_name site.ru; root /var/www/site.ru/; index index.php; location ~ \.php$ { include /etc/nginx/fastcgi_params; fastcgi_pass unix:/var/run/php/php7.0-fpm.sock; fastcgi_index index.php; fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name; fastcgi_intercept_errors on; } if ($request_uri ~ ^(.*)/index.(html|php)) { return 301 $1/$is_args$args; } location / { try_files $uri $uri/ /bitrix/urlrewrite.php$is_args$args; } location ~* @.*\.html$ { internal; } }


Quiero enfatizar que esta es una configuración minimalista sin reglas de estática, compresión, y allí cubrí solo los archivos de caché compuestos de acceso directo. La configuración que cubre ciertos lugares desde el acceso directo a través de nginx es bastante individual. Tengo este kit que podría adaptarse a alguien. Pero debes usarlo con cuidado y conciencia. Tenga en cuenta que ingresar datos de ubicación en su configuración puede provocar la inoperancia del sitio o parte de sus funciones.

Ahora pasemos a la configuración para que el compuesto funcione con el retorno de los archivos de caché a través de nginx. En primer lugar, es necesario determinar si es posible asignar el caché compuesto a esta solicitud o si debe enviarse para su procesamiento a través de php. Para hacer esto, en Nginx, en la sección http, agregue algunos mapas, así como varias directivas: 

 modern_browser_value "modern"; modern_browser msie 10.0; modern_browser unlisted; map "$cookie_BITRIX_SM_LOGIN:$cookie_BITRIX_SM_UIDH:$cookie_BITRIX_SM_CC" $storedAuth { default ""; "~*:*:Y" ""; "~*:*:*" 1; "~*:*:" 1; } map "$request_method:$http_bx_action_type:$cookie_BITRIX_SM_NCC:$http_x_forwarded_scheme:$modern_browser:$storedAuth" $usecache { default "1"; "~GET:::*https*" "1"; "~GET:::*:*:" ""; }

A continuación, escribimos directamente en la sección del servidor
