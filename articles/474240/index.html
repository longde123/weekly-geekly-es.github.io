Orleans 3.0 lanzado

Este es un post invitado del equipo de Orleans. Orleans es un marco multiplataforma para crear aplicaciones distribuidas usando .NET. Consulte https://github.com/dotnet/orleans para obtener más información.

Nos complace anunciar el lanzamiento de Orleans 3.0. En comparación con Orleans 2.0, tiene muchas mejoras y correcciones, así como varias características nuevas. Estos cambios están dictados por la experiencia de muchas personas que usan aplicaciones basadas en Orleans en la producción en una amplia gama de escenarios y entornos, así como el entusiasmo de la comunidad global de Orleans, que ayuda a que el marco sea mejor, más rápido y más flexible. ¡Muchas gracias a todos los que contribuyeron a este lanzamiento de varias maneras!



Cambios importantes en comparación con Orleans 2.0


Orleans 2.0 se lanzó hace poco más de 18 meses, y el marco ha progresado significativamente desde entonces. Algunos cambios en comparación con 2.0:

  • Transacciones ACID distribuidas
  • Un nuevo planificador que en algunos casos ha aumentado la productividad en más del 30%.
  • Nuevo generador de código basado en Roslyn
  • Reescritura de membresía de clúster para mejorar la velocidad de recuperación
  • Soporte de cohospedaje

Así como muchas otras mejoras y correcciones.

Sustitución de la capa de red con ASP.NET Bedrock


Mantener comunicaciones seguras usando TLS ha sido una preocupación importante por algún tiempo. Además, tanto de la comunidad como de los socios internos. En la versión 3.0, agregamos compatibilidad con TLS, disponible a través del paquete Microsoft.Orleans.Connections.Security . Vea el ejemplo de TransportLayerSecurity para más información.

La implementación del soporte TLS fue una preocupación importante debido a cómo se implementó la capa de red en versiones anteriores de Orleans: no podía adaptarse fácilmente para usar SslStream , que es el método más común para implementar TLS. Usando TLS como fuerza impulsora, reescribimos la capa de red de Orleans.

En Orleans 3.0, toda la capa de red se reemplaza con una capa creada sobre la base de Project Bedrock , una característica del grupo ASP.NET. El objetivo de Bedrock es ayudar a los desarrolladores a crear clientes y servidores de red rápidos y confiables.

El equipo de ASP.NET y el equipo de Orleans trabajaron juntos para crear abstracciones que admitan tanto los clientes como los servidores de la red, sean independientes del transporte y se puedan configurar mediante middleware. Estas abstracciones nos permiten cambiar el "transporte" de la red a través de la configuración sin cambiar el código de red interno específico de Orleans. El soporte Orleans TLS se implementa como middleware Bedrock, y tenemos la intención de hacerlo universal para que pueda compartirse con otros en el ecosistema .NET.

Aunque el incentivo para esta innovación fue la incorporación del soporte TLS, en nuestras pruebas de carga por la noche, el rendimiento promedio aumenta en aproximadamente un 30%.

La reescritura de la capa de red también incluyó la sustitución de nuestra agrupación de búferes personalizada con un MemoryPool<byte> , y con este cambio, la serialización ahora aprovecha más Span<T> . Algunas rutas de código que anteriormente dependían del bloqueo a través de hilos dedicados que llaman a BlockingCollection<T> ahora usan el Channel<T> para pasar mensajes asincrónicos. Esto da como resultado hilos menos dedicados, en lugar de mover el trabajo al grupo de hilos .NET.

Cohospedaje a través del host genérico


Coorganizar Orleans con otras plataformas como ASP.NET Core en el mismo proceso ahora es más fácil que antes gracias a .NET Generic Host .

Aquí hay un ejemplo de UseOrleans agregar Orleans junto con ASP.NET Core a un host usando UseOrleans : 
 var host = new HostBuilder() .ConfigureWebHostDefaults(webBuilder => { //  ASP.NET Core webBuilder.UseStartup<Startup>(); }) .UseOrleans(siloBuilder => { //  Orleans siloBuilder.UseLocalHostClustering(); }) .ConfigureLogging(logging => { /*   ,    */ }) .ConfigureServices(services => { /*    */ }) .UseConsoleLifetime() .Build(); //      . await host.RunAsync();

Esta funcionalidad se puede usar para simplificar la topología de implementación o para agregar características adicionales a una aplicación existente. Algunos equipos usan alojamiento compartido internamente para agregar pruebas de Kubernetes a sus proyectos de Orleans utilizando las comprobaciones de estado de ASP.NET Core .

Mejoras de confiabilidad


Los clústeres ahora se recuperan más rápido de las fallas gracias a los chismes avanzados. Además, los errores de mensajería ahora se manejan de manera más consistente, lo que resulta en errores que se devuelven a la persona que llama. Esto ayuda a los desarrolladores a encontrar errores más rápido. Por ejemplo, cuando un mensaje no se puede serializar o deserializar por completo, se devolverá una excepción detallada al llamante original.

Extensibilidad mejorada


Las transmisiones ahora pueden tener sus propios adaptadores de datos, lo que les permite recibir datos en cualquier formato. Esto brinda a los desarrolladores más control sobre cómo se representan los elementos de flujo en el repositorio. También le da al proveedor de flujo control sobre cómo se escriben los datos, permitiendo que los flujos se integren con los sistemas existentes y / o servicios que no son de Orleans.

El estado personalizado de la transacción ahora puede declarar qué roles puede jugar en la transacción. Por ejemplo, una implementación de un estado de transacción que escribe eventos del ciclo de vida de la transacción en la cola del bus de servicio no puede cumplir con las obligaciones de un administrador de transacciones porque es de solo escritura.
