Nueva serialización puede aparecer en Java

En el sitio web de OpenJDK apareció un nuevo documento de investigación que describe la idea de introducir una nueva serialización mejorada en el lenguaje para reemplazar el antiguo.

La serialización en Java existe desde la versión 1.1, es decir, casi desde el momento en que nació. Por un lado, la serialización es un mecanismo muy conveniente que le permite hacer rápida y fácilmente cualquier clase serializable heredando esta clase de la interfaz java.io.Serializable. Quizás incluso esta simplicidad se haya convertido en una de las razones clave por las que Java ha ganado tanta popularidad en el mundo, porque le permite escribir aplicaciones de red de manera rápida y eficiente.

Por otro lado, la forma en que se implementa la serialización en Java implica una gran cantidad de problemas que aumentan el costo de soporte de aplicaciones, reducen su seguridad y ralentizan la evolución de la plataforma.

¿Qué hay de malo con la serialización en Java? Enumeramos los problemas más serios:

  • La serialización (y la deserialización) evita los mecanismos del lenguaje. Ignora los modificadores de acceso de campo (privados, protegidos) y crea objetos sin usar constructores, lo que significa que ignora los invariantes que pueden estar presentes en estos constructores. Un atacante podría explotar tal vulnerabilidad al sustituir los datos con datos no válidos, y se los tragaría con éxito durante la deserialización.
  • Al escribir clases serializables, el compilador no ayuda de ninguna manera y no detecta errores. Por ejemplo, no puede garantizar estáticamente que todos los campos de una clase serializable sean ellos mismos serializables. O puede hacer un error tipográfico en los nombres de los métodos readObject, writeObject, readResolve, etc., y estos métodos simplemente no se utilizarán durante la serialización.
  • La serialización no admite el mecanismo normal de versiones, por lo que es muy difícil modificar las clases serializables para que sigan siendo compatibles con sus versiones anteriores.
  • La serialización está fuertemente ligada a la codificación / decodificación de transmisión, lo que significa que cambiar el formato de codificación a uno diferente del estándar es muy difícil. Además, el formato estándar no es compacto, ni eficiente, ni legible para humanos.

El error fundamental de la serialización existente en Java es que intenta ser demasiado "invisible" para el programador. Simplemente hereda de java.io.Serializable y recibe algún tipo de magia implícita que es ejecutada por la máquina virtual.
Por el contrario, el programador debe escribir explícitamente construcciones responsables de construir y deconstruir objetos. Estas construcciones deben estar en el nivel del lenguaje y deben escribirse a través del acceso de campo estático, no de reflexión.

Otro error de serialización es que está intentando hacer demasiado. Se establece la tarea de poder serializar cualquier gráfico arbitrario de objetos (que puede contener bucles) y deserializarlo de nuevo sin romper su estado.

Este error se puede corregir simplificando la tarea y serializando no un gráfico de objetos, sino un árbol de datos en el que no habrá un concepto de identidad (como en JSON).

¿Cómo hacer que la serialización se ajuste naturalmente al modelo de objetos, use constructores para la deserialización, se separe del formato de codificación y admita el control de versiones? Con este fin, las anotaciones rescatan la posibilidad de un lenguaje que aún no está incluido en Java: coincidencia de patrones . Por ejemplo: 

public class Range { int lo; int hi; private Range(int lo, int hi) { if (lo > hi) throw new IllegalArgumentException(String.format("(%d,%d)", lo, hi)); this.lo = lo; this.hi = hi; } @Serializer public pattern Range(int lo, int hi) { lo = this.lo; hi = this.hi; } @Deserializer public static Range make(int lo, int hi) { return new Range(lo, hi); } }

En este ejemplo, se declara la clase Range, que está lista para la serialización a través de dos miembros especiales de la clase: un serializador y un deserializador marcado con anotaciones @Serializer y @Deserializer. El serializador se implementa a través del deconstructor del patrón, y el deserializador se implementa a través del método estático en el que se llama al constructor. Por lo tanto, durante la deserialización, inevitablemente se verifica el hi> = lo especificado en el constructor.
No hay magia en este enfoque, y se utilizan anotaciones regulares, por lo que cualquier marco puede hacer la serialización, y no solo la plataforma Java en sí. Esto significa que el formato de codificación también puede ser absolutamente cualquier cosa (binario, XML, JSON, YAML, etc.).

Dado que los serializadores y los deserializadores son métodos comunes, el programador tiene una gran libertad en su implementación. Por ejemplo, puede elegir una representación de un objeto diferente de la forma en que se representa el objeto en la memoria. Por ejemplo, LinkedList se puede serializar no en una cadena de enlaces, sino en una matriz continua, lo que hará que la presentación sea más simple, más eficiente y más compacta.

La versión en este enfoque se implementa utilizando el campo de versión especial de las anotaciones @Serializer y @Deserializer: 

 class C { int a; int b; int c; @Deserializer(version = 3) public C(int a, int b, int c) { this a = a; this.b = b; this.c = c; } @Deserializer(version = 2) public C(int a, int b) { this(a, b, 0); } @Deserializer(version = 1) public C(int a) { this(a, 0, 0); } @Serializer(version = 3) public pattern C(int a, int b, int c) { a = this.a; b = this.b; c = this.c; } }

En este ejemplo, se llamará a uno de los tres deserializadores, según la versión.
¿Qué sucede si no queremos que los serializadores y deserializadores estén disponibles para nadie más que para fines de serialización? Para hacer esto, podemos hacerlos privados. Sin embargo, en este caso, un marco de serialización específico no podrá acceder a ellos a través de la reflexión si dicho código está dentro del módulo en el que el paquete no está abierto para un acceso reflexivo profundo. Para tal caso, se propone introducir otra nueva construcción en el lenguaje: miembros de clase abierta. Por ejemplo: 

 class Foo { private final InternalState is; public Foo(ExternalState es) { this(new InternalState(es)); } @Deserializer private open Foo(InternalState is) { this.is = is; } @Serializer private open pattern serialize(InternalState is) { is = this.is; } }

Aquí, los serializadores y los deserializadores están marcados con la palabra clave abierta, lo que los abre a setAccessible.

Por lo tanto, el nuevo enfoque es fundamentalmente diferente del anterior: en él, las clases se diseñan como serializables y no se dan a la plataforma tal como están. Esto requiere un esfuerzo adicional, pero hace que la serialización sea más predecible, más segura e independiente del formato de codificación y el marco de serialización.
