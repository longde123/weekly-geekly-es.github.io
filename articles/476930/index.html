Extensión en Dart (Flutter)

En una versión reciente de Dart 2.6, el lenguaje introdujo una nueva función, extensión estática o métodos de extensión estática, que le permite agregar nuevos métodos a los tipos existentes. ¿Por qué necesitamos extensión? ¿Cómo usarlos y para qué sirven?



Introduccion


Para empezar, ¿qué es la extensión en general? La extensión es azúcar sintáctica que extiende una clase existente en un lugar diferente del módulo de declaración de clase.

En programación, los métodos de extensión han existido durante mucho tiempo, por lo que tuvieron que lanzarse. La extensión se usa activamente en lenguajes como C #, Java a través de Manifold, Swift, Kotlin y muchos otros.

El problema


Digamos que tenemos un método catchError, que es simplemente horrible y necesita reescribirse para una nueva función genial. Suponga que usa una función de cualquier tipo como argumento para un lugar de una función estrictamente tipada o para verificar el tipo de una función, y esto sucede porque hace 8 meses cuando desarrollaba esta funcionalidad era lógico en ese momento.

Lo primero que viene a la mente es reescribir esta función, pero aquí nos enfrentamos con el problema de que ocurre con demasiada frecuencia en el proyecto, y cambiar la función conducirá a la inoperancia de todo el proyecto.

Bueno, si la primera opción no es para nosotros. adecuado, por razones lógicas, entonces puedo implementar una nueva función Futuro que cumpla con todos mis requisitos. 

abstract class Future<T> { ... /// Catches any [error] of type [E]. Future<T> onError<E>(FutureOr<T> handleError(E error, StackTrace stack)) => this.catchError(...   -  ...); } ... }

y la llamaré así: 

 Future<String> someString = ...; someString.onError((FormatException e, s) => ...).then(...);

Desafortunadamente, no puedo agregar esta función a la clase Future. Si hago esto, también lo agregaré a la interfaz Future, y cualquier otra clase que implemente esta interfaz estará incompleta y ya no se compilará.

Bueno, otra opción es implementar una función de terceros que se verá así: 

 Future<T> onFutureError<T, E>(Future<T> source, FutureOr<T> handleError(E error, StackTrace stack)) => source.catchError(... - ...);

Y su llamada se vería así: 

 Future<String> someString = ...; onFutureError(someString, (FormatException e, s) => ...).then(...);

¡Genial, todo funciona! Pero es triste que haya comenzado a leerse terriblemente. Usamos métodos que se implementan dentro de la clase, por lo que se llaman -.doingSomething (); Este código es comprensible, lo acabo de leer de izquierda a derecha y tengo en mi cabeza una secuencia de eventos. El uso de una función auxiliar hace que el código sea engorroso y menos legible.

Bueno, entonces puedo implementar una nueva clase y dejar que los usuarios envuelvan su interfaz anterior con una funcionalidad mejorada. 

 class CustomFuture<T> { CustomFuture(Future<T> future) : _wrapper = future; Future<T> _wrapper; Future<T> onError<E>(FutureOr<T> handleError(E error, StackTrace stack)) => _wrapper.catchError(...-     ...); }

y la llamada se verá así: 

 Future<String> someString = ...; CustomFuture(someString).onError((FormatException e, s) => ...).then(...);

Se ve genial!

Resolviendo un problema con la extensión


Tan pronto como detengamos la programación en pascal y regresemos a 2019, la implementación de esta funcionalidad se reducirá a este tamaño: 

 extension CustomFuture <T> on Future<T> { Future<T> onError<E>( FutureOr<T> handleError(E error, StackTrace stack)) => this.catchError(...something clever...); }

y así es como se verá la llamada: 

 Future<String> someString = ...; someString.onError((FormatException e, s) => ...).then(...);

Eso es todo! La solución a este problema tomó solo 5 líneas de código. Usted Te preguntarás qué tipo de magia es y cómo funciona.

De hecho, se comporta de la misma manera que una clase contenedora, aunque en realidad es solo una función estática auxiliar. La extensión le permite dejar de lado la escritura explícita de envoltorios.

Esto no es un envoltorio


El diseño de la extensión funciona de tal manera que parece una declaración de una clase existente, pero actúa como si fuera un contenedor con un _wrapper privado. Pero hay una ventaja en comparación con una clase de contenedor: esto es acceder a la clase directamente, en lugar de acceder a la clase de contenedor _wrapper.

Esta característica no se hizo por el bien de las características, pero como dije antes, las extensiones son de hecho una forma más conveniente de llamar a funciones estáticas. Esto significa que no hay ningún objeto contenedor.

Todo es estático


Dije "métodos de extensión estática" arriba, ¡y lo hice por una razón!

Dart está estáticamente escrito. El compilador conoce el tipo de cada expresión en el momento de la compilación, por lo que si escribe user.age (19) y age es una extensión, entonces el compilador debe averiguar qué tipo está envuelto en el objeto dado para encontrar el tipo de la llamada completa.

¿Qué problemas pueden surgir?


El ejemplo más simple de problemas con la extensión es cuando tiene más de una extensión en su alcance. Básicamente, el ganador es la extensión más cercana al tipo real de expresión que está llamando al miembro, con algunas reservas.

La forma más fácil de resolver el problema es conectar estrictamente la extensión que necesita, o puede usar la extensión explícitamente: 

 ... List list = ...; MyList(list).printlist(); SomeList(list).printlist(); ... extension MyList on List { void printlist() { print(...- ...); } } extension SomeList on List { void printlist() { print(...-  ...); } }

Resumen


  • El lenguaje dardo tiene una herramienta conveniente para expandir la funcionalidad existente.
  • Puede ampliar métodos, operadores, establecedores y captadores, pero no campos.
  • Puede invocar métodos de extensión explícitamente o, cuando no haya conflicto con un miembro de la interfaz u otra extensión, implícitamente.
  • Las llamadas implícitas funcionan igual que las llamadas explícitas.
  • Las extensiones son estáticas. Todo sobre ellos se resuelve sobre la base de tipos estáticos.

Si la salida de la extensión falla debido a extensiones en conflicto, puede hacer lo siguiente:

  1. Aplica la extensión explícitamente.
  2. Importe la extensión conflictiva con el prefijo, porque no está disponible para llamadas implícitas.
  3. No importe una extensión conflictiva en absoluto.

Eso es todo! Puede usar la extensión en todo su potencial.
