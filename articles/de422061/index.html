Kompakter Server mit angepasstem Code-Reload und SQL-Protokollen von einem Maven-Team für eine produktive Entwicklung

Ich möchte die Erfahrungen mit Payara-Micro bei der Entwicklung von Unternehmensanwendungen teilen. Ich hoffe, dass jemand Zeit spart, da eine solche Entscheidung nicht sofort getroffen wurde. Wenn Sie Payara oder Glassfish bereits als Industrieserver verwenden oder einfach nur in die Welt von Javaee eintauchen möchten, dann ist payara-micro und dieser Artikel genau das Richtige für Sie.


Sie benötigen eine Webanwendung, die über maven in einem Kriegsarchiv zusammengestellt wird, und Java 8 (ich habe nicht nach älteren gesucht).


Zuerst werde ich den vollständigen Text des Profils angeben, danach werden wir ihn in Teile zerlegen und die fehlenden Details hinzufügen, die im Quellcode nicht sichtbar sind.

Als erstes müssen Sie das Payara-Micro-Archiv selbst herunterladen. Genau das macht das erste Plugin ( Maven-Dependency-Plugin ). Das Archiv ist ein vollständiger Anwendungsserver, der über die Befehlszeile als einfache Java-Anwendung gestartet werden kann.


Das neueste Plugin ( Payara-Micro-Maven-Plugin ) hat keine Funktionen und ist für eine stabile Arbeitsversion von Payra-Micro 4.1.2.173 konfiguriert . Von den Funktionen des Plugins wird hier die Funktion zum Packen der Webanwendung in ein spezielles, sofort einsatzbereites Bundle verwendet. Es gibt andere Möglichkeiten, Payara-Micro zu starten, aber hier werden sie für uns nicht nützlich sein.


Es bleibt die gewünschte Befehlszeile zu generieren und alles zusammenzusetzen. Das exec-maven-Plugin wird verwendet, um von der Befehlszeile aus zu starten. Lassen Sie uns die Befehlszeilenoptionen genauer betrachten: 


 <configuration> <executable>java</executable> <arguments> <!--http://blog.payara.fish/using-hotswapagent-to-speed-up-development--> <argument>-XXaltjvm=dcevm</argument> <argument>-javaagent:hotswap/hotswap-agent.jar</argument> <argument>-agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=9009</argument> <argument>-Duser.language=en</argument> <argument>-Duser.region=US</argument> <argument>-Ddb.ora.url=jdbc:p6spy:oracle:thin:@localhost:1521:XE</argument> <argument>-Ddb.pg.url=jdbc:p6spy:postgresql://localhost:5432/postgres</argument> <argument>-jar</argument> <argument>${project.build.directory}/${project.build.finalName}-microbundle.jar</argument> <argument>--domainConfig</argument> <argument>src/main/setup/domain.xml</argument> <argument>--deploy</argument> <argument>${project.build.directory}/${project.build.finalName}.war</argument> <argument>--rootDir</argument> <argument>${project.build.directory}/payaramicro</argument> </arguments> </configuration>

Die ersten beiden Argumente ( -XXaltjvm = dcevm -javaagent: hotswap / hotswap-agent.jar ) werden benötigt, um den Quellcode zu ändern, ohne die Anwendung neu zu starten. Dank des zweiten Arguments ( -agentlib: jdwp = transport = dt_socket, server = y, suspend = n, address = 9009 ) starten wir die Anwendung sofort im Debug-Modus, wodurch wir nicht nur den Hauptteil der Methoden selbst ändern können, indem wir Codeänderungen beim Debuggen in der IDE anwenden , sondern auch neue Felder und mehr hinzufügen. Sie müssen nur über die IDE eine Verbindung zum Debugger herstellen, die geänderte Quelldatei kompilieren und die Änderungen übernehmen.


Die folgenden zwei Argumente ( -Duser.language = de -Duser.region = US ) werden benötigt, damit der jdbc-Client ordnungsgemäß mit Oracle funktioniert. Die Einstellungen für die Verbindung zum Test-DBMS werden über die Argumente -D übergeben . In meinen Anwendungen verwende ich einen Link wie $ {db.ora.url} zu diesen Parametern in der Ressourcendatei, der sie beim Start automatisch erstellt. Das Ersetzen einer Datei, die Ressourcen mit Test- oder Industrieeinstellungen erstellt, wird auch durch das Profil in maven und dessen Fähigkeit zum Filtern von Textdateien implementiert.


Die wichtigsten Argumente beim Start geben den Pfad ( --domainConfig src / main / setup / domain.xml ) zur Domäneneinstellungsdatei sowie den Pfad zur Kriegsdatei ( --deploy $ {project.build.directory} / $ {project an. build.finalName} .war ), das auf dem Server bereitgestellt werden soll. Die Domain-Einstellungen wurden von mir basierend auf einer Standarddatei im Archiv erhalten. Ich werde den Inhalt als Beispiel geben. Die Verwendung bei der Entwicklung dieser Einstellungen ist mehr als ausreichend.
