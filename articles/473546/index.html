Hay muchos manuales sobre cómo crear un servicio HTTP en Internet. Por lo tanto, describiré de inmediato la implementación. Nuestro servicio consta de tres patrones de URL:

  • Directorios - / v1 / catalogs / {Directory} / {Action}
  • Documentos - / v1 / documents / {Document} / {Action}
  • Servicios - / v1 / services / {Action}


Para no inflar el artículo, considere el ejemplo con el libro de referencia "Nomenclatura". 1C no nos permite tener rutas diferentes para los métodos HTTP. Por lo tanto, el patrón URL será (/ v1 / catalogs / wares / {Action}, para el cual se definen dos métodos HTTP GET, POST. En otros patrones también usaremos GET, POST.
  • / v1 : nos permitirá admitir varias opciones de API en el futuro. Podremos agregar la plantilla de URL v_V2_Directories y, en consecuencia, la ruta para comenzar con / v2
  • / catalogs / wares / - catalogs integra todos los directorios. Esta es la parte estática del camino. wares cumple con el Directorio de nomenclatura. 1C pondrá "mercancías" en los parámetros de URL. Cabe señalar que nosotros mismos encontramos formas. Y luego también pensamos cómo procesarlos en 1C nosotros mismos
  • {Acción} : aquí pasaremos lo que queremos obtener de 1C. Por ejemplo getByParam o getAll


Desde el punto de vista de Retrofit, baseUrl se verá así "http://192.168.0.1/unf/hs/inntsd/v1/" donde:

  • 192.168.0.1 : servidor donde se publica el servicio HTTP
  • / unf : nombre de la base de datos 1C
  • / hs : parte estática para servicios HTTP
  • / inntsd es la URL raíz de todo el servicio. Llama como queramos
