Apakah semua orang pergi ke DevOps?


Hari ini, DevOps adalah penyebutan khusus dari saya - sebuah pendekatan baru untuk pengembangan dan pengoperasian perangkat lunak, sebagai Sebenarnya ada banyak artikel dan perdebatan tentang topik ini tahun ini.



Istilah DevOps diartikan cukup luas saat ini. Seseorang di bawah DevOps memahami pendekatan khusus untuk pengembangan dan pengoperasian perangkat lunak, ketika orang-orang terlibat dalam pekerjaan, yang bisa sedikit kode dan admin. Bagi yang lain, ini adalah, pertama-tama, keberadaan administrator sistem pribadi dalam tim, yang memungkinkan Anda untuk menghapus bagian dari beban non-inti dari pengembang perangkat lunak dalam bentuk pengaturan lingkungan sistem, menciptakan lingkungan pengujian, mengimplementasikan integrasi dengan layanan internal dan eksternal, serta menulis skrip otomatisasi. Untuk yang ketiga, itu hanya seperangkat teknologi dan alat modis yang perlu Anda gunakan agar selalu tetap muda dan sukses. Untuk empat, ini adalah CI \ CD dan segala sesuatu yang berkaitan dengannya. Memang, DevOps memiliki banyak interpretasi, jadi siapa pun dapat menemukan di dalamnya apa yang lebih sesuai dengan keinginan mereka.

Interpretasi yang berbeda dari DevOps menimbulkan diskusi panas, yang mengarah pada munculnya lebih banyak artikel tentang topik ini. Saya bahkan mem-bookmark beberapa dari mereka sendiri:

  1. Siapa itu DevOps?
  2. Cara masuk ke DevOps, cara belajar, dan apa yang harus dibaca .
  3. Mengapa administrator sistem harus menjadi insinyur DevOps .

Jika Anda membaca cukup banyak artikel yang memuji DevOps, Anda mungkin mendapat kesan bahwa setiap administrator sistem / insinyur hanya perlu mengubah posisi saat ini di profilnya di LinkedIN dari administrator / insinyur ke DevOps, segera setelah undangan SDM untuk wawancara mulai mengalir padanya dari perusahaan besar dan sukses yang akan menjanjikan gaji 2 kali lipat dari yang sekarang akan diberikan Macbook baru, skuter gyro dan tidak akan melupakan berlangganan untuk pengisian bahan bakar vape gratis dan smoothie dalam jumlah tak terbatas. Secara umum, surga IT-shny akan datang.

Jika Anda membaca artikel yang meremehkan manfaat DevOps, maka Anda mendapat kesan berbeda bahwa DevOps adalah jenis perbudakan baru, di mana orang harus kode hampir pada tingkat yang sama dengan pengembang, membantu mereka memperbaiki bug, melakukan otomatisasi dan CI \ CD, menyebarkan Jira dengan Wiki, memuntir awan, mengumpulkan wadah dan mengelolanya saat melakukan pekerjaan admin, sambil juga mengingat bagaimana cara mengisi ulang kartrid, crimp twisted pair dan penyiraman warna kantor.

Admin tidak lagi dibutuhkan?


Sebagai administrator / insinyur sistem yang telah bekerja dengan produk Microsoft dan VMware selama beberapa waktu, saya mulai memperhatikan bahwa dalam beberapa tahun terakhir telah ada pembicaraan berkala bahwa administrator sistem akan segera dibutuhkan oleh siapa pun, karena:

  1. Seluruh infrastruktur akan berubah dan berubah menjadi IaaC (Infrastruktur sebagai kode). Sekarang tidak akan ada GUI dengan tombol, tetapi hanya akan ada PowerShell, file yaml, konfigurasi, dan banyak lagi. Jika ada layanan atau komponennya rusak, maka tidak perlu lagi memperbaikinya, karena Sebarkan salinan baru dari kondisi kerja terakhir dengan lebih cepat.
  2. Seluruh infrastruktur TI akan segera berpindah ke cloud, dan secara lokal (di lokasi) hanya akan ada kabel jaringan ke router terdekat, yang akan menghubungkan kita dengan semua sumber daya perusahaan lain yang berada di cloud. Nah, printer maksimum masih tetap lokal, sehingga gadis-gadis akuntansi dapat mencetak gambar kucing dari internet di atasnya. Segala sesuatu yang lain harus berada di awan.
  3. Seorang guru DevOps akan datang dan mengotomatiskan semua yang ada di sekitarnya, jadi admin hanya perlu mengingat dengan hangat bagaimana di masa lalu mereka mulai ping dan jejak untuk mendiagnosis masalah dasar pada jaringan dan pada server.
  4. Saya juga mendengar tentang fenomena seperti "Vendekapets", tetapi itu sudah lama sekali, pada awal karir saya, ketika saya baru mulai mengambil langkah pertama menuju administrasi sistem. Tetapi untuk beberapa alasan, "Vendekapets" tidak pernah datang, seperti akhir dunia menurut kalender Maya. Kebetulan? Saya kira tidak. :)

Instalasi lokal (di tempat)


Microsoft Exchange Server adalah server surat multifungsi yang mencakup tidak hanya bekerja dengan surat, tetapi juga dengan kontak, kalender, tugas, dan banyak lagi. Exchange Server adalah salah satu produk andalan Microsoft, yang secara de facto menjadi standar perusahaan di banyak perusahaan. Ini memiliki integrasi yang erat tidak hanya dengan produk Microsoft, tetapi juga dengan solusi vendor pihak ketiga. Exchange populer baik di perusahaan menengah (dari 100 orang) dan perusahaan besar.

Saat ini, versi saat ini dianggap sebagai Exchange Server 2019. Sebelumnya, produk ini dikembangkan cukup aktif, tetapi sejak rilis Exchange 2013 perkembangan ini telah melambat secara signifikan, sehingga Exchange 2016 dapat disebut Paket Layanan 1 (SP1) untuk Exchange 2013, dan karenanya Exchange 2019 adalah Layanan Paket 2 (SP2) untuk Exchange 2013. Nasib versi on-premise berikutnya (Exchange 2022) masih dipertanyakan.

Sekarang Microsoft secara aktif mempromosikan Exchange Online sebagai bagian dari layanan cloud Office 365, sehingga semua fitur baru muncul terutama di sana. Exchange Online bukan hanya yang pertama menerima fitur baru, tetapi juga untuk memperoleh fitur tambahan yang tidak akan ditransfer ke instalasi di tempat dalam waktu dekat. Ini dilakukan untuk mempercepat transisi sejumlah perusahaan ke cloud, karena Model berlangganan lebih menguntungkan secara finansial untuk Microsoft daripada penjualan satu kali.

Jika Anda saat ini sedang melayani instalasi lokal Exchange Server (2013 - 2019), maka Anda dapat terus melakukan ini selama 3-5 tahun ke depan. Sepanjang jalan, Anda harus mulai menjelajahi peluang yang disediakan Exchange Online; dan konfigurasi hibrid, ini adalah ketika versi lokal dan cloud ada secara bersamaan. Bahkan jika kami berasumsi bahwa tidak akan ada lagi versi Exchange lokal (on-premise) berikutnya, maka pengetahuan yang diperoleh di Exchange Server akan terus relevan untuk beberapa waktu karena sejumlah alasan:

  • Jumlah instalasi lokal saat ini cukup besar, oleh karena itu, diperlukan administrator yang memenuhi syarat yang dapat menemani mereka. Tidak semua organisasi akan dapat mentransfer email mereka ke cloud dalam waktu dekat karena satu dan lain alasan.
  • Proyek migrasi awan belum sepele, jadi pengetahuan tentang spesifikasi solusi lokal dan cloud diperlukan untuk memotong sebagian besar jebakan dan berhasil menyelesaikan migrasi.
  • Pengetahuan tentang smtp \ imap \ mapi \ pop3, aliran surat, dkim, dmark, spf, antivirus, protokol antispam bersifat universal dan akan berlaku untuk sistem surat apa pun.
  • Pengalaman yang diperoleh saat bekerja dengan Exchange Server di tempat akan memungkinkan Anda untuk dengan cepat memahami Exchange Online dan mengonfigurasi konfigurasi yang diinginkan.
  • E-mail adalah salah satu saluran komunikasi terpenting dengan dunia luar, sehingga kebutuhan akan hal itu tetap ada. Penganut "rasul dan obrolan bot akan mengganti surat" tidak dapat didengarkan, karena mereka "mengubur" surat berkali-kali dan sejauh ini tidak berhasil.

SharePoint adalah portal internal perusahaan tempat perusahaan dapat memposting layanan Web mereka yang bermanfaat (jadwal liburan, daftar karyawan dengan foto dan telepon, pengingat ulang tahun, berita perusahaan, dll.). Pengguna dapat menyimpan, mengedit, dan berbagi file yang dihosting di pustaka SharePoint mereka.

SharePoint seperti Bitrix24, hanya lebih besar, lebih fungsional, lebih mahal dan berkali-kali lebih sulit untuk dikonfigurasikan dan didukung. Fitur Killer adalah kemampuan untuk secara bersamaan mengedit satu dokumen dengan sejumlah besar karyawan, yang sangat nyaman ketika 100 orang mencoba mengisi jadwal liburan, dan integrasi dengan Office Online Server dan MS Office lokal.

Sharepoint adalah produk yang besar, kompleks, dan mahal, oleh karena itu biasanya digunakan hanya di perusahaan besar. Perusahaan kecil menggunakan Bitrix24 atau analognya, atau hanya menyimpan file di server file, dan layanan Web yang berguna didistribusikan di situs internal yang berbeda.

Farm SharePoint (cluster) biasanya ditangani oleh pengembang dengan fungsi administrator, daripada administrator sistem "murni", seperti agar SharePoint lepas landas dan bermanfaat bagi perusahaan, perlu banyak penyelesaian kode.

Office 365 mencakup layanan SharePoint Online, yang merupakan versi disederhanakan SharePoint di tempat, yaitu. ia memiliki sejumlah kecil kemungkinan untuk penyesuaian dan “menyelesaikannya sendiri”, tetapi menghilangkan banyak sakit kepala dari pengembang / administrator untuk operasinya. Vonis saya adalah bahwa kompleksitas dan biaya tinggi untuk mendukung versi SharePoint di tempat akan melakukan pekerjaan mereka dan perusahaan akan dengan senang hati mulai merangkak ke SharePoint Online, atau meninggalkan Sharepoint sama sekali demi solusi yang lebih sederhana. Saya pribadi tidak melihat kehidupan pelangi dan riang untuk SharePoint di pemasangan lokal.

System Center adalah keluarga produk untuk menyebarkan, mengonfigurasi, mengelola, dan memantau infrastruktur Windows besar. Dinilai oleh: Manajer Konfigurasi Pusat Sistem (SCCM), Manajer Mesin Virtual Pusat Sistem (SCVMM), Manajer Operasi Pusat Sistem (SCOM), Manajer Perlindungan Data Pusat Sistem (SCDPM), Manajer Layanan Pusat Sistem (SCSM), System Center Orchestrator (SCORCH) )



Seluruh jajaran produk System Center biasanya hanya diminati di perusahaan besar, sedangkan perusahaan menengah biasanya hanya menggunakan satu atau dua produk.

Karena produk-produk System Center cukup sulit untuk dipelajari dan biasanya hanya digunakan dalam infrastruktur besar, sudah lazim untuk memilih individu untuk bekerja dengannya, misalnya, administrator sistem pemantauan (SCOM), administrator layanan stasiun kerja (SCCM), dan administrator sistem virtualisasi (Hyper -V + SCVMM), Administrator Otomasi Infrastruktur (SCORCH + SCSM).

Microsoft dengan cepat mengembangkan layanan cloud-nya, sehingga fungsionalitas System Center secara bertahap mengalir ke cloud. Semua ini akan berdampak signifikan pada produk-produk System Center di tempat dalam waktu dekat.

Fungsionalitas System Center Orchestrator (SCORCH) di masa mendatang akan menggantikan layanan Azure Automation ( https://docs.microsoft.com/en-us/azure/automation/automation-intro ).

Fungsi Manajer Operasional Pusat Sistem (SCOM) di masa mendatang akan menggantikan layanan Monitor Azure ( https://docs.microsoft.com/en-us/azure/azure-monitor/overview ).

Fungsionalitas Manajer Perlindungan Data Pusat Sistem (SCDPM) akan menggantikan layanan Cadangan Azure ( https://docs.microsoft.com/en-us/azure/backup/backup-overview ) di masa mendatang.

Fungsi Manajer Sistem Pusat Layanan (SCSM) akan berhenti diminati atau akan diganti oleh sistem tiket lainnya, misalnya, Jira.

System Center Virtual Machine Manager (SCVMM) akan tetap dengan perusahaan yang menggunakan virtualisasi Hyper-V secara lokal. Instalasi Hyper-V kecil (10-15 server) dapat berhasil dikelola tanpa SCVMM hanya menggunakan alat standar - Failover Cluster Manager, Hyper-V Manager, Windows Admin Center.

System Center Configuration Manager (SCCM) - digunakan untuk penyebaran massal sistem operasi, pemasangan aplikasi perusahaan dari satu direktori, pemasangan pembaruan Windows pada server dan akhir workstation, inventaris aplikasi dan penghitungan lisensi. Tampaknya ini adalah satu-satunya produk dari seluruh lini System Center yang akan tetap bersama kami dalam infrastruktur on-premise, seperti untuk sepenuhnya menggantinya dengan sesuatu yang mendung saat ini tidak berfungsi.

Jika saat ini Anda mendukung instalasi lokal Pengelola Konfigurasi Pusat Sistem (SCCM), Anda dapat terus melakukannya, sebagai Produk akan bersama kami setidaknya selama 3-5 tahun ke depan. Selain itu, saya akan merekomendasikan mulai menjelajahi kemungkinan Office 365, sebagai ini akan cocok dengan posisi Enterprise Desktop Administrator.

Peran administrator sebagian besar produk System Center lainnya akan dihentikan. Layanan Azure secara signifikan menyederhanakan pekerjaan mereka, menyembunyikan semua kerumitan dari pengintaian. Ambil contoh administrator otomatisasi (SCORCH + SCSM). SCORCH akan diganti oleh Azure Automation. Pengetahuan tentang proses otomasi, PowerShell, SQL akan tetap dan akan berguna untuk Azure Automation, tetapi pengetahuan tentang membangun kluster SCORCH, memastikan ketersediaan tinggi, ukuran sumber daya, pembaruan, migrasi ke versi baru, pencadangan dan pemantauan akan kehilangan relevansinya, karena Semua pekerjaan ini akan dilakukan oleh cloud Azure. Administrator otomasi hanya akan fokus pada proses otomasi itu sendiri, seperti dia akan meninggalkan semua pekerjaannya untuk mempertahankan pengoperasian infrastruktur otomasi.

Server Windows dan perannya


Active Directory (AD) adalah tempat penyimpanan akun pengguna dan komputer. Jika perusahaan memiliki lebih dari 20 komputer, maka kemungkinan besar sudah ada beberapa domain Direktori Aktif di sana. Pengetahuan tentang Active Directory, kemampuan untuk membedakan domain dari hutan, kemampuan untuk bekerja dengan kebijakan grup adalah wajib untuk setiap administrator Windows. Pengetahuan ini akan relevan untuk 20 tahun ke depan. Selain itu, saya akan merekomendasikan agar Anda membiasakan diri dengan Azure AD (AAD), lihat opsi untuk menyinkronkan pengguna antara on-premise dan infrastruktur cloud.

DNS, DHCP adalah layanan jaringan, pemahaman yang berguna di semua bidang TI, dari administrasi hingga pemrograman, jadi Anda perlu mengetahuinya.Memahami pengoperasian jaringan, protokol perutean, model OSI dan TCP \ IP akan menjadi nilai tambah yang pasti untuk setiap spesialis TI.

Hyper-V adalah nama untuk seluruh tumpukan teknologi virtualisasi dari Microsoft dan hypervisor khususnya. Ini berkembang cukup cepat, meskipun menurut saya, sebagian besar fitur baru (VM Terlindung, Subnet Terenkripsi, Ruang Penyimpanan Langsung) difokuskan terutama pada penyedia cloud lokal (Penyedia Layanan Cloud) dan global (Azure), dan bukan pada segmen korporasi (Perusahaan). Ini umumnya dapat dipahami, karena Microsoft pertama kali mengimplementasikan dan meluncurkan fungsionalitas baru di cloud Azure-nya, dan hanya kemudian mentransfernya ke Windows Server dan Hyper-V.

Hyper-V masih mengalami kekurangan satu konsol gratis di mana semua fitur yang diperlukan disajikan. Sekarang kami memiliki Failover Cluster Manager, Hyper-V Manager, Windows Admin Center. SCVMM seharusnya menjadi konsol seperti itu, tetapi dibayar dan agak sulit dipelajari.

Jika saat ini Anda mendukung instalasi lokal Hyper-V dengan \ tanpa SCVMM, Anda dapat terus melakukannya. Secara paralel, saya akan merekomendasikan mulai belajar tentang Azure IaaS dan mekanisme untuk memigrasi mesin virtual antara cloud dan infrastruktur lokal.

Di antara lingkungan saya (bank, telekomunikasi, perusahaan asuransi, kepemilikan industri besar), semua virtualisasi produktif biasanya dikelola oleh VMware vSphere, bukan Hyper-V dengan SCVMM, jadi saya dapat merekomendasikan administrator Hyper-V untuk melihat ke arah VMware dan produk-produknya .

Layanan cloud


Office 365 adalah layanan cloud yang menyediakan paket berlangganan aplikasi Microsoft Office (lokal dan versi Web-nya), dan juga termasuk produk server utama - Exchange, Tim, OneDrive, dan Sharepoint.

Office 365 saat ini adalah layanan mandiri yang hampir sepenuhnya mencakup kebutuhan untuk komunikasi kantor. Karena kemudahan pengaturan, itu sempurna untuk perusahaan kecil dan menengah dan besar.

Kehadiran layanan Exchange, Tim, OneDrive, dan Sharepoint yang sudah dikerahkan di cloud secara signifikan mengurangi beban pada administrator sistem, karena semua prosedur untuk instalasi, ukuran sumber daya, pembaruan, dan migrasi ke versi baru sekarang sepenuhnya berada di tangan Microsoft. Sebelumnya, untuk mendukung Exchange, Tim, OneDrive, dan Sharepoint di infrastruktur lokal, diperlukan 4-6 administrator terpisah, tetapi sekarang Office 365 hanya membutuhkan 1 administrator berukuran sedang. Jika sesuatu tidak berfungsi atau tidak berfungsi dengan benar, maka Anda dapat membuat tiket ke dukungan teknis Microsoft langsung dari antarmuka Office 365, yang sangat nyaman.

Jika Anda sekarang administrator sistem yang menyertai versi Exchange, Skype for Business, atau produk Sharepoint di tempat, maka saya akan merekomendasikan untuk melihat versi cloud mereka sebagai bagian dari Office 365 untuk memahami bagaimana mereka cocok dengan Anda dan fungsi apa yang mereka berikan dibandingkan dengan versi di tempat.

Azure adalah platform cloud global dari Microsoft, yang mencakup serangkaian layanan cloud yang terus berkembang yang membantu organisasi memecahkan masalah bisnis mereka. Saat ini, Azure mencakup lebih dari 300 layanan yang dikelompokkan ke dalam berbagai kategori (komputasi, jaringan, penyimpanan, basis data, analitik, Internet, keamanan, devOps, kontainer, dll.).

Setelah muncul untuk pertama kalinya pada tahun 2009, Microsoft Azure sekarang menempati salah satu posisi terkemuka di pasar layanan cloud global, yang berhasil bersaing di sana dengan Amazon AWS.

Menurut laporan keuangan terbaru ( https://www.microsoft.com/en-us/Investor/earnings/FY-2019-Q4/press-release-webcast ) triwulanan (Q4 2019) Laba Microsoft meningkat sebesar 49% karena keberhasilan Office 365 dan bisnis cloud. Pendapatan dari Azure tumbuh 64%.

Azure, bersama dengan Office 365, adalah area utama di mana Microsoft menempatkan sumber daya keuangan dan organisasinya.

Banyaknya layanan pada platform Azure dapat membingungkan bahkan seorang spesialis TI yang berpengalaman, jadi di bawah ini akan menjadi deskripsi infrastruktur Windows berbasis server yang khas, di mana saya akan menunjukkan dalam kurung analog perkiraan mereka di awan Azure. Saya harap ini akan berfungsi sebagai titik awal untuk studi Azure, karena, seperti yang Anda tahu, Anda harus memulai dari yang kecil, secara bertahap bergerak lebih dalam.

Infrastruktur Windows sisi server yang khas terlihat seperti ini:

  • Active Directory (AD) dengan Kebijakan Grup dan DNS. (Azure Active Directory (AAD), Azure DNS) .
  • DHCP
  • Tukar server surat. (Exchange online sebagai bagian dari Office 365) .
  • Peternakan RDS dengan beberapa server terminal. (Mesin virtual Azure + Jaringan Virtual Azure + Penyimpanan Azure) .
  • Server file tempat karyawan menyimpan file mereka. (Penyimpanan File Azure, mesin virtual Azure + Jaringan Virtual Azure + Penyimpanan Azure)
  • Server dengan aplikasi dan database (1C, situs internal \ portal, CRM, dll.). (Azure SQL Database, Situs Web Azure, Microsoft Dynamics 365, mesin virtual Azure + Jaringan Virtual Azure + Penyimpanan Azure)

Tugas administrasi utama adalah:

  • Buat cadangan. (Cadangan Azure) .
  • Pengumpulan dan analisis log. (Azure Log Analytics) .
  • Otomatisasi tugas-tugas rutin. (Azure Automation) .
  • Memantau status layanan dan menerima pemberitahuan kegagalan (Monitor Azure) .

Saya akan menyarankan administrator Windows yang menyertai infrastruktur lokal untuk pertama-tama mencari analog layanan favorit mereka di cloud Azure untuk bekerja sedikit dengan mereka, menentukan kegunaannya bagi perusahaan, dan, mungkin, mengatur opsi hibrida, memilih yang terbaik dari kedua dunia.
