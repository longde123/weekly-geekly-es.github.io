Mi experiencia de salir y trabajar con Robot Framework

Hace poco más de un año, probé por primera vez el Robot Framework. Durante mi participación en un proyecto a gran escala, experimenté dos enfoques diferentes para la automatización de pruebas con esta herramienta: escribir pruebas en un marco de robot DSL puro y trabajar en conjunto con Python. Si el primer camino tiene un umbral de entrada bajo, entonces el segundo, en mi opinión, es más conveniente desde el punto de vista de apoyar grandes proyectos. Aunque no hay una diferencia fundamental entre los enfoques. De una forma u otra, todo se reduce a encontrar bibliotecas.

Sin embargo, vale la pena hablar sobre las características de los enfoques.



¿Quién es Robot y con qué come?


Probablemente valga la pena comenzar con la introducción de esta poderosa herramienta.

Robot Framework es un marco para pruebas basadas en palabras clave. Se utiliza para automatizar las pruebas de aceptación y ATDD (enfoque de desarrollo a través de las pruebas de aceptación). Este sistema tiene una sintaxis de datos de prueba fácil de usar y le permite automatizar las pruebas usando palabras clave. Además, Robot Framework tiene excelentes bibliotecas integradas y de terceros que le permiten iniciar e integrar rápidamente la automatización en los procesos de trabajo sin inventar sus propias bicicletas, el umbral de entrada muy bajo que mencioné anteriormente.

La estructura básica de Robot Framework se implementa usando Python y también funciona en Jython (JVM) y IronPython (.NET).

Se pueden encontrar ejemplos de uso, así como documentación completa sobre el marco y las bibliotecas internas en el sitio web oficial del proyecto: http://robotframework.org/ .

Mis primeros pasos Primer acercamiento


Por primera vez, me encontré con el Robot Framework hace un año, después de cambiar mi trabajo. Antes de eso, solo había automatizado en Java y C #.

Eligiendo por mí mismo las herramientas con las que tengo que lidiar con las pruebas existentes, entrevisté a nuevos colegas sobre sus preferencias. No acordaron el mejor IDE para trabajar con Robot Framework. Los complementos para varios editores de texto e IDE, como PyCharm, permiten trabajar principalmente con Robot Framework. Y las recomendaciones que recopilé se dividieron 50/50 entre Atom y PyCharm. Por supuesto, hay RIDE, pero esto no es una panacea. En ese momento (hace un año), no encontré la documentación normal en él, y en el que encontré, no vi grandes ventajas para mi tarea. Por lo tanto, para empezar, decidí probar Atom con complementos. Al clonar el repositorio, comencé a estudiar lentamente lo que estaba sucediendo en nuestras pruebas y en el propio Robot Framework.

Me he conectado al proyecto para prácticamente todo. Un montón de Jython + Robot Framework ya funcionaba allí, se ensambló una enorme base de código (escrita en el propio DSL Robot Framework) a partir de más de 1000 pruebas y varios miles de líneas de código de bibliotecas auxiliares en Java.

Según tengo entendido, cuando comenzó el proyecto, es decir incluso antes de llegar al departamento, eran principalmente especialistas en Java quienes trabajaban en él, y el producto que se estaba probando estaba en Java, por lo que al elegir un enfoque, nos centramos en los recursos disponibles. En general, el cálculo fue algo como esto: después de resolver algunos problemas relacionados con la integración de Robot Framework y Java (principalmente debido al hecho de que Java es un lenguaje compilado, pero se interpretan el mismo Python y las pruebas en el paquete Python + RF), luego Es fácil atraer a expertos de terceros que solo conocen el DSL Robot Framework y escribir pruebas silenciosamente sobre palabras clave. Es cierto que los colegas tuvieron que dedicar mucho esfuerzo a crear bibliotecas en Java, por lo tanto, sin condiciones del cliente, no recomendarían ese camino.

Después de refactorizar un poco, como parte de mi primera tarea, primero realicé las pruebas. Como se usó el paquete Jython + RF, todo fue recopilado por maven, y los archivos del robot simplemente se copiaron en el directorio de destino para su posterior ejecución.

Las pruebas se ejecutaron mediante scripts (archivos .bat o .sh) que tomaron la ruta a un caso de prueba separado (un archivo .robot separado) o un plan de prueba (un archivo con una lista de rutas relativas a los casos de prueba).

La refactorización afectó una gran cantidad de pruebas, por lo que la primera ejecución tardó unos 15 minutos. Después de su finalización, es hora de echar un vistazo a los informes proporcionados por Robot Framework.
El informe estándar (en la captura de pantalla anterior) consta de los archivos report.html y log.html:

  • el informe contiene un resumen general de la pasada ejecución, donde puede ver los resultados de la superficie de todas las pruebas (Aprobado o Fallido);
  • en el archivo de registro puede ver información más detallada: la ejecución de cada prueba paso a paso. Allí puede mostrar todo lo que necesita para depurar las pruebas.

Honestamente, desde el primer vistazo al informe de Robot Framework, el ojo comienza a temblar un poco: se muestra una gran cantidad de información y toma algo de tiempo comprender la estructura de extremo a extremo de las pruebas y desarrollar la habilidad para leer dicho registro. Pero este negocio no es tan complicado. Dentro de un par de meses podría citar The Matrix: “Tu cerebro hace la traducción misma. Ni siquiera veo el código. Veo una rubia, una morena y una pelirroja ". Entonces, vi toda la información necesaria en un archivo sin herramientas adicionales.

La ventaja es que la salida se puede controlar: existen diferentes niveles de registro que determinan qué información se mostrará y cuál no. El nivel se puede ajustar incluso para cada línea individualmente a través del método de biblioteca incorporado. Al mismo tiempo, conocer el orden de las llamadas dentro de las bibliotecas de prueba y auxiliares no es superfluo, es más fácil detectar el momento del error.

Utilizando DSL Robot Framework como nuestra herramienta principal, trabajamos durante unos seis meses. Durante este tiempo, cambié de preferencias personales de Atom a VSCode, pero esto no cambió la esencia del enfoque.

Sin embargo, el proyecto se estaba desarrollando. En la iteración final, la biblioteca auxiliar para trabajar con la base de datos totalizó 6.700 líneas de código en un Robot Framework puro. Con tal escala de la base del código, se ha vuelto difícil de mantener, refactorizando los recursos requeridos que no fueron asignados.
La última palabra en la aplicación del primer enfoque pertenecía a los negocios. El cliente de nuestro proyecto también trabajó con otros equipos en tareas relacionadas. En una de las pistas paralelas, vio, desde su punto de vista, una opción más efectiva e ilustrativa para usar Robot Framework, que se comenzó a implementar específicamente para la administración.

Segundo enfoque


El segundo enfoque fue el desarrollo de pruebas en Python junto con Robot Framework. En lugar de crear todo en la sintaxis de DSL Robot Framework, comenzamos a escribir bibliotecas auxiliares y otras interacciones de bajo nivel con el producto de prueba en Python. Y el Robot Framework, de hecho, se convirtió en un corredor.

Dado que Python es un lenguaje puro de alto nivel, no DSL, hay más opciones de estructuración, es más fácil resolverlo todo. Como mínimo, puede usar el IDE de Python para ayudarlo a encontrar los mismos métodos (hacen lo mismo, pero se les llama de manera diferente) o incluso escribir un código para usted. Algunos datos podrían estar envueltos en generadores, colgar decoradores en funciones, etc. En este contexto, las herramientas del primer enfoque (Robot Framework puro) parecen bastante duras; de hecho, era el Bloc de notas con resaltado de sintaxis. Sin setters ni getters que IntelliJ escribe para ti. Así que estaba feliz de volver a un idioma de nivel superior. Trabajar con este enfoque es más como un desarrollo normal. Es cierto que hay una mosca en la pomada. Sin baile adicional, es imposible entender lo que cayó dentro de Python, llamado desde RF.

Poco a poco, las primeras pruebas y bibliotecas auxiliares para nuestro subsistema comenzaron a escribirse. Durante el tiempo que pude trabajar con un nuevo enfoque, sentí que con otra herramienta realmente hay más oportunidades. Pero, de hecho, la redacción de las pruebas no ha cambiado mucho. En este proyecto en particular, dentro del código de Python, los métodos de biblioteca integrados de Robot Framework todavía se llamaban. Esto se debió a los requisitos particulares del cliente para el desarrollo de pruebas. Simplemente separamos la parte ejecutable del algoritmo del caso de prueba.

Cual es mejor


Personalmente me gusta el segundo enfoque. Sin embargo, al elegir la ruta de su proyecto, vale la pena comenzar desde la tarea: quién escribe las pruebas y cómo.

Como dije anteriormente, Python (el segundo enfoque) brinda más oportunidades, pero en este caso, se necesitan personas que estén familiarizadas con este lenguaje en el proyecto. El Robot Framework en sí (y el primer enfoque) es menos exigente: puede abordarlo leyendo la documentación oficial en su DSL. Después de estudiar la guía del usuario, puede crear cualquier prueba; se verán bastante limpias.

Como resultado, Robot Framework y cómo lo usamos al principio son más adecuados para los probadores manuales de ayer sin ninguna experiencia de programación explícita en lenguajes de alto nivel. Incluso una persona que no está muy involucrada en la programación puede escribir una prueba, simplemente llamando las palabras clave necesarias.

Sin embargo, si desea mantener la parte ejecutable separada, siguiendo el ejemplo de nuestro proyecto, y al mismo tiempo refactorizar el código en IDE amigables, el segundo enfoque es para usted.
