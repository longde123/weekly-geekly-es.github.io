为什么（今天）返回444并不总是有用的

Nginx Web服务器具有出色的444响应代码，可以“关闭”连接而不发送数据。 此功能在过滤虚假流量方面非常有用-如果我们确定客户端在某些条件下无效，则无需通知他（例如，第403个答案）。 简单地停止数据传输会更有效，这通常可以大大减少服务器上的负载。

在阻止点击流行站点链接和引荐垃圾邮件，防止DDoS防护等说明中，随处可见使用这种答案的建议。

而且，通常来说，这些年来，这些技巧几乎可以不经使用就可以使用，但是...现代浏览器并不会停滞不前，并定期为我们带来新的惊喜。

因此，我们有一个网站，并且我们希望在没有任何条件且不传输任何数据的情况下限制对位置的访问： 

 server { ... location = /code/444 { return 444; } ... }

等待中：

  • 浏览器将发送一个请求；
  • 服务器将收到一个请求，关闭连接，并将一行代码为444的行写入日志；
  • 不接收数据，浏览器将显示错误并停止与服务器通信。

因此，期望收到来自客户端的一个请求，实际上，我们收到了3-20个请求。 除了我们完全不需要的呼叫外，还存在将这样的日志馈入反DDoS脚本并阻止该IP的风险。 这似乎是麻烦的一半，因为我们已经“给”了444给他，也就是说，我们不为他感到难过，但是事情可能反过来发生-我们将向客户展示我们根本没有计划的事情。

修改配置： 

 map $http_referer $code_if { "~*https://habr.com/post/415565/" 1; "http://tison.ru/ref" 1; default 0; } ... server { ... location = /code/444 { return 444; } ... location = /codeif/444 { if ( $code_if = 1 ) { return 444; } add_header "Content-Type" "text/html; charset=UTF-8" always; return 200 "Expected code 444"; } ... }
