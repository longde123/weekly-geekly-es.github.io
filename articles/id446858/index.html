Cara menggunakan SAP HANA: parsing metode yang berbeda

SAP HANA adalah sistem manajemen basis data dalam-memori yang populer yang mencakup layanan dan analisis Gudang Data, middleware tertanam, server aplikasi, platform untuk mengonfigurasi atau mengembangkan utilitas baru. Dengan menghilangkan latensi DBMS tradisional dengan SAP HANA, Anda dapat sangat meningkatkan kinerja sistem, pemrosesan transaksi (OLTP), dan intelijen bisnis (OLAP).



Anda dapat menggunakan SAP HANA dalam mode Peralatan dan TDI (dalam hal lingkungan produksi). Untuk setiap opsi, pabrikan memiliki persyaratannya sendiri. Dalam posting ini kita akan berbicara tentang kelebihan dan kekurangan dari berbagai opsi, serta kejelasan, tentang proyek nyata kami dengan SAP HANA.

SAP HANA terdiri dari 3 komponen utama - host, instance dan sistem.

Host adalah server atau lingkungan operasi untuk database SAP HANA. Komponen yang diperlukan adalah CPU, RAM, penyimpanan, jaringan dan OS. Host menyediakan tautan ke direktori instalasi, data, log, atau langsung ke sistem penyimpanan. Dalam hal ini, penyimpanan untuk pemasangan SAP HANA tidak harus berada di host. Jika sistem memiliki beberapa host, Anda memerlukan penyimpanan bersama, atau yang tersedia sesuai permintaan dari semua host.

Instance adalah seperangkat komponen sistem SAP HANA yang diinstal pada satu host. Komponen utama adalah Index Server dan Name Server. Yang pertama, yang juga disebut "server produksi", memproses permintaan, mengelola gudang data aktual dan kernel basis data. Name Server menyimpan informasi tentang topologi instalasi SAP HANA - di mana komponen bekerja dan data apa yang ada di server.

Suatu sistem adalah satu atau lebih contoh dengan nomor yang sama. Sebenarnya, ini adalah elemen terpisah yang dapat Anda aktifkan, nonaktifkan, atau salin (buat cadangan). Data didistribusikan dalam memori berbagai server yang membentuk sistem SAP HANA.


Sistem dapat dikonfigurasi sebagai host tunggal (satu instance pada satu host) atau multi-host, didistribusikan (beberapa instance SAP HANA didistribusikan di beberapa host, setiap host memiliki satu instance). Dalam sistem multi-host, setiap instance harus memiliki nomor yang sama. Sistem SAP HANA diidentifikasi oleh ID Sistem (SID), jumlah unik dari tiga karakter alfanumerik.

Virtualisasi SAP HANA


Salah satu batasan utama SAP HANA adalah dukungan hanya satu sistem - satu contoh dengan server SID yang unik. Anda dapat menggunakan virtualisasi untuk membuat penggunaan perangkat keras yang lebih efisien atau mengurangi jumlah server di pusat data. Dengan demikian, lanskap lain dapat hidup berdampingan di server yang sama dengan sistem yang memiliki persyaratan lebih rendah (sistem tidak produktif). Untuk server HA / DR siaga, virtualisasi dapat meningkatkan kecepatan switching antara mesin virtual produktif dan non-produktif.

SAP HANA mencakup dukungan untuk hypervisor VMWare ESX. Ini berarti bahwa sistem SAP HANA yang berbeda - instalasi SAP HANA dengan SID yang berbeda - dapat hidup berdampingan pada host yang sama (server fisik bersama) di mesin virtual yang berbeda. Setiap mesin virtual harus bekerja dalam OS yang didukung.

Untuk lingkungan yang produktif, virtualisasi SAP HANA memiliki batasan parah:

  • Scale-out scaling tidak didukung - virtualisasi hanya dapat digunakan dengan sistem Scale-Up, apakah itu BwoH / DM / SoH atau SoH murni;
  • Virtualisasi harus dilakukan dalam aturan yang ditetapkan untuk perangkat Appliance atau TDI.
  • hanya ada satu mesin virtual di General Availability (GA) - perusahaan yang ingin menggunakan virtualisasi dengan lingkungan produktif HANA harus berpartisipasi dalam program Ketersediaan Terkendali dengan SAP.

Dalam lingkungan non-produktif di mana keterbatasan yang tercantum tidak ada, virtualisasi dapat digunakan untuk mengoptimalkan penggunaan peralatan.

Topologi SAP HANA


Mari kita beralih ke penerapan SAP HANA. Dua topologi didefinisikan di sini.

  • Scale-up adalah satu server besar. Ketika basis HANA tumbuh, server itu sendiri tumbuh: jumlah CPU dan jumlah memori meningkat. Dalam solusi dengan Ketersediaan Tinggi (HA) dan Pemulihan Bencana (DR), server redundan atau toleran kesalahan harus cocok dengan server produktif dalam hal kinerja.
  • Scale-out - seluruh volume sistem SAP HANA didistribusikan di beberapa server yang identik. Server master berisi informasi untuk Server Indeks dan Server Nama. Server slave tidak mengandung data ini - kecuali server, yang mengambil alih fungsi Master jika terjadi kegagalan pada server utama. Server yang berfungsi (Server Indeks) mengelola segmen data yang ditugaskan kepadanya, dan juga menanggapi permintaan. Server Nama menyadari bagaimana data didistribusikan antara server produksi. Dalam kasus pertumbuhan HANA, node lain hanya ditambahkan ke konfigurasi server saat ini. Dalam topologi seperti itu, cukup memiliki satu simpul cadangan untuk memastikan keamanan seluruh server.


Persyaratan perangkat keras SAP


SAP memiliki persyaratan perangkat keras untuk HANA. Mereka berhubungan dengan lingkungan produktif - karena karakteristik minimal non-prod sudah mencukupi. Jadi, berikut adalah persyaratan untuk lingkungan produksi:

  • CPU Intel Xeon v5 (SkyLake) / 8880/90/94 v4 (Broadwell)
  • dari 128 GB RAM untuk aplikasi BW dengan 2 CPU, 256 GB dengan 4+ CPU;

Menyebarkan SAP HANA dalam Peralatan dan Mode TDI


Sekarang mari kita berlatih dan berbicara tentang bagaimana menerapkan SAP HANA dalam mode Appliance dan TDI. Kami menggunakan platform SAP HANA kami untuk ini berdasarkan server BullSequana S dan Bullion S, yang disertifikasi SAP untuk bekerja dalam mode ini.

Sedikit bantuan pada produk. BullSequana S yang ditenagai oleh Intel Xeon Scalable mencakup berbagai model, hingga 32 CPU dalam satu server. Server dibangun pada desain modular, memberikan skalabilitas hingga 32 CPU dan jumlah GPU yang sama. RAM - dari 64 GB hingga 48 TB. Di antara fitur-fitur BullSequana S adalah dukungan untuk AI perusahaan untuk peningkatan kinerja, analitik data yang lebih cepat, peningkatan komputasi dalam memori, dan peningkatan menggunakan virtualisasi dan teknologi cloud.

Bullion S hadir dengan Intel Xeon E7 v4 Family CPU. Jumlah prosesor maksimum adalah 16. Skala RAM dari 128 GB hingga 24 TB. Sejumlah besar fitur RAS menyediakan ketersediaan tinggi untuk infrastruktur penting seperti SAP HANA. Bullion S cocok untuk konsolidasi pusat data secara besar-besaran, bekerja dengan aplikasi dalam Memori, memigrasi mainframe atau sistem lama.

SAP HANA Appliance


Appliance adalah solusi pra-konfigurasi yang mencakup server, penyimpanan, dan paket perangkat lunak untuk implementasi turnkey, dengan layanan dukungan terpusat dan tingkat kinerja yang disepakati. Di sini HANA hadir dalam perangkat keras dan lunak yang telah dikonfigurasi sebelumnya, terintegrasi penuh dan bersertifikat. Perangkat dalam mode Peralatan siap dipasang di pusat data, dan sistem operasi, SAP HANA, dan (jika perlu) instance VMWare tambahan sudah dikonfigurasikan dan diinstal.

Sertifikasi SAP menentukan tingkat kinerja yang dijamin, serta model CPU, jumlah RAM dan penyimpanan. Setelah sertifikasi, Anda tidak dapat mengubah konfigurasi tanpa kehilangan garansi. HANA SAP menawarkan tiga opsi untuk meningkatkan skala platform.

  • Scale-Up BWoH / DM / SoH - penskalaan vertikal, yang cocok untuk sistem tunggal (satu SID). Perangkat peralatan tumbuh pada 256/384 GB dimulai dengan SAP HANA SPS 11. Rasio ini menunjukkan kapasitas maksimum yang didukung oleh satu CPU dan umum untuk seluruh daftar perangkat Peralatan bersertifikat. Appliance scalable BWoH / DM / SoH paling cocok untuk BW pada aplikasi HANA (BWoH), Data Mart (DM) dan SAP Suite on HANA (SoH).
  • Scale-Up SoH adalah versi ringan dari model sebelumnya, dengan pembatasan lebih sedikit pada jumlah RAM. Ini masih merupakan server yang dapat diukur secara vertikal, tetapi jumlah maksimum RAM untuk 2 prosesor sudah 1536 GB (hingga versi SPS11) dan 3 TB (SPS12 +). Hanya cocok untuk SoH.
  • Scale-Out adalah opsi dengan penskalaan horizontal, sistem yang mendukung konfigurasi multi-server. Penskalaan horizontal paling cocok untuk BW dan, dengan beberapa keterbatasan, SoH.

Di server BullSequana S dan Bullion S, penskalaan vertikal sangat penting karena memiliki batasan operasional yang lebih sedikit dan memerlukan administrasi yang lebih sedikit. Untuk mode Appliance, ada garis besar perangkat yang berbeda.

Berikut adalah contoh konfigurasi penyimpanan EMC Unity 450F dalam pengaturan kami:

  • Pengendali: 2

  • RAID: level 5 (8 +1), 4 grup RAID
  • Antarmuka: 4 FC per controller, 8 atau 16 Gb / s

Appliance adalah opsi penyebaran yang andal, tetapi memiliki kelemahan besar: ada sedikit kebebasan dalam mengkonfigurasi perangkat keras . Selain itu, opsi ini mungkin memerlukan perubahan dalam proses departemen TI.

SAP HANA TDI


Alternatif untuk Appliance adalah mode TDI (Tailored Data center Integration), di mana Anda dapat memilih produsen dan komponen infrastruktur tertentu tergantung pada keinginan pelanggan - dengan mempertimbangkan tugas dan beban kerja. Misalnya, SAN dapat digunakan kembali di pusat data, dengan beberapa disk disediakan untuk instalasi HANA.

Dibandingkan dengan Appliance, dalam mode TDI, pengguna diberi lebih banyak kebebasan untuk memenuhi persyaratan. Ini sangat menyederhanakan integrasi HANA ke pusat data - Anda dapat membangun infrastruktur khusus Anda sendiri. Misalnya, variasikan jenis dan jumlah prosesor tergantung pada beban.


Untuk menghitung kapasitas, disarankan untuk menggunakan SAP Quick Sizer, alat sederhana yang menyediakan kebutuhan CPU dan memori untuk beban kerja yang berbeda di SAP HANA. Kemudian, Anda dapat menghubungi Dukungan Global SAP Aktif untuk merencanakan lanskap TI Anda. Setelah itu, mitra perangkat keras SAP HANA mengubah hasil perhitungan menjadi berbagai kemungkinan konfigurasi sistem - baik di perangkat keras teratas maupun yang lebih sederhana. Dalam mode TDI, server diperbolehkan untuk menggunakan Intel E7 CPU, termasuk Intel Broadwell E7 dan Skylake-SP (Platinum, Gold, Silver dengan 8 atau lebih core per prosesor), serta IBM Power8 / 9.

Server dikirimkan tanpa penyimpanan, sakelar, dan rak, tetapi persyaratan perangkat keras tetap sama seperti dalam mode Peralatan - node tunggal yang sama, solusi dengan penskalaan vertikal atau horizontal. SAP hanya membutuhkan server, penyimpanan, dan sakelar bersertifikasi untuk digunakan , tetapi ini bukan masalah - sebagian besar produsen memiliki hampir semua peralatan bersertifikat.

Pengujian kinerja harus dilakukan menggunakan tes HWCCT (Perangkat Konfigurasi Konfigurasi Perangkat Keras) , yang memungkinkan Anda memverifikasi kepatuhan dengan KPI SAP tertentu. Dan ada persyaratan yang tidak terkait dengan perangkat keras: HANA, OS, dan hypervisor (opsional) harus diinstal oleh spesialis dengan sertifikasi SAP . Hanya sistem yang mematuhi semua aturan ini yang dapat menerima dukungan SAP untuk kinerja.

Garis server BullSequana S dalam mode TDI mirip dengan garis dalam mode Appliance, tetapi tanpa penyimpanan, sakelar, dan rak. Anda dapat menginstal sistem penyimpanan apa pun dari daftar SAP bersertifikat - VNX, XtremIO, NetApp, dan lainnya. Misalnya, jika VNX5400 memenuhi persyaratan kinerja SAP HANA, Anda dapat menghubungkan Dell EMC Unity 450F sebagai bagian dari konfigurasi TDI. Jika perlu, adaptor FC (1 atau 10 Gbit / dtk), serta sakelar Ethernet, dipasang.

Sekarang, agar Anda dapat lebih jelas menyajikan mode yang dijelaskan, kami akan berbicara tentang beberapa kasus nyata kami.

Peralatan + TDI: HANA untuk toko online


Toko online Mall.cz, bagian dari Grup Mall, didirikan pada tahun 2000. Ini memiliki cabang di Republik Ceko, Slovakia, Polandia, Hongaria, Slovenia, Kroasia dan Rumania. Ini adalah toko online terbesar di negara ini, menjual hingga 75 ribu barang per hari, pendapatannya pada tahun 2017 berjumlah sekitar 280 juta euro.

Upgrade ke infrastruktur pusat data diperlukan karena migrasi ke SAP HANA. Perkiraan ukuran adalah 2x6 TB untuk lingkungan prod dan 6 TB untuk lingkungan tes / dev. Ini memerlukan solusi pemulihan bencana untuk lingkungan produksi SAP HANA di kluster aktif-aktif.

Pada saat pengumuman tender, pelanggan memiliki sistem untuk SAP berdasarkan server rak dan blade standar. Dua pusat data, yang terletak pada jarak sekitar 10 km dari satu sama lain, dilengkapi dengan berbagai sistem penyimpanan - IBM SVC, HP dan Dell. Sistem utama bekerja dalam mode pemulihan bencana.

Pertama, pelanggan meminta solusi tersertifikasi dalam mode Peralatan untuk SAP HANA untuk semua sistem (Lingkungan produksi dan pengujian / pengembangan) dengan pertumbuhan hingga 12 TB. Tetapi karena keterbatasan anggaran, opsi lain mulai dipertimbangkan - misalnya, jumlah CPU yang lebih besar dengan modul RAM yang lebih kecil (modul 64 GB, bukan modul 128 GB). Selain itu, sistem penyimpanan bersama untuk lingkungan Produksi dan pengujian / pengembangan dipertimbangkan untuk optimasi harga.



Kami menyetujui 4 CPU dan 6 TB RAM untuk lingkungan Produksi, dengan kemungkinan pertumbuhan. Untuk lingkungan uji / dev dalam mode TDI, kami memutuskan untuk melakukannya dengan CPU yang lebih murah - kami mendapatkan 8 CPU dan 6 TB RAM. Karena semakin banyak fungsi yang diminta oleh pelanggan - replikasi, cadangan, produksi, dan lingkungan pengujian / pengembangan pada platform kedua - sistem penyimpanan DellEMC Unity digunakan sebagai ganti disk internal dalam konfigurasi flash penuh. Selain itu, pelanggan meminta solusi pemulihan bencana berdasarkan Replikasi Sistem HANA (HSR) dengan simpul kuorum di situs ketiga.

Konfigurasi akhir untuk lingkungan Prod terdiri dari server BullSequana S400 pada Intel Xeon P8176M (28 core, 2,10 GHz, 165 W) dan dengan 6 TB RAM. Penyimpanan - Unity 450F 10x 3,84 TB. Untuk pemulihan bencana untuk lingkungan Prod, kami menggunakan BullSequana S400 pada Intel Xeon P8176M (28 core, 2,10 GHz, 165 W) dengan 6 TB RAM. Untuk lingkungan pengujian / pengembangan, kami mengambil server BullSequana S800 dengan Intel Xeon P8153 (16 core, 2,00 GHz, 125 W) dan 6 TB RAM plus penyimpanan Unity 450F 15x 3,84 TB. Sebagai kuorum, server aplikasi (Solusi VxRail) dan solusi cadangan (DataDomain), para ahli kami menginstal dan mengkonfigurasi server DellEMC.


Peralatan siap untuk peningkatan di masa mendatang. Pelanggan mengharapkan pertumbuhan ukuran HANA pada tahun 2019, dan ia hanya dapat menginstal modul baru di rak.

Peralatan: HANA untuk integrator perjalanan utama


Kali ini, klien kami adalah penyedia layanan TI besar yang terlibat dalam pengembangan solusi teknologi untuk perusahaan perjalanan. Pelanggan meluncurkan proyek SAP HANA yang ambisius untuk memperkenalkan sistem penagihan baru. Solusi Peralatan diperlukan dengan 8 TB RAM untuk lingkungan Produksi dan PreProd. Sesuai dengan rekomendasi SAP, pelanggan memilih opsi dengan penskalaan vertikal.

Tantangan utama adalah pengenalan infrastruktur perangkat keras berdasarkan perangkat bersertifikasi Appliance untuk SAP HANA. Kriteria prioritas adalah efisiensi biaya, kinerja tinggi, skalabilitas, dan ketersediaan data yang tinggi.

Kami telah mengusulkan dan menerapkan solusi yang disertifikasi oleh SAP, yang mencakup dua server Bullion S16 - untuk lingkungan Prod dan PreProd. Peralatan ini berjalan pada prosesor Intel Xeon E7-v4 8890 (24 core, 2,20 GHz, 165 W) dan dilengkapi dengan 16 TB RAM. Sembilan server Bullion S4 (22 core, 2,20 GHz, 150 W) untuk 4 TB RAM dipasang untuk BW dan lingkungan Dev / Test. Sebagai tempat penyimpanan digunakan hybrid EMC Unity.

Solusi ini memberikan dukungan untuk penskalaan untuk semua elemen perangkat - misalnya, hingga 16 soket dengan CPU Intel Xeon E7-v4. Administrasi dalam konfigurasi ini disederhanakan - khususnya, untuk mengkonfigurasi ulang atau mempartisi server menjadi partisi.

Appliance + TDI: HANA untuk Ahli Metalurgi


MMC Norilsk Nickel, salah satu produsen nikel dan paladium terbesar, memutuskan untuk meningkatkan platform perangkat keras SAP HANA untuk mendukung aplikasi dan proyek bisnis penting. Perluasan lansekap yang ada dalam hal daya komputasi diperlukan. Salah satu kondisi utama yang diajukan oleh pelanggan adalah ketersediaan platform yang tinggi - terlepas dari keterbatasan perangkat keras.



Untuk lingkungan yang produktif, kami menggunakan server dan penyimpanan Bullion S8 dalam mode SAP HANA Appliance. Untuk HA dan test / dev, platform dikerahkan dalam mode TDI. Kami menggunakan satu server Bull Bullion S8, dua server Bull Bullion S6 dan sistem penyimpanan hybrid. Kombinasi ini memungkinkan untuk secara signifikan meningkatkan kecepatan aplikasi lansekap SAP, meningkatkan jumlah daya komputasi dan sumber daya penyimpanan dan meminimalkan biaya pengoperasian. Adalah penting bahwa klien memiliki kemampuan untuk meningkatkan hingga 16 CPU.
