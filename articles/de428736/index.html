Verwenden von Retrofit 2.x als REST-Client - Lernprogramm

1. Nachrüstung


1.1. Was ist Nachrüstung?


Retrofit ist ein REST-Client für Java und Android. Es macht es einfach, JSON (oder andere strukturierte Daten) über einen REST-basierten Webdienst abzurufen und zu laden. In Retrofit konfigurieren Sie, welcher Konverter zum Serialisieren von Daten verwendet wird. Normalerweise wird GSon für JSON verwendet, aber Sie können Ihre eigenen Konverter hinzufügen, um XML oder andere Protokolle zu verarbeiten. Retrofit verwendet die OkHttp-Bibliothek für HTTP-Anforderungen.

Sie können JSON-basierte Java-Objekte mit dem folgenden Tool erstellen : www.jsonschema2pojo.org Dies kann nützlich sein, um komplexe Java-Datenstrukturen aus vorhandenem JSON zu erstellen.


1.2. Nachrüstung verwenden


Um mit Retrofit arbeiten zu können, benötigen Sie die folgenden drei Klassen:

  • Modellklasse, die als JSON-Modell verwendet wird
  • Schnittstellen, die mögliche HTTP-Vorgänge identifizieren
  • Die Retrofit.Builder-Klasse ist eine Instanz, die die Builder-Schnittstelle und die API verwendet, um eine URL-Endpunktdefinition für HTTP-Vorgänge zu definieren.

Jede Schnittstellenmethode repräsentiert einen der möglichen API-Aufrufe. Es muss eine HTTP-Annotation (GET, POST usw.) enthalten, um den Anfragetyp und die relative URL anzugeben. Der Rückgabewert vervollständigt die Antwort im Call-Objekt mit dem erwarteten Ergebnistyp. 

@GET("users") Call<List<User>> getUsers();

Sie können die Ersatzblöcke und Abfrageparameter verwenden, um die URL zu konfigurieren. Der Ersetzungsblock wird mit {} an die relative URL angehängt. Bei Verwendung der Annotation @ Path für einen Methodenparameter wird der Wert dieses Parameters an einen bestimmten Ersetzungsblock gebunden. 

 @GET("users/{name}/commits") Call<List<Commit>> getCommitsByName(@Path("name") String name);

Abfrageparameter werden mithilfe der Annotation @ Query zum Methodenparameter hinzugefügt. Sie werden automatisch am Ende der URL hinzugefügt. 

 @GET("users") Call<User> getUserById(@Query("id") Integer id);

Die Annotation @ Body zum Methodenparameter weist Retrofit an, das Objekt als aufrufenden Anforderungshauptteil zu verwenden. 

 @POST("users") Call<User> postUser(@Body User user)

In dieser Übung erstellen Sie einen eigenständigen REST-Client. Antworten werden vom Mock-Server generiert.

Erstellen Sie ein neues Gradle-Projekt mit dem Namen com.vogella.retrofitgerrit. Fügen Sie src / main / java ein neues Paket mit dem Namen com.vogella.retrofitgerrit hinzu.

Fügen Sie der Datei build.gradle die folgenden Abhängigkeiten hinzu. 

 // retrofit implementation 'com.squareup.retrofit2:retrofit:2.1.0' implementation 'com.squareup.retrofit2:converter-gson:2.1.0' // Junit testImplementation("org.junit.jupiter:junit-jupiter-api:5.0.0") testRuntime("org.junit.jupiter:junit-jupiter-engine:5.0.0") // to run JUnit 3/4 tests: testImplementation("junit:junit:4.12") testRuntime("org.junit.vintage:junit-vintage-engine:4.12.0")

3.2. API und Nachrüstadapter definieren


In Gerrits JSON-Antwort interessiert uns nur die Frage der Änderungen. Erstellen Sie daher die folgende Datenklasse im zuvor hinzugefügten Standardpaket. 

 package com.vogella.java.retrofitgerrit; public class Change { String subject; public String getSubject() { return subject; } public void setSubject(String subject) { this.subject = subject; } }

Definieren Sie die REST-API für die Nachrüstung über die folgende Schnittstelle. 

 package com.vogella.java.retrofitgerrit; import java.util.List; import retrofit2.Call; import retrofit2.http.GET; import retrofit2.http.Query; public interface GerritAPI { @GET("changes/") Call<List<Change>> loadChanges(@Query("q") String status); }

Erstellen Sie die folgende Controller-Klasse. Diese Klasse erstellt einen Retrofit-Client, ruft die Gerrit-API auf und verarbeitet das Ergebnis (druckt das Ergebnis des Aufrufs an die Konsole). 

 package com.vogella.java.retrofitgerrit; import java.util.List; import com.google.gson.Gson; import com.google.gson.GsonBuilder; import retrofit2.Call; import retrofit2.Callback; import retrofit2.Response; import retrofit2.Retrofit; import retrofit2.converter.gson.GsonConverterFactory; public class Controller implements Callback<List<Change>> { static final String BASE_URL = "https://git.eclipse.org/r/"; public void start() { Gson gson = new GsonBuilder() .setLenient() .create(); Retrofit retrofit = new Retrofit.Builder() .baseUrl(BASE_URL) .addConverterFactory(GsonConverterFactory.create(gson)) .build(); GerritAPI gerritAPI = retrofit.create(GerritAPI.class); Call<List<Change>> call = gerritAPI.loadChanges("status:open"); call.enqueue(this); } @Override public void onResponse(Call<List<Change>> call, Response<List<Change>> response) { if(response.isSuccessful()) { List<Change> changesList = response.body(); changesList.forEach(change -> System.out.println(change.subject)); } else { System.out.println(response.errorBody()); } } @Override public void onFailure(Call<List<Change>> call, Throwable t) { t.printStackTrace(); } }

Erstellen Sie eine Klasse mit einer Hauptmethode, um den Controller zu starten. 

 package com.vogella.java.retrofitgerrit; public class Main { public static void main(String[] args) { Controller controller = new Controller(); controller.start(); } }

4. Wandler und Adapter nachrüsten


4.1. Nachrüstkonverter


Die Nachrüstung kann für die Verwendung eines bestimmten Konverters konfiguriert werden. Dieser Konverter übernimmt die (De-) Serialisierung von Daten. Für verschiedene Serialisierungsformate stehen bereits mehrere Konverter zur Verfügung.

  • So konvertieren Sie nach JSON und umgekehrt:
    • Gson: com.squareup.retrofit: converter-gson
    • Jackson: com.squareup.retrofit: Konverter-Jackson
    • Moshi: com.squareup.retrofit: Konverter-Moshi

  • So konvertieren Sie in Protokollpuffer und umgekehrt:
    • Protobuf: com.squareup.retrofit: converter-protobuf
    • Draht: com.squareup.retrofit: Konverterdraht

  • So konvertieren Sie nach XML und umgekehrt:
    • Einfaches XML: com.squareup.retrofit: converter-simplexml


Zusätzlich zu den aufgelisteten Konvertern können Sie auch eigene erstellen, um andere Protokolle zu verarbeiten, indem Sie die Converter.Factory-Klasse erweitern.

4.2. Nachrüstadapter


Retrofit kann auch mit Adaptern für die Schnittstelle mit anderen Bibliotheken wie RxJava 2.x, Java 8 und Guava erweitert werden.

Eine Übersicht der verfügbaren Adapter finden Sie unter Github square / retrofit / retrofit-adapters / .

Beispielsweise kann der RxJava 2.x-Adapter mit Gradle verbunden werden: 

 compile 'com.squareup.retrofit2:adapter-rxjava2:latest.version'

oder mit Apache Maven: 

 <dependency> <groupId>com.squareup.retrofit2</groupId> <artifactId>adapter-rxjava2</artifactId> <version>latest.version</version> </dependency>

Um einen Adapter hinzuzufügen, müssen Sie die Methode retrofit2.Retrofit.Builder.addCallAdapterFactory (Factory) verwenden. 

 Retrofit retrofit = new Retrofit.Builder() .baseUrl("https://api.example.com/") .addCallAdapterFactory(RxJava2CallAdapterFactory.create()) .build();

Mit diesem Adapter können Retrofit-Schnittstellen RxJava 2.x-Typen zurückgeben, z. B. Observable, Flowable oder Single usw. 

 @GET("users") Observable<List<User>> getUsers();

5. Authentifizierung nachrüsten


Retrofit unterstützt API-Aufrufe, die eine Authentifizierung erfordern. Die Authentifizierung kann mit einem Benutzernamen und einem Kennwort (HTTP-Basisauthentifizierung) oder einem API-Token erfolgen.

Es gibt zwei Möglichkeiten, die Authentifizierung zu verwalten. Die erste Methode besteht darin, den Anforderungsheader mithilfe von Anmerkungen zu verwalten. Eine andere Möglichkeit besteht darin, dafür einen OkHttp-Interceptor zu verwenden.

5.1. Authentifizierung mit Anmerkungen


Angenommen, Sie möchten Informationen zu einem Benutzer anfordern, für den eine Authentifizierung erforderlich ist. Sie können dies tun, indem Sie der API-Definition einen neuen Parameter hinzufügen, zum Beispiel: 

 @GET("user") Call<UserDetails> getUserDetails(@Header("Authorization") String credentials)


Mit der Annotation @ Header ("Authorization") weisen Sie Retrofit an, den Authorization-Header mit dem von Ihnen übergebenen Wert zur Anforderung hinzuzufügen.

Um Anmeldeinformationen für die Standardauthentifizierung zu generieren, können Sie die OkHttps Credentials-Klasse mit ihrer Basismethode (String, String) verwenden. Die Methode verwendet einen Benutzernamen und ein Kennwort und gibt Authentifizierungsdaten für das HTTP-Basisschema zurück. 

 Credentials.basic("ausername","apassword");

Wenn Sie das API-Token und nicht das Basisschema verwenden möchten, rufen Sie einfach die Methode getUserDetails (String) mit Ihrem Token auf.

5.2. Authentifizierung mit OkHttp-Interceptors.


Mit der obigen Methode werden Anmeldeinformationen nur hinzugefügt, wenn Sie Benutzerdaten anfordern. Wenn Sie mehr Anrufe haben, für die eine Authentifizierung erforderlich ist, können Sie dazu einen Interceptor verwenden. Ein Interceptor wird verwendet, um jede Anforderung zu ändern, bevor sie ausgeführt wird, und um den Anforderungsheader festzulegen. Der Vorteil ist, dass Sie nicht jeder API-Methodendefinition @ Header ("Authorization") hinzufügen müssen.

Um einen Interceptor hinzuzufügen, müssen Sie die okhttp3.OkHttpClient.Builder.addInterceptor (Interceptor) -Methode in OkHttp Builder verwenden. 

 OkHttpClient okHttpClient = new OkHttpClient().newBuilder().addInterceptor(new Interceptor() { @Override public okhttp3.Response intercept(Chain chain) throws IOException { Request originalRequest = chain.request(); Request.Builder builder = originalRequest.newBuilder().header("Authorization", Credentials.basic("aUsername", "aPassword")); Request newRequest = builder.build(); return chain.proceed(newRequest); } }).build();

Der von OkHttp erstellte Client muss mit der Methode retrofit2.Retrofit.Builder.client (OkHttpClient) zu Ihrem Retrofit-Client hinzugefügt werden. 

 Retrofit retrofit = new Retrofit.Builder() .baseUrl("https://api.example.com/") .client(okHttpClient) .build();

Wie Sie bemerkt haben, wird hier die Klasse Anmeldeinformationen für die Basisautorisierung verwendet.
Wenn Sie das API-Token verwenden möchten, verwenden Sie stattdessen einfach das Token.

In dieser Übung wird davon ausgegangen, dass Sie mit Gradle und Buildship for Eclipse vertraut sind.

Erstellen Sie ein neues Gradle-Projekt mit dem Namen com.vogella.java.retrofitxml. Fügen Sie src / main / java ein neues Paket mit dem Namen com.vogella.java.retrofitxml hinzu.

Fügen Sie der Datei build.gradle die folgenden Abhängigkeiten hinzu.

7.2. Definieren einer XML-Ansicht


Ein RSS-Feed lautet wie folgt: 

 <?xml version="1.0" encoding="UTF-8"?> <rss version="2.0"> <channel> <title>Eclipse and Android Information</title> <link>http://www.vogella.com</link> <description>Eclipse and Android Information</description> <language>en</language> <copyright>Creative Commons Attribution-NonCommercial-ShareAlike 3.0 Germany (CC BY-NC-SA 3.0)</copyright> <pubDate>Tue, 03 May 2016 11:46:11 +0200</pubDate> <item> <title>Android user interface testing with Espresso - Tutorial</title> <description> This tutorial describes how to test Android applications with the Android Espresso testing framework.</description> <link>http://www.vogella.com/tutorials/AndroidTestingEspresso/article.html</link> <author>lars.vogel@vogella.com (Lars Vogel)</author> <guid>http://www.vogella.com/tutorials/AndroidTestingEspresso/article.html</guid> </item> <item> <title>Using the Gradle build system in the Eclipse IDE - Tutorial</title> <description>This article describes how to use the Gradle tooling in Eclipse</description> <link>http://www.vogella.com/tutorials/EclipseGradle/article.html</link> <author>lars.vogel@vogella.com (Lars Vogel)</author> <guid>http://www.vogella.com/tutorials/EclipseGradle/article.html</guid> </item> <item> <title>Unit tests with Mockito - Tutorial</title> <description>This tutorial explains testing with the Mockito framework for writting software tests.</description> <link>http://www.vogella.com/tutorials/Mockito/article.html</link> <author>lars.vogel@vogella.com (Lars Vogel)</author> <guid>http://www.vogella.com/tutorials/Mockito/article.html</guid> </item> </channel> </rss>

Neben dem XML-Header besteht diese Datei aus verschiedenen XML-Elementen. Ein RSS-Element enthält ein Kanalelement, das andere Elemente (z. B. Titel, Beschreibung, pubDate) und mehrere Elementelemente (Artikel) enthält.

Erstellen Sie die folgenden zwei Datenklassen: RSSFeed und Article. 

 package com.vogella.java.retrofitxml; import org.simpleframework.xml.Element; import org.simpleframework.xml.Root; @Root(name = "item", strict = false) public class Article { @Element(name = "title") private String title; @Element(name = "link") private String link; public String getTitle() { return title; } public void setTitle(String title) { this.title = title; } public String getLink() { return link; } public void setLink(String link) { this.link = link; } } 

 package com.vogella.java.retrofitxml; import java.util.List; import org.simpleframework.xml.Element; import org.simpleframework.xml.ElementList; import org.simpleframework.xml.Path; import org.simpleframework.xml.Root; @Root(name="rss", strict=false) public class RSSFeed { @Element(name="title") @Path("channel") private String channelTitle; @ElementList(name="item", inline=true) @Path("channel") private List<Article> articleList; public String getChannelTitle() { return channelTitle; } public void setChannelTitle(String channelTitle) { this.channelTitle = channelTitle; } public List<Article> getArticleList() { return articleList; } public void setArticleList(List<Article> articleList) { this.articleList = articleList; } }

Die Artikelklasse stellt einen Artikel dar und behält nur den Titel und den Link zum Artikel bei. Dies sind die einzigen Bereiche, die uns interessieren.

Annotation @ Root markiert eine Klasse als Gegenstand einer (De-) Serialisierung. Optional können Sie in der Annotation @ Root einen Namen angeben, der mit dem Namen des XML-Elements übereinstimmt. Wenn kein Name angegeben wird, wird der Klassenname als Name des XML-Elements verwendet. Da sich der Klassenname (RSSFeed) vom XML-Elementnamen (rss) unterscheidet, müssen wir den Namen angeben.

Wenn strict auf false gesetzt ist, ist das strikte Parsen deaktiviert. Dies weist den Parser an, keine Ausnahme zu unterbrechen oder auszulösen, wenn ein XML-Element oder -Attribut gefunden wird, für das keine Zuordnung bereitgestellt wird. Da das rss-Element ein Versionsattribut hat, für das es kein entsprechendes Feld gibt, generiert die Anwendung einen Fehler, wenn der strenge Parameter nicht auf false gesetzt ist.

Mit der Annotation @ Element wird ein XML-Element dargestellt. Bei Bedarf können Sie den XML-Namen des durch dieses Feld dargestellten Elements angeben. Wenn kein Name angegeben ist, wird der Feldname verwendet.

Das Feld articleList ist mit @ ElementList versehen. Dies zeigt, dass in diesem Feld eine Sammlung (in unserem Fall: Liste) von XML-Elementen mit demselben Namen gespeichert wird. Wenn inline auf true gesetzt ist, bedeutet dies, dass die Elemente in der Sammlung unmittelbar innerhalb des angegebenen Elements nacheinander aufgelistet werden und kein übergeordnetes Zwischenelement haben.

Mit der Annotation @ Path können Sie den Pfad zu einem XML-Element im XML-Baum angeben. Dies ist nützlich, wenn Sie keinen vollständigen XML-Baum mit Java-Objekten modellieren möchten. Für den Namen des Kanals und mehrerer Elementelemente können wir direkt auf bestimmte Elemente im Kanalelement verweisen.

7.3. Definieren einer API und eines Nachrüstadapters


Definieren Sie die REST-API für die Nachrüstung über die folgende Schnittstelle. 

 package com.vogella.java.retrofitxml; import retrofit2.Call; import retrofit2.http.GET; public interface VogellaAPI { @GET("article.rss") Call<RSSFeed> loadRSSFeed(); }

Erstellen Sie die folgende Controller-Klasse. Diese Klasse erstellt einen Retrofit-Client, ruft die Vogella-API auf und verarbeitet das Ergebnis. 

 package com.vogella.java.retrofitxml; import retrofit2.Call; import retrofit2.Callback; import retrofit2.Response; import retrofit2.Retrofit; import retrofit2.converter.simplexml.SimpleXmlConverterFactory; public class Controller implements Callback<RSSFeed> { static final String BASE_URL = "http://vogella.com/"; public void start() { Retrofit retrofit = new Retrofit.Builder().baseUrl(BASE_URL) .addConverterFactory(SimpleXmlConverterFactory.create()).build(); VogellaAPI vogellaAPI = retrofit.create(VogellaAPI.class); Call<RSSFeed> call = vogellaAPI.loadRSSFeed(); call.enqueue(this); } @Override public void onResponse(Call<RSSFeed> call, Response<RSSFeed> response) { if (response.isSuccessful()) { RSSFeed rss = response.body(); System.out.println("Channel title: " + rss.getChannelTitle()); rss.getArticleList().forEach( article -> System.out.println("Title: " + article.getTitle() + " Link: " + article.getLink())); } else { System.out.println(response.errorBody()); } } @Override public void onFailure(Call<RSSFeed> call, Throwable t) { t.printStackTrace(); } }

Der letzte Schritt besteht darin, eine Klasse mit einer Hauptmethode zum Starten des Controllers zu erstellen. 

 package com.vogella.java.retrofitxml; public class Application { public static void main(String[] args) { Controller ctrl = new Controller(); ctrl.start(); } }
