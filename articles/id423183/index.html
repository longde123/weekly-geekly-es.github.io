Memperkenalkan Azure


Apa itu Azure?


Azure adalah platform cloud Microsoft.

Teknologi Cloud - Umum


Cloud computing adalah alternatif modern untuk pusat data lokal yang sudah dikenal. Semua tugas pengadaan dan pemeliharaan peralatan sepenuhnya menjadi tanggung jawab penyedia layanan cloud publik. Ini memberi pelanggan akses ke berbagai layanan platform. Pelanggan menyewa sumber daya perangkat keras dan perangkat lunak yang hanya mereka butuhkan dari waktu ke waktu. Jadi mereka mengubah biaya modal pembelian peralatan menjadi biaya operasi. Selain itu, pendekatan ini memungkinkan pelanggan untuk menyewa akses ke sumber daya perangkat keras dan perangkat lunak, yang pembeliannya akan terlalu mahal. Hanya perangkat yang ditawarkan vendor platform cloud yang tersedia untuk pelanggan, tetapi mereka hanya membayar sesuai penggunaannya.

Portal web digunakan untuk mengelola lingkungan cloud (sumber daya komputasi, penyimpanan, jaringan, dan aplikasi). Misalnya, pada portal Azure, pengguna dapat membuat konfigurasi mesin virtual (VM) di mana karakteristik berikut akan didefinisikan: konfigurasi mesin virtual (prosesor, RAM dan disk lokal), sistem operasi, perangkat lunak yang digunakan sebelumnya, konfigurasi jaringan, dan lokasi mesin virtual. Setelah itu, pengguna dapat menggunakan mesin virtual berdasarkan konfigurasi ini dan mulai bekerja dengannya dalam beberapa menit. Sebelumnya, akan perlu untuk menggunakan mesin fisik, pembelian yang saja bisa memakan waktu beberapa minggu, sehingga kemungkinan penyebaran cepat terlihat sangat menarik.

Kami berbicara tentang awan publik. Ada juga awan pribadi dan hibrida. Cloud pribadi adalah lingkungan cloud yang dibuat di pusat pemrosesan data perusahaan sendiri, dan pengguna (karyawan perusahaan) diberikan alat untuk penggunaan sumber dayanya secara independen. Pengguna bekerja dengan lingkungan seperti itu dengan cara yang hampir sama dengan cloud publik, tetapi semua tugas yang terkait dengan pembelian dan pemeliharaan peralatan, pemilihan sumber daya perangkat keras dan perangkat lunak sepenuhnya berada di bawah tanggung jawab perusahaan. Awan hibrid adalah lingkungan yang menggabungkan awan publik dan pribadi. Pendekatan ini memungkinkan Anda memilih opsi yang paling cocok untuk menempatkan beban kerja. Misalnya, jika beban di situs web sangat bervariasi, ia dapat ditempatkan di cloud publik dan terhubung ke database aman di cloud pribadi Anda (atau di pusat data lokal).

Solusi Microsoft mendukung cloud publik, pribadi, dan hybrid. Platform Microsoft Azure yang didedikasikan untuk buku ini adalah cloud publik. Microsoft Azure Stack adalah ekstensi untuk Windows Server 2016 yang memungkinkan Anda untuk menyebarkan banyak layanan Azure dasar di pusat data lokal dan menyediakan portal layanan mandiri bagi pengguna. Komponen-komponen ini dapat diintegrasikan dengan cloud hybrid melalui virtual private network (VPN).

Perbandingan Lingkungan dan Azure Lokal


Dengan infrastruktur lokal, komponen perangkat keras dan perangkat lunak yang Anda gunakan sepenuhnya berada di bawah kendali Anda. Oleh karena itu, ketika membeli peralatan, mereka biasanya bertujuan penskalaan vertikal (mis., Mereka mencoba membeli server dengan sejumlah besar inti untuk memberikan kinerja yang diperlukan). Jika Anda menggunakan Azure, Anda hanya dapat menggunakan perangkat yang ditawarkan Microsoft. Dalam hal ini, untuk meningkatkan produktivitas, penskalaan horizontal digunakan: Anda cukup menggunakan node komputasi tambahan. Fitur ini harus diperhitungkan ketika merancang arsitektur sistem perangkat lunak, namun, seperti yang telah ditunjukkan oleh praktiknya, penskalaan horisontal (penyebaran perangkat berbiaya rendah) jauh lebih hemat biaya daripada vertikal (membeli perangkat berkinerja tinggi).

Pada saat penulisan ini, pusat data Microsoft Azure terbuka di lebih dari 22 wilayah di dunia: dari Melbourne ke Amsterdam, dari São Paulo ke Singapura. Selain itu, Microsoft telah menandatangani perjanjian dengan 21Vianet, dan sekarang platform Azure tersedia di dua wilayah Cina. Microsoft telah mengumumkan penyebaran Azure di delapan wilayah lainnya. Hanya perusahaan terbesar di dunia yang dapat membuka pusat data dengan cakupan seperti itu. Oleh karena itu, dengan bantuan Azure, perusahaan dari skala apa pun dapat mengerahkan layanan mereka di area di mana pelanggan mereka terkonsentrasi di setiap wilayah di Bumi. Dan semua ini - bahkan tanpa meninggalkan kantor!

Azure memungkinkan perusahaan muda untuk memulai dengan biaya yang sangat rendah dan dengan cepat meningkatkan infrastruktur mereka saat pelanggan baru tiba. Menjalankan satu atau lebih mesin virtual baru tidak memerlukan uang muka dalam jumlah besar. Perusahaan muda saat ini cenderung untuk berkembang dengan cepat dan belajar dengan cepat dari kesalahan mereka. Penggunaan teknologi cloud sepenuhnya konsisten dengan prinsip-prinsip ini.

Azure membantu Anda dengan cepat dan fleksibel meluncurkan pengembangan baru dan lingkungan pengujian. Anda dapat menggunakan skrip untuk menggunakan lingkungan ini. Jadi Anda dapat, jika perlu, menjalankan lingkungan pengembangan atau pengujian, melakukan pengujian dan menghapusnya. Akibatnya, perusahaan menghemat banyak dan menghabiskan hampir tidak ada uang untuk mendukung infrastruktur.

Keuntungan lain Azure adalah kemampuan untuk menguji versi baru perangkat lunak tanpa mengganti perangkat keras lokal. Misalkan Anda perlu tahu bagaimana aplikasi Anda akan berubah ketika beralih dari Microsoft SQL Server 2014 ke Microsoft SQL Server 2016. Untuk melakukan ini, Anda cukup membuat instance dari SQL Server 2016 dan memulai salinan layanan Anda dengan menghubungkannya ke database baru - Anda tidak perlu memilih peralatan atau peregangan kabel. Atau Anda dapat memulai mesin virtual yang menjalankan Microsoft Windows Server 2012 R2 daripada Microsoft Windows Server 2008 R2.

Penawaran cloud


Layanan cloud biasanya termasuk dalam salah satu dari tiga kategori: SaaS, PaaS, atau IaaS. Namun, dengan perkembangan teknologi cloud, batas antara keduanya menjadi kurang jelas.

SaaS: Perangkat Lunak sebagai Layanan


Perangkat lunak, yang di-host di lingkungan terpusat dan dikelola atas nama klien, disebut SaaS (perangkat lunak sebagai layanan). Biasanya, pendekatan ini menggunakan arsitektur multi-tenant, mis., Semua klien diberikan aplikasi versi yang sama. Itu dapat diskalakan di beberapa contoh untuk memberikan kinerja yang optimal terlepas dari lokasi. Untuk lisensi SaaS, biaya berlangganan bulanan atau tahunan biasanya digunakan.

Salah satu contoh SaaS adalah Microsoft Office 365. Pengguna membayar langganan bulanan atau tahunan dan mendapatkan akses ke beberapa produk: Tukarkan sebagai layanan (klien Web Outlook dan / atau aplikasi desktop), Penyimpanan sebagai Layanan (OneDrive) dan komponen paket Microsoft lainnya Office (versi desktop dan / atau web). Pada saat yang sama, pelanggan selalu diberikan versi produk terbaru. Sehingga Anda dapat, pada kenyataannya, siap melayani Anda server Microsoft Exchange tanpa perlu membeli, menginstal, dan mendukungnya - yang lain akan mengurus pengelolaan server Exchange, termasuk menginstal tambalan dan pembaruan. Opsi ini jauh lebih murah dan lebih mudah dalam hal pemeliharaan daripada menginstal paket perangkat lunak Microsoft Office dan memperbaruinya setiap tahun.

Contoh lain dari produk SaaS adalah Dropbox, WordPress, dan Amazon Kindle.

PaaS: platform sebagai layanan


Sebagai bagian dari pendekatan PaaS, Anda menyebarkan aplikasi Anda dalam lingkungan hosting aplikasi khusus yang diselenggarakan oleh penyedia layanan cloud. Pengembang membuat aplikasi, dan penyedia PaaS menyediakan kemampuan untuk menyebarkan dan menjalankannya. Akibatnya, pengembang tidak perlu berurusan dengan pemeliharaan infrastruktur, yang berarti mereka dapat mencurahkan seluruh waktu mereka untuk pengembangan.

Di dalam Azure, beberapa penawaran PaaS tersedia, yang mencakup komponen aplikasi web layanan aplikasi Azure, serta layanan cloud Azure (peran web dan peran kerja). Dalam semua kasus, pengembang memiliki banyak cara untuk menyebarkan aplikasi tanpa harus memahami detail infrastruktur pendukung. Pengembang tidak harus membuat mesin virtual, atau menghubungkan ke mereka menggunakan Remote Desktop Protocol (RDP), atau menginstal aplikasi. Cukup bagi mereka untuk mengklik tombol (atau melakukan tindakan lain yang sama sederhananya), dan alat Microsoft akan menyiapkan mesin virtual, menyebarkannya dan menginstal aplikasi pada mereka.

IaaS: infrastruktur sebagai layanan


Penyedia layanan cloud IaaS memantau dan mengelola server farm yang menjalankan sistem perangkat lunak virtualisasi. Dalam sistem ini, pelanggan membuat mesin virtual yang beroperasi di infrastruktur penyedia. Klien menciptakan mesin virtual yang menjalankan Windows atau Linux (opsi yang tersedia tergantung pada penyedia layanan) dan menginstal semua yang diperlukan di dalamnya. Azure memungkinkan Anda mengonfigurasi jaringan virtual, memuat penyeimbang dan penyimpanan, dan menggunakan banyak layanan lain yang berjalan di lingkungan ini. Klien tidak dapat mengontrol perangkat virtualisasi atau sistem perangkat lunak, tetapi hampir semua yang lain sudah penuh
pembuangan. Dengan pendekatan ini (tidak seperti PaaS), perangkat lunak mengontrol pelanggan.

Mesin Virtual Azure (penawaran IaaS Azure) adalah alat yang sangat populer untuk memigrasi layanan ke Azure, karena pada dasarnya memungkinkan Anda untuk dengan mudah memigrasikan solusi Anda. Anda dapat membuat mesin virtual yang mirip dengan infrastruktur pusat data Anda, di mana layanan berjalan sekarang, dan mentransfer aplikasi Anda ke sana. Dalam beberapa kasus, langkah-langkah tambahan mungkin diperlukan (misalnya, mengubah URL sehingga mengarah ke layanan atau toko baru), tetapi pendekatan ini memungkinkan Anda untuk memindahkan banyak aplikasi.

Azal Virtual Machine Scalable Sets (VMSS) berdasarkan mesin virtual Azure memungkinkan Anda dengan cepat membuat sekelompok mesin virtual yang identik. Selain itu, VMSS mendukung penskalaan otomatis (penyebaran otomatis mesin virtual baru sesuai kebutuhan). Karena itu, VMSS adalah platform ideal untuk hosting cluster komputasi berdasarkan pada layanan microser tingkat tinggi: misalnya, untuk Kain Layanan Azure dan Layanan Wadah Azure.

Layanan Biru


Platform cloud Azure mencakup banyak layanan. Mari kita pertimbangkan beberapa di antaranya.

  • Layanan Komputasi. Kategori ini mencakup mesin virtual Azure (menjalankan Linux dan Windows), layanan cloud, layanan aplikasi (aplikasi web, aplikasi seluler, Aplikasi Logika, aplikasi API dan aplikasi fitur), layanan batch (untuk melakukan tugas komputasi paralel dan batch yang besar) ), RemoteApp, Fabric Layanan, dan Layanan Container Azure.
  • Layanan data. Ini termasuk Microsoft Azure Storage (yang mencakup blob, antrian, tabel, dan layanan file Azure), Azure SQL Database, DocumentDB, StorSimple, dan Redis Cache.
  • Layanan aplikasi. Kategori ini mencakup layanan yang digunakan untuk membangun dan menjalankan aplikasi, termasuk Azure Active Directory (Azure AD), bus layanan untuk menghubungkan sistem terdistribusi, HDInsight untuk memproses data besar, Azure Scheduler, dan Layanan Media Azure.
  • Layanan jaringan. Grup ini mencakup layanan Azure seperti jaringan virtual, ExpressRoute, Azure DNS, manajer traffic Azure, dan jaringan pengiriman konten Azure.

Saat memigrasi aplikasi, penting untuk memahami layanan mana yang tersedia sebagai bagian dari Azure, karena pengetahuan ini akan membantu menyederhanakan migrasi aplikasi dan meningkatkan fleksibilitasnya. Tidak mungkin menggambarkan secara rinci tentang semua layanan yang tersedia dalam kerangka buku ini, jadi kami memilih beberapa yang paling menarik. Bab 9 (Layanan Lanjutan Azure) memberikan daftar layanan ini dan penjelasan singkat tentang kemampuan mereka.

Dunia Baru: Manajer Sumber Daya Azure


Azure Resource Manager adalah pendekatan baru untuk menggunakan sumber daya.

Apa ini


Model penyebaran Azure Service Management (ASM) telah digunakan untuk menyebarkan layanan sejak peluncuran versi evaluasi publik. Layanan yang dikelola oleh ASM di portal Azure disebut klasik. Pada 2015, Microsoft memperkenalkan model penyebaran menggunakan Azure Resource Manager (pengganti ASM yang modern dan lebih fungsional), yang direkomendasikan untuk mengelola beban kerja baru.

Mode penyebaran ini sering disebut "pesawat kontrol" karena mereka digunakan tidak hanya untuk menyebarkan layanan, tetapi juga untuk mengelolanya. Mereka tidak harus bingung dengan "pesawat data" - alat manajemen data yang digunakan oleh layanan.

Infrastruktur di Azure biasanya berisi banyak sumber daya, beberapa di antaranya saling berhubungan: misalnya, layanan komponen diperlukan untuk menjalankan aplikasi web. Misalkan Anda memerlukan dua mesin virtual yang menjalankan aplikasi web. Untuk menyimpan data, database yang terletak di jaringan virtual yang sama akan digunakan. Resource Manager memungkinkan Anda untuk menyebarkan semua ini dalam satu grup sumber daya yang dapat dikelola secara keseluruhan. Menyebarkan, memperbarui, dan menghapus semua sumber daya yang dimiliki grup yang sama dilakukan dalam satu tindakan.

Dalam contoh ini, grup sumber daya akan berisi yang berikut ini:

  • Mesin virtual 1
  • Mesin virtual 2
  • Jaringan virtual
  • Akun penyimpanan
  • Database SQL Azure

Selain itu, Anda bisa membuat templat yang menjelaskan secara akurat semua sumber daya di Resource Manager yang terkait dengan penyebaran. Setelah itu, cukup melakukan satu operasi di bidang kontrol untuk menggunakan templat Resource Manager ini sebagai grup sumber daya. Pada saat yang sama, Manajer Sumber Daya Azure akan memastikan bahwa semua sumber daya digunakan dengan benar. Selain itu, Manajer Sumber Daya mendukung berbagai kemampuan untuk bekerja dengan sumber daya yang digunakan: keamanan, audit, dan pemberian tag.

Apa manfaat dari Resource Manager?


Menggunakan Resource Manager menawarkan sejumlah manfaat. Ini memungkinkan Anda untuk menggunakan sumber daya lebih cepat, karena ia melakukan operasi tidak secara berurutan (seperti ASM), tetapi secara paralel. Model penyebaran menggunakan Azure Resource Manager memungkinkan setiap layanan untuk bekerja dengan penyedia layanannya dan, jika perlu, memperbaruinya secara terpisah dari layanan lain. Penyimpanan Azure memiliki satu penyedia layanan, mesin virtual memiliki yang lain, dan sebagainya. Saat menggunakan model ASM, semua layanan harus diperbarui pada saat yang sama, jadi jika satu layanan menyelesaikan pembaruan lebih awal daripada yang lain, masih harus menunggu yang lain dirilis. Berikut adalah beberapa manfaat lebih penting dari model penyebaran dengan Azure Resource Manager:

Kemampuan untuk menggunakan templat

  • Anda dapat membuat template JSON dan menggunakannya kembali untuk menggunakan semua sumber daya untuk solusi sekaligus. Anda tidak perlu lagi membuat satu mesin virtual melalui portal, menunggu prosesnya selesai, mulai membuat yang berikutnya, dan seterusnya.
  • Anda dapat menggunakan kembali kumpulan sumber daya yang sama menggunakan templat. Misalkan Anda mengatur sumber daya di lingkungan pengujian dan menyadari bahwa mereka tidak sesuai dengan kebutuhan Anda. Anda dapat menghapus grup sumber daya ini (ini akan secara otomatis menghapus semua sumber daya yang termasuk di dalamnya), mengubah template dan coba lagi. Untuk memodifikasi set sumber daya yang digunakan, cukup ubah templat dan mulai kembali penggunaannya: Manajer Sumber Daya akan mengubah sumber daya agar sesuai dengan templat baru.
  • Menggunakan templat, Anda dapat dengan cepat membuat beberapa versi infrastruktur: misalnya, lingkungan kerja menengah dan. Berbagai bidang (misalnya, nama mesin virtual, nama jaringan, akun penyimpanan, dll.) Dapat diparameterisasi dan kemudian diunduh template beberapa kali dengan nilai parameter yang berbeda.

Resource Manager dapat mendeteksi dependensi dalam template dan memungkinkan Anda untuk menentukan dependensi tambahan. Misalnya, menggunakan mesin virtual sebelum membuat akun penyimpanan untuk file VHD yang menyimpan disk dengan sistem operasi dan data adalah ide yang buruk.

Keamanan

  • Untuk mengontrol akses ke sumber daya grup, Anda dapat menggunakan mekanisme baru yang disebut Role-Based Access Control (RBAC). Misalnya, Anda dapat menetapkan pengguna peran "pemilik", dan kemudian ia akan memiliki hak administrator penuh sehubungan dengan sumber daya grup ini - tetapi tidak sumber daya berlangganan lainnya. Peran lain juga disediakan, misalnya, "pembaca" (memungkinkan Anda untuk membaca semuanya kecuali data rahasia) dan "peserta" (memungkinkan Anda untuk melakukan hampir semua operasi, kecuali untuk menambah dan menghapus akses).

Tagihan

  • Untuk mengatur semua sumber daya grup untuk penagihan yang nyaman, Anda dapat menetapkan tag ke setiap sumber daya, dan kemudian menerima data pembayaran untuk penggunaan sumber daya dengan tag tertentu.

Model Penerapan Klasik


Mari kita bicara sedikit tentang apa yang terjadi sebelum Resource Manager. Sekarang sumber daya seperti itu disebut klasik. Misalnya, klien mungkin memiliki akun penyimpanan, mesin virtual, dan jaringan virtual yang dikelola menggunakan model penyebaran klasik. Model klasik dan model penerapan yang menggunakan Azure Resource Manager tidak kompatibel - Sumber daya Manajer Sumber Daya tidak dapat berinteraksi dengan sumber daya klasik dan sebaliknya. Misalnya, komponen Layanan Cloud PaaS Azure adalah klasik, sehingga Anda hanya dapat bekerja dengannya melalui akun penyimpanan klasik. Ada satu pengecualian untuk aturan ini: Anda bisa meng-host mesin virtual Resource Manager di akun penyimpanan klasik. Fitur ini menyederhanakan migrasi mesin virtual dari model penyebaran klasik ke model penggunaan menggunakan Azure Resource Manager.

Harap dicatat: selama migrasi seperti itu, Anda mungkin harus masuk ke portal Azure klasik, yang menampilkan sumber daya klasik, tetapi tidak memiliki sumber daya Manajer Sumber Daya, dan sebaliknya.
Catatan Ada dua versi portal. Saat ini adalah portal Azure, tersedia di portal.azure.com . Sebagian besar fitur telah dipindahkan ke portal Azure, tetapi ada beberapa pengecualian, seperti Azure Active Directory (Azure AD). Versi portal sebelumnya disebut "portal Azure klasik" (https://manage.windowsazure.com). Sekarang dapat digunakan untuk mengelola layanan Azure AD, serta untuk mengkonfigurasi dan skala sumber daya klasik (seperti layanan cloud).
Anda dapat mentransfer sumber daya Anda dari model penyebaran klasik ke model penyebaran menggunakan Azure Resource Manager.

  • File, tabel, dan blob dapat dimigrasikan dari akun penyimpanan klasik ke akun penyimpanan Manajer Sumber Daya baru menggunakan AzCopy. Harap dicatat: tabel harus diekspor dari akun klasik dan kemudian diimpor ke akun Resource Manager.
  • Mesin virtual dapat dimigrasi sebagai berikut: lepaskan mereka, salin file VHD ke akun penyimpanan Manajer Sumber Daya baru, dan kemudian buat ulang mesin virtual menggunakan file VHD.
  • Jaringan virtual dapat dibuat sebagai objek Vnet di Resource Manager.
  • Selain itu, layanan migrasi diluncurkan (sekarang sedang dalam tahap versi evaluasi publik). Saat ini, tidak disarankan untuk menggunakannya untuk beban produksi. Lihat artikel untuk lebih jelasnya.

Pertimbangan Model Penerapan PowerShell


Bab 8 ("Alat Manajemen") membahas beberapa alat Azure, termasuk cmdlet Azure PowerShell dan Azure CLI.

Saat mengembangkan model penyebaran menggunakan Manajer Sumber Daya, spesialis Azure berusaha membuat cmdlet PowerShell yang hanya akan berfungsi untuk model penempatan menggunakan Manajer Sumber Daya. Alih-alih kata Azure, nama AzureRm ditunjukkan dalam nama cmdlet tersebut. Misalnya, untuk membuat akun penyimpanan klasik, Anda dapat menggunakan cmdlet New-AzureStorageAccount. Untuk membuat akun penyimpanan di Resource Manager, Anda perlu menjalankan cmdlet New-AzureRmStorageAccount.

Ini dilakukan agar pengguna dapat segera memahami jenis sumber daya apa yang ia buat. Selain itu, ini memberikan kemampuan untuk mengeksekusi skrip yang dibuat sebelumnya dengan benar. Setiap kali Anda menggunakan sumber daya Resource Manager, Anda harus menentukan grup sumber daya di mana Anda ingin menempatkannya. Selain itu, beberapa cmdlet Resource Manager (misalnya, cmdlet untuk membuat mesin virtual) mendukung parameter yang lebih rinci daripada cmdlet model klasik.

Satu catatan terakhir: hanya cmdlet PowerShell yang terkait dengan akun penyimpanan yang terkait dengan bidang manajemen yang dipengaruhi oleh perubahan (misalnya, cmdlet untuk membuat, menghapus, dan mendaftarkan akun penyimpanan). Untuk cmdlet PowerShell, yang digunakan untuk mengakses konten repositori (gumpalan, tabel, antrian, dan file), tidak ada yang berubah.
