Servidor compacto con recarga de código configurada y registros de SQL por un equipo de Maven para un desarrollo productivo

Me gustaría compartir la experiencia de usar payara-micro en el desarrollo de aplicaciones empresariales. Espero que alguien ahorre tiempo, ya que tal decisión no llegó de inmediato. Si ya está utilizando Payara o Glassfish como servidor industrial, o simplemente se sumerge en el mundo de Java, entonces payara-micro y este artículo es para usted.


Necesitará una aplicación web, que se ensambla a través de Maven en un archivo de guerra, y Java 8 (no verificó las anteriores).


Primero, daré el texto completo del perfil, después de lo cual lo analizaremos en partes y agregaremos los detalles faltantes que no están visibles en el código fuente.

Lo primero que debe hacer es descargar el archivo payara-micro. Esto es exactamente lo que hace el primer complemento ( maven-dependency-plugin ). El archivo es un servidor de aplicaciones completo que se puede iniciar desde la línea de comandos como una simple aplicación java.


El último complemento ( payara-micro-maven-plugin ) no tiene características y está configurado para una versión de trabajo estable de payra-micro 4.1.2.173 . De las capacidades del complemento, aquí se usa la función de empaquetar la aplicación web en un paquete especial, listo para ejecutarse. Hay otras formas de lanzar payara-micro, pero aquí no nos serán útiles.


Queda por generar la línea de comando deseada y poner todo junto. El complemento exec-maven se utiliza para comenzar desde la línea de comandos. Consideremos las opciones de la línea de comandos con más detalle: 


 <configuration> <executable>java</executable> <arguments> <!--http://blog.payara.fish/using-hotswapagent-to-speed-up-development--> <argument>-XXaltjvm=dcevm</argument> <argument>-javaagent:hotswap/hotswap-agent.jar</argument> <argument>-agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=9009</argument> <argument>-Duser.language=en</argument> <argument>-Duser.region=US</argument> <argument>-Ddb.ora.url=jdbc:p6spy:oracle:thin:@localhost:1521:XE</argument> <argument>-Ddb.pg.url=jdbc:p6spy:postgresql://localhost:5432/postgres</argument> <argument>-jar</argument> <argument>${project.build.directory}/${project.build.finalName}-microbundle.jar</argument> <argument>--domainConfig</argument> <argument>src/main/setup/domain.xml</argument> <argument>--deploy</argument> <argument>${project.build.directory}/${project.build.finalName}.war</argument> <argument>--rootDir</argument> <argument>${project.build.directory}/payaramicro</argument> </arguments> </configuration>

Los dos primeros argumentos ( -XXaltjvm = dcevm -javaagent: hotswap / hotswap-agent.jar ) son necesarios para cambiar el código fuente sin reiniciar la aplicación. Gracias al segundo argumento ( -agentlib: jdwp = transport = dt_socket, server = y, suspend = n, address = 9009 ), iniciamos inmediatamente la aplicación en modo de depuración, lo que nos permite cambiar no solo el cuerpo de los métodos mismos a través de los medios de aplicar cambios de código en la depuración en el IDE , pero también agrega nuevos campos y más. Solo necesita conectarse con el depurador desde el IDE, compilar el archivo fuente modificado y aplicar los cambios.


Los siguientes dos argumentos ( -Duser.language = en -Duser.region = US ) son necesarios para que el cliente jdbc funcione correctamente con Oracle. La configuración para conectarse al DBMS de prueba se pasa a través de los argumentos -D . En mis aplicaciones, uso un enlace como $ {db.ora.url} a estos parámetros en el archivo de recursos, que los crea automáticamente al inicio. La sustitución de un archivo que crea recursos con configuraciones industriales o de prueba también se implementa a través del perfil en maven y su capacidad para filtrar archivos de texto.


Los argumentos clave al inicio indican la ruta ( --domainConfig src / main / setup / domain.xml ) al archivo de configuración del dominio, así como la ruta al archivo war ( --deploy $ {project.build.directory} / $ {project. build.finalName} .war ) que se implementará en el servidor. La configuración del dominio fue obtenida por mí en base a un archivo estándar dentro del archivo. Daré los contenidos como ejemplo. Puede descargar la última versión del agente para la modificación ilimitada del código fuente, así como la recarga de código dentro de marcos como hibernar desde aquí .


Las ideas para configurar la recarga de código para payara surgieron de aquí . Pero la verdad es solo la idea en sí, ya que la implementación de payara-micro no se considera allí.


Gracias a la biblioteca p6spy, puede encontrar sus registros sql en la carpeta raíz del proyecto: el archivo spy.log. La biblioteca p6spy se puede personalizar si es necesario, pero este es el tema de un artículo completamente diferente.


Después de colocar el perfil descrito anteriormente en su pom.xml y los archivos adicionales discutidos anteriormente en las carpetas apropiadas, puede iniciar su aplicación con el siguiente comando: 


 mvn install exec:exec -P micro

Y no se requieren complementos adicionales para el servidor de aplicaciones en el IDE ...
