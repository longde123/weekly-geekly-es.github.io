H2 evolution: funciones de ventana, CTE, JSON / XML en una base de datos integrada

Hay proyectos de código abierto que se han convertido en una corriente principal comercialmente exitosa, como PostgreSQL / Elasticsearch. Otros, por ejemplo RethinkDB, perdieron el mercado y detuvieron el desarrollo. Y el proyecto de base de datos H2 de una base de datos integrada escrita en Java se está desarrollando y viviendo en su nicho.


Para demostrar la funcionalidad de SonarQube, Jira, Confluence, la base de datos H2 se utiliza en el primer inicio. H2 es la base para ejecutar pruebas SQL en memoria en casi cualquier proyecto JVM. Hay un ejemplo de una aplicación menos conocida por los usuarios: este es el uso de H2 en un encendido ignite-sql distribuido y este ya es un script listo para producción para usar una base de datos integrada como parte de otra solución. Hace menos de un mes, se lanzó la versión 1.4.199 en la que ahora puede escribir consultas SQL bastante complejas.

En proyectos, nunca confié en H2 como una base de datos completa para guardar datos en el disco. Más bien, como un módulo para transformar datos en memoria JVM, con buen soporte de SQL. Pero para esta aplicación, estaba muy limitada por la falta de funciones de ventana . Y ahora, después de más de medio año desde el comienzo del desarrollo de la funcionalidad , ahora H2database se ha puesto al día con SQLite . Y esto se debe al gran mérito de Yevgeny Riazanov de Irkutsk: no he visto un ritmo de desarrollo como el suyo antes en proyectos de código abierto sin fines de lucro. Además, los compromisos de otros contribuyentes de habla rusa aparecen regularmente en el repositorio del proyecto. Y en los momentos de lanzamiento: el fundador del proyecto Thomas Mueller .

H2 tiene soporte para consultas recursivas (CTE) . Esta es la forma estándar en SQL para trabajar con datos jerárquicos en tablas y descomposición de consultas (aquí puede pisar el rastrillo del planificador). Las consultas recursivas se describen en la publicación con ejemplos.

Para trabajar con datos mal estructurados, aparecerá una próxima implementación del estándar SQL / JSON . Mientras tanto, expandí H2 para mis necesidades con la ayuda de una función personalizada del procesador XQuery 3.1, basada en BaseX . El código está disponible en el proyecto github H2XQueryAdapter . Esta es una función de tabla que puede extraer datos del formato XML o JSON usando XQuery y verificar el tipo y la restricción de no nulo para los valores devueltos por la función. Además, toda la potencia de las expresiones SQL está disponible para transformar el resultado de una transformación XQuery en la memoria del proceso JVM.

La función de tabla xquery () está sobrecargada y tiene dos opciones: con una consulta xQuery de parámetro y otra con la consulta xQuery y la segunda cadena de consulta sql para formar los parámetros para xQuery.

Un enfoque con tales transformaciones demostró ser excelente en un proyecto para procesar un volumen de datos en bruto de petabytes en un proyecto de almacenamiento de datos biomédicos . 

create table xresult (GR VARCHAR(500) not null,AR varchar, VER VARCHAR(50)) as select * from xquery('declare variable $getHeader as xs:boolean external := false(); declare variable $getData as xs:boolean external := true(); <csv> { if($getHeader) then( <record><mavengr>VARCHAR(500) not null</mavengr><artifactname>varchar</artifactname><versionname>VARCHAR(50)</versionname></record> ), if($getData) then(( for $row in doc("http://central.maven.org/maven2/org/springframework/spring-context/5.1.4.RELEASE/spring-context-5.1.4.RELEASE.pom")//*:dependency return <record><mavengr>{$row/*:groupId/text()}</mavengr><artifactname>{$row/*:artifactId/text()}</artifactname><versionname>{$row/*:version/text()}</versionname></record> )) } </csv>')

Esta implementación tiene limitaciones en el formato de la consulta xquery.

  • En primer lugar, debe declarar dos variables externas getHeader y getData; esto es necesario para que la función de tabla no provoque la transformación de datos varias veces, descartando resultados innecesarios para obtener solo los nombres y el tipo de columnas. Las llamadas a funciones múltiples son características de la base H2 que funciona con funciones java que devuelven una lista de valores.
  • En segundo lugar, el formato del resultado debe ser como para la serialización csv 

     <csv> <record><1> </1>...<N> </N></record> <record><1></1>...<N></N></record> </csv>
  • En tercer lugar, debe declarar el tipo de datos, la dimensión y los campos obligatorios, por ejemplo: decimal (20,4) no nulo

Con el análisis de tipo de datos de columna, intenté reutilizar el analizador SQL desde H2, pero resultó estar tan fuertemente conectado con otros objetos de base de datos que se negó a funcionar sin crear una base de datos y una sesión. Me consuelo pensando que los desarrolladores lo hicieron para simplificar el diseño de la aplicación y no para hacer un analizador en todas las ocasiones para las gramáticas BNF.

Puede ejecutar este ejemplo en modo de depuración de Java. Se pueden desarrollar nuevas reglas de transformación en el conocido editor de xquery o en la GUI de BaseX de código abierto.

Es posible convertir casi cualquier colección de Java o POJO en una tabla H2 virtual. Código de proyecto H2POJOTable en github. En el ejemplo, por referencia, el MemoryManagerMXBeans basado en la plataforma se convierte en una función de tabla H2. Quizás el enfoque será un poco de consuelo para aquellos que están tristes por la falta de LINQ y soporte para operaciones en sets en Java. 

 try (Statement statement = connection.createStatement()) { String pojoTableAlias = "create alias MemoryManagerMXBeans as $$ \n" + "import java.lang.management.ManagementFactory;\n" + "import java.lang.management.MemoryManagerMXBean;\n" + "import org.h2.expression.function.pojo.*;\n" + "import java.sql.*;\n" + "import java.util.Collections;\n" + "@CODE\n" + " ResultSet getRuntimeStat(Connection connection) throws Exception{\n" + " return H2PojoAdapter.toTable(connection, new CollectionWraper<>(MemoryManagerMXBean.class," + " ManagementFactory::getMemoryManagerMXBeans, Collections.emptyMap()));\n" + " }\n" + "\n$$"; statement.executeUpdate(pojoTableAlias); } try (Statement statement = connection.createStatement()) { try (ResultSet resultSet = statement.executeQuery("select * from MemoryManagerMXBeans()")) { int columnCount = assertResultSet(resultSet, new String[]{"memoryPoolNames", "name", "valid"}); assertThat(columnCount).isGreaterThan(1); } }

A veces, la antigua funcionalidad en H2 se rompe y los usuarios no la exigen. Como ejemplo, en mis proyectos de trabajo, los programas leen datos de la URL de AWS S3 . Por lo tanto, espero que el error conocido aún se arregle al aceptar mi solicitud de extracción . En el camino para solucionar este error hay pruebas inestables para TLS, que tampoco funcionan en Java 11.

H2 le permite utilizar el controlador ODBC PostgreSQL emulando un subconjunto de su protocolo de red. Lo que teóricamente también te permite vincularlo a través de FDW en PostgreSQL.

Además de la base de datos en sí, la entrega de H2 también incluye una consola web minimalista con soporte para autocompletar al editar, servlet o opción de inicio independiente. H2 parece un "cuchillo suizo" para desarrolladores: una herramienta compacta y versátil si su proyecto ya utiliza la JVM. Al intentar usar esta consola con un controlador jdbc "torcido", el DBMS Redshift realizó su primera solicitud de extracción de ojos rojos en el proyecto. Noel Grandin , uno de los participantes del proyecto, me ayudó con la revisión del código y aceptó las correcciones.

Si necesita un análogo de Berkeley DB Java Edition, el proyecto tiene MVStore , un almacenamiento persistente para datos de valor clave y, en combinación con MVCC, el "motor" por defecto en las versiones recientes de la base de datos. Es sorprendente que la base de datos incluso tenga soporte básico para geofunciones y búsqueda de texto completo.

Gracias a los colaboradores de H2database, ¡todos usan esta base de datos e informan errores! La base de datos H2 se ha desarrollado desde 2005, y ahora admite funciones de ventana, consultas recursivas, es uno de los "motores" SQL más potentes para procesar datos en la memoria JVM y se expande mediante funciones de tabla para trabajar con datos estructurados libremente.
