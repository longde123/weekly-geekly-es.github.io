Est-ce que tout le monde va à DevOps?


Aujourd'hui, DevOps est une mention spéciale de ma part - une nouvelle approche pour le développement et l'exploitation de logiciels, comme Il y a eu beaucoup d'articles et de débats sur ce sujet cette année.



Le terme DevOps est interprété assez largement aujourd'hui. Quelqu'un sous DevOps comprend une approche spéciale pour le développement et l'exploitation de logiciels, lorsque les gens sont impliqués dans le travail, qui peuvent à la fois coder un peu et administrer. Pour d'autres, c'est avant tout la présence d'un administrateur système personnel dans l'équipe, qui vous permet de supprimer une partie de la charge non principale des développeurs de logiciels sous la forme de la configuration de l'environnement système, de la création d'environnements de test, de la mise en œuvre de l'intégration avec les services internes et externes et de l'écriture de scripts d'automatisation. Pour le troisième, ce n'est qu'un ensemble de technologies et d'outils à la mode que vous devez utiliser pour rester toujours jeune et performant. Pour quatre, c'est CI \ CD et tout ce qui s'y rapporte. En effet, DevOps a beaucoup d'interprétations, donc tout le monde peut y trouver ce qui lui plaît le plus.

Différentes interprétations de DevOps donnent lieu à des discussions animées, ce qui conduit à l'apparition de plus d'articles sur ce sujet. J'ai même mis en signet certains d'entre eux moi-même:

Les administrateurs ne sont plus nécessaires?


En tant qu'administrateur système / ingénieur qui travaille avec les produits Microsoft et VMware depuis un certain temps, j'ai commencé à remarquer qu'au cours des dernières années, il y a eu une discussion périodique sur le fait que les administrateurs système ne seront bientôt nécessaires à personne, car:

  1. L'ensemble de l'infrastructure est sur le point de changer et de se transformer en IaaC (Infrastructure as a code). Désormais, il n'y aura plus d'interface graphique avec des boutons, mais il n'y aura que PowerShell, des fichiers yaml, des configurations, etc. Si un service ou son composant est défectueux, il n'est plus nécessaire de le réparer, car Déployez plus rapidement une nouvelle copie du dernier état de fonctionnement.
  2. L'ensemble de l'infrastructure informatique passera bientôt aux nuages ​​et, localement (sur site), il n'y aura que des câbles réseau vers le routeur le plus proche, ce qui nous connectera à toutes les autres ressources de l'entreprise situées dans le cloud. Eh bien, l'imprimante maximale reste toujours localement, de sorte que les comptables peuvent imprimer des images de chats à partir d'Internet. Tout le reste devrait être dans le cloud.
  3. Un gourou DevOps viendra et automatisera tout ce qui se passe, donc les administrateurs n'auront qu'à se rappeler chaleureusement comment ils ont commencé dans les temps anciens les pings et les traces pour diagnostiquer les problèmes de base sur le réseau et sur les serveurs.
  4. J'ai également entendu parler d'un phénomène tel que «Vendekapets», mais il y a longtemps, à l'aube de ma carrière, je commençais à peine à faire les premiers pas vers l'administration système. Installations locales (sur site)


Microsoft Exchange Server est un serveur de messagerie multifonctionnel qui comprend non seulement le travail avec le courrier, mais aussi avec les contacts, les calendriers, les tâches et bien plus encore. Exchange Server est l'un des produits phares de Microsoft, qui est devenu de facto la norme d'entreprise dans de nombreuses entreprises. Il a une intégration étroite non seulement avec les produits Microsoft, mais aussi avec des solutions de fournisseurs tiers. L'échange est populaire aussi bien dans les moyennes (à partir de 100 personnes) que dans les grandes entreprises.

À l'heure actuelle, la version actuelle est considérée comme Exchange Server 2019. Auparavant, le produit s'était développé assez activement, mais depuis la sortie d'Exchange 2013, ce développement s'est considérablement ralenti, de sorte qu'Exchange 2016 peut être appelé Service Pack 1 (SP1) pour Exchange 2013, et Exchange 2019 est donc Service Pack 2 (SP2) pour Exchange 2013. Le sort de la prochaine version sur site (Exchange 2022) est toujours en question.

Désormais, Microsoft promeut activement Exchange Online dans le cadre du service cloud Office 365, de sorte que toutes les nouvelles fonctionnalités y apparaissent principalement. Exchange Online n'est pas seulement le premier à recevoir de nouvelles fonctionnalités, mais également à acquérir des fonctionnalités supplémentaires qui ne seront pas transférées vers des installations sur site dans un avenir proche. Cela est fait pour accélérer la transition d'un certain nombre d'entreprises vers les nuages, car Le modèle d'abonnement est plus avantageux financièrement pour Microsoft qu'une vente unique.

Si vous effectuez actuellement la maintenance d'une installation locale d'Exchange Server (2013-2019), vous pouvez continuer à le faire pendant les 3 à 5 prochaines années. En cours de route, vous devriez commencer à explorer les opportunités qu'offre Exchange Online; et les configurations hybrides, c'est lorsque les versions locale et cloud existent simultanément. Même si nous supposons qu'il n'y aura plus la prochaine version locale (sur site) d'Exchange, les connaissances acquises sur Exchange Server continueront d'être pertinentes pendant un certain temps pour un certain nombre de raisons:

  • Le nombre d'installations locales à l'heure actuelle est assez important, par conséquent, des administrateurs qualifiés capables de les accompagner seront nécessaires. Toutes les organisations ne seront pas en mesure de transférer leur courrier vers les nuages ​​dans un avenir proche pour une raison ou une autre.
  • Les projets de migration vers le cloud ne sont pas encore triviaux, il est donc nécessaire de connaître les spécificités des solutions locales et cloud afin de contourner la plupart des écueils et de mener à bien la migration.
  • La connaissance des protocoles smtp \ imap \ mapi \ pop3, flux de messagerie, dkim, dmark, spf, antivirus et antispam est universelle et sera applicable à tout système de messagerie.
  • L'expérience acquise en travaillant avec le serveur Exchange local vous permettra de comprendre rapidement Exchange Online et de configurer la configuration souhaitée.
  • Le courrier électronique est l'un des canaux de communication les plus importants avec le monde extérieur, de sorte que le besoin en restera. Les adhérents «les messagers et les robots de chat remplaceront le courrier» ne peuvent pas être écoutés, car ils ont "enterré" le courrier plusieurs fois et jusqu'à présent sans succès.

Skype Entreprise (SfB) (anciennement Lync) est un messager d'entreprise doté de fonctionnalités avancées. Il a une intégration étroite avec le serveur Exchange, mais est nettement inférieur à ce dernier en termes de popularité. Skype Entreprise n'est généralement utilisé que dans les grandes entreprises, comme petites et moyennes entreprises, ce n'est pas très intéressant.

La version actuelle est désormais Skype Entreprise 2019, qui présente des différences minimes par rapport à la version précédente de Skype Entreprise 2016, de sorte que SfB 2019 peut être conditionnellement considéré comme le Service Pack 1 pour SfB 2016, et non une nouvelle version à part entière.

Dans le cloud Office 365, ce produit a été introduit par Skype Entreprise Online, qui après un certain temps a été complètement remplacé par Microsoft Teams, c'est-à-dire Skype Entreprise n'est actuellement pas dans le cloud Office 365. Pour cette raison, il ne vaut guère la peine d'attendre l'apparition de la prochaine version locale de Skype Entreprise 2022, car la priorité pour Microsoft est le développement et le développement du messager Teams, qui est devenu la réponse du fournisseur à l'émergence d'un messager Slack réussi.

Si vous administrez actuellement Skype Entreprise local et que vous aimez le concept de messager d'entreprise, je vous conseille de regarder les équipes dans le cadre d'Office 365, sinon il est préférable de choisir un produit différent pour pomper vos connaissances, car Skype Entreprise local attend l'oubli. Contrairement à Exchange, qui est devenu la norme de facto dans le créneau des serveurs de messagerie, Skype Entreprise propose aujourd'hui des alternatives. Team et Slack pour les grandes et moyennes entreprises. Telegram, Viber, Whatsapp - pour les petites entreprises.

SharePoint est un portail d'entreprise interne où les entreprises peuvent publier leurs services Web utiles (calendrier des vacances, liste des employés avec photos et téléphones, rappels d'anniversaire, actualités de l'entreprise, etc.). Les utilisateurs peuvent stocker, modifier et partager des fichiers qu'ils hébergent dans leurs bibliothèques SharePoint.

SharePoint est comme Bitrix24, seulement plus grand, plus fonctionnel, plus cher et plusieurs fois plus difficile à configurer et à prendre en charge. Les fonctionnalités de Killer sont la possibilité de modifier simultanément un document avec un grand nombre d'employés, ce qui est très pratique lorsque 100 personnes essaient de remplir un calendrier de vacances et l'intégration avec Office Online Server et MS Office local.

Sharepoint est un produit volumineux, complexe et coûteux, c'est pourquoi il n'est généralement utilisé que dans les grandes entreprises. Les petites entreprises utilisent Bitrix24 ou ses analogues, ou stockent simplement des fichiers sur des serveurs de fichiers, et des services Web utiles sont distribués sur différents sites internes.

Les batteries de serveurs SharePoint (clusters) sont généralement traitées par des développeurs dotés de fonctions d'administrateur, plutôt que par des administrateurs système «purs», comme pour que SharePoint décolle et soit utile à l'entreprise, il a besoin de beaucoup de complétion de code.

Office 365 comprend le service SharePoint Online, qui est une version simplifiée de SharePoint local, c'est-à-dire il a un nombre minimal de possibilités de personnalisation et «l'a terminé pour lui-même», mais il supprime beaucoup de maux de tête de son développeur / administrateur pour son fonctionnement. Mon verdict est que la complexité et le coût élevé de la prise en charge des versions sur site de SharePoint feront leur travail et les entreprises commenceront volontiers à ramper progressivement vers SharePoint Online, ou à abandonner complètement Sharepoint au profit d'une solution plus simple. Personnellement, je ne vois pas d'arc-en-ciel et de vie insouciante pour SharePoint dans les installations locales.

System Center est une famille de produits pour le déploiement, la configuration, la gestion et la surveillance de grandes infrastructures Windows. A en juger par: System Center Configuration Manager (SCCM), System Center Virtual Machine Manager (SCVMM), System Center Operations Manager (SCOM), System Center Data Protection Manager (SCDPM), System Center Service Manager (SCSM), System Center Orchestrator (SCORCH )



La gamme complète des produits System Center n'est généralement demandée que dans les grandes entreprises, tandis que les moyennes entreprises n'utilisent généralement qu'un ou deux produits.

Étant donné que les produits System Center sont assez difficiles à apprendre et ne sont généralement utilisés que dans les grandes infrastructures, il est habituel de distinguer les individus pour travailler avec eux, par exemple, un administrateur de système de surveillance (SCOM), un administrateur de service de poste de travail (SCCM) et un administrateur de système de virtualisation (Hyper -V + SCVMM), administrateur d'automatisation d'infrastructure (SCORCH + SCSM).

Microsoft développe rapidement ses services cloud, de sorte que les fonctionnalités de System Center affluent progressivement dans les clouds. Tout cela aura un impact significatif sur les produits System Center sur site dans un avenir très proche.

À l'avenir, la fonctionnalité de System Center Orchestrator (SCORCH) remplacera le service Azure Automation ( https://docs.microsoft.com/en-us/azure/automation/automation-intro ).

À l'avenir, la fonctionnalité de System Center Operations Manager (SCOM) remplacera le service Azure Monitor ( https://docs.microsoft.com/en-us/azure/azure-monitor/overview ).

La fonctionnalité de System Center Data Protection Manager (SCDPM) remplacera le service Azure Backup ( https://docs.microsoft.com/en-us/azure/backup/backup-overview ) à l'avenir.

La fonctionnalité de System Center Service Manager (SCSM) cessera d'être demandée ou sera remplacée par tout autre système de ticket, par exemple Jira.

System Center Virtual Machine Manager (SCVMM) restera avec les entreprises qui utilisent la virtualisation Hyper-V localement. Les petites installations Hyper-V (10 à 15 serveurs) peuvent être gérées avec succès sans SCVMM en utilisant uniquement des outils standard - Gestionnaire de cluster de basculement, Gestionnaire Hyper-V, Centre d'administration Windows.

System Center Configuration Manager (SCCM) - utilisé pour le déploiement en masse de systèmes d'exploitation, l'installation d'applications d'entreprise à partir d'un répertoire unique, l'installation de mises à jour Windows sur les serveurs et les postes de travail finaux, l'inventaire des applications et le comptage des licences. Il semble que ce soit le seul produit de toute la gamme System Center qui restera avec nous dans l'infrastructure sur site, comme le remplacer complètement par quelque chose de nuageux pour le moment ne fonctionne pas.

Si vous prenez actuellement en charge une installation locale de System Center Configuration Manager (SCCM), vous pouvez continuer à le faire, comme Le produit sera avec nous pendant au moins les 3-5 prochaines années. De plus, je recommanderais de commencer à explorer les possibilités d'Office 365, comme cela ira bien avec le poste d'administrateur de bureau d'entreprise.

Le rôle d'administrateur de la plupart des autres produits System Center sera supprimé. Les services Azure simplifient considérablement leur travail, cachant toute complexité aux regards indiscrets. Prenons par exemple l'administrateur d'automatisation (SCORCH + SCSM). SCORCH sera remplacé par Azure Automation. La connaissance du processus d'automatisation, PowerShell, SQL restera et sera utile pour Azure Automation, mais la connaissance de la création de clusters SCORCH, assurant leur haute disponibilité, le dimensionnement des ressources, la mise à jour, la migration vers de nouvelles versions, la sauvegarde et la surveillance perdront leur pertinence, car Tous ces travaux seront réalisés par le cloud Azure. L'administrateur d'automatisation se concentrera uniquement sur le processus d'automatisation lui-même, comme il laissera tout le travail pour maintenir l'opérabilité de l'infrastructure d'automatisation.

Serveur Windows et ses rôles


Active Directory (AD) est l'endroit où les comptes d'utilisateurs et d'ordinateurs sont stockés. Si la société compte plus de 20 ordinateurs, il est fort probable qu'il existe déjà un domaine Active Directory. La connaissance d'Active Directory, la capacité de distinguer un domaine d'une forêt, la capacité de travailler avec des stratégies de groupe sont obligatoires pour tout administrateur Windows. Ces connaissances seront pertinentes pendant encore 20 ans. En outre, je vous recommande de vous familiariser avec Azure AD (AAD), d'examiner les options de synchronisation des utilisateurs entre les infrastructures sur site et cloud.

DNS, DHCP sont des services réseau dont la compréhension est utile dans tous les domaines de l'informatique, de l'administration à la programmation, vous devez donc les connaître.Comprendre le fonctionnement des réseaux, les protocoles de routage, les modèles OSI et TCP \ IP sera un atout certain pour tout spécialiste informatique.

Hyper-V est le nom de toute la pile des technologies de virtualisation de Microsoft et de son hyperviseur en particulier. Il se développe assez rapidement, bien qu'à mon avis, la plupart des nouvelles fonctionnalités (Shielded VM, Encrypted Subnets, Storage Spaces Direct) se concentrent principalement sur les fournisseurs de cloud local (Cloud Service Providers) et global (Azure), et non sur le segment d'entreprise (Enterprise). Cela est généralement compréhensible, car Microsoft implémente et déploie d'abord de nouvelles fonctionnalités dans son cloud Azure, puis ne les transfère qu'à Windows Server et Hyper-V.

Hyper-V souffre toujours de l'absence d'une seule console gratuite où toutes les fonctionnalités nécessaires sont présentées. Nous avons maintenant le gestionnaire de cluster de basculement, le gestionnaire Hyper-V et le centre d'administration Windows. SCVMM était censé être une telle console, mais elle est payante et quelque peu difficile à apprendre.

Si vous prenez actuellement en charge une installation locale d'Hyper-V avec \ sans SCVMM, vous pouvez continuer à le faire. En parallèle, je recommanderais de commencer à en savoir plus sur Azure IaaS et les mécanismes de migration des machines virtuelles entre le cloud et l'infrastructure locale.

Dans mon environnement (banques, télécoms, compagnies d'assurance, grandes entreprises industrielles), toute la virtualisation productive est généralement gérée par VMware vSphere et non Hyper-V avec SCVMM, je peux donc recommander à l'administrateur Hyper-V de se tourner vers VMware et ses produits .

Services cloud


Office 365 est un service cloud qui fournit un package d'abonnement d'applications Microsoft Office (local et sa version Web), ainsi que les principaux produits serveur - Exchange, Teams, OneDrive et Sharepoint.

Office 365 est actuellement un service autonome qui couvre presque entièrement les besoins en communications de bureau. En raison de la facilité d'installation, il est parfait pour les petites entreprises et les moyennes et grandes entreprises.

La présence de services Exchange, Teams, OneDrive et Sharepoint déjà déployés dans le cloud réduit considérablement la charge de l'administrateur système, car toutes les procédures d'installation, de dimensionnement des ressources, de mise à jour et de migration vers de nouvelles versions incombent désormais entièrement à Microsoft. Auparavant, pour prendre en charge Exchange, Teams, OneDrive et Sharepoint dans l'infrastructure locale, 4-6 administrateurs distincts auraient été nécessaires, mais maintenant Office 365 n'a besoin que d'un administrateur moyen. Si quelque chose ne fonctionne pas ou ne fonctionne pas correctement, vous pouvez créer un ticket pour le support technique Microsoft directement à partir de l'interface Office 365, ce qui est très pratique.

Si vous êtes maintenant l'administrateur système accompagnant les versions locales des produits Exchange, Skype Entreprise ou Sharepoint, je recommanderais de regarder leurs versions cloud dans le cadre d'Office 365 pour comprendre comment elles vous conviennent et quelles fonctionnalités elles fournissent par rapport aux versions locales.

Azure est une plate-forme cloud mondiale de Microsoft, qui comprend un ensemble toujours croissant de services cloud qui aident les organisations à résoudre leurs problèmes commerciaux. Actuellement, Azure comprend plus de 300 services regroupés en différentes catégories (informatique, réseaux, stockage, bases de données, analytique, Internet des objets, sécurité, devOps, conteneurs, etc.).

Apparu pour la première fois en 2009, Microsoft Azure occupe désormais l'une des positions de leader sur le marché mondial des services cloud, y rivalisant avec succès avec Amazon AWS.

Selon le dernier rapport financier ( https://www.microsoft.com/en-us/Investor/earnings/FY-2019-Q4/press-release-webcast ) trimestriel (T4 2019), le bénéfice Microsoft a augmenté de 49% en raison du succès d'Office 365 et le cloud. Le chiffre d'affaires d'Azure a augmenté de 64%.

Azure, avec Office 365, sont les principaux domaines dans lesquels Microsoft place ses ressources financières et organisationnelles.

L'abondance de services sur la plate-forme Azure peut dérouter même un spécialiste informatique expérimenté, donc ci-dessous sera une description d'une infrastructure Windows basée sur serveur typique, où j'indiquerai leurs analogues approximatifs dans le cloud Azure entre parenthèses. J'espère que cela servira de point de départ pour l'étude d'Azure, car, comme vous le savez, vous devez commencer petit, aller progressivement plus loin.

Une infrastructure Windows côté serveur typique ressemble à ceci:

  • Active Directory (AD) avec stratégie de groupe et DNS. (Azure Active Directory (AAD), Azure DNS) .
  • DHCP
  • Serveur de messagerie Exchange. (Exchange en ligne dans le cadre d'Office 365) .
  • Ferme RDS avec plusieurs serveurs de terminaux. (Machine virtuelle Azure + Réseau virtuel Azure + Stockage Azure) .
  • Un serveur de fichiers où les employés stockent leurs fichiers. (Stockage de fichiers Azure, machine virtuelle Azure + réseau virtuel Azure + stockage Azure)
  • Serveurs avec applications et bases de données (1C, site \ portail interne, CRM, etc.). (Base de données Azure SQL, sites Web Azure, Microsoft Dynamics 365, machine virtuelle Azure + réseau virtuel Azure + stockage Azure)

Les principales tâches administratives sont:

  • Créez des sauvegardes. (Sauvegarde Azure) .
  • Collecte et analyse des journaux. (Azure Log Analytics) .
  • Automatisation des tâches de routine. (Azure Automation) .
  • Surveillance de l'état des services et réception des notifications d'échec (Azure Monitor) .

Je conseillerais aux administrateurs Windows accompagnant les infrastructures locales de rechercher d'abord des analogues de leurs services préférés dans le cloud Azure pour travailler un peu avec eux, de déterminer leur utilité pour l'entreprise et, éventuellement, d'organiser des options hybrides, en choisissant le meilleur des deux mondes.
