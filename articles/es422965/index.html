UniSharping: Convertir código C # en Java y Python

Introduccion


Desde los años 70, se ha desarrollado el inglés simplificado , cuyo propósito es determinar un subconjunto del idioma que sea comprensible para una amplia gama de hablantes no nativos del idioma. Recomendado, por ejemplo, para documentación técnica. Los traductores automáticos en un subconjunto de este tipo funcionarán obviamente de manera más correcta, idealmente generando texto que no requiere revisión manual.
Si aplica este enfoque a C # para la tarea de convertir automáticamente el código a otros lenguajes de programación, puede seleccionar un subconjunto de construcciones de lenguaje, bibliotecas del sistema y tecnologías que podrían traducirse a una amplia gama de otros lenguajes. Además, la conversión no es una sola vez (migración), sino constante para expandir las capacidades de integración del proyecto en C #, para que en cualquier momento pueda obtener un código de trabajo en otro idioma sin la necesidad de ninguna edición.


Déjame presentarte: UniSharping


La restricción de C # .NET para resolver este problema se llamó U # (Universal Sharp), y el proceso de conversión y su herramienta se llamaron UniSharping . Los módulos ejecutables, la configuración y la documentación están disponibles en GitHub , el sistema es gratuito para uso no comercial (Freeware no comercial).


Para multiplataforma, Microsoft ya ha hecho una limitación de .NET Framework en términos de bibliotecas y tecnologías: .NET Core. Es como el primer paso en la dirección correcta, U # da el segundo paso hacia la "capacidad de programación cruzada".


Hubo pocas limitaciones de U # en las construcciones del lenguaje: estos son atavismos goto y case goto, así como el rendimiento, que no se modela adecuadamente en modo automático. No se recomienda (aunque es posible) usar struct, hay matices con nombres; todo esto se describe en detalle en un documento separado. El analizador U # proporciona errores y advertencias, y para garantizar la generación correcta, debe ajustar el código fuente de C # para que, idealmente, desaparezcan por completo. Si aún necesita conservar la versión original, puede usar las directivas de preprocesador #if JAVA || PHP ... #else ... #endif. Estas restricciones se aplican a nivel del motor U # y no están sujetas a corrección externa, así como a la lista de idiomas admitidos.


Pero las restricciones a nivel de las bibliotecas del sistema no se establecen rígidamente y se configuran externamente a través de archivos de texto especiales que determinan cómo traducir una clase particular y sus miembros al idioma correspondiente. Si hay un análogo directo, entonces se indica, si la situación es más complicada, entonces se escribe un código del lenguaje final o, en general, una clase especial (de servicio) que resuelve el problema deseado. En casos muy difíciles, debe "codificar" en el nivel del motor, pero tales situaciones son bastante raras (alrededor de una docena). El procedimiento para configurar las clases del sistema y sus miembros se describe en un documento separado. Aquí hay una lista de clases de C # compatibles y sus miembros con análogos en Java y Python en la versión actual del sitio, también hay una demostración en línea .


En cuanto a la tecnología, ahora la lista se limita a la aplicación de consola y las pruebas unitarias (UnitTest). Bueno, los proyectos individuales de Lib, como un caso especial, se traducen en las construcciones correspondientes del idioma deseado.


Para una traducción exitosa, el proyecto de C # de origen (solución) debe tener una parte de inicio que verifique la funcionalidad dentro del C # de origen. Es bueno si se trata de un extenso sistema de pruebas automáticas (UnitTest estándar en diferentes implementaciones o autoescrito), pero al menos debe haber al menos una aplicación de consola que funcione correctamente cuando se inicia sin intervención del usuario. La necesidad de esto es obvia: después de la generación en el lenguaje final, puede verificar inmediatamente el rendimiento. Idealmente, todas las pruebas deberían funcionar de manera similar a C #.


Historia del proyecto


La idea de tal convertidor ha existido durante mucho tiempo. Mi principal proyecto de procesamiento de lenguaje natural SDK Pullenti es un candidato ideal para la conversión: una gran cantidad de código complejo y en constante mejora. Para integrarme con Java, tuve que envolverlo en servicios web, servidores tcp, etc.
El verano pasado, encontré el tiempo y la energía para crear la primera opción. Tradujo el proyecto Pullenti a Java, así como a sí mismo en Java.
Durante los siguientes seis meses, el convertidor se desarrolló en varios proyectos internos que estaban en la compañía, principalmente a través de la expansión de las clases de sistemas.
En la primavera de 2018, la idea surgió para apoyar Python, que se implementó en el verano. Pero la inclusión de un segundo idioma no se proporcionó en la versión inicial y resultó torpemente. En el verano tuve que rehacer completamente el motor por el potencial de varios idiomas finales. Además, la configuración de las clases del sistema del código rígido se movió a archivos de texto externos. Espero que este conjunto no se expanda sin su ayuda.


Otros planes son los siguientes:


  • tire de Python al nivel de Java. Python ahora es compatible con el nivel Pullenti, pero Java ha avanzado mucho en otros proyectos en comparación con él.
  • admite PHP al menos en el nivel de proyecto Pullenti.
  • Soporta C ++. Sí, me doy cuenta de que esto es muy difícil, ya que no está claro al liberar memoria qué puntero es un enlace y para qué eliminación debe hacerse. Pero hay ideas ...

¿Quién puede ser útil?


Principalmente aquellos que desarrollan SDK potencialmente multiplataforma en C #. Gracias al convertidor UniSharping, su SDK también puede convertirse en "software cruzado", lo que ampliará el círculo de usuarios potenciales.
Recientemente, las posiciones de STR se han fortalecido en Rusia, que se han convertido en obligatorias en la mayoría de las agencias gubernamentales y algunas grandes empresas. Explique que .NET Core tampoco siempre funciona, porque es "Microsoft". Deje que alguna compañía desarrolle su sistema de información en C #. Para introducir el producto en la "empresa de código abierto", puede seleccionar la parte lógica del proyecto (back-end), convertirlo automáticamente a versiones según sea necesario y convertir la parte visual (front-end) en software de código abierto. Es decir, para continuar el desarrollo en C #, y en Java solo front-end.


No excluyo que, en principio, la conversión de proyectos web sea posible (con limitaciones, por supuesto), pero no tengo las habilidades e información necesarias para esto. Si alguien ve esa oportunidad, es muy posible implementarla en UniSharping.


Observo que para un proyecto de C # realmente complejo, admitir Java u otro lenguaje requerirá cierto esfuerzo para modificar el código, resaltar la parte portátil en el proyecto y "rodearlo" con pruebas unitarias. Además, configurar clases y métodos de sistema aún no compatibles y corregir errores de UniSharping en sí (con mi ayuda) sigue siendo el trabajo. Pero el proceso es convergente, al final del cual el proyecto espera una bonificación entre programadores.
