Tidak hanya uBlock Origin akan menderita API baru di Chromium, tetapi juga ekstensi lainnya

Selama tiga bulan, pengembang Chromium telah membahas API deklaratifNetRequest yang baru, yang membuatnya tidak mungkin untuk sepenuhnya menggunakan API webRequest . Ekstensi menggunakan data API untuk memblokir konten dengan cepat selama pemuatan halaman. Dalam sistem baru, pemblokir tidak dapat memblokir acara, tetapi hanya melihatnya. Alih-alih, ekstensi ditawarkan untuk beralih ke API DeclarativeNetRequest - dan memberi tahu browser tentang peristiwa yang ingin mereka blokir. Ini seharusnya mempercepat pemuatan halaman di browser (karena ekstensi tidak lagi memperlambat aliran utama), dan juga melindungi privasi pengguna, menurut Google.

Yang pertama membunyikan alarm adalah Raymond Hill, penulis uBlock Origin dan uMatrix. Dia menyatakan bahwa ekstensi pemblokiran iklannya â€œtidak akan adaâ€ jika perubahan diterima.

Pesimisme kemudian diungkapkan oleh pengembang ekstensi lain, termasuk F-Secure, NoScript, dan Ermes Cyber â€‹â€‹Security. Misalnya, NoScript untuk Firefox tidak dapat diangkut ke Chrome.

Raymond Hill mencatat bahwa declarativeNetRequest API menggunakan sistem pemfilteran gaya Adblock Plus yang tidak kompatibel dengan uBlock Origin. Dia percaya bahwa ini adalah kesalahan mendasar: "Selain fakta bahwa uBlock Origin dan uMatrix tidak ada, mengkhawatirkan bahwa deklaratifNetRequest API yang diusulkan memblokir penerapan mesin penyaringan konten baru pada arsitektur inovatif, karena declarativeNetRequest API tidak lebih dari penerapan satu spesifik mesin penyaringan, dan implementasi yang agak terbatas (batas 30.000 pembatasan tidak cukup untuk menyelesaikan daftar EasyList yang terkenal saja). " Raymond Hill juga mencatat bahwa API baru tidak mendukung beberapa fitur lain, termasuk memblokir elemen multimedia yang lebih besar dari ukuran yang ditentukan, menonaktifkan eksekusi JavaScript dengan memperkenalkan arahan Kebijakan-Keamanan-Konten, dan menghapus header cookie yang keluar. Raymond percaya bahwa perubahan ini bukan untuk kepentingan pengguna.

Menurut Andrei Meshkov , salah satu pendiri pemblokir iklan AdGuard Chrome lainnya, perubahan ini kemungkinan akan mempengaruhi semua pemblokir iklan lainnya.

Bukan hanya blocker


Selain pemblokir, ekstensi anti-virus akan sangat menderita. "Selain pemblokiran iklan, itu mungkin akan mempengaruhi perangkat lunak, yang bergantung pada pemblokiran dinamis lalu lintas https, yang dinilai berbahaya," kata Juni Korte, insinyur perangkat lunak utama di produsen antivirus Finlandia F-Secure. "Ini adalah halaman yang menyebarkan malware, serta, misalnya, fungsi kontrol orang tua, yaitu, melindungi pengguna dari konten yang diklasifikasikan sebagai berbahaya / tidak pantas baginya."

Claudio Guarnieri, spesialis terkemuka organisasi hak asasi manusia Amnesty International, mendukung pendapat pengembang: "Saya ingin mengulangi apa yang dikatakan Junie. Saya percaya bahwa perubahan ini akan menghambat berfungsinya banyak ekstensi keamanan, â€ tulisnya .

"Jika perubahan ini dipublikasikan, ekstensi [saya] akan berhenti berfungsi," Brandon Dixon, penulis ekstensi Blockade.io, yang memblokir serangan drive-by dan mencegah akses ke situs phishing, bergabung dengan rekan-rekannya .

Pendapat serupa diungkapkan oleh Christoph Kovacs , salah satu pengembang ekstensi kontrol orang tua, pencipta ekstensi Privowny , yang menyediakan berbagai fungsi untuk meningkatkan privasi di Internet, serta tim Ermes Cyber â€‹â€‹Security , pencipta ekstensi keamanan berbasis Chrome lainnya.

Penulis ekstensi NoScript populer untuk Firefox mengatakan bahwa jika Anda menerima perubahan ini, ia tidak akan dapat mem-porting NoScript ke Chrome.

Para kritikus percaya bahwa Google, dengan dalih palsu, sebenarnya mencoba membatasi fungsionalitas pemblokir iklan pihak ketiga untuk mempromosikan pemblokir browser yang baru-baru ini diumumkan oleh perusahaan. Dan juga untuk mengontrol iklan mana yang dapat diblokir oleh pengguna dan mana yang tidak.

Berita baiknya adalah bahwa kritik terhadap API DeclarativeNetRequest yang baru datang pada saat yang tepat ketika pengembang Google terbuka untuk menerima umpan balik. Ada harapan bahwa mereka akan berubah pikiran dan menolak untuk mengimplementasikan API baru dalam kode Chromium, yang menjadi basis Chrome, Vivaldi, Opera, Brave, dan browser lainnya.
