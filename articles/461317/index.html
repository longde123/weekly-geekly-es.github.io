9 principios para crear aplicaciones iOS de calidad

Los principios se basan en la conocida metodología de heroku , adaptada a las realidades del desarrollo de ayios (falta de contenedores que tardan varios días en completarse y ralentizar la implementación, Xcode solo funciona en una amapola).


Este artículo es una breve introducción, la serie completa se puede encontrar en el factor iOS , la traducción al ruso también está disponible. El proyecto de código abierto del factor iOS en github se refina constantemente y da la bienvenida a nuevas ideas. También participé en su desarrollo. El proyecto fue fundado por Felix, quien es el creador de Fastlane .


TL; DR


  • Dependencias : debe especificarse explícita y específicamente (versión de Xcode, CocoaPods, versión de dependencias en podfile). Esto hace posible que un nuevo desarrollador juegue la compilación en cualquier Mac. Repita también el ensamblaje que se utilizó hace 6 meses.
  • Configuración : no hay configuración en el código, viene con la aplicación y la capacidad de actualizar por aire
  • Desarrollo de aplicaciones / paridad de trabajo : mantenga los entornos de desarrollo, preparación y producción lo más similares posible
  • Implementación : automatice la implementación para que pueda liberarla desde cualquier máquina.
  • Local versus Beck : mantenga su aplicación iOS universal para que pueda trabajar sin backend siempre que sea posible
  • API compatibles con versiones anteriores : no piense que cada usuario está actualizando a la última versión
  • Control de versiones de la aplicación : automatice la versión y cree actualizaciones
  • Rollback : montaje de Rollback que causa problemas
  • Almacenamiento de datos : siga las recomendaciones de Apple para el almacenamiento de datos


Dependencias


La aplicación declara todas sus dependencias de forma completa y precisa utilizando el manifiesto de declaración de dependencia.
Esto incluye una versión específica de Swift, Xcode, CocoaPods, Carthgae y Fastlane. Además, todas las dependencias en podfile y cartfile deben especificar una versión específica.


Una de las ventajas de declarar explícitamente dependencias es que facilita la configuración de aplicaciones para nuevos desarrolladores.


Con la ayuda de especificar dependencias específicas, puede reproducir la compilación que se usó hace 6 meses, sabiendo que se compilará porque la compilación usará la misma versión de Xcode, CocoaPods y Swift.


Limitación: dado que el desarrollo de iOS no puede estar contenido en un contenedor, ya que ya se usa para el desarrollo web, estamos limitados a herramientas de terceros que intentan cumplir este requisito hasta que Apple proporcione una solución oficial. Existe una solución comercial (cerrada) de terceros llamada Veertu que le permite crear entornos virtuales de MacOS en el hardware de Apple.



Configuracion


El código no depende del entorno, pero la configuración depende. Por lo tanto, el código debe almacenarse en el repositorio y la configuración en el entorno. Verificar si la configuración y el código de la aplicación están separados correctamente es el hecho de que la base del código de la aplicación puede ser accesible libremente en cualquier momento sin comprometer ningún dato privado.


Hay muchas formas de ingresar valores de configuración en el momento de la compilación.


  • Archivos de configuración (archivos JSON o YAML)
  • cocoapods-keys para ocultar las claves y aplicarlas a su aplicación iOS durante la compilación
  • Solución personalizada (por ejemplo, usando la fase de construcción)

Dado que las implementaciones en la plataforma iOS son significativamente más lentas que en los servidores, es posible que necesite una forma de actualizar la configuración inalámbrica (OTA) para responder rápidamente a los problemas.


Las actualizaciones de configuración de OTA le permiten al instante:


  • Ejecute pruebas A / B para habilitar ciertas funciones o cambios en la interfaz de usuario solo para una parte de los usuarios activos.
  • Cambiar claves de API
  • Actualice los servidores web u otras URL que hayan cambiado
  • Desactivar de forma remota funciones u ocultar botones

Algunos enfoques potenciales al implementar actualizaciones de OTA son:


  • Implementa tu propia decisión
  • Soluciones llave en mano como la configuración remota de Firebase
  • Un servicio de código abierto como FeatureFlags


Desarrollo de aplicaciones / paridad laboral


Históricamente, existen diferencias significativas entre el desarrollo (el desarrollador realiza cambios en vivo en la implementación local de la aplicación) y la operación de la aplicación (implementación de la aplicación en la App Store con acceso a ella por parte de los usuarios finales).
Estas diferencias aparecen en tres áreas:


  • Diferencia horaria
  • Diferencia personal
  • Brecha de herramienta:

Solución:


  • Reduzca la diferencia horaria: el desarrollador puede escribir el código y se implementará en unas pocas horas o incluso minutos.
  • Haga pequeñas las diferencias de personal: el desarrollador que escribió el código participa activamente en su implementación y monitorea su comportamiento mientras se ejecuta la aplicación.
  • Reduzca las diferencias de herramientas: mantenga el entorno de trabajo y desarrollo de su aplicación lo más similar posible.


Despliegue


Como se describe en el principio de dependencias , el repositorio de código debe incluir todas las dependencias necesarias para crear, probar e implementar la aplicación iOS.


Desafortunadamente, debido al hecho de que Xcode debería funcionar en MacOS, no podemos usar contenedores como en la web. Ejecutar macOS en un entorno virtual está lleno de problemas técnicos y legales.


En este momento, el mejor enfoque que podemos usar los desarrolladores de iOS es:


  • Automatice la instalación de Xcode con xcode-install
  • Utilice el archivo .xcode-version para especificar la versión exacta de Xcode
  • Defina todas las dependencias en los archivos de configuración (consulte Factor de dependencias)
  • Automatice todo el proceso de implementación utilizando fastlane
  • Automatizar la firma de la aplicación (codesigning.guide)
  • Suelte a menudo, idealmente en un horario semanal

Muchas compañías usan el concepto de trenes de lanzamiento: un cronograma que lanza una nueva versión de su aplicación. Todo el código que se fusionó con su sucursal principal (maestra o versión) en el momento en que el tren de lanzamiento "se va" se enviará a la App Store. Este enfoque es implementado por la mayoría de las grandes aplicaciones de iOS.



Local versus Beck


En los últimos años, algunos equipos de desarrollo han comenzado a utilizar enfoques que requieren menos esfuerzo de desarrollo al reducir la calidad de la experiencia del usuario, trasladar más lógica al backend y hacer que la aplicación iOS sea un cliente ligero que muestre los resultados del servidor.


Una aplicación debe hacer la mayor cantidad posible de lógica y computación de negocios en el dispositivo por varias razones:


  • Confidencialidad: evite enviar datos a un servidor remoto
  • Velocidad: enviar datos al servidor y esperar una respuesta lleva tiempo y puede provocar un bloqueo (por ejemplo, WiFi deficiente)
  • Uso de datos: los usuarios a menudo tienen límites de datos mensuales
  • Escalado: si su aplicación se vuelve popular, usted es responsable de ampliar los servicios de back-end.
  • Duración de la batería: el móvil es caro
  • Fiabilidad: las conexiones LTE / 3G siguen siendo pobres en algunos países

Si su aplicación requiere una conexión a Internet para todas las funciones (por ejemplo, una aplicación de red social o una aplicación para compartir viajes), debería funcionar (en modo de solo lectura) sin una conexión a Internet para acceder a datos históricos (por ejemplo, recientes viajes, comunicaciones directas recientes).



API compatibles con versiones anteriores


Aunque la mayoría de sus usuarios se actualizarán a la última versión en unas pocas semanas, siempre habrá usuarios que no lo harán. Esto puede tener varias razones. A menudo, esto se debe a la versión de iOS utilizada, que no siempre pueden actualizarse debido a la antigüedad del dispositivo.


El concepto básico es que no está actualizando la API existente, sino que está agregando una nueva y permitiéndoles trabajar en paralelo 


https://your-api.com/1.0/drivers.json https://your-api.com/1.1/drivers.json


Versiones de aplicaciones


La versión y el número de compilación se usan juntos para identificar una aplicación específica en la aplicación.


  • Número de versión (CFBundleShortVersionString) : se (CFBundleShortVersionString) como versión en Xcode
  • Número de compilación (CFBundleVersion) : se (CFBundleVersion) como Compilar en Xcode

En el desarrollo actual de iOS, no hay ninguna razón por la que deba cambiar manualmente estos números. En cambio, necesita un sistema confiable y automatizado para mantener las versiones actualizadas.


Xcode tiene una herramienta incorporada llamada agvtool .



Contragolpes


La App Store inicialmente no permite retrocesos, por lo que esta sección describe cómo lograr resultados similares utilizando los métodos disponibles.


Lanzamientos de hitos: al usar lanzamientos de hitos, puede implementar lentamente el ensamblado en el producto, comenzando con un pequeño número de usuarios activos


Sin embargo, incluso en el caso de las versiones por fases, es imposible recuperar por completo el ensamblaje: después de instalar el ensamblaje en el dispositivo del usuario, la única forma de cambiar este ensamblaje es distribuir una nueva versión con un número de versión / compilación actualizado.


Al igual que con los lanzamientos por fases, los ensamblados de App Store y TestFlight solo pueden actualizarse siguiendo estos pasos:


  1. Regrese a su sistema de control de versiones al estado al que desea volver
  2. Aumente la versión y / o el número de compilación de su proyecto
  3. Crea y firma tu solicitud
  4. Distribuya su aplicación a través del servicio beta o App Store
  5. El usuario debe actualizar la aplicación en su teléfono.

Los pasos anteriores se pueden realizar manualmente, sin embargo, se recomienda que el proceso esté completamente automatizado para poder responder rápidamente.


Alternativa: volver a firmar la compilación anterior


  1. Acceda al ensamblaje anterior (archivo .ipa) antes de introducir la regresión
  2. Actualice el número de versión / compilación en el archivo Info.plist
  3. Construcción antigua "Firmar de nuevo"
  4. Distribuirlo como una nueva compilación

Sin embargo, la "firma" de aplicaciones de iOS a menudo crea más problemas, especialmente porque las herramientas de línea de comandos de Xcode no ofrecen una buena manera de hacerlo.



Almacenamiento de datos


Almacenar datos y configuraciones de acuerdo con las recomendaciones de Apple es fundamental para el ciclo de vida de su aplicación, en particular cuando se trata de la sincronización de iCloud, la actualización a un nuevo teléfono y la restauración de su teléfono desde la copia de seguridad.


Asegúrese de seguir las Pautas oficiales de almacenamiento de datos de Apple iOS :


  • Documents : use este directorio para contenido personalizado, se archivará
  • Caches : use este directorio para los datos que se pueden recuperar.
  • tmp : use este directorio para archivos temporales
  • Utilice la propiedad de do not back up para archivos

Nunca almacene información confidencial del usuario (como contraseñas o sesiones) en estos directorios. Utilice la API de llavero en su lugar.
