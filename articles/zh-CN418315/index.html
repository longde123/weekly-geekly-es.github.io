没有一个“哎呀”：DevOops 2017的十大报告



如果您还没有去过DevOops 2017，那么您现在的想法可能是这样的：“希腊toga与它有什么关系？ 那么开发者大会上最好的报告是什么样的？ 该报告是什么？为什么被认为是最好的报告？ 普通衣服有严肃的表演吗？”

严肃的绰绰有余：如果您对与Kubernetes，Istio或Zipkin合作的细微差别感兴趣，那么在帖子中，您会发现所有这些，并且togas不会干扰说实际的话。 “最佳”的头衔取决于听众：我们根据他们的收视率来排名前十名。 第一个问题的答案已被削减。


（主题演讲）大规模的DevOps：希腊悲剧三幕


演讲者： Baruch Sadogursky和Leonid Igolnik
报告简报

在这种情况下，了解上下文很重要。 一天结束了。 观看者已经观看了六份报告，但最后的主题演讲仍然存在。 紧接它之后将有一个聚会，并且在一天的积极吸收信息之后，许多人可能想迅速参加它。

那时很难感觉到一些顽固的东西-但是关于devops的讽刺服装表演，像希腊的悲剧一样，完美地融入了语境，使整个大厅充满活力。 同时，“一个童话是一个谎言，但却是一个谎言”：关于一个虚拟公司如何在成长过程中遇到各种困难的故事，不仅使他笑起来，也使他思考了。

结果，该性能被DevOops评为最高评价。 为了让您分享他们的感受，我们建议您在类似的情况下观看视频：在困难的一天晚上，喝杯自己喜欢的东西。




扩展k8s


演讲者： 尼古拉·里芝科夫（Nikolay Ryzhikov）
报告简报

在上一段之后立即与之形成了鲜明的对比。 在这份报告中排名第二的是扎实的技术细节，而不是笑话和娱乐。

Kubernetes已经成为我们时代的关键工具（由于它，GitLab从那开始从Azure 迁移到GCP）。 尽管有些人已经将它与强大和主要功能结合使用，但其他人却对此感兴趣。 我们会在报告的开头“仔细看”并提供背景信息，使用它会变得更加有趣：Nikolai谈论“内部的一切”和“如何尽可能简化使用k8的开发人员的操作”（理想情况下，他想将来到NoOps）。




一片云：Odnoklassniki中的数据中心操作系统


演讲者： Oleg Anastasiev
报告简报

如今，随着Kubernetes的流行，谁又有理由不使用它呢？ 在Odnoklassniki：看着他，他们得出的结论是：“我们自己裁员比适应自己更容易。” 但是他们在那里有了Docker。 所有这一切都是在“创建自己的云”的框架中发生的：以前，社交网络使用了“每台服务器仅忙于一项任务”的原则，但是随着时间的推移，他们决定在一台服务器上执行不同的任务。 报告对此进行了详细介绍。

如果不是每个人都在拥有多个数据中心的公司中工作，并且大多数人具有完全不同的特殊性，那么Odnoklassniki的经验是否对其他人有用？ 只是说您不太可能以“我们需要做同样的事情”的感觉观看录音，但是其中一些报告可能在另一种情况下有用，并且了解“其他人的生活”对于对该行业的一般理解很有用。




（主题演讲）跟我一起走：解决您没有的问题


演讲者： Corey Quinn
报告简报

Corey Quinn 在AWS Weekly Newsletter中被称为Amazon AWS Last Week的创建者。 在他的文章中，他不断地进行熨烫，这与IT材料通常的干燥样式截然不同。 但是，尽管开了所有玩笑，但他的时事通讯是如此的翔实和有用，以至于亚马逊员工也经常关注新闻。

科里在开幕式上发表了主旨演讲，演讲中同样体现了“讽刺与行为”：人们可以听到既充满笑声又冒犯罪恶的情况。 “我曾经坐在Netflix的一份报告中，他们说他们向所有开发人员提供一般的制作访问权限。 坐在我旁边的那个人热情地同意了这些话。 我看着他的徽章，惊恐地发现他在银行工作。”




硅油锯大屠杀（我周末如何工作）


主讲人： 巴鲁克·萨多古尔斯基和列昂尼德·伊戈尔尼克
报告简报

Baruch Sadogursky和Leonid Igolnik再次提出，但是没有希腊的托加斯，并且有关于如何正确建立职责流程（以及通常如何生成）的报告。 “规模化开发运营”不再是纯粹有趣的事情，而是具有特定结论的更加实用的报告。 但是，如果您了解Baruch，您已经了解到该演出的地点在这里。 首先，演讲者不仅列举了这些“如何正确地做”的文章，还以假设的故事为例-您可以担心她的角色，认识自己和其中的其他人。 其次，所有这些都用gif进行说明-在录像中，您可以清楚地从听众的反应中听到听众对他们选择的准确性。




在Kubernetes中对生产应用程序进行故障排除和调试（aka The Failing Demo Talk）


主讲人： Baruch Sadogur和Ray Tsang

还有巴鲁克（Baruch），但是这次是另一位发言者（而且，您可能猜到了，是另一种语言）。

对于带有“现场演示”的报告，总是存在“舞台上的演示可能突然不起作用”的风险。 在这里，我们通过故意使演示“好像突然被拒绝”来克服了这一难题，并逐步了解需要做什么，以便一切都变得很好。 这与故障排除和调试的主题非常吻合：通过一个具体的可视示例来理解调试很有趣。




成功案例，或“ Dev + DevOps + Ops”


演讲者： Artyom Kalichkin和Sergey Berdnikov
报告简报

在对话中赞扬开发人员的好处很容易。 当一家大型公司的既定流程无法立即更改时，要真正找他们就困难得多。 而且，如果出现诸如“这是金融科技，那么安全至上”的情况，任务将变得更加复杂。

这是已经走过这条路的人的一份报告，他们在这里详细分享了自己的经验，从“过去”变为“成为”。 事实证明，除其他事项外，对于开发者而言，您不仅可以采取并下令切换到其他工具的方式-变革必须由人来做，并要从中来。
