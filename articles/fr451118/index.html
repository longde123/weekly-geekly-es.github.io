Les tests ne sont pas pour les débutants

Je suis un grand fan de tests. J'en parle dans mon blog et ma liste de diffusion, j'en discute avec d'autres développeurs pendant mon temps libre, je suis même allé jusqu'à créer un cours de formation sur les tests dans Go.

Mais malgré tout mon amour des tests, je ne le recommande pas aux débutants.

Cela semble sauvage, non? Dans cet article, je vais clarifier mon point de vue plus en détail, mais tout le point se résume finalement à deux points:

  1. Les débutants n'ont pas suffisamment de connaissances pour écrire autre chose que les tests les plus simples. Cela mène inévitablement au point suivant ...
  2. Essayer de former les compétences nécessaires pour écrire des tests réalistes tout en apprenant la programmation est extrêmement difficile

Je comprends que c'est, en principe, un point. En tout cas, je l'ai divisé en deux afin de le rendre plus facile à comprendre.

Je sais que beaucoup d'entre vous seront en désaccord avec moi, mais veuillez lire l'article, et si après lecture vous n'êtes pas convaincu, je serai heureux d'en discuter avec vous. En fin de compte, je suis ici pour étudier

Les débutants manquent de connaissances pour écrire autre chose que les tests les plus simples


Chaque fois qu'un novice écrit du code, son objectif principal n'est pas de décomposer les tâches, d'éviter les variables globales ou d'écrire du code de test. Honnêtement, la plupart des débutants ont probablement peu d'idée de tout cela. En fait, leur objectif principal est simple - faire fonctionner cette fichue chose. C’est tout.

Confirmer que cela ne nécessitera probablement aucun effort, il suffit de regarder le code écrit par le débutant.

Allez application web? Ils écrivent sûrement des requêtes SQL partout et partout dans le code, et la connexion à la base de données est presque garantie d'être stockée dans une variable globale.

Application Rails? Vous verrez sûrement la logique métier dans les vues et des tonnes de bourrages logiques dans les contrôleurs.

Application web PHP? Je ne serai pas surpris si TOUTE la logique se trouve dans un seul fichier php - analyse de formulaire, interaction avec la base de données, etc.

Même si nous prenons quelque chose d'élémentaire, disons, une calculatrice avec des fonctionnalités limitées, nous rencontrons toujours des exemples comme ceux que j'ai décrits. Ce n'est pas du tout dû au fait que les débutants s'en foutent, ils ne savent tout simplement pas comment faire mieux.

Les débutants ne savent pas ce qu'est l'injection de dépendance, ils ne comprennent pas comment les variables globales compliquent les tests, très probablement ils ne savent pas ce qu'est la simulation, il est donc assez ridicule de s'attendre à ce qu'ils comprennent comment concevoir du code facile à coder.

Par conséquent, pour un débutant, seuls les tests les plus simples comme ceux-ci ont du sens: 

func Add(a, b int) int { return a+b } // And a test... func TestAdd(t *testing.T) { got := Add(2, 4) want := 6 if got != want { t.Errorf("Add() = %d; want %d", got, want) } }

Bien que je n'aie aucun problème à le montrer aux débutants et à leur donner une idée de ce que sont les tests, je pense qu'il est assez ridicule de leur montrer ce code et de prétendre que c'est quelque chose comme un vrai test.

En fin de compte, cela conduit au fait que nous commençons à leur enseigner davantage. Nous essayons d'expliquer ce qu'est l'injection de dépendances, pourquoi les variables globales compliquent les tests, comme time.Now () peut rendre difficile la vérification des cas limites. Et à cet endroit, je commence à m'inquiéter, car nous n'enseignons plus au débutant comment écrire du code. En ce moment, nous lui apprenons comment écrire du code ET COMMENT TESTER en même temps. Et cela m'amène au deuxième point ...

Essayer de former les compétences nécessaires pour écrire des tests réalistes tout en apprenant la programmation est extrêmement difficile


Comme auparavant, je veux que vous pensiez au code écrit par le débutant, mais cette fois, je veux que vous vous souveniez de certains des premiers programmes que vous avez écrits.

Si vous étiez comme moi, votre première application Web pourrait ressembler à ceci: 

 <p> <?php // This may not work. I don't know PHP anymore. $name = $_GET['name']; echo "Hello, " . $name; ?> </p>

Une œuvre d'art, non?

Imaginez maintenant que vous venez d'écrire ce code pour la première fois et que quelqu'un vous dit de le tester. Et vous devez utiliser React. Et le cadre. Oh, et cela ne vous ferait pas de mal de modifier la base de données et éventuellement GraphQL pour travailler avec.

Je ne sais pas pourquoi, mais nous, les développeurs, avons l’habitude d’utiliser les compétences acquises au cours de nombreuses années d’expérience et de pratique et nous attendons à ce que d’autres, en particulier les débutants, fassent de même . C'est tout simplement ridicule! Il faut s'attendre à ce que quelqu'un comprenne les mathématiques uniquement parce que vous avez vous-même déjà étudié la trigonométrie, l'algèbre et bien plus encore, et vous pouvez utiliser les mathématiques pour résoudre des problèmes spécifiques.

Si quelque chose fonctionne bien pour vous, cela ne signifie pas qu'il conviendra aux débutants. Ils peuvent être hors contexte, sans l'expérience et la pratique nécessaires pour bénéficier de ce que vous utilisez. Ou peut-être que les problèmes avec lesquels ils se débattent sont en fait trop simples pour utiliser toutes ces solutions complexes.

C’est comme si nous avions oublié comment nous avons étudié pas à pas le fonctionnement des requêtes HTTP. Fonctionnement des en-têtes. Les cookies Formulaires. Comment fonctionne une requête POST sur le serveur - ou même qu'il existe des méthodes HTTP EXTRÊMEMENT différentes. Et nous avons appris tout cela avec la bonne vieille méthode d'essai et d'erreur.

En fait, je ne pense pas que les tests soient à blâmer, le vrai problème est que nous croyons fermement que vous devriez apprendre la programmation, les tests, la création de sites Web et un million d'autres choses en même temps. Je ne sais pas vraiment comment cela s’est produit, mais je suppose que le problème vient en partie du fait que nous ne prenons pas la peine de réfléchir à ce qu’il en coûtera pour étudier tout cela. Un débutant demande "que dois-je apprendre?" et nous commençons "apprendre à tester, réagir, graphql et c'est parti, mais n'utiliser que la bibliothèque standard ..."

Non, non et encore non. Arrête.



C'est ridicule, car dans d'autres conditions, l'absurdité d'une telle approche est évidente. Si vous voulez apprendre à quelqu'un à jouer au football, vous commencerez par les bases comme l'emballage et le dribble. Vous ne commencerez pas par dire «C'est comme ça que les professionnels procèdent», allumez la vidéo de Ronaldo et partez. Alors que diable faisons-nous avec les nouveaux venus dans la programmation?

Nous essayons de nous justifier en disant: «Eh bien, bien sûr, ils comprennent qu'il est impossible d'apprendre tout cela en même temps», mais ils ne comprennent pas! La situation est aggravée par le fait que les développeurs novices, tombant dans ce piège, se sentent comme de la merde parce qu'ils sont coincés. Il leur semble qu’ils n’ont tout simplement pas ce qu’il faut pour devenir développeur et c’est dommage pour vous et moi, car beaucoup d’entre eux adoreraient programmer s’ils ne s’écrasaient pas sur ce mur de briques.

Et cela m'amène à ce que je voulais réellement dire - beaucoup d'entre nous étudient mieux si nous nous concentrons sur l'étude de plusieurs choses à la fois. Nous voulons de nouveaux défis, nous voulons essayer de nouvelles choses, mais nous ne voulons pas nous embrouiller et tomber dans la stupeur. Apprendre les tests et toute autre chose - par exemple, comment créer une application Web, comment fonctionne http, cookie, etc. - est un moyen facile de tomber dans ce piège. Et par conséquent, je recommande généralement d'étudier le reste en premier et de tester en dernier. Vous pouvez toujours revenir à vos anciens projets et voir comment vous les reconstruiriez maintenant, lorsque vous avez étudié les tests, mais cela n'est possible que si vous n'êtes pas surchargé, pas contrarié et, finalement, n'avez pas abandonné cette activité.

Mais que faire si je veux apprendre à tester?! (et un million d'autres «et si»)


Cool en avant! Bien sûr, vous pouvez d'abord étudier les tests, puis seulement le développement Web ou un autre sujet. Je suis sûr que certains l'ont fait et vous aimerez peut-être davantage ainsi. Quand je dis que les tests ne sont pas pour les débutants, je ne dis pas que c'est un mauvais sujet à étudier, je veux dire qu'essayer d'apprendre les tests et tout le reste est une erreur.

La plupart des gens veulent d'abord apprendre à créer des applications Web ou quelque chose de visuel, mais cela ne signifie pas que vous devez commencer par cela, vous pouvez très bien commencer par les tests. Vous ne commencerez probablement à rien comprendre avant d'avoir les bosses, mais ne me laissez pas vous empêcher d'apprendre ce que vous voulez.

Cela ne signifie pas non plus que vous ne pouvez pas étudier pendant la programmation de paires ou quelque chose comme ça. Avec un mentor, vous pouvez apprendre beaucoup de choses en même temps et réussir à supporter cette charge, car vous avez quelqu'un qui vous guidera à travers tout cela. Dans cette situation, si vous vous perdez, vous n'entrerez pas dans une impasse et vous ne vous sentirez pas comme un échec, car il y aura quelqu'un qui dira: «Vous allez bien, c'est tout simplement trop difficile pour l'instant. La prochaine fois, essayez plutôt X et Y! » Autrement dit, dans ce scénario, vous ne serez pas confus, déçu et quittez.
