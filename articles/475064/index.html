.NET Core 3 para escritorio de Windows

En septiembre, lanzamos la compatibilidad con .NET Core para crear aplicaciones de escritorio de Windows, incluidos WPF y Windows Forms. Desde entonces, nos ha encantado ver a tantos desarrolladores compartir sus historias sobre la migración de aplicaciones de escritorio (y bibliotecas de control) a .NET Core. Constantemente escuchamos historias de desarrolladores de escritorio .NET Windows que impulsan su negocio con WPF y Windows Forms, especialmente en escenarios donde el escritorio brilla, incluyendo:

  • Aplicaciones de formularios densos en UI sobre datos (FOD)
  • IU de baja latencia receptiva
  • Aplicaciones que deben ejecutarse sin conexión / desconectadas
  • Aplicaciones con dependencias en controladores de dispositivos personalizados

Este es solo el comienzo para el desarrollo de aplicaciones de Windows en .NET Core. Siga leyendo para obtener más información sobre los beneficios de .NET Core para crear aplicaciones de Windows.



¿Por qué el escritorio de Windows en .NET Core?


.NET Core (y en el futuro .NET 5 que se construye sobre .NET Core) será el futuro de .NET. Estamos comprometidos a admitir .NET Framework en los años venideros, sin embargo, no recibirá nuevas funciones, solo se agregarán a .NET Core (y eventualmente .NET 5). Para mejorar las pilas de escritorio de Windows y permitir que los desarrolladores de escritorio .NET se beneficien de todas las actualizaciones del futuro, llevamos Windows Forms y WPF a .NET Core. Seguirán siendo tecnologías exclusivas de Windows porque existen dependencias estrechamente acopladas a las API de Windows. Pero .NET Core, además de ser multiplataforma, tiene muchas otras características que pueden mejorar las aplicaciones de escritorio.

En primer lugar, todas las mejoras de tiempo de ejecución y las características del lenguaje se agregarán solo a .NET Core y en el futuro a .NET 5. Un buen ejemplo aquí es C # 8 que estuvo disponible en .NET Core 3.0. Además, las versiones .NET Core de Windows Forms y WPF se convertirán en parte de la plataforma .NET 5. Entonces, al portar su aplicación a .NET Core hoy, los está preparando para .NET 5.

Además, .NET Core brinda flexibilidad de implementación para sus aplicaciones con nuevas opciones que no están disponibles en .NET Framework, como:

  • Despliegue de lado a lado. Ahora puede tener múltiples versiones de .NET Core en la misma máquina y puede elegir a qué versión debe apuntar cada una de sus aplicaciones.
  • Despliegue autónomo. Puede implementar la plataforma .NET Core con sus aplicaciones y volverse completamente independiente de su entorno de usuarios finales: su aplicación tiene todo lo que necesita para ejecutarse en cualquier máquina con Windows.
  • Tamaños de aplicación más pequeños. En .NET Core 3 presentamos una nueva característica llamada enlazador (también conocida como trimmer), que analizará su código e incluirá en su implementación autónoma solo aquellos ensamblados de .NET Core que sean necesarios para su aplicación. De esa forma, se recortarán todas las partes de la plataforma que no se usen para su caja.
  • Archivos individuales .exe. Puede empaquetar su aplicación y la plataforma .NET Core en un solo archivo .exe.
  • Rendimiento mejorado en tiempo de ejecución. .NET Core tiene muchas optimizaciones de rendimiento en comparación con .NET Framework. Cuando piensa en el historial de .NET Core, creado inicialmente para cargas de trabajo web y de servidor, ayuda a comprender si su aplicación puede ver beneficios notables de las optimizaciones de tiempo de ejecución. Específicamente, las aplicaciones de escritorio con grandes dependencias en las operaciones de E / S de archivos, redes y bases de datos probablemente verán mejoras en el rendimiento para esos escenarios . Algunas áreas en las que puede que no note muchos cambios son en el rendimiento de representación de la interfaz de usuario o el rendimiento de inicio de la aplicación.

Al establecer las propiedades <PublishSingleFile> , <RuntimeIdentifier> y <PublishTrimmed> en el perfil de publicación, podrá implementar una aplicación autónoma recortada como un único archivo .exe como se muestra en el siguiente ejemplo. 

 <PropertyGroup> <OutputType>Exe</OutputType> <TargetFramework>netcoreapp3.0</TargetFramework> <PublishSingleFile>true</PublishSingleFile> <RuntimeIdentifier>win-x64</RuntimeIdentifier> <PublishTrimmed>true</PublishTrimmed> </PropertyGroup>

Diferencias entre el escritorio de .NET Framework y el escritorio de .NET Core


Al desarrollar aplicaciones de escritorio, no notará mucha diferencia entre las versiones .NET Framework y .NET Core de WPF y Windows Forms. Una parte de nuestro esfuerzo fue proporcionar una paridad funcional entre estas plataformas en el área de escritorio y mejorar la experiencia de .NET Core en el futuro. Las aplicaciones WPF son totalmente compatibles con .NET Core y están listas para su uso, mientras estamos trabajando en actualizaciones y mejoras menores. Para Windows Forms, la parte de tiempo de ejecución está totalmente portada a .NET Core y el equipo está trabajando en el Diseñador de Windows Forms. Estamos planeando tenerlo listo para el cuarto trimestre de 2020 y, por ahora, puede consultar la versión preliminar del diseñador en Visual Studio 16.4 Preview 3 o posterior. No olvide establecer la casilla de verificación en Herramientas-> Opciones-> Características de vista previa-> Usar la vista previa del diseñador de formularios Windows Forms para aplicaciones .NET Core y reinicie Visual Studio. Tenga en cuenta que la experiencia es limitada por ahora ya que el trabajo está en progreso.

Rompiendo cambios


Hay algunos cambios importantes entre .NET Framework y .NET Core, pero la mayoría del código relacionado con Windows Forms y las áreas de WPF se transfirió a Core tal como está. Si estaba utilizando componentes como WCF Client, Code Access Security, App Domains, Interop y Remoting, necesitará refactorizar su código si desea cambiar a .NET Core.

Otra cosa a tener en cuenta: las rutas de salida predeterminadas en .NET Core son diferentes de en .NET Framework, por lo que si tiene algunas suposiciones en su código sobre la estructura de archivos / carpetas de la aplicación en ejecución, entonces probablemente fallará en el tiempo de ejecución.

Además, hay cambios en la forma de configurar las características .NET. .NET Core en lugar del archivo <something>.runtimeconfig.json usa el archivo <something>.runtimeconfig.json que viene con una aplicación y tiene el mismo propósito general e información similar. Algunas configuraciones como system.diagnostics , system.net o system.net no son compatibles, por lo que un archivo de configuración de la aplicación no se cargará si contiene alguna de estas secciones. Este cambio afecta al sistema, el seguimiento de System.Diagnostics y los escenarios de clientes WCF que comúnmente se configuraron utilizando la configuración XML anteriormente. En .NET Core necesitará configurarlos en código. Para cambiar los comportamientos sin volver a compilar, considere configurar el rastreo y los tipos de WCF utilizando valores cargados desde una fuente Microsoft.Extensions.Configuration o desde appSettings .

Puede encontrar más información sobre las diferencias entre .NET Core y .NET Framework en la documentación .

Portado de .NET Framework a .NET Core


En primer lugar, ejecute el Analizador de portabilidad y, si es necesario, actualice su código para obtener una compatibilidad del 100% con .NET Core. Aquí hay instrucciones sobre el uso del analizador de portabilidad . Recomendamos utilizar un control de origen o hacer una copia de seguridad de su código antes de realizar cambios en su aplicación en caso de que la refactorización no se realice de la manera deseada y decida volver a su estado inicial.

Cuando su aplicación es totalmente compatible con .NET Core, está listo para portarla. Como punto de partida, puede probar una herramienta que creamos para ayudar a automatizar la conversión de sus proyectos de .NET Framework a .NET Core - try-convert .

Es importante recordar que esta herramienta es solo un punto de partida en su viaje a .NET Core. Tampoco es un producto de Microsoft compatible. Aunque puede ayudarlo con algunos de los aspectos mecánicos de la migración, no manejará todos los escenarios o tipos de proyectos. Si su solución tiene proyectos que la herramienta rechaza o no puede convertir, deberá realizar el puerto a mano. No se preocupe, tenemos muchos tutoriales sobre cómo hacerlo (al final de esta sección).

La herramienta try-convert intentará migrar sus archivos de proyecto de estilo antiguo al nuevo estilo SDK y volver a diseñar los proyectos aplicables a .NET Core. Para sus bibliotecas, dejamos que haga una llamada con respecto a la plataforma: el clima al que le gustaría apuntar .NET Core o .NET Standard. Puede especificarlo en su archivo de proyecto actualizando el valor de <TargetFramework> . Las bibliotecas sin dependencias específicas de .NET Core como WPF o Windows Forms pueden beneficiarse de la orientación de .NET Standard: 

 <TargetFramework>netstandard2.1</TargetFramework>

para que puedan ser utilizados por las personas que llaman a muchas plataformas .NET diferentes. Por otro lado, si una biblioteca usa una característica que requiere .NET Core (como las API de IU de escritorio de Windows), está bien que la biblioteca apunte a .NET Core: 

 <TargetFramework>netcoreapp3.0</TargetFramework>

try-convert es una herramienta global que puede instalar en su máquina, luego puede llamar desde CLI: 

 C:\> try-convert -p <path to your project>

o 

 C:\> try-convert -w <path to your solution>

Como se mencionó anteriormente, si la herramienta try-convert no funcionó para usted, aquí hay materiales sobre cómo portar su aplicación a mano.
