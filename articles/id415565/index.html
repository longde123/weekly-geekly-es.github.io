Mengapa (hari ini) kembali 444 tidak selalu berguna

Server web Nginx memiliki kode respons 444 yang luar biasa yang "menutup" koneksi tanpa mengirim data. Fungsionalitas ini sangat berguna dalam memfilter traffic palsu - jika kami yakin bahwa klien tidak valid dengan beberapa kriteria, maka tidak perlu memberi tahu dia, misalnya, dengan jawaban ke-403. Lebih efisien untuk menghentikan transfer data, yang, seringkali, dapat secara signifikan mengurangi beban pada server.

Rekomendasi untuk menggunakan jawaban semacam itu dapat ditemukan di mana-mana dalam instruksi untuk memblokir klik pada tautan dari situs populer dan spam rujukan, perlindungan terhadap DDoS, dll.

Dan, secara umum, selama bertahun-tahun, tips ini dapat digunakan hampir tanpa melihat, tetapi ... browser modern tidak tinggal diam dan secara berkala menghadirkan kejutan baru bagi kita.

Server: 

$ uname -orm FreeBSD 11.1-STABLE amd64 

 $ nginx -v nginx version: nginx/1.15.0

Pelanggan: 

 >ver Microsoft Windows [Version 10.0.15063]

Saya hanya menggunakan browser Chrome dan Firefox. Bukan karena sisanya tidak berdosa, hanya perilaku yang dijelaskan di bawah ini cukup stabil pada keduanya.

Jadi, kami memiliki situs web dan kami ingin membatasi akses ke lokasi tanpa persyaratan apa pun dan tanpa mengirimkan data apa pun: 

 server { ... location = /code/444 { return 444; } ... }

Menunggu:

  • browser akan mengirim satu permintaan;
  • server akan menerima satu permintaan, menutup koneksi dan menulis satu baris dengan kode 444 ke log;
  • tidak menerima data, browser akan menampilkan kesalahan dan berhenti berkomunikasi dengan server.

Judul spoiler


Dengan demikian, mengharapkan untuk menerima satu permintaan dari klien, pada kenyataannya, kami mendapatkan 3-20 permintaan. Selain panggilan yang sama sekali tidak perlu bagi kami, ada risiko memasukkan log seperti itu, misalnya, ke skrip anti-DDoS dan memblokir ip ini. Dan itu kelihatannya setengah dari masalah, karena kita sudah "memberi" 444 kepadanya, yaitu, kita tidak merasa kasihan padanya, tetapi itu bisa terjadi sebaliknya - kita akan menunjukkan kepada klien apa yang tidak kita rencanakan sama sekali.

Ubah konfigurasi: 

 map $http_referer $code_if { "~*https://habr.com/post/415565/" 1; "http://tison.ru/ref" 1; default 0; } ... server { ... location = /code/444 { return 444; } ... location = /codeif/444 { if ( $code_if = 1 ) { return 444; } add_header "Content-Type" "text/html; charset=UTF-8" always; return 200 "Expected code 444"; } ... }

Di sini kami telah melarang pengiriman konten ke semua klien dengan perujuk "
https://habr.com/post/415565/ "saat beralih ke halaman tison.ru/codeif/444
