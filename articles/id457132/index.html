Substitusi impor dalam praktek. Bagian 3.1. "QP OS". Ulasan pertama dari jenisnya tentang sistem operasi


Sampai saat itu, tidak ada ulasan OS ini. Penulis dan pengembang menyimpan rahasia mereka, dan tidak ingin semua pekerjaan mereka mengalir ke jaringan dan menjadi publik. Dalam hal ini, pengujian OS QP hanya dimungkinkan berdasarkan kesepakatan, dan, pada saat ini, hanya untuk badan hukum. Saya diberi kesempatan untuk menyentuh OS ini, dan menulis artikel tentang itu.

Siapa pun yang tertarik dalam pengembangan perangkat lunak domestik - tolong, di bawah kucing.

Sebelumnya, saya menarik perhatian pada fakta bahwa file yang dapat dieksekusi di QP OS, seperti pada Windows, memiliki ekstensi .exe, sementara pengembang mengklaim bahwa mereka memiliki format sendiri untuk file yang dapat dieksekusi.

Komentar Pengembang:
File yang dapat dieksekusi di QP OS ada dalam 2 format: CMF (ekstensi .qpx, .qps, .qpl) - format kami untuk file yang dapat dieksekusi dan PE (ekstensi .exe, .dll) - format file yang dapat dieksekusi Microsoft. Struktur file CMF dapat dilihat dengan memilih file yang sesuai di editor konten file manajer file dengan menekan F4. Dan pastikan bahwa format internal dari file yang dapat dieksekusi ini eksklusif. Format file PE harus didukung untuk mengatur peluncuran aplikasi untuk platform .Net.

7. QP VMM


Hypervisor Dia sedang bekerja. Apa yang ada di bawah tenda tidak diketahui. Tapi ini jelas merupakan hypervisor tipe 2. Meskipun:



Itu diinstal bersama dengan OS, dan dari bawah itu dikendalikan menggunakan aplikasi terpisah. Dari fitur (dari situs pengembang):

  • mengimplementasikan platform perangkat keras IA32 dan x64;
  • menggunakan virtualisasi perangkat keras VTX, VTD (menurut pengembang, hypervisor meneruskan perangkat periferal langsung ke VM, dan memberikan kontrol ke driver di dalam OS tamu);
  • mendukung multiprosesing;
  • menyediakan hibernasi independen mesin virtual;
  • menurut perwakilan pengembang, ini mendukung migrasi dinamis dari mesin virtual (pindah dari satu host ke yang lain tanpa menghentikan mesin virtual)

Lebih detail di sini

Dinilai oleh laporan V.Yu. Egorova, Cryptosoft berencana untuk mengimplementasikan pengelompokan hypervisor yang toleran terhadap kesalahan, dan pengembangan QP VMM bergerak ke arah ini.

Itu diperlakukan dengan entri manual di bidang pemilihan jalur "C: \". Trivia, tapi tidak menyenangkan.

Ada beberapa parameter saat membuat mesin virtual baru. Kami dengan cepat membuat, melampirkan gambar instalasi Windows 7 dan mulai.



Saya memberi VM semua 4 core. Secara sadar.



Dan ... Kami melihat sisi kanan layar ... Apa-apaan ini ?! Boot sistem 100% ?! Apauuuuu ?! Manajer tugas! Saya menantang Anda!



Itu tidak menjadi lebih jelas ... Mode lanjutan!



Yaitu, tuan rumah memberikan semua sumber daya yang dipilih ke OS tamu, dan menunjukkan kepada saya bahwa itu dilakukan dengan baik, dan bahwa beban CPU adalah 100%. Tuan rumah itu sendiri tidak ketinggalan, atau tidak diperhatikan.



Yah, dia mengalokasikan memori tidak secara dinamis, tetapi ketika mengatur VM tidak ada pilihan alokasi memori dinamis. Masuk akal di sini. Tapi apa yang terjadi pada CPU, saya masih tidak mengerti. Tetapi mengapa dalam kasus ini tuan rumah menunjukkan pemanfaatan CPU pada level 100% tidak jelas. Artinya, jelas tidak mungkin untuk melakukan analisis dan mengidentifikasi hambatan dalam produktivitas. Pada prinsipnya, ini tidak begitu penting untuk penggunaan di rumah, tetapi jika Anda berpikir lebih jauh dan mempertimbangkan operasi industri, itu tidak mungkin.

8. Kesimpulan


Basah. Itu kosong. Mengerikan tentang apa pun. Ini adalah kata-kata pertama yang muncul di pikiran saya.

Saya masih tidak mengerti situasinya dengan file .exe. Mereka hadir di OS, mereka berlari di dalamnya. Tetapi tidak satu pun dari file yang dapat dieksekusi yang ditujukan untuk Windows memulai atau melaporkan kesalahan. Pengembang mengatakan bahwa mereka memiliki format file yang dapat dieksekusi sendiri. Baiklah Lalu mengapa mereka memiliki .exe dalam pekerjaan? Ini menciptakan kebingungan liar di kepalaku. Selain itu, tidak ada perangkat lunak untuk OS Anda ini. Tidak semuanya. Tidak mungkin untuk bekerja dengan perangkat lunak yang dibangun di dalamnya dari kata sama sekali. Aplikasi di dalamnya adalah non-fungsional, keju, tidak lengkap, menakutkan persegi.

OS - basis di mana aplikasi pihak ketiga kacau, saya tidak bisa membayangkan bagaimana cara mengacaukan server 1C yang sama pada QP OS; Untuk melakukan ini, pengembang 1C ingin menulis ulang kodenya untuk OS baru ini. Entah bagi pengembang QP OS untuk menulis ulang kode 1C (ha ha), atau mengubah OS-nya sehingga 1C dapat bekerja di bawahnya dalam bentuk saat ini. Namun, ternyata, tidak satu pun dari peristiwa ini yang akan terjadi. Sejak 1C tidak masuk akal untuk menulis ulang kode untuk OS baru, yang belum diimplementasikan di mana saja dalam jumlah yang cukup, dan repot-repot dengan porting platform hanya tidak layak. Demi 100-1000 orang? Tidak serius. Dan akan menjadi masalah untuk mengimplementasikan QP OS pada sejumlah besar mesin, karena orang normal tidak akan berhenti total dalam pekerjaan perusahaan mereka sampai perangkat lunak ditulis ulang untuk OS yang dibeli.

Saya melihat di sini hanya satu jalan keluar bagi pengembang QP OS - untuk mengubah strategi dan menggunakan perangkat lunak yang ada untuk produk saya. Kalau tidak, tidak akan ada yang datang darinya. Kecuali, tentu saja, mereka memiliki perjanjian dengan eselon kekuasaan tertinggi pada implementasi OS mereka dengan kekuatan pesanan dari atas. Tapi ini semua ...

Anda menulis ulang tumpukan protokol jaringan, seperti yang Anda katakan. Wow! Anda yakin bahwa koneksi jaringan Anda sepenuhnya aman. Ini merupakan nilai tambah. Tetapi pengaturan jaringan Anda melalui DHCP tidak diterima. Server DNS harus terdaftar secara manual. Nyaman Ini adalah minus. Jika kami mempertimbangkan opsi untuk menggunakan PC khusus yang menjalankan OS Anda, maka umumnya akan menjadi bencana jika Anda memiliki kusen dengan jaringan atau pengaturan telah berubah. Penting untuk mengubah pengaturan untuk semua orang dan semua orang. Saya pikir ini bisa diselesaikan di masa depan, tetapi sejauh ini belum dilaksanakan.

OS hanya bekerja dengan hypervisor tertentu. Ini juga minus besar. Dengan semua keinginan, itu tidak bisa disebut "universal" dengan pendekatan ini. Ya, mungkin suatu hari nanti, itu akan bekerja dengan Hyper-V dan VirtualBox, tetapi apa yang akan terjadi dan kapan itu tidak diketahui. Pada tahap ini, OS hanya berfungsi dengan VMware, dan ini juga minus yang sangat besar.

Saya tidak melihat di mana Anda dapat menggunakan OS QP. Sebagai firewall? Gerbang? Untuk melakukan ini, Anda dapat membeli perangkat keras yang lengkap, lebih dapat diandalkan dan lebih mudah disesuaikan. Sebagai OS untuk PC pengguna, dalam kondisi saat ini ia sama sekali tidak cocok. Ketidakmampuan untuk menggunakan aplikasi Office dan browser membuatnya sama sekali tidak berguna. Artinya, bahkan tidak bisa dalam aplikasi web 1C karena kurangnya dukungan untuk teknologi modern di browser. Ini semua minus besar lainnya.

Antarmuka pengguna masih diselesaikan dan diselesaikan. Ada terobosan dalam pekerjaan. Saya tidak berbicara tentang bug visual, saya berbicara tentang fungsionalitas dan kepenuhan. Anda telah mengambil tombol "perkecil semua jendela". Dia melipatnya. Mengapa Anda tidak bisa membuatnya sehingga ketika Anda menekannya lagi, itu menyebarkannya kembali? Ya, Anda jelas-jelas mengambilnya dari Windows. Mekanisme semacam itu diterapkan di sana. Minus lainnya.

Keuntungan utama OS adalah dilindungi. Pengembang itu 100% yakin bahwa tidak ada "bookmark" di dalamnya yang semua orang sangat takuti. Dia aman. Menurut pengembang, beberapa modulnya ditulis ulang ...

Di sini saya hanya punya satu pertanyaan - dan modul-modul yang tidak "ditulis ulang", dari mana asalnya? .. Dari Windows?

Hypervisor tidak memenuhi harapan bahkan dengan 10 persen. Tidak mungkin mengoperasikannya di lingkungan produk karena kinerja. Ya, ia telah mengumumkan peluang yang menyenangkan, tetapi itu tidak diekspor dengan latar belakang kekurangan dan kekurangan yang mengerikan.

TOTAL:

Jika Anda menyangkal fakta bahwa saat ini OS tidak kompetitif, keju dan belum selesai ... hmm ... bukan itu intinya. Singkatnya, jika Anda berpikir tentang perspektif kecil, maka OS mungkin setara dengan pesaing. Karena fakta bahwa itu dibuat lagi, dan dibuat bersama kami. Tetapi sampai saat ini, pengembang akan memiliki banyak pekerjaan yang harus dilakukan. Sangat, sangat banyak. Pangkalan itu diletakkan tidak buruk, tetap hanya untuk memecahkan masalah dengan pengisian, perangkat lunak, fungsi dan peralatan.

9. Opini


Pengembangan OS di awal. Saya tidak tahu apa yang diambil oleh para pengembang sebagai dasar OS QP mereka, tetapi bagi saya sepertinya itu Windows. Bahkan jika mereka hanya terinspirasi oleh sistem yang dibuat oleh Bill Gates, dan pinjaman itu bukan salinan kasar, tetapi kemiripan, itu masih sangat dekat dengan versi Windows yang sangat awal. Tombol pintas, nama layanan dan aplikasi, penampilan. Tapi ini, menurut saya, juga merupakan nilai tambah yang pasti dari sistem, ambang untuk memasukkannya tidak akan sebesar seperti ketika beralih dari Windows ke Linux.

Komentar Pengembang:
Dari Windows kami mengambil antarmuka agar tidak menemukan kembali roda, untuk mengatur lingkungan yang akrab bagi pengguna. Kami tidak menggunakan binari Windows, karena API tingkat rendah tidak kompatibel dengan keluarga OS Windows.

Ada beberapa video di Internet yang didedikasikan untuk QP OS. Salah satunya adalah laporan oleh V.Yu. Egorova. Pada awalnya, dia mengatakan yang berikut: “Kami, tidak seperti yang lain, tidak memiliki sistem waktu nyata, kami adalah sistem tujuan umum, dan kami menulis sistem ini untuk tawon dari awal di Rusia di Penza ... Tidak ada hubungannya dengan OS Linux »Dan tidak sepatah kata pun tentang sikap terhadap Windows. Juga V.Yu. Egorov berfokus pada kemampuan bertahan sistem, dan bukan pada keandalan.

Secara umum, konsep OS terlihat rapi dan jika Anda berpikir sedikit, Anda dapat berharap bahwa pengembang akan menyelesaikan antarmuka dan bahkan dapat digunakan secara normal. Saya suka penekanan pada antarmuka grafis dari segala sesuatu yang mungkin. Ya, lawan X akan menghukum saya, tetapi saya jauh lebih senang melihat keseluruhan gambar di jendela terpisah daripada menggali ke dalam kiloton baris teks untuk mengkonfigurasi beberapa variabel. Antarmuka FireWall membuat saya senang. Jika GUI dari semua aplikasi sistem sama menyenangkan dan informatifnya, itu akan bagus.

Tidak perlu berbicara tentang penggunaan OS dalam kondisi saat ini di lingkungan produk. Tidak mungkin untuk meningkatkan layanan utama di dalamnya. Dan apa yang dapat Anda tingkatkan tidak sebanding dengan masalahnya, karena lebih mudah untuk melakukan ini pada distribusi Linux atau Windows yang sudah dikenal. Untuk mulai menggunakan QP OS, Anda perlu menulis ulang sebagian besar perangkat lunak yang ada untuk itu, dan di sini kita mendapatkan lingkaran setan. Dalam kesimpulan, saya sudah membicarakan hal ini. Situs ini memiliki skema yang berisi "di mana Anda dapat menerapkan QP OS", tetapi ...

Singkatnya, ini dia:



Saya akan membuat reservasi tentang gambar:

  • Naikkan server terminal tanpa otorisasi melalui layanan direktori - kerja monyet dan neraka untuk administrator.
  • Server mesin virtual dengan apa yang saat ini tersedia bukanlah solusi produk.
  • Server email tanpa dukungan otorisasi melalui layanan direktori - lihat paragraf tentang server terminal.
  • Server file tanpa dukungan layanan direktori - lihat paragraf tentang server terminal.
  • Klien Tipis? Mungkin saya tidak melihat implementasi dari solusi ini.
  • Firewall? Ya, tetapi ada solusi besi yang jauh lebih produktif dan lebih fungsional. Niche ini sudah lama ditempati.
  • Workstation tanpa dukungan otorisasi melalui layanan direktori ... tanpa kebijakan grup ... Ini adalah neraka nyata bagi administrator sistem.

Intinya, saya tidak dapat memverifikasi stabilitas sistem. Bagaimana cara memeriksa sistem yang tidak mungkin menjalankan uji waktu yang telah teruji? Suatu kali, dia meninggal dengan BSOD (dalam kasus QP OS itu RSOD, karena layar kematian mereka merah), yang disebabkan oleh kesalahan terkait dengan driver video (menurut pengembang). Itu di bawah Hyper-V, jadi ...

Rencana pengembangan QP VMM menyenangkan saya. Jika Cryptosoft mengimplementasikan ide-idenya ke arah pengelompokan, itu akan setidaknya menarik. Benar, sampai mereka menyelesaikan kinerja VMM mereka, dan mendukung berbagai perangkat keras, itu tidak akan sangat produktif, tetapi tetap saja. Saya ingin percaya bahwa para pria mengetahui pekerjaan mereka, dan mereka akan berhasil.
