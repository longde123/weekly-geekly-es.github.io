Lombok, sources.jar y depuración conveniente

En nuestro equipo, realmente amamos a Lombok . Le permite escribir menos código y refactorizar menos, lo cual es ideal para desarrolladores perezosos. Pero si, además del artefacto del proyecto, publica el código fuente con la documentación también, entonces puede encontrarse con un problema: el código fuente no coincidirá con el código de bytes. Hablaré sobre cómo resolvimos este problema y qué dificultades encontramos en el proceso en esta publicación.



Por cierto, si escribe en Java y, por alguna razón, todavía no utiliza Lombok en su proyecto, le recomiendo que lea los artículos sobre Habré ( uno y dos ). Estoy seguro de que te gustará!

El problema


El proyecto en el que estamos trabajando consta de varios módulos. Algunos de ellos (llamémoslos condicionalmente backend) cuando se lanza la versión se empaquetan en el archivo (entrega), se cargan en el repositorio y posteriormente se implementan en los servidores de aplicaciones. La otra parte es la llamada Módulo cliente: publicado en el repositorio como un conjunto de artefactos, incluidos sources.jar y javadoc.jar. Usamos Lombok en todas partes, y Maven lo hará todo.

Hace algún tiempo, uno de los consumidores de nuestro servicio abordó un problema: intentó depurar nuestro módulo, pero no pudo hacerlo, porque sources.jar carecía de métodos (e incluso clases) en los que le gustaría establecer un punto de interrupción. ¡En nuestro equipo creemos que un intento de identificar y resolver un problema de forma independiente, en lugar de hacer un defecto sin pensar, es un acto de un marido digno que debe ser alentado! :-) Por lo tanto, se decidió traer sources.jar de acuerdo con el código de bytes.

Ejemplo


Imaginemos que tenemos una aplicación simple que consta de dos clases:

SomePojo.java

Main.java 
 package com.github.monosoul.lombok.sourcesjar; import lombok.val; public final class Main { public static void main(String[] args) { if (args.length != 2) { throw new IllegalArgumentException("Wrong arguments!"); } val pojo = SomePojo.builder() .someStringField(args[0]) .anotherStringField(args[1]) .build(); System.out.println(pojo); } }


Y nuestra aplicación está construida con Maven:

Bastante diferente de lo que entra en nuestras fuentes .jar, ¿no? ;) Como puede ver, si conectó el código fuente para la depuración de SomePojo y quería establecer un punto de interrupción, por ejemplo, en el constructor, entonces se encontraría con un problema: no hay ningún lugar para establecer el punto de interrupción, y la clase SomePojoBuilder no está allí en absoluto.

¿Qué hacer al respecto?


Como suele suceder, este problema tiene varias soluciones. Veamos cada uno de ellos.

No uses Lombok


Cuando encontramos este problema por primera vez, estábamos hablando de un módulo en el que solo había un par de clases que usaban Lombok. Por supuesto, no quería rechazarlo, así que inmediatamente pensé en hacer delombok. Después de investigar esta pregunta, encontré algunas soluciones extrañas con el complemento Lombok de Maven : lombok-maven-plugin . En uno de ellos se sugirió, por ejemplo, mantener las fuentes en las que se utiliza Lombok en un directorio separado para el que se ejecutará delombok, y las fuentes generadas irán a las fuentes generadas, desde donde se compilará y se dirigirá a sources.jar. Esta opción probablemente esté funcionando, pero en este caso el resaltado de sintaxis en las fuentes originales no funcionará en el IDE, ya que un directorio con ellos no se considerará un directorio fuente. Esta opción no me convenía, y dado que el costo de abandonar Lombok en este módulo era bajo, se decidió no perder tiempo en esto, deshabilitar Lombok y simplemente generar los métodos necesarios a través del IDE.

En general, me parece que esta opción tiene derecho a la vida, pero solo si las clases que usan Lombok son realmente pequeñas y rara vez cambian.


Delombok plugin + sources.jar build usando Ant


Después de un tiempo, tuve que volver a este problema nuevamente, cuando ya se trataba de un módulo en el que Lombok se usaba mucho más intensamente. Volviendo nuevamente al estudio de este problema, me encontré con una pregunta sobre stackoverflow , que sugería ejecutar delombok para las fuentes y luego usar fuentes en Ant generate sources.jar.
Aquí debe divagar acerca de por qué necesita generar sources.jar usando Ant, en lugar de usar el complemento Source ( maven-source-plugin ). El hecho es que no puede configurar el directorio de origen para este complemento. Siempre usará el contenido de la propiedad sourceDirectory del proyecto.

Entonces, en el caso de nuestro ejemplo, pom.xml se verá así:

Como puede ver, la configuración ha crecido mucho, y está lejos de ser solo lombok-maven-plugin y maven-antrun-plugin . ¿Por qué sucedió esto? El hecho es que, dado que ahora recopilamos sources.jar con Ant, Maven no sabe nada sobre este artefacto. Y tenemos que decirle claramente cómo instalar este artefacto, cómo implementarlo y cómo empaquetar recursos en él.

Además, descubrí que al ejecutar delombok por defecto, Lombok agrega un comentario al encabezado de los archivos generados. En este caso, el formato de los archivos generados se controla no usando las opciones en el archivo lombok.config , sino usando las opciones del complemento. La lista de estas opciones no fue fácil de encontrar. Era posible, por supuesto, llamar al apodo jar de Lombok con las teclas delombok y --help , pero soy demasiado vago para este programador, así que los encontré en los códigos fuente en el github .

Pero ni el volumen de la configuración ni sus características se pueden comparar con el principal inconveniente de este método. El no resuelve el problema . Bytecode se compila a partir de una fuente, mientras que otras ingresan en sources.jar. Y a pesar del hecho de que delombok es ejecutado por el mismo Lombok, todavía habrá diferencias entre el código de bytes y las fuentes generadas, es decir. Todavía no son adecuados para la depuración. Para decirlo suavemente, me molesté cuando me di cuenta de esto.


Complemento Delombok + perfil en maven


Entonces que hacer? Tenía sources.jar con las fuentes "correctas", pero aún eran diferentes del bytecode. En principio, el problema podría resolverse compilando a partir de códigos fuente generados como resultado de delombok. Pero el problema es que maven-compiler-plugin 'no puede especificar la ruta a la fuente. Siempre usa las fuentes especificadas en el sourceDirectory proyecto, como el maven-source-plugin . Podríamos indicar el directorio donde se generan las fuentes no combinadas, pero en este caso, al importar un proyecto al IDE, el directorio con fuentes reales no se considerará como tal y el resaltado de sintaxis y otras características no funcionarán para los archivos que contiene. Esta opción tampoco me convenía.

¡Puedes usar perfiles! ¡Cree un perfil que se usaría solo al construir el proyecto y en el que se reemplazaría el valor de sourceDirectory ! Pero hay un matiz. La etiqueta sourceDirectory solo se puede declarar dentro de la etiqueta de build en la raíz del proyecto.

Afortunadamente, hay una solución para este problema. ¡Puede declarar una propiedad que se sustituirá en la etiqueta sourceDirectory y cambiar el valor de esta propiedad en el perfil!

En este caso, la configuración del proyecto se verá así:

pom.xml 
 <?xml version="1.0" encoding="UTF-8"?> <project xmlns="http://maven.apache.org/POM/4.0.0" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/xsd/maven-4.0.0.xsd"> <modelVersion>4.0.0</modelVersion> <artifactId>lombok-sourcesjar</artifactId> <groupId>com.github.monosoul</groupId> <version>1.0.0</version> <packaging>jar</packaging> <properties> <maven.compiler.source>1.8</maven.compiler.source> <maven.compiler.target>1.8</maven.compiler.target> <lombok.version>1.18.2</lombok.version> <origSourceDir>${project.basedir}/src/main/java</origSourceDir> <sourceDir>${origSourceDir}</sourceDir> <delombokedSourceDir>${project.build.directory}/delombok</delombokedSourceDir> </properties> <dependencies> <dependency> <groupId>org.projectlombok</groupId> <artifactId>lombok</artifactId> <version>${lombok.version}</version> <scope>provided</scope> </dependency> </dependencies> <profiles> <profile> <id>build</id> <properties> <sourceDir>${delombokedSourceDir}</sourceDir> </properties> </profile> </profiles> <build> <sourceDirectory>${sourceDir}</sourceDirectory> <plugins> <plugin> <addOutputDirectory>false</addOutputDirectory> <encoding>UTF-8</encoding> <formatPreferences> <generateDelombokComment>skip</generateDelombokComment> <javaLangAsFQN>skip</javaLangAsFQN> </build> </project>


Funciona de la siguiente manera. En la propiedad origSourceDir sustituimos la ruta al directorio con las fuentes originales, y en la propiedad sourceDir sustituimos el valor de origSourceDir . En la propiedad delombokedSourceDir especificamos la ruta a las fuentes generadas por delombok. Por lo tanto, al importar un proyecto en el IDE, origSourceDir el directorio de origSourceDir , y al origSourceDir el proyecto, si especifica el perfil de build , se delombokedSourceDir directorio delombokedSourceDir .

Como resultado, obtenemos un código de bytes compilado de las mismas fuentes que caerá en sources.jar, es decir. la depuración finalmente funcionará. En este caso, no necesitamos configurar la instalación y la implementación del artefacto con las fuentes, ya que usamos el maven-source-plugin para generar el artefacto. Es cierto que la magia con variables puede confundir a una persona que no está familiarizada con los matices de Maven.

También puede agregar la opción lombok.addJavaxGeneratedAnnotation = true en lombok.config , luego la anotación @javax.annotation.Generated("lombok") ubicará por encima del código generado en las fuentes generadas, lo que ayudará a evitar preguntas como "¿Por qué su código se ve tan extraño? ! " :)


Utilizar gradle


Creo que si ya estás familiarizado con Gradle , entonces no deberías explicar todas sus ventajas sobre Maven. Si aún no está familiarizado con él, entonces hay un centro completo en el centro ¡Una gran razón para investigarlo! :)
En general, cuando pensé en usar Gradle, esperaba que fuera mucho más fácil hacer lo que necesitaba en él, porque sabía que en él podía señalar fácilmente de qué construir sources.jar y qué compilar a bytecode. El problema me estaba esperando donde menos esperaba: no hay un complemento delombok que funcione para Gradle.

Existe este complemento , pero parece que no puede especificar opciones para formatear fuentes no combinadas en él, lo que no me convenía.

Hay otro complemento , genera un archivo de texto a partir de las opciones que se le pasan y luego lo pasa como argumento a lombok.jar. No pude lograr que coloque las fuentes generadas en el directorio correcto, parece que esto se debe al hecho de que la ruta en el archivo de texto con los argumentos no se cita y no se escapa correctamente. Tal vez más tarde haré una solicitud de extracción al autor del complemento con una sugerencia para una solución.

Al final, decidí ir hacia otro lado y acabo de describir una tarea con Ant llamando a ejecutar delombok, en Lombok solo hay una tarea Ant para esto , y se ve bastante bien:

build.gradle.kts 
 group = "com.github.monosoul" version = "1.0.0" plugins { java } java { sourceCompatibility = JavaVersion.VERSION_1_8 targetCompatibility = JavaVersion.VERSION_1_8 } dependencies { val lombokDependency = "org.projectlombok:lombok:1.18.2" annotationProcessor(lombokDependency) compileOnly(lombokDependency) } repositories { mavenCentral() } tasks { "jar"(Jar::class) { manifest { attributes( Pair("Main-Class", "com.github.monosoul.lombok.sourcesjar.Main") ) } } val delombok by creating { group = "lombok" val delombokTarget by extra { File(buildDir, "delomboked") } doLast({ ant.withGroovyBuilder { "taskdef"( "name" to "delombok", "classname" to "lombok.delombok.ant.Tasks\$Delombok", "classpath" to sourceSets.main.get().compileClasspath.asPath) "mkdir"("dir" to delombokTarget) "delombok"( "verbose" to false, "encoding" to "UTF-8", "to" to delombokTarget, "from" to sourceSets.main.get().java.srcDirs.first().absolutePath ) { "format"("value" to "generateDelombokComment:skip") "format"("value" to "generated:generate") "format"("value" to "javaLangAsFQN:skip") } } }) } register<Jar>("sourcesJar") { dependsOn(delombok) val delombokTarget: File by delombok.extra from(delombokTarget) archiveClassifier.set("sources") } withType(JavaCompile::class) { dependsOn(delombok) val delombokTarget: File by delombok.extra source = fileTree(delombokTarget) } }
