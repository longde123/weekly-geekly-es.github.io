Poca conveniencia en la vida estudiantil.

Una historia simple sobre cómo me daba vergüenza preguntar constantemente a los compañeros por información faltante y decidí hacer nuestra vida un poco más fácil.



Creo que muchos de mis compañeros están familiarizados con la situación cuando en una sala de chat general, donde a menudo se muestra información importante, hay unos 30 interlocutores activos que constantemente cargan las bases de datos Vkontakte con sus mensajes. En tales condiciones, es poco probable que todos vean esta información importante. A mí también me pasa. Hace un año, se decidió corregir este malentendido.


Pregunto a los que están listos para no indignarse con el próximo artículo sobre el bot.


Como soy un estudiante de primer nivel, los ejemplos estarán relacionados con este tema.
Entonces, hay una tarea: hacer que la transferencia de información del director a los estudiantes sea conveniente tanto para el director como para los estudiantes. Gracias a las características relativamente nuevas de Vkontakte (es decir, mensajes personales de las comunidades), la decisión me llamó la atención de inmediato. Un bot sentado en un grupo debe recibir mensajes del anciano (ancianos si hay muchos grupos en la transmisión) y enviarlos a las partes interesadas (estudiantes).


La tarea está establecida, proceda.


Necesitaremos:


  1. vk_api biblioteca para usar vk api
  2. orbe orm para trabajar con la base de datos
  3. y módulos integrados de python

Además, antes de leer, propongo actualizar la memoria de los patrones "Observador" ( Habr , Wiki ) y "Fachada" ( Habr , Wiki )

Parte 3. "¿Qué escribiste sobre el mío ...?"


El reconocimiento de comandos es manejado por un subsistema separado implementado a través del patrón Observador.
Atención CommandObserver: 


 class CommandObserver(AbstractObserver): def execute(self, member, message, attachments, group, **kwargs): for command in self.commands: for trigger in command.triggers: body = command.get_body(trigger, message) if body is not None: group.api.messages.setActivity(user_id=member.id, type="typing") if command.system: kwargs.update({"trigger": trigger, "commands": self.commands}) else: kwargs.update({"trigger": trigger}) return command.proceed(member, body, attachments, group, **kwargs)

Nuevamente, el renderizado está hecho para una posible expansión futura.

Pero, ¿qué reconocerá este observador?
Así que llegamos a la parte más interesante: el equipo.
Cada equipo es una clase independiente, un descendiente de la clase de Comando base.
Todo lo que se requiere del comando es ejecutar el método proceder () si su palabra clave se encuentra al comienzo del mensaje del usuario. Las palabras clave de comando se definen en la variable desencadenantes de la clase de comando (cadena o lista de cadenas) 


 class Command(metaclass=ABCMeta): def __init__(self): self.triggers = [] self.description = "Empty description." self.system = False self.privilege = False self.activate_times = [] self.activate_days = set() self.autostart_func = self.proceed def proceed(self, member, message, attachments, group, **kwargs): raise NotImplementedError() @staticmethod def get_body(kw, message): if not isinstance(kw, list): kw = [kw, ] for i in kw: reg = '^ *(\\{}) *'.format(i) if re.search(reg, message): return re.sub(reg, '', message).strip(' ')

Como puede ver en la firma del método Proceder (), cada comando recibe un enlace a una instancia de un miembro del grupo, su mensaje (sin una palabra clave), aplicaciones y un enlace a una instancia de grupo. Es decir, toda interacción con un miembro del grupo descansa en el equipo. Creo que esta es la solución más correcta, ya que de esta manera es posible crear un shell (Shell) para una mayor interactividad.
(En verdad, para hacer esto, necesitará agregar asíncrono, porque el procesamiento es sincrónico, o cada mensaje recibido debe procesarse en un nuevo hilo, que no es rentable)
