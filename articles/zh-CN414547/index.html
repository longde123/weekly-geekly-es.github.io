虚拟机即时恢复的基础架构规划即时VM恢复：第2部分

我们继续该主题，该主题已在第一部分开始考虑。 今天，我们将讨论网络连接和目标服务器，介绍可能的选项和基础架构计划选项，以实现Instant VM Recovery的最佳恢复。 所以欢迎猫。


关于网络连接


当然，最好有一个带宽为10 Gbit / s的通道，在备份过程中通过该通道传输数据。 但是，从备份还原通道的方式较为适中，但建议将NIC分组与LACP或SMB多通道一起使用，或者将其他选项与带宽聚合一起使用。 例如，您可以使用4x1 Gbit / s版本的LOM端口。 对于“多个源设备-1个目标备用设备”连接，即连接“多对一”时，建议使用此配置。 （同样，从一个备份存储到目标设备的并行恢复（通常是与执行备份的初始设备相同）是一对多连接。）
例如，您可以从多个Hyper-V / LUN主机配置多个备份作业，并将备份保存到同一目标存储。 如果您有10个这样的主机，总通道带宽为4x1 Gbit / s，那么如果目标设备上有10Gbit / s的管道，则这是一个足够的配置。

在备份存储为SMB共享的情况下，多通道可以很好地工作（如果您配置了具有RDMA支持的NIC，则可以通过SMB Direct进行补充）。 现在，许多Hyper-V群集部署都支持这些功能。 但是，仅当使用存储在SMB文件共享上的VM进行备份的情况下，负责数据移动器的Veeam解决方案组件才可以使用SMB Multichannel和SMB Direct（同样，具有RDMA支持的已配置NIC）脱离主机代理。 这些Veeam数据移动器分别在脱离主机备份代理和存储库上工作。 这里将详细描述这种情况。

另一个重要点：在独立于交换机的模式下使用Windows NIC分组时，允许所有参与者进行数据传输，并且仅从一个参与者接收数据。 如果要在一个过程中获得双向最佳吞吐量，则无需使用LACP。 但是在这种情况下，您需要确保对同一主机进行多个还原。

如您所见，带宽聚合具有许多限制，并且与拥有一个好的信道并不完全相同。 无论如何，您都需要基于计划的使用方案。

总结：根据您的基础架构，您可以在LACP或“独立于交换机”模式/“ SMB多通道”模式下使用Windows NIC分组。 如果您正在使用SMB文件共享并且想要使用SMB Direct（请不要忘记上面提到的工作的功能），则后一个选项很有用。



为了在虚拟磁盘安装期间，即时VM恢复期间访问和复制数据时提供最佳性能，需要高带宽和低延迟。

您可以同时执行多个恢复操作，但仍不能停止备份作业。 也就是说，同样，在存在体面的通道的情况下，计算资源和存储将扮演主要角色。 如果所有这些都正确地设计用于备份，那么恢复将是有效的。

针对目标设备的建议


考虑几种选择，其中很可能您会为自己选择最好的。

选项1：恢复到Hyper-V主机并直接恢复到生产基础结构中的LUN


即使您具有配置了读/写缓存的高性能存储系统或配置了1级，那么，如前一篇文章中所述 ，您也需要注意不要溢出。 否则，生产虚拟机将受到影响。 例如，如果您尝试尽快将大量数据写入存储系统，则会发生这种情况-在迁移存储系统时会发生这种情况。 在此类操作中，我们尝试避免使用1级存储系统。 类似的注意事项适用于大型VM的还原。

您可以建议恢复到具有不同配置文件的单独LUN。 然后，可以将还原的VM缓慢迁移到生产CSV。 为了确保高可用性，您可以使用存储实时迁移（“实时”存储迁移功能）使用群集。 自然，您需要关注存储阵列的性能。

选项2：使用本地SSD / NMVe驱动器恢复到Hyper-V主机


生产中的另一个恢复方案非常有效：从本地存储到SSD或NVMe使用Hyper-V主机。 磁盘空间的大小取决于您要在特定时间内恢复多少个VM，以及这些VM的大小。

从理论上讲，您不太可能需要恢复所有人和所有内容，因此此配置在成本上应该非常经济。 例如，您可以在每个群集节点中使用一个SSD，或仅在几个群集节点中使用，或通常仅在一个群集节点中使用。 您使用的SSD / NVMe越多，它们的预算就越多，同时在主机之间保持相当有效的负载分配。 在即时恢复过程的最后阶段，可以使用相同的存储实时迁移功能轻松地将虚拟机传输到生产CSV。


该图显示了基础架构计划选项。 当然，您可以酌情结合以上方法。

选项3：使用本地SSD / NVMe驱动器恢复到专用的Hyper-V主机


在此选项中，我们专门分配一个或多个主机以支持恢复。 这样可以避免可能的资源短缺以及对集群中生产主机运行的影响。 您可以使用NVMe驱动器。 我们建议您事先测试此配置中的恢复功能，以了解资源如何耗尽。


如果您打算增加其消耗量，则对于将经过修复的计算机最终迁移到生产中，可以使用所谓的迁移而无需共享资源“无共享实时迁移”。 （为此，您将必须设置其他安全设置。）对于网络资源，您可以使用例如SMB多通道和SMB Direct功能来迁移到CSV /实时迁移/ S2D Hyper-V。
是的，存储系统的迁移（存储实时迁移）不是最快的过程，而是一个负数。 但是还有一个好处-您的虚拟机将还原并在此过程中继续工作。

总结


当然，每个人都根据特定基础结构（源服务器，目标服务器，网络资源）的瓶颈来选择首选选项。 此外，很有可能只需要为计划最关键的VM的恢复或为使用此类服务​​付费的消费者进行仔细研究。

无论如何，主要目标始终是尽可能快的恢复。
之后，已经可以安全地迁移到群集存储系统，从而确保高可用性和容错能力。 并且，当然，虚拟机应该以备份\复制的形式受到保护，以防万一需要再次还原它们。
