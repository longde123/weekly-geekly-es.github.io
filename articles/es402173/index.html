Firefox 52 lanzado con soporte para WebAssembly


Representación 3D de la demostración de Zen Garden en Firefox 52 con soporte para WebAssembly

Mozilla ha lanzado Firefox 52 , el último navegador con soporte para el sistema operativo Windows XP. Se han realizado varios cambios importantes : se ha simplificado la conexión a puntos de acceso, donde primero debe iniciar sesión en el navegador, aparecen advertencias de peligro si la página solicita una contraseña para una conexión insegura (no HTTPS), el soporte para los complementos NPAPI ha desaparecido (excepto Flash, y el soporte completo permanecerá en la compilación ESR) cerró 28 vulnerabilidades .

Pero nada puede compararse con la innovación principal y fundamental: el soporte para el lenguaje de programación de bajo nivel WebAssembly (wasm) como el ensamblador, que se llama una de las innovaciones más significativas de la plataforma web en la última década. Esto es lo que faltaba JavaScript.

Montaje web



Los desarrolladores explican por qué era necesario crear WebAssembly . El hecho es que JavaScript fue originalmente concebido como un lenguaje liviano para scripts simples. Nadie imaginó en qué se convertiría y cómo comenzaría a aplicarse. Fue inventado para principiantes en programación, para cosas simples como escribir un formulario en una página web.

Desde entonces, mucho ha cambiado. Las aplicaciones web modernas son programas informáticos complejos con código de cliente y servidor, la mayoría de los cuales está escrito en JavaScript. A pesar del progreso en el desarrollo de JavaScript en sí mismo y de todos los intentos de los desarrolladores para crear motores efectivos para la ejecución rápida de JavaScript, no surgió nada, es simplemente físicamente imposible. JavaScript tiene limitaciones inherentes. El navegador simplemente no puede ejecutar este código al menos tan rápido como el código nativo en el sistema operativo.

Mozilla fue el primero en madurar antes de desarrollar una especie de máquina virtual en un navegador donde puede ejecutar código de bajo nivel, y hace unos años se lanzó asm.js como demostración (Google experimentó con la API Native Client ). El sublenguaje asm.js demostró ser tan bueno que quedó claro: necesitamos unir fuerzas con todas las compañías de desarrollo más grandes para un proyecto conjunto que haga avanzar la web.

El lenguaje WebAssembly de bajo nivel puede funcionar junto con JavaScript y permitir que las aplicaciones web se ejecuten con un rendimiento mucho mayor, casi como las aplicaciones nativas en el sistema operativo.

Ahora, en un navegador, puede ejecutar juegos 3D de alto rendimiento, sistemas de diseño asistido por computadora (CAD), editores de video, editores gráficos, visualizaciones científicas, computación intensiva en recursos, codificación de video, cualquier cosa.

Con el tiempo, es más probable que muchas aplicaciones web existentes (correo, redes sociales, editores de texto) y marcos de JavaScript comiencen a usar WebAssembly, lo que aumentará significativamente la velocidad de descarga y aumentará considerablemente su rendimiento mientras trabajan.

A diferencia de otros enfoques, como Flash, que requieren la instalación de un complemento en un navegador para ejecutar aplicaciones a una velocidad comparable a las aplicaciones nativas, WebAssembly encaja completamente en la plataforma web estándar. Es un estándar abierto y compatible integrado en los navegadores. Esto significa que los desarrolladores pueden integrar bibliotecas WebAssembly para computación intensiva en CPU (compresión, detección de rostros, física) directamente en aplicaciones web existentes que usan JavaScript.

WebAssembly es un estándar abierto desarrollado por Mozilla, Google, Microsoft y Apple. Como puede ver, este grupo representa a los desarrolladores de los cuatro navegadores más comunes, por lo que puede contar con convertirse en wasm como estándar universal. Google promete implementar soporte para WebAssembly en la próxima versión de Chrome (57), Microsoft ya está trabajando en una implementación en Edge.

El lenguaje de bajo nivel se convertirá en una especie de adición a JavaScript y, al final, debería funcionar en todos los lugares donde funciona JS: en todos los navegadores y en todos los entornos de tiempo de ejecución como Node.js.

¿Quién se beneficiará con el uso de WebAssembly? No se trata solo de escribir nuevas aplicaciones en wasm. A través de compiladores como Emscripten, se pueden portar juegos completos y aplicaciones nativas listas para usar a la web. El código C / C ++ portado usando este compilador se ejecutará en el navegador a casi la misma velocidad que la aplicación nativa. Además de C / C ++, Rust también proporciona soporte preliminar para WebAssembly.

Por ejemplo, puede reproducir la demostración de Zen Garden (requiere el navegador Firefox 52, solo la versión de escritorio es compatible actualmente).


Las funciones de JavaScript llamarán a funciones de WebAssembly y viceversa. Es decir, puede escribir en un solo programa en un lenguaje JavaScript de alto nivel y, a veces, cambiar a C / C ++ / Rust según sea necesario.

Los desarrolladores comenzarán a distribuir y reutilizar módulos WebAssembly de bajo nivel sin tener que comprender su dispositivo, ya que ahora usan bibliotecas JavaScript minificadas.

Mozilla señala que, por el nivel de reutilización de código y arquitectura de software, el muro entre las aplicaciones nativas y web está empezando a colapsar, y esto es solo el comienzo. Las herramientas de desarrollo, los depuradores y los compiladores continuarán evolucionando, al igual que la compatibilidad, el rendimiento y la funcionalidad de WebAssembly. Por ejemplo, el plan de Mozilla ahora es implementar soporte para SIMD multihilo y concurrencia.

"En cierto modo, WebAssembly está cambiando lo que significa un desarrollador web", escribe David Bryant, gerente de desarrollo de plataformas en Mozilla, "ya que está cambiando las propiedades fundamentales de la web".

De hecho, ahora es posible portar programas C / C ++ para su ejecución en un navegador, y en el futuro cercano se puede hacer lo mismo para los lenguajes en los que se escriben las aplicaciones móviles: Java, Swift, C #. Todos serán compatibles con la plataforma web estándar. Resulta que, en cierto sentido, todos los programadores eventualmente se convertirán en desarrolladores web.
