Protocolos


Ni siquiera sé si este criterio debe tomarse por separado. Las aplicaciones web viven en la web y están obligadas a cumplir con las leyes de la web (protocolos). Los principales protocolos en la Web son TCP e IP . Muchos otros protocolos se basan en ellos, pero para las aplicaciones web considero que HTTP es el más importante (o más bien, su extensión HTTPS basada en TLS ). Es decir, una hermosa aplicación web está disponible a través de HTTPS / TLS (como opción, a través de HTTP), y otros protocolos (LDAP, RPC, IMAP4, POP3, SMTP, FTP, NNTP, ...) la hacen menos bella con cada uno Protocolo opcionalmente compatible. La aplicación en sí misma puede usar recursos externos usando estos protocolos adicionales.


En cuanto a WebSocket , no tengo suficiente experiencia en el uso de este protocolo con aplicaciones web. Se ve hermoso y prometedor, pero no puedo decir qué tan estable y práctico es.


Navegadores


Una aplicación web con un solo pie se encuentra en el lado del servidor y el otro en el lado del cliente. El lado del cliente es el navegador. Un navegador moderno ofrece muchas cosas que una aplicación web moderna puede y debe usar para su beneficio. Una aplicación web hermosa utiliza las capacidades modernas de los navegadores y no se requiere que funcione en aquellos navegadores que no ofrecen capacidades modernas. Entiendo que los polifilos son una medida necesaria , pero esto es feo. Al final, no solo los desarrolladores deben seguir el ritmo de las tecnologías modernas, los usuarios y las empresas también se ven afectados.

Existen muchas tecnologías para el lado del cliente de las aplicaciones web que permiten al desarrollador facilitar la creación de la tríada HTML / CSS / JS (algo que todos los navegadores modernos entienden). Pero una vez tuve un contacto cercano con GWT y lo considero hermoso cuando un desarrollador ve el código original en un navegador, y no el resultado de la compilación o la transpilación. Por lo tanto, usar webpack y productos similares para generar código de cliente, en mi humilde opinión, es feo. Cuanto más similar sea el código ejecutado en el navegador al código fuente creado por el desarrollador, mejor. No lo creo? Intente desviar en producción el código creado por GWT.


Hay más libertad en el lado del servidor (Java, PHP, perl, python, C #, Ruby, ...), pero me parece que es hermoso cuando tanto el lado del servidor como el navegador usan un lenguaje de programación: JavaScript. Aún así, el lenguaje determina el pensamiento , y los equipos de ideas afines son más productivos.

Escalabilidad inversa


Érase una vez, los programas podían transferirse en disquetes, ahora en unidades flash o descargarse inmediatamente de la Web. Copiar una aplicación normal y ejecutarla en otra máquina es una tarea trivial. Con las aplicaciones web, la situación es algo especial. Una red es un entorno global en el que no es necesario tener clones de la misma aplicación web. Un Facebook, Twitter, Instagram, Mail.ru o Yandex es suficiente en la Web. Puede tener diferentes aplicaciones web en el mismo nicho temático, pero con diferentes audiencias (como Facebook y Vkontakte, Mail.ru y Gmail, Google Maps y Azure Maps). Se necesitan recursos de hardware para garantizar la disponibilidad global de tales aplicaciones web, digamos, no triviales .


Nunca he trabajado con aplicaciones web de tal nivel como desarrollador y no puedo imaginar cómo están organizadas en su interior. Para garantizar la operatividad de tales aplicaciones web, se necesitan equipos de especialistas relevantes y centros de datos separados. Admiro la capacidad de las personas de cooperar a tal escala y crear productos similares, pero mi estándar de belleza es una aplicación web que se puede ejecutar en una computadora portátil separada.


Una hermosa aplicación web escala no solo hacia arriba y en amplitud (para usuarios), sino hacia abajo y hacia adentro (para desarrolladores).

Intercambio cruzado


Ya he señalado en el párrafo "Escalabilidad inversa" que la naturaleza global de la red hace posible tener una aplicación web en el planeta. Por lo tanto, cada aplicación web debe ser al menos algo diferente de las demás para garantizar su supervivencia. Sin embargo, mis muchos años de experiencia con Magento (el marco para construir tiendas de comercio electrónico) dice que puede haber más en común entre las aplicaciones web individuales que diferencias. Una hermosa aplicación web no solo debe ser modular, sino que también debe compartir sus módulos con otras aplicaciones web. Hasta cierto punto, esta idea se refleja en las especificaciones de JSR 168 y JSR 286 y en marcos como WordPress , Django y el mismo Magento. Cuanto mayor sea el número de módulos de aplicaciones web utilizados por otras aplicaciones web, más hermoso es desde mi punto de vista. El intercambio cruzado le permite crear mejores módulos y, como resultado, aplicaciones web más estables.


Por módulo, no me refiero a bibliotecas como jQuery o RequireJS, sino a formaciones más grandes, como complementos en WordPress y Django . Pero para las bibliotecas, la tesis también es cierta de que la amplia distribución de la biblioteca hace que sea mejor y más estable.


Arquitectura de Harvard


La arquitectura de Harvard , a diferencia de la actual bola de Princeton , implica la separación de código y datos. La arquitectura no despegó, pero la idea misma me parece hermosa. Especialmente para aplicaciones web. Cualquier estática (HTML / CSS / JS / Images / ...) es código. Puede y debe almacenarse en caché al menos en el lado del servidor, al menos en el lado del cliente. Y los datos son REST / JSON (hermoso) o SOAP / XML (un poco menos hermoso). O WebSockets / JSON (puede ser la mejor opción, pero no lo he intentado).

Forjas en lugar de estrellas de la muerte


Érase una vez, cada ciudad más o menos grande tenía su propia fragua. Quizás no uno. Algunos son mejores, otros son peores. Había maestros de herrería conocidos en todo el mundo, y había quienes no tenían otra alternativa. Guerras, epidemias, desastres naturales arrasaron. Algunas ciudades desaparecieron junto con la población. Pero la herrería se quedó para vivir. En lugar de las ciudades desaparecidas, se erigieron otras nuevas y también aparecieron forjas en ellas.


Ahora mira un servicio como DNS . Cuando los servidores raíz se acuestan, todo el mundo tiene fiebre .


En mi opinión, una hermosa aplicación web no puede ser tan grande como Facebook o Mail.ru. Esto está más cerca de la Estrella de la Muerte , tanto en términos de los recursos necesarios para la construcción como en los recursos necesarios para mantener la operatividad. Sí, en caso de que Facebook sea destruido, la humanidad no desaparecerá, sus funciones serán asumidas rápidamente por otras aplicaciones (el mismo VK en la Federación de Rusia y adyacentes, Instagram, Twitter, ...). Sin embargo, encerrar a una parte importante de la población del planeta en una sola aplicación es feo. Además, en presencia de alternativas mucho más estables (por ejemplo, torrentes ).
