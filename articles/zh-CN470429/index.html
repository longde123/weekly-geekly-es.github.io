拟议的计划模型：ERP + SPM


以下是可以相对简单，快速地实施的计划模型的摘要。 而且，重要的是，在实施“生产ERP”的“会计部分”时不会浪费现金和时间。

规划和监控系统（SPM）步阶的概念



计划系统





一级规划


负责建模/计划（发布）和平衡（容量）。 给定容量。 但是，为了保留系统的外部简单性和可控性-通过规划，仅在瓶颈中考虑资源限制。 和/或-考虑到DSE购买和/或生产的关键库存，和/或考虑了订单履行的关键周期。 如下图所示。





备注 。 第一级计划，很少执行“诚实”。 因此，此级别上所有未解决的问题或冲突（资源和需求）“下降”，“跌落”（进入生产）。 在哪里“解决”：
1）通过每天两次或多次，计划会议，
2）或者，在企业“强大”的IT部门在场的情况下-通过尝试通过引入详细计划的系统/功能来清除它们（顶级问题），包括 考虑容量（包括“现成的” MES系统）。 不知道或不注意这些功能/系统仅在可靠（适当）的顶层计划下工作的事实。 也许不是很准确，但是已经通过平衡的“筛选器”进行了筛选，从而消除了主要的资源冲突，包括 并通过会议“生产与 销售。”
重要！ 不，即使是最出色的计划系统，也能够消除对客户（市场）的错误承诺对生产造成的损害。 承诺是计划的第一层，无论如何都不是“车间”。

第二层计划和执行
第二级计划（生产/ MTO中需求与内部状况的同步）以及进一步的执行：
a）基于每个订单的PSI的“正常”形成模型，
b）通过（订单-PSI）顺序（优先）计划，
c）考虑到订单的日期和优先级，
d）根据订单需求考虑库存和预期收据（RFP，PP）的软/硬/有条件（重新）分配，
e）可以合并执行（使用自定义计划！）



随着计划的“指示性”版本的不断更新（不少于每晚）（从命令的日期开始，考虑到优先级，“从左向下”-“应为应该”），并进行估计-“结果如何……”。 通过比较整个定制和生产“监控”，可以制定操​​作决策。

一个 在物理级别，解决了各个特定企业的低级问题：
1）交换机制：
通过某种集成总线/现成的ETL应用程序，
直接在系统之间；
2）匹配参考代码的规则：
adopted采用统一编纂
每个系统都以“自己的”代码工作，并且存在某种MDM系统，用于将源系统代码转换为接收器系统代码。
b。 每个系统具有什么集成功能：
1）是否提供了REST API
2）系统使用的数据格式。

SPM已准备好用于几乎所有集成方案：
1）JMP提供了REST API，用于对其对象执行CRUD操作，
2）每个PSD对象都有一个mdm_code字段，用于存储对MDM系统中记录的符合性，
3）在JMP中有其自己的内部任务队列供卸载：
a）将上传数据的任务排队等待系统中的事件（例如，创建对象，更改状态等）。 有一个设置可以确定应为哪些对象卸载哪些事件，
b）队列是由后台的一个独立进程异步处理的，
c）处理任务的结果可能是：
将http请求发送到特定地址，
将文件保存到目录。
d）还为每种类型的对象配置了发生作业执行错误时的系统行为。 例如，在发送http请求时，外部系统不可用。 行为选项：
1）忽略错误，继续执行其他任务，
2）停止队列，直到错误修复，
3）等待操作员的手动决定，
4）在m分钟后尝试再次发送请求，在n个失败的请求后停止尝试；
4）上传并加载到JMP中的数据的格式为JSON。 您可以自定义转换为其他格式。

结论


我们的实践表明了该方案的可行性。 完整的ERP + SPM系统可以被足够快地实施，同时切实保证收到运营（充分的生产计划和MTO，运营和可靠的监控）以及业务成果（提高生产吞吐量等）。 然而，后者取决于改变管理概念的愿望和能力。 其中一个重要的部分是拒绝（分阶段）计件工资，拒绝定期（月度）计划范式，“通过签名”确定和发布月度计划以及其他一些可能的更改。
