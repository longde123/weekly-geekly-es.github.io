Extensión PHP y Kotlin Native. Primera parte, ingenua

Este artículo analiza el enfoque más ingenuo y sencillo para crear una extensión PHP usando Kotlin Native. Le llamo la atención sobre el hecho de que no con , sino con el uso .

Es más bien una especie de tutorial con una descripción de los problemas que surgieron al cruzar un erizo con un erizo y las formas de resolverlos. No habrá revelaciones, pero tal vez alguien sea útil.

Entonces, si está interesado, bienvenido a cat.

El objetivo es escribir una extensión con una función `hello ($ name)` que tome una cadena e imprima `Hello, $ name!`.

Decidiremos ingenuamente, como su nombre lo dice.

  1. Escribamos una función en Kotlin
  2. Compilar en la biblioteca compartida
  3. En la forma clásica (en C), escribimos una extensión que redirigirá una llamada de función a esta biblioteca

Parece simple, pero un rastrillo ya ha acechado en la densa hierba:

  1. Hay una serie de ejemplos del uso de bibliotecas C en Kotlin, pero no pude encontrar nada adecuado para usar las funciones de la biblioteca Kotlin en C (bueno, tal vez estaba mirando mal, lo que realmente)
  2. El compilador kotlinc carga las dependencias la primera vez que se inicia. Pero la mala suerte es mi máquina virtual Linux en la nube corporativa. Sin, incluso teórica, la capacidad de llegar a Internet.
  3. Este rastrillo es más probable debido a la falta de experiencia en C, pero te diré cómo el enlazador me trolleó notablemente.

Todo sucedió en Red Hat Enterprise Linux Server versión 7.5.

Vamos!


Para comenzar, instale ... No, no el compilador de Kotlin. Primero, instale el JDK. Bueno asi.
Luego instale el compilador Kotlin. Simplemente descargue el archivo del github y descomprímalo, por ejemplo, en el directorio de inicio.

En un mundo ideal, al principio, él descarga todas las dependencias él mismo, pero, en ausencia de Internet, procedemos de la siguiente manera (los empleados de JetBrains obtienen un conocimiento secreto:

  • Creamos cualquier secuencia de comandos Kotlin simple para que haya algo que hacer en el compilador en el siguiente paso
  • Comenzamos $ KOTLIN_HOME / bin / kotlinc SimpleScript.kt, espere un poco y presione CTRL + C: esto es para crear una estructura de carpetas en el directorio de inicio
  • Buscamos en el archivo $ KOTLIN_HOME / konan / konan.properties, en la sección con nuestra arquitectura, y buscamos el elemento: 

dependencies.linux_x64 = \ clang-llvm-5.0.0-linux-x86-64 \ target-gcc-toolchain-3-linux-x86-64 \ libffi-3.2.1-2-linux-x86-64

  • Vamos a un repositorio especial y descargamos todo lo anterior.
  • Agregamos todo esto a ~ / .konan / cache (le recuerdo que el directorio de inicio de Linux es una tilde)

Ahora, en el primer inicio, el compilador usará estas distribuciones y no se conectará a Internet.

Tenga en cuenta que las dependencias no son muy pequeñas y, después de instalarlas, mi directorio de inicio se volvió 3.4 GB más pesado. Para aquellos que tienen un volumen separado debajo de su tarea, puede ser crítico.

Luego, con el administrador de paquetes estándar, instale php y el correspondiente php-devel.
Sobre esto, las medidas preparatorias han terminado.

Como no vale la pena hablar de escribir extensiones PHP, nos las arreglaremos con el código más corto posible.

Comencemos con Kotlin


hellokt.kt 

 fun kt_print(string:String){ println("Hello, $string!!!") }

En varios manuales y tutoriales, kotlinc o konanc se usan como compiladores, lo cual es algo confuso. Entonces, es lo mismo. Aquí está la prueba:


Compilando 

 # $KOTLINC_HOME/kotlinc -opt ./hellokt.kt -o hellokt -produce dynamic

Con el modificador -opt , la biblioteca es más pequeña. -produce dynamic le dice al compilador que haga una biblioteca compartida para la plataforma actual.

Después de la ejecución, tendremos dos archivos: libhellokt.so y hellokt_api.h. Biblioteca y archivo de encabezado para ello. Tenga en cuenta que los prefijos y sufijos se generan automáticamente y no podemos influir en ellos (probablemente).

En nuestro caso, obtenemos un archivo de encabezado. 

 #ifndef KONAN_HELLOKT_H #define KONAN_HELLOKT_H #ifdef __cplusplus extern "C" { #endif #ifdef __cplusplus typedef bool hellokt_KBoolean; #else typedef _Bool hellokt_KBoolean; #endif typedef char hellokt_KByte; typedef unsigned short hellokt_KChar; typedef short hellokt_KShort; typedef int hellokt_KInt; typedef long long hellokt_KLong; typedef float hellokt_KFloat; typedef double hellokt_KDouble; typedef void* hellokt_KNativePtr; struct hellokt_KType; typedef struct hellokt_KType hellokt_KType; typedef struct { /* Service functions. */ void (*DisposeStablePointer)(hellokt_KNativePtr ptr); void (*DisposeString)(const char* string); hellokt_KBoolean (*IsInstance)(hellokt_KNativePtr ref, const hellokt_KType* type); /* User functions. */ struct { struct { void (*kt_print)(const char* string); } root; } kotlin; } hellokt_ExportedSymbols; extern hellokt_ExportedSymbols* hellokt_symbols(void); #ifdef __cplusplus } /* extern "C" */ #endif #endif /* KONAN_HELLOKT_H */

El acceso a nuestra función kt_print irá de esta manera. 

 hellokt_symbols()->kotlin.root.kt_print(char *);

Te contaré sobre las clases y paquetes a continuación: hay matices.

La biblioteca está lista, ve a C


config.m4 ( cómo crearlo ) 

 PHP_ARG_ENABLE(hello, whether to enable hello support,[ --enable-hello Enable hello support]) if test "$PHP_HELLO" != "no"; then PHP_ADD_LIBRARY_WITH_PATH(hellokt, /path/to/compiled/library, HELLO_SHARED_LIBADD) PHP_NEW_EXTENSION(hello, hello.c, $ext_shared) PHP_SUBST(HELLO_SHARED_LIBADD) fi

¡Se requiere la primera línea!
Lo primero que se ve en el ejemplo config.m4 anterior, aparte de un par de comentarios, son tres líneas que usan PHP_ARG_WITH () y PHP_ARG_ENABLE ().
...
Cada extensión debe proporcionar al menos uno u otro con el nombre de la extensión, para que los usuarios puedan elegir si construir o no la extensión en PHP.

Tenga en cuenta que en PHP_ADD_LIBRARY_WITH_PATH, el primer argumento es el nombre de la biblioteca. No libhellokt, es decir, hellokt. Maté dos horas hasta que descubrí por qué ld no puede encontrar la biblioteca. (Aquí está la historia prometida sobre la intimidación de enlazadores).

Ahora, de hecho, el código de extensión en sí

hola.c 

 #include "php.h" //    #include "hellokt_api.h" #define PHP_MY_EXTENSION_VERSION "1.0" #define PHP_MY_EXTENSION_EXTNAME "hello" PHP_FUNCTION(hello); static zend_function_entry hello_functions[] = { PHP_FE(hello, NULL) }; zend_module_entry hello_module_entry = { #if ZEND_MODULE_API_NO >= 20010901 STANDARD_MODULE_HEADER, #endif PHP_MY_EXTENSION_EXTNAME, hello_functions, NULL, NULL, NULL, NULL, NULL, #if ZEND_MODULE_API_NO >= 20010901 PHP_MY_EXTENSION_VERSION, #endif STANDARD_MODULE_PROPERTIES }; ZEND_GET_MODULE(hello) PHP_FUNCTION(hello) { char * name; size_t name_len; if (zend_parse_parameters(ZEND_NUM_ARGS() TSRMLS_CC, "s", &name, &name_len) == FAILURE) { RETURN_NULL(); } hellokt_symbols()->kotlin.root.kt_print(name); //   efree(name); }

Bueno, entonces todo está en el camino rodado: 

 # $PHP_PATH/phpize # ./configure --with-php-config=$PHP_PATH/php-config # make # $PHP_PATH/php -dextension=./modules/hello.so -r "echo hello('World');" Hello, World!!!

Bono, sobre clases y paquetes


Supongamos que queremos hacer feng shui y cegar este código. 

 package hello.kt; public class HelloKt { fun kt_print(string: String) { println("Hello, $string!!!") } }

No hay forma de hacer una llamada a un método habitual aquí: primero tendrá que crear una clase y pasarle el primer argumento a la función llamada. 

 hellokt_kref_hello_kt_HelloKt helloKt = { 0 }; if(!helloKt.pinned){ helloKt = hellokt_symbols()->kotlin.root.hello.kt.HelloKt.HelloKt(); } hellokt_symbols()->kotlin.root.hello.kt.HelloKt.kt_print(helloKt, name);
