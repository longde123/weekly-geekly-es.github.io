SQLite y NW.js: instrucciones paso a paso para crear fuertes amistades

¡Hola, Habr y sus habitantes! Quería esculpir algo en HTML y JS para que fuera de escritorio y conveniente. ¿Qué se necesita para esto? Derecho Node.js y NW.js o Electron. Al observar los débiles intentos de NW vs Electron holivar y leer la documentación de ambos, se tomó la decisión de comenzar a sentir NW.js.

Pero aquí hay una emboscada. La ejecución de npm i sqlite3 instala solo aquellas bibliotecas que se pueden ejecutar cuando se inicia directamente a través del nodo. Y si intenta conectar el script en html y ejecutar este html en NW.js, entonces no saldrá nada. Google Associates ofrece solo una receta para incluir sqlite en un proyecto en nw.js, e incluso esa debe sacarse del caché. Sí, y ya está desactualizado. Por lo tanto, pasamos un par de días estudiando los problemas y buscando una solución que funcione. Pido gato.

Entonces Para empezar, la máquina fue construida "a partir de lo que era" e instaló Windows 7 x32 con el paquete .NET 4.7.1 (requerido para Visual C ++). ¿Por qué 32 bits? Decidí comenzar de a poco. Aunque, como se vio después, crear un módulo para x64 no requiere ejecución en un sistema x64.

Luego se instaló Node.js versión 10.8.0, aunque 10.9.0 ya había aparecido al final de los experimentos. Pero luego decidí escuchar a personas con conocimientos:



Después de pensarlo, decidí que estaba haciendo lo mismo para LTS, aunque no está del todo claro dónde tienen LTS y dónde no. Sugirió que los no LTS son versiones beta, alfa y nocturnas.

Aunque, como cualquier abandono, hizo todo a la mitad y cumplió la siguiente recomendación:

Recomendación de documentación para lanzamientos de LTS
En Windows, debe reemplazar el archivo
<npm-path> \ node_modules \ node-gyp \ src \ win_delay_load_hook.cc con el que está en github.com/nwjs/nw.js/blob/nw18/tools/win_delay_load_hook.cc antes de instalar módulos con node-gyp o npm.

Bueno, no encontré este archivo en mi computadora. Probablemente porque no instalé node-gyp.

Lo configuramos para el nodo (es absolutamente necesario hacerlo desde la línea de comandos con derechos de administrador)

npm i -g nw-gyp

De hecho, el reemplazo de nodo-gyp
nw-gyp es un truco en node-gyp para admitir encabezados y bibliotecas específicas de NW.js.

A continuación, descargue Visual Studio Community Edition (el enlace actual se encuentra fácilmente en su motor de búsqueda favorito), al momento de escribir este artículo es la versión actual de 2017. Lo que es importante para nosotros, contiene Build Tools 2015, mientras que los módulos nativos en el nodo solo se ensamblan con esta versión y más Fresco no es compatible. Descargue el instalador, inicie y coloque una marca de verificación en la pestaña "Componentes individuales" - Toolbox VC ++ 2015.3 v14.00 (v140) para PC



El SDK de Windows 8.1 agregado también es necesario, por lo que instalamos los tres puntos. La instalación de esta forma conduce a un salto de 2.89 gigabytes de todo lo que necesita.

Vamos más allá, balanceamos una serpiente. Y la versión 2.7. Instalar por defecto en C: \ Python27.
Y ahora, ¡atención! Emboscada

Por defecto, el instalador cree que no es necesario agregar una ubicación de Python a la RUTA, por lo tanto, eliminamos este malentendido en la etapa de instalación



Una emboscada similar persigue a muchos desarrolladores al construir una amplia variedad de módulos para un nodo. Y todo porque la corporación a pequeña escala tampoco considera necesario prescribir los caminos hacia sus creaciones. El github simplemente está abrumado con errores de compilación debido al hecho de que después de la instalación, la ruta a las herramientas de ensamblaje no llega a la RUTA. También corregimos este malentendido: corregimos la variable de entorno para que permita que el ensamblaje se ejecute, si no con mucha suavidad, al menos sin errores. Elemento obligatorio: reiniciar .

Bueno, ahora la asamblea real. Usé el administrador FAR, pero nadie prohíbe el cmd de la vieja escuela

c:\
md app
cd \app
npm init
npm install sqlite3 --build-from-source --runtime=node-webkit --target_arch=ia32 --target=0.32.1 --msvs_version=2015

para x64 cambiamos --target_arch = x64.

Bueno eso es todo. Si todo se hace correctamente, el ensamblado pasa con un cierto número de mensajes como

..\src\database.cc(672): warning C4996: 'Nan::MakeCallback': deprecate [C:\app\node_modules\sqlite3\build\node_sqlite3.vcxproj]
C:\app\node_modules\nan\nan.h(929): note: . "Nan::MakeCallback"
c:\app\node_modules\nan\nan_new.h(208): warning C4244: : "sqlite3_int64" "double", ( ..\src\database.cc) [C:\app\node_modules\sqlite3\build\node_sqlite3.vcxproj]

pero al final el módulo está ensamblado y ubicado en
C:/app/node_modules/sqlite3/lib/binding/node-webkit-v0.32.1-win32-ia32\node_sqlite3.node

Matiz de ensamblaje: con cada ensamblaje, el recopilador elimina la carpeta node_modules / sqlite3 / lib. Por lo tanto, si es necesario ensamblar dos módulos (para cada arquitectura), guarde el resultado intermedio.
