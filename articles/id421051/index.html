Bagaimana Saya Meluncurkan Proyek SaaS Pertama Saya untuk Mempekerjakan Sepanjang Hari

Halo, Habr! Saya mempersembahkan kepada Anda terjemahan artikel Bagaimana Saya Mengirimkan Proyek Samping SaaS Pertama Saya Saat Bekerja Purna Waktu oleh Tigran Hakobyan, yang, yang bekerja di layanan posting posting Buffer yang terkenal, dapat meluncurkan proyeknya dan bahkan menguangkannya.




Ini adalah kisah pribadi saya tentang bagaimana saya meluncurkan proyek SaaS pertama saya, bekerja 40 jam seminggu di Buffer. Tujuan artikel ini adalah untuk menginspirasi Anda. Jika Anda seperti saya, yang memiliki pekerjaan penuh waktu, dan Anda ingin membangun bisnis sampingan yang menguntungkan sebagai sumber pendapatan, maka kisah ini akan beresonansi dengan Anda. Dalam artikel ini, saya ingin menunjukkan bahwa saya tidak berkeringat atau memproses sama sekali dan, bagaimanapun, dapat menawarkan produk SaaS yang benar-benar berfungsi.

Pendahuluan


Saya seorang pengembang situs web dan saya sangat beruntung bahwa selain bermain sepak bola di waktu luang, saya juga menikmati pengkodean dan membuat proyek untuk hiburan. Baru-baru ini, saya membuat Rak Buku , yang membantu banyak orang mengatur buku mereka di Web. Meskipun pekerjaan penuh waktu memiliki dampak besar pada pertumbuhan saya sebagai seorang insinyur, saya dapat memperoleh beberapa keterampilan pengembang dengan mengerjakan proyek-proyek pribadi saya.

Hanya tahun lalu saya mulai berpikir tentang menciptakan sumber penghasilan lain selain pekerjaan utama saya. Gagasan untuk hanya mengandalkan satu gaji sedikit menakutkan. Saya tahu bahwa saya memiliki keterampilan dan hasrat untuk merenungkan sesuatu. Saya memutuskan bahwa saya ingin memulai bisnis, mungkin online, mengingat keterampilan yang saya miliki. Dorongan lain untuk pikiran-pikiran ini adalah apa yang ingin saya alami dan belajar bagaimana membangun bisnis. Saya tidak pernah terlibat dalam bisnis apa pun dalam hidup saya, jadi saya melihat ini sebagai kesempatan belajar yang hebat, cara saya bisa menguasai keterampilan yang tidak saya miliki saat ini. Hal terburuk yang dapat terjadi adalah bahwa saya akan gagal, tetapi saya akan memiliki pengalaman.

Ide


Jelas, hal pertama yang harus dilakukan pengembang adalah mulai memikirkan ide. Gagasan tidak pernah menjadi masalah bagi saya, jadi saya selalu harus mengidentifikasi ide yang sesuai dengan saya. Kali ini, saya memutuskan untuk mencoba pendekatan yang berbeda dan benar-benar memikirkan ide yang cocok untuk saya sebelum saya memikirkannya. Ada beberapa kriteria yang dengannya saya ingin menggambar setiap ide.

  • Saya ingin menyelesaikan beberapa masalah nyata, mungkin yang saya temui secara pribadi
  • Seharusnya untuk pasar yang saya kenal baik
  • Ini seharusnya bukan ide baru (itu tidak akan mengubah dunia)
  • Itu bisa menjadi semacam bisnis

Aturan emas dari setiap ide adalah bahwa ia harus menyelesaikan masalah yang dihadapi orang. Saya biasa menambahkan begitu banyak ide ke dalam catatan saya, jadi itulah alasan saya kembali ke tumpukan ide yang saya miliki.


Catatan saya di mana saya menyimpan semua ide

Sejak awal, saya tahu bahwa saya mungkin bisa lebih sukses jika saya menciptakan sesuatu untuk pengembang, karena saya tahu pasar dengan cukup baik, dan sebagian besar teman dekat dan pengikut saya secara teknis cerdas. Saya bisa menggunakan kontak dan audiens saya untuk mengonfirmasi ide dan mendapatkan umpan balik yang berharga sebelum memutuskan apa pun. Itu benar-benar mengurangi semua ide saya menjadi daftar 2-3 hal yang bisa saya kerjakan. Salah satu idenya adalah apa yang terus-menerus saya kembali lagi dan lagi. Ini adalah sesuatu yang saya temui dalam layanan Buffer dan saat mengerjakan proyek paralel saya sebelumnya. Cara mudah untuk mengontrol tugas yang dijadwalkan di cron. Karena salah satu bidang yang saya tangani dengan Buffer adalah infrastruktur data analitik, saya telah meluncurkan selusin pekerjaan cron di latar belakang untuk mengumpulkan data analitik harian untuk klien kami. Seharusnya semuanya relevan. Layanan pemantauan Datadog yang kami gunakan di Buffer benar-benar hebat, tetapi pada awalnya dirancang untuk memantau layanan atau server yang sudah berjalan lama. Saya membutuhkan bilah alat sederhana di mana saya dapat melihat daftar semua tugas saya di cron, status dan log mereka. Setiap hari saya mendapatkan laporan tentang semua tugas yang sedang berjalan, jadi saya tahu bahwa semuanya berjalan sesuai rencana.

Setelah memilih ide ini, saya ingin mencari tahu apakah ada solusi yang berfungsi di pasar. Jika ada solusi, ini pertanda baik bahwa ada permintaan untuk alat tertentu. Bahkan, ada beberapa solusi dengan berbagai paket berbayar di pasaran. Saya tidak perlu ingin membuat sesuatu yang sama sekali baru, karena jika saya lakukan, akan lebih sulit untuk mengidentifikasi dan menguji pasar. Semua solusi yang ada telah membayar rencana, jadi saya tahu bahwa orang bersedia membayar untuk itu. Tujuan selanjutnya adalah mengevaluasi kebenaran pikiran saya dengan membuat dan menjalankan MVP.

MVP


Saya menghabiskan 2 bulan membuat versi pertama Cronhub (ya, saya memberinya nama). Sesuatu yang layak yang bisa saya kirimkan dalam uji coba ke sekelompok teman dan pengikut saya di Twitter. Untuk MVP, saya membutuhkan sesuatu yang sangat sederhana, tetapi juga cukup berharga, yang akan dibayar orang. Saya tahu Anda mungkin berpikir bahwa 2 bulan adalah waktu yang lama untuk membuat MVP, tapi saya belum mengadopsi pendekatan "keringat-keringat" tradisional dan sebagai gantinya:

  • Bekerja hanya 1-2 jam setiap hari
  • Tidur 8 jam setiap hari.
  • Saya menonton Netflix kapan pun saya mau.
  • Benar-benar santai di akhir pekan.
  • Menggunakan semua solusi teknis yang membuat saya nyaman.

Karena saya bekerja penuh waktu, saya bekerja di Cronhub biasanya dari 7 hingga
8:30 malam, saya juga bisa bekerja di pagi hari, tetapi saya menghabiskan sebagian besar pagi hari di gym. Ada beberapa hari ketika saya merasa lelah secara mental setelah bekerja, dan saya melambat, tetapi sebagian besar waktu saya mengikuti jadwal harian saya. Saya tahu bahwa jika saya ingin menyelesaikan proyek ini, saya harus menghemat biaya dan melakukan setiap hari, bahkan jika itu adalah komitmen kecil (bahkan mungkin satu baris komit). Konsentrasi selalu sangat membantu saya untuk terus maju. Saya menggunakan Trello untuk memecah tugas proyek saya menjadi langkah-langkah kecil.


Papan Trello saya untuk Cronhub

Saya mencoba membuat setiap tugas menjadi sangat kecil sehingga saya dapat memulai dan menyelesaikannya dalam satu hari. Menjaga tugas-tugas kecil membantu saya meluncurkan produk lebih cepat dan melihat kemajuan harian saya. Ketika Anda melihat semacam kemajuan, itu memotivasi dan mendukung Anda dalam banyak hal. Mungkin ini tipuan pikiran? Mengerjakan tugas-tugas besar memperlambat kita dan, pada akhirnya, kita menyerah, karena kita bosan dan kita ingin mengerjakan sesuatu yang lain. Saya tidak pernah bekerja di malam hari. Saya pergi tidur sekitar jam 10:30 setiap hari dan bangun jam 7. Tidur yang benar adalah prioritas nomor satu saya. Dia menentukan energi mental yang ada di siang hari, dan saya tidak bisa menyumbang untuk itu. Selain tidur, saya memutuskan untuk menghabiskan sebagian besar akhir pekan saya melakukan sesuatu yang sama sekali berbeda, seperti bermain sepak bola, menonton film, atau mengobrol dengan teman dan keluarga. Terlepas dari kenyataan bahwa saya suka pengkodean, saya tahu mudah membakar diri. Akhir pekan membantu saya menyegarkan otak saya.

Saya pikir sebagai pengembang Anda selalu ingin menggunakan teknologi paling canggih dan keren. Ini normal. Saya juga menginginkannya. Namun, tujuan saya berbeda, dan saya ingin membangun dan meluncurkan Cronhub secepat mungkin, mengandalkan teknologi yang sudah saya ketahui. Saya fokus pada tujuan saya dan menggunakan Laravel dan Vue.js. Cronhub adalah aplikasi satu halaman menggunakan Laravel sebagai backend.

Peluncuran Beta Tertutup


Pada 20 Februari, saya menyelesaikan pengembangan Cronhub yang diperlukan minimum dan siap mengundang grup pengguna pertama untuk mencoba Cronhub. Setelah tweet saya, sekitar 20-25 orang menoleh ke saya di Twitter untuk meminta undangan, dan umpan balik yang saya terima dari mereka sangat berharga.


Tweet ini adalah undangan untuk menutup beta

Ada beberapa laporan bug dan beberapa saran fitur hebat yang saya tambahkan ke dokumen umpan balik. Melacak umpan balik pengguna adalah langkah penting, karena membantu mengidentifikasi pola yang jelas yang dapat Anda ikuti ketika memutuskan pengembangan produk. Secara umum, kesan pertama dan ulasan sangat menggembirakan. Sekarang saya perlu terus meningkatkan produk dan mempersiapkannya untuk peluncuran publik pertama. Saya merencanakan peluncuran publik pertama dalam sebulan.

Peluncuran publik


Tiga bulan kemudian, hari ini, saya secara terbuka meluncurkan proyek SaaS pertama saya. Hore!

Jelas, saya gugup dan tidak tahu apakah ini akan berhasil atau tidak. Namun, saya tahu bahwa ini akan membawa saya selangkah lebih dekat ke tujuan saya. Tujuannya adalah menjadikan Cronhub bisnis online yang menguntungkan di mana saya bisa belajar dan mengalami semua rahasia dalam berbisnis. Pada akhirnya, apa yang lebih buruk bisa terjadi? Saya akan belajar banyak!

Saya tahu, mungkin saya terlalu fokus untuk memikirkan keuntungan, tetapi setelah membuat beberapa produk sampingan gratis, saya tahu bahwa inilah saatnya untuk melakukan sesuatu yang lebih serius. Waktu adalah aset paling berharga yang saya miliki, dan saya ingin menghabiskannya secara sadar. Membangun produk berbayar adalah hal yang paling memotivasi, dan mendorong Anda untuk maju. Juga, mendukung produk sampingan gratis tidaklah murah, dan saya tahu ini dari pengalaman saya sendiri.

Pelajaran yang dipetik


3 bulan terakhir adalah waktu yang tepat untuk berpikir, dan juga untuk mengevaluasi apa yang bekerja dengan baik dan apa yang tidak. Setiap kali saya membuat proyek baru, ini adalah pengalaman baru untuk belajar. Setiap proyek adalah unik dan membutuhkan proses pemikiran yang berbeda tentang produk. Sebagai seorang insinyur produk, saya ingin mengembangkan pemahaman tentang produk saya, dan ini membantu.

Secara keseluruhan, saya mendapat banyak pelajaran yang benar-benar membantu saya memulai dan memulai ide. Saya ingin berbagi dengan Anda yang paling penting.

  • Selesaikan masalah yang Anda hadapi secara pribadi. Ini sangat penting, karena pada dasarnya Anda sedang membangun produk untuk diri Anda sendiri, selalu ingat ini. Ini sangat memudahkan pengambilan keputusan produk. Anda tahu pertanyaan apa yang harus Anda tanyakan, dan peluangnya lebih tinggi ketika Anda mengajukan pertanyaan yang tepat.
  • Kurangi tugas Anda. Saat Anda memecah proyek Anda menjadi beberapa bagian, cobalah membuatnya lebih kecil. Cara yang baik untuk mengukur ukuran tugas adalah dengan bertanya pada diri sendiri: "Bisakah saya menyelesaikan tugas ini dalam sehari?" Jika jawabannya "Tidak," maka ini mungkin tugas yang besar, dan Anda dapat memecahnya lebih lanjut.
  • Tidur nyenyak dan istirahat. Saya tidak bisa menekankan betapa pentingnya tidur yang layak. Anda tidak perlu bekerja di malam hari. Fokus pada kemajuan bertahap dan prestasi harian kecil. Jika Anda tidak menjaga diri sendiri, Anda akan segera lelah dan akhirnya menyerah.
  • Pilih pasar yang Anda kenal baik. Saya seorang pengembang dan saya tahu pasar ini dengan baik. Saya tahu apa yang diperlukan untuk menjadi pengembang dan bagaimana tim pengembangan bekerja bersama. Ini memberi saya gambaran tentang apa yang akan dan tidak akan berhasil di pasar ini. Tentu saja, saya masih bisa salah, tetapi kemungkinan kesalahannya jauh lebih sedikit.
  • Ceritakan tentang proyek Anda kepada kami. Ini adalah tugas yang sulit bagi saya, dan saya masih menyesuaikannya. Saya tidak terlalu suka berbicara tentang diri saya sendiri. Saya lebih suka mendengarkan. Tidak mudah bagi saya untuk berbicara tentang proyek yang saya buat, karena saya sedikit pemalu dan tidak ingin menciptakan kesan bahwa saya terus-menerus berbicara tentang diri saya. Namun, saya tahu bahwa saya perlu bicara dan menjual proyek saya. Ini adalah cara orang lain mengetahui tentang produk saya. Artikel ini adalah contohnya.

Kesimpulannya
