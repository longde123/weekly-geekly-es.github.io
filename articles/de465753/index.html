Vergleich von Komparatoren zum Synchronisieren von Schemas und Datenbankdaten von MS SQL Server

Beschreibung des allgemeinen Bedarfs an Änderungssynchronisation


Wenn Sie mit Datenbanken arbeiten, müssen Sie häufig das Problem der Synchronisierung von Änderungen lösen.
Wenn das Unternehmen nur eine industrielle Umgebung verwendet, ist zusätzlich zu dieser mindestens eine zusätzliche Umgebung erforderlich, um die vorgenommenen Änderungen zu testen. In diesem Fall müssen Änderungen von der Testumgebung in die industrielle Umgebung übertragen werden.

Mit dem Wachstum des Unternehmens kann die Anzahl der erforderlichen Server und virtuellen Umgebungen zunehmen, und die Anzahl der Instanzen von Datenbankverwaltungssystemen kann ebenfalls zunehmen. Dann ist eine komplexere Synchronisierung erforderlich.

Datenbanken werden in der Regel in einer speziellen Umgebung entwickelt. Anschließend werden die Änderungen für verschiedene Tests (Komponententests, Autotests, Belastungstests usw.) in die Testumgebung übertragen. Und erst nach allen Entwicklungs- und Inspektionsphasen werden Änderungen in das industrielle Umfeld übertragen.

Bei einer ausreichend großen Infrastruktur müssen Änderungen gleichzeitig auf mehrere Industrieumgebungen übertragen werden. Darüber hinaus kann die Datenbankentwicklung nach Funktionen in verschiedene Umgebungen unterteilt werden. Anschließend müssen regelmäßig Änderungen zwischen diesen Umgebungen übertragen werden.

Es gibt Fälle, in denen Änderungen außerhalb der Entwicklungsumgebung angewendet wurden und auf diese übertragen werden müssen. Solche Situationen müssen jedoch minimiert werden, damit der ordnungsgemäße Betrieb des Systems nicht gefährdet wird.


Im Allgemeinen kann die Reihenfolge der Übertragung von Änderungen wie folgt dargestellt werden:
  1. Zwischen Entwicklungsumgebungen.
  2. Von Entwicklungsumgebungen zu Testumgebungen.
  3. Zwischen Testumgebungen. Wenn Sie Änderungen an einer der Testumgebungen vornehmen, die nicht von der Entwicklungsumgebung stammen, werden diese Änderungen auch auf die Entwicklungsumgebung übertragen. In der Regel sollten alle Änderungen von der Entwicklungsumgebung in die Testumgebung erfolgen, nicht jedoch umgekehrt.
  4. Von der Testumgebung bis zur vorindustriellen Umgebung, in der Änderungen für jede spezifische industrielle Umgebung erfasst werden, werden die neuesten Integrationstests durchgeführt sowie Tests zum Zurücksetzen von Änderungen bei vordefinierten inszenierten Fehlern.
  5. Vom vorindustriellen Umfeld zum industriellen Umfeld mit der Möglichkeit, die vorgenommenen Änderungen schnell rückgängig zu machen.

Es gibt verschiedene Tools zum Synchronisieren von Änderungen zwischen Umgebungen, zum Beispiel:
  1. Visual Studio-Vergleicher
  2. SQL Server Integration Services
  3. dbForge Comparer von Devart
  4. Quest Software ApexSQL
  5. RedGate-Lösungen

In der Regel funktionieren in .NET geschriebene Lösungen (z. B. in C #) mit MS SQL Server.

In diesem Artikel wird ein Beispiel für die Übertragung von Änderungen an einer MS SQL Server-Datenbank von einem Server auf einen anderen mithilfe von Tools von Devart, Quest Software und RedGate erläutert. Am Ende wird ein Vergleich dieser Werkzeuge durchgeführt.

In diesem Artikel wird beispielsweise die SRV-Datenbank betrachtet, die für das MS SQL Server-DBMS ausgelegt ist. Sie können die Quellen hier herunterladen.
Diese SRV-Datenbank wird für jeden Zweck frei verteilt.

Beachten Sie auch die folgenden Funktionen der oben beschriebenen Tools:
  1. Die Quelle für den Vergleich kann sein:
    • Datenbank
    • Schnappschuss
    • Skriptordner
    • Backup
    • Versionskontrollsystem
  2. Sie können mit diesen Tools über die Befehlszeile arbeiten
  3. Die Synchronisation kann direkt durchgeführt werden oder ein Synchronisationsskript kann zur späteren Ausführung gespeichert werden. Sie können auch als Fledermausdatei speichern

Die offizielle Website des ApexSQL Diff-Tools bietet einen Vergleich dieses Programms mit dem SQL Compare-Tool.

In ähnlicher Weise wird für das ApexSQL Data Diff-Tool ein Vergleich mit dem SQL Data Compare-Tool bereitgestellt.

Aus den obigen Vergleichen ist leicht ersichtlich, dass es in ApexSQL etwas gibt, das nicht in Compare enthalten ist, und etwas im Gegenteil in Compare, aber nicht in ApexSQL. Alle betrachteten Dienstprogramme verfügen jedoch über die erforderlichen Grundfunktionen, um sowohl die Schemas als auch die Datenbankdaten selbst zu vergleichen.

In Bezug auf Funktionalität und Preis sind ApexSQL Diff- und ApexSQL Data Diff-Produkte unter Berücksichtigung von Rabatten und des oben diskutierten Materials die beste Wahl.

Fazit


Beliebte Dienstprogramme zum Synchronisieren von Datenbankschemata und -daten wurden berücksichtigt:
  1. dbForge Schema Compare für SQL Server von Devart
  2. dbForge Data Compare für SQL Server von Devart
  3. Quest Software ApexSQL Diff
  4. ApexSQL Data Diff von Quest Software
  5. RedGate SQL Compare
  6. RedGate SQL Data Compare


Ein kurzer Vergleich wurde auch über den Preis und die Funktionalität dieser Produkte durchgeführt.

Das Ergebnis der Überprüfung war die Schlussfolgerung, dass ApexSQL Diff und ApexSQL Data Diff von Quest Software die beste Wahl in Bezug auf Preis und Funktionalität sind. Die grundlegenden Funktionen sind jedoch auch in Devart-Produkten ausreichend verfügbar: dbForge Schema Compare für SQL Server und dbForge Data Compare für SQL Server. Diese Produkte können viel billiger als ihre Gegenstücke ApexSQL Diff und ApexSQL Data Diff von Quest Software gekauft werden.
