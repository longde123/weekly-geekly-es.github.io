Ahora oficialmente: TLS 1.3 reconocido como estándar

Anteriormente escribimos que el Consejo de Ingeniería de Internet (IETF) aprobó una nueva versión de TLS - 1.3. La semana pasada, el protocolo fue reconocido como un estándar. Hoy, hablemos de sus capacidades.


/ foto Charles Dyer CC

Características de TLS 1.3


Comenzaron a trabajar en la actualización del protocolo en 2014. Extraoficialmente, el trabajo en TLS 1.3 finalizó en marzo de este año, pero los ingenieros necesitaron unos meses más para realizar verificaciones adicionales.

Los creadores afirman que la versión final de TLS 1.3 es más segura y productiva: todas las vulnerabilidades conocidas (hoy) de TLS 1.2 están cerradas en sus algoritmos de cifrado, y el proceso de "apretón de manos" es el doble de rápido que su predecesor. Los desarrolladores también agregaron confidencialidad y nuevas características como 0-RTT.

TLS 1.3 realizó el mayor número de cambios significativos en la historia del protocolo. Por esta razón, algunos incluso sugirieron llamarlo TLS 2.0.

Ahora que la nueva versión del protocolo TLS 1.3 ( RFC 8446 ) está oficialmente aprobada, queda por implementarlo para todas las conexiones de red.

Dificultades de implementación


TLS tiene una especie de compatibilidad con versiones anteriores. Cuando se establece una conexión entre el cliente y el servidor, se intercambian las versiones compatibles del protocolo y se selecciona aquella con la que ambas partes pueden trabajar. Sin embargo, esta característica no se usa en todas partes. Con la llegada de TLS 1.3, más del 3% de los servidores con soporte TLS 1.2 simplemente se desconectaron en lugar de enviar al cliente el número de versión que era compatible.

Se ha producido un problema similar con los nodos intermedios ( middlebox ). Debido al hecho de que TLS no cambió mucho, cosas como firewalls, NAT y equilibradores de carga se negaron a trabajar con la nueva versión del protocolo.

Los ingenieros denominaron este fenómeno osificación (osificación). El hecho de que algunos desarrolladores no utilicen las características flexibles del protocolo dificulta la introducción de nuevas implementaciones del mismo. Como analogía, los participantes de la industria citan el ejemplo de la vieja puerta. Si no lo toca durante mucho tiempo, los bucles se oxidan y se abre con un crujido.


/ foto Christopher Sessums CC

Resulta que el protocolo anterior está desactualizado, pero no funcionará introducir uno nuevo por defecto. Se puede encontrar más información sobre el tema, por ejemplo, en el estudio del año pasado de IEEE ( PDF ).

La solución fue encontrada por David Benjamin, trabajando en Chromium. Sugirió disfrazar el primer mensaje de un cliente que admite TLS 1.3 como un mensaje TLS 1.2. Y funcionó: el 3% mencionado de los servidores dejó de desconectarse. Para los sitios intermedios, Kyle Nekritz de Facebook sugirió usar el mismo enfoque. Esto redujo el número de bloqueos en un 6.5% en Chrome y en un 2% en Firefox.

Para verificar si los middleboxes son compatibles con la nueva versión del protocolo, puede usar la prueba desarrollada en Cloudflare.

Cómo simplificar la implementación


Según Eric Rescorla, uno de los desarrolladores de especificaciones para TLS y HTTPS, en general, implementar TLS 1.3 no es tan difícil. El consejo de ingeniería trató de hacer este proceso lo más simple posible. TLS 1.3 utiliza las mismas claves y certificados que TLS 1.2. Esto permite que el cliente y el servidor establezcan automáticamente una conexión a través de TLS 1.3 si ambos admiten la nueva versión del protocolo.

Además, hay varias bibliotecas que pueden ayudarlo a implementar el protocolo más rápido. Por ejemplo, a principios de la semana pasada, Facebook transfirió su biblioteca TLS 1.3 Fizz a código abierto . Fizz reduce la latencia al transmitir datos, así como la carga en la CPU.

Los desarrolladores han preparado una guía sobre cómo comenzar a usar Fizz en Ubuntu 16.04 LTS. Está en el repositorio oficial en GitHub (también tiene una guía para MacOS).

Primero necesita instalar las dependencias necesarias de locura y libsodium : 

sudo apt-get install \ g++ \ cmake \ libboost-all-dev \ libevent-dev \ libdouble-conversion-dev \ libgoogle-glog-dev \ libgflags-dev \ libiberty-dev \ liblz4-dev \ liblzma-dev \ libsnappy-dev \ make \ zlib1g-dev \ binutils-dev \ libjemalloc-dev \ libssl-dev \ pkg-config \ libsodium-dev

A continuación, debe compilar e instalar locura: 

 git clone https://github.com/facebook/folly mkdir folly/build_ && cd folly/build_ cmake configure .. make -j $(nproc) sudo make install

Luego puede proceder a instalar Fizz: 

 cd ../.. git clone https://github.com/facebookincubator/fizz mkdir fizz/build_ && cd fizz/build_ cmake configure ../fizz make -j $(nproc) sudo make install

Además de Fizz, hay otras bibliotecas en la red, por ejemplo, wolfSSL , GnuTLS o susurros .

Protocolo futuro


Para finalmente resolver el problema con la "osificación" del protocolo, David Benjamin propuso, además de la versión oficial de la norma, usar varias de sus variaciones, que se lanzarán cada seis semanas (junto con el lanzamiento de nuevas versiones de Chrome). Por lo tanto, los servidores y los nodos intermedios deberán cumplir con todas las reglas para establecer una conexión; de lo contrario, la mayoría de los clientes no podrán conectarse a los servicios.

Debido a esto, los desarrolladores esperan evitar posibles bloqueos al cargar páginas web, así como problemas similares con futuras versiones de TLS.

Se espera que la seguridad general de la red aumente significativamente después de la introducción de TLS 1.3. Las bibliotecas que faciliten el despliegue de una nueva versión del protocolo deberán contribuir a la distribución masiva.
