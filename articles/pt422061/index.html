Servidor compacto com recarga de código personalizada e logs sql por uma equipe preparada para desenvolvimento produtivo

Eu gostaria de compartilhar a experiência de usar o payara-micro no desenvolvimento de aplicativos corporativos. Espero que alguém economize tempo, pois essa decisão não chegou imediatamente. Se você já está usando o Payara ou o Glassfish como servidor industrial ou está apenas mergulhando no mundo do javaee, então payara-micro e este artigo é para você.


Você precisará de um aplicativo da Web, que é montado através do maven em um arquivo de guerra e do Java 8 (não foi verificado nos mais antigos).


Primeiro, darei o texto completo do perfil, após o qual o analisaremos em pedaços e adicionaremos os detalhes ausentes que não são visíveis no código-fonte.

A primeira coisa a fazer é baixar o próprio arquivo payara-micro. É exatamente isso que o primeiro plug-in faz ( maven-dependency-plugin ). O archive é um servidor de aplicativos completo que pode ser iniciado a partir da linha de comando como um aplicativo java simples.


O último plugin ( payara-micro-maven-plugin ) não possui recursos e está configurado para uma versão estável do payra-micro 4.1.2.173 . Dos recursos do plug-in, a função de empacotar o aplicativo da web em um pacote especial, pronto para execução, é usada aqui. Existem outras maneiras de lançar o payara-micro, mas aqui elas não serão úteis para nós.


Resta gerar a linha de comando desejada e juntar tudo. O exec-maven-plugin é usado para iniciar na linha de comando. Vamos dar uma olhada nos parâmetros da linha de comando: 


 <configuration> <executable>java</executable> <arguments> <!--http://blog.payara.fish/using-hotswapagent-to-speed-up-development--> <argument>-XXaltjvm=dcevm</argument> <argument>-javaagent:hotswap/hotswap-agent.jar</argument> <argument>-agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=9009</argument> <argument>-Duser.language=en</argument> <argument>-Duser.region=US</argument> <argument>-Ddb.ora.url=jdbc:p6spy:oracle:thin:@localhost:1521:XE</argument> <argument>-Ddb.pg.url=jdbc:p6spy:postgresql://localhost:5432/postgres</argument> <argument>-jar</argument> <argument>${project.build.directory}/${project.build.finalName}-microbundle.jar</argument> <argument>--domainConfig</argument> <argument>src/main/setup/domain.xml</argument> <argument>--deploy</argument> <argument>${project.build.directory}/${project.build.finalName}.war</argument> <argument>--rootDir</argument> <argument>${project.build.directory}/payaramicro</argument> </arguments> </configuration>

Os dois primeiros argumentos ( -XXaltjvm = dcevm -javaagent: hotswap / hotswap-agent.jar ) são necessários para alterar o código-fonte sem reiniciar o aplicativo. Graças ao segundo argumento ( -agentlib: jdwp = transport = dt_socket, servidor = y, suspende = n, endereço = 9009 ), lançamos imediatamente o aplicativo no modo de depuração, o que nos permite alterar não apenas o corpo dos métodos por meio da aplicação de alterações de código na depuração no IDE , mas também adicione novos campos e muito mais. Você só precisa se conectar ao depurador a partir do IDE, compilar o arquivo de origem modificado e aplicar as alterações.


Os dois argumentos a seguir ( -Duser.language = en -Duser.region = US ) são necessários para que o cliente jdbc funcione corretamente com o Oracle. As configurações para conectar-se ao DBMS de teste são passadas pelos argumentos -D . Nos meus aplicativos, uso um link como $ {db.ora.url} para esses parâmetros no arquivo de recurso, que os cria automaticamente na inicialização. A substituição de um arquivo que cria recursos com configurações industriais ou de teste também é implementada através do perfil no maven e sua capacidade de filtrar arquivos de texto.


Os principais argumentos na inicialização indicam o caminho ( --domainConfig src / main / setup / domain.xml ) para o arquivo de configurações do domínio, bem como o caminho para o arquivo war ( --deploy $ {project.build.directory} / $ {project. build.finalName} .war ) a ser implantado no servidor. As configurações de domínio foram obtidas por mim com base em um arquivo padrão dentro do arquivo morto. Vou dar o conteúdo como um exemplo. Para uso no desenvolvimento dessas configurações é mais do que suficiente.

Você pode instalar o dcevm aqui . Certifique-se de instalá-lo como uma JVM alternativa, para que você sempre tenha a oportunidade de trabalhar em um padrão regular.


Você pode fazer o download da versão mais recente do agente para modificação ilimitada do código-fonte, bem como recarregar o código dentro de estruturas como o hibernar a partir daqui .


As idéias para configurar o recarregamento de código para payara vieram daqui . Mas a verdade é apenas a ideia em si, já que a implementação do payara-micro não é considerada lá.


Graças à biblioteca p6spy, você pode encontrar seus logs sql na pasta raiz do projeto - o arquivo spy.log. A biblioteca p6spy pode ser personalizada, se necessário, mas este é o tópico de um artigo completamente diferente.


Depois de colocar o perfil descrito acima em seu pom.xml e os arquivos adicionais discutidos acima nas pastas apropriadas, você pode iniciar seu aplicativo com o seguinte comando: 


 mvn install exec:exec -P micro

E nenhum plug-in adicional para o servidor de aplicativos no IDE é necessário ...
