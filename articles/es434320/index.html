Cómo probamos Sberbank Online en iOS



En el artículo anterior, nos familiarizamos con la pirámide de pruebas y los beneficios de las pruebas automatizadas. Pero la teoría suele ser diferente de la práctica. Hoy queremos hablar sobre nuestra experiencia en probar el código de aplicación utilizado por millones de usuarios de iOS. Y también sobre el difícil camino que nuestro equipo tuvo que recorrer para lograr un código estable.

La situación es la siguiente: supongamos que los desarrolladores lograron convencerse a sí mismos y al negocio de la necesidad de cubrir la base del código con pruebas. Con el tiempo, el proyecto se ha convertido en más de una docena de miles de unidades y más de mil pruebas de IU. Una base de prueba tan grande dio lugar a varios problemas, cuya solución queremos contar.

En la primera parte del artículo nos familiarizaremos con las dificultades que surgen al trabajar con pruebas unitarias limpias (sin integración), en la segunda parte consideraremos las pruebas de IU. Para saber cómo estamos mejorando la estabilidad de las pruebas, bienvenido a Cat.

En un mundo ideal, con un código fuente sin cambios, las pruebas unitarias siempre deben mostrar el mismo resultado, independientemente del número y la secuencia de los inicios. Y las pruebas de caída constante no deben pasar por la barrera del servidor de integración continua (CI).


En realidad, uno puede encontrar el hecho de que la misma prueba unitaria mostrará un resultado positivo o negativo, lo que significa "parpadeo". La razón de este comportamiento radica en la implementación deficiente del código de prueba. Además, dicha prueba puede pasar CI con una ejecución exitosa, y más tarde comenzará a recaer en la solicitud de extracción (PR) de otras personas. En una situación similar, existe el deseo de deshabilitar esta prueba o jugar a la ruleta y ejecutar la ejecución de CI nuevamente. Sin embargo, este enfoque es antiproductivo, ya que socava la credibilidad de las pruebas y carga a CI con un trabajo sin sentido.

Este tema se destacó este año en la conferencia internacional WWDC de Apple:

  • Esta sesión habla sobre pruebas paralelas, análisis de la cobertura de un código objetivo individual con pruebas y también sobre el orden de lanzamiento de las pruebas.
  • Aquí Apple habló sobre probar solicitudes de red, piratería, notificaciones de prueba y la velocidad de las pruebas.

Pruebas unitarias


Para combatir las pruebas de parpadeo, utilizamos la siguiente secuencia de acciones:



0. Evaluamos el código de prueba de calidad de acuerdo con criterios básicos: aislamiento, corrección de moka, etc. Seguimos la regla: con una prueba parpadeante, cambiamos el código de prueba y no el código de prueba.

Si este elemento no ayuda, proceda de la siguiente manera:

1. Arreglamos y reproducimos las condiciones bajo las cuales cae la prueba;
2. Encuentra la razón por la caída;
3. Cambie el código de prueba o el código de prueba;
4. Vaya al primer paso y verifique si se ha eliminado la causa de la caída.

Jugar otoño


La opción más simple y obvia es ejecutar una prueba de problema en la misma versión de iOS y en el mismo dispositivo. Como regla general, en este caso la prueba es exitosa y aparece la idea: "Todo funciona para mí localmente, reiniciaré el ensamblaje en CI". Eso es, de hecho, el problema no se ha resuelto, y la prueba continúa cayendo con otra persona.

Por lo tanto, en el siguiente paso de verificación, debe ejecutar localmente todas las pruebas unitarias de la aplicación para identificar el impacto potencial de una prueba en otra. Pero incluso después de dicha verificación, el resultado de su prueba puede ser positivo, pero el problema permanece sin ser detectado.

Si toda la secuencia de prueba fue exitosa y no se registró la caída esperada, puede repetir la ejecución un número significativo de veces.
Para hacer esto, en la línea de comando, debe ejecutar un bucle con xcodebuild: 

#! /bin/sh x=0 while [ $x -le 100 ]; do xcodebuild -configuration Debug -scheme "TargetScheme" -workspace App.wcworkspace -sdk iphonesimulator -destination "platfrom=iOS Simulator, OS=11.3, name=iPhone 7" test >> "report.txt"; x=$(( $x +1 )); done

Como regla, esto es suficiente para reproducir la caída y pasar al siguiente paso: identificar la causa de la caída registrada.

Motivos de la caída y posibles soluciones.


Considere las causas principales de las pruebas unitarias intermitentes que puede encontrar en su trabajo, las herramientas para identificarlas y las posibles soluciones.

Hay tres grupos principales de razones para la caída de las pruebas:

Pobre aislamiento

Por aislamiento queremos decir un caso especial de encapsulación, a saber: un mecanismo de lenguaje que permite restringir el acceso de algunos componentes del programa a otros.

El aislamiento del entorno juega un papel importante, ya que para la pureza de la prueba, nada debería afectar a las entidades evaluadas. Se debe prestar especial atención a las pruebas que tienen como objetivo verificar el código. Utilizan entidades de estado globales como variables globales, llavero, red, CoreData, Singleton, NSUserDefaults, etc. Es en estas áreas donde surge el mayor número de lugares potenciales para la manifestación de un aislamiento deficiente. Supongamos que, al crear un entorno de prueba, se establece un estado global, que se usa implícitamente en otro código de prueba. En este caso, la prueba que verifica el código bajo prueba puede comenzar a "parpadear", porque, dependiendo de la secuencia de pruebas, pueden surgir dos situaciones, cuando el estado global está configurado y cuando no está configurado. A menudo, las dependencias descritas son implícitas, por lo que puede olvidarse accidentalmente de configurar / restablecer dichos estados globales.

Para que las dependencias sean claramente visibles, puede usar el principio de Inyección de dependencias (DI), a saber: pasar la dependencia a través de los parámetros del constructor o la propiedad del objeto. Esto facilitará la sustitución de dependencias simuladas en lugar de un objeto real.

Llamada asincronía

Todas las pruebas unitarias se realizan sincrónicamente. La dificultad de probar la asincronía surge porque la llamada del método de prueba en la prueba se "congela" en anticipación de la finalización del alcance de la prueba unitaria. El resultado será una caída estable en la prueba. 

 //act [self.testService loadImageFromUrl:@"www.google.ru" handler:^(UIImage * _Nullable image, NSError * _Nullable error) { //assert OCMVerify([cacheMock imageAtPath:OCMOCK_ANY]); OCMVerify([cacheMock dateOfFileAtPath:OCMOCK_ANY]); OCMVerify([imageMock new]); [imageMock stopMocking]; }]; [self waitInterval:0.2];

Para probar dicha prueba, hay varios enfoques:

  1. Ejecute NSRunLoop
  2. waitForExpectationsWithTimeout

Ambas opciones requieren que especifique un argumento con un tiempo de espera. Sin embargo, no se puede garantizar que el intervalo seleccionado sea suficiente. A nivel local, su prueba pasará, pero en un CI con mucha carga puede que no haya suficiente energía y se caerá; desde aquí aparecerá un "parpadeo".

Tengamos algún tipo de servicio de procesamiento de datos. Queremos verificar que después de recibir una respuesta del servidor, transfiera estos datos para su posterior procesamiento.

Para enviar solicitudes a través de la red, el servicio utiliza el cliente para trabajar con él.

Dicha prueba se puede escribir de forma asincrónica utilizando un servidor simulado para garantizar respuestas de red estables. 

 @interface Service : NSObject @property (nonatomic, strong) id<APIClient> apiClient; @end @protocol APIClient <NSObject> - (void)getDataWithCompletion:(void (^)(id responseJSONData))completion; @end - (void)testRequestAsync { // arrange __auto_type service = [Service new]; service.apiClient = [APIClient new]; XCTestExpectation *expectation = [self expectationWithDescription:@"Request"]; // act id receivedData = nil; [self.service receiveDataWithCompletion:^(id responseJSONData) { receivedData = responseJSONData; [expectation fulfill]; }]; [self waitForExpectationsWithTimeout:10 handler:^(NSError * _Nullable error) { expect(receivedData).notTo.beNil(); expect(error).to.beNil(); }]; }

Pero la versión síncrona de la prueba será más estable y le permitirá deshacerse de los tiempos de espera.

Para él necesitamos un simulacro síncrono APIClient 

 @interface APIClientMock : NSObject <APIClient> @end @implementation - (void)getDataWithCompletion:(void (^)(id responseJSONData))completion { __auto_type fakeData = @{ @"key" : @"value" }; if (completion != nil) { completion(fakeData); } } @end

Entonces la prueba se verá más simple y funcionará más estable 

 - (void)testRequestSync { // arrange __auto_type service = [Service new]; service.apiClient = [APIClientMock new]; // act id receivedData = nil; [self.service receiveDataWithCompletion:^(id responseJSONData) { receivedData = responseJSONData; }]; expect(receivedData).notTo.beNil(); expect(error).to.beNil(); }

La operación asincrónica se puede aislar encapsulando una entidad separada, que se puede probar de forma independiente. El resto de la lógica necesita ser probada sincrónicamente. Este enfoque evitará la mayoría de los escollos provocados por la asincronía.

Como opción, en el caso de actualizar la capa de interfaz de usuario desde el subproceso de fondo, puede verificar si estamos en el subproceso principal y qué sucederá si realizamos una llamada desde la prueba: 

 func performUIUpdate(using closure: @escaping () -> Void) { // If we are already on the main thread, execute the closure directly if Thread.isMainThread { closure() } else { DispatchQueue.main.async(execute: closure) } }

Prueba de código más allá de su control
A menudo nos olvidamos de las siguientes cosas:

  • la implementación de los métodos puede depender de la localización de la aplicación,
  • Hay métodos privados en el SDK que pueden ser llamados por las clases marco,
  • La implementación de los métodos puede depender de la versión del SDK


Los casos anteriores introducen incertidumbre al escribir y ejecutar pruebas. Para evitar consecuencias negativas, debe ejecutar pruebas en todas las configuraciones regionales, así como en las versiones de iOS compatibles con su aplicación. Por separado, debe tenerse en cuenta que no es necesario probar el código cuya implementación está oculta para usted.
