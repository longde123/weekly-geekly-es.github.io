DevOps会议DevOops 2018的公告


自从每个人听到“ DevOps”一词以来，至少已经过去了几年。 谁没有实施它，什么才没有实施。


同时，该地区尚未开发，充满了许多发现。 例如，俄语社区仍未决定术语：有人已经雇用了“ devops”这个职位，有人总是说“ devops”是一种文化和实践，旨在将开发，运营和开发结合在一起。因此，也称这个职位不正确。


许多人正在寻找书籍中的答案，因为最近有很多书。 例如，在我看来，最重要的一本是由我们的演讲者约翰·威利斯（John Willis）撰写的《 Devops手册》和《 Google SRE Book》，该书可在互联网上免费获得。 但是，阅读这些书后，我发现以下内容：枯燥乏味的文字不太适合知识的传播，而很大程度上是基于在职人员的真实工作。 事实证明，知识太抽象了。


例如，我们采用第14章“管理事件” 。 我们举两个例子：首先，一个事件的故事被错误地处理了。 然后讲同样的故事，但结构正确且结果良好。 如果遵循以下重要做法，将会产生良好的结果：


  • 明确划分角色，并分配负责人：
    • 整个事件（“指挥官”）；
    • 操作部分；
    • 沟通交流
    • 工作计划；
  • 突出团队帖子（包括物理聊天和聊天）；
  • 不断更新的文档，描述了事件的当前状态；
  • 及时且可理解的权限转移（例如，轮班结束时）。

最后，对世界上的所有事物都给出了很多好的建议。 从原则上讲，这很好，但是有一个问题：如何将其付诸实践？ 每个项目都适用于整本书，其中一些要求具备无法在书籍中描述的软技能。 想象一下，在事件的中间，CTO会来找您，并开始提供过去IT生涯的无用建议：它会增加Linux中的内存页面大小（尽管完全不同），或者关闭ext4障碍（尽管打开了缓存）。 以他在团队中没有任何角色为借口，将他踢出我们指挥所的屁股容易吗？ 撰写本文的人是如何做到的？


理想情况下，我想要以下几点：首先，对于同一问题要有不止一种观点，包括从各个团队的经验中得出的观点。 并非所有公司都像Google。 即使如此：类似Google的公司也可以指望。 其次，我想在现场与这些神圣知识的作者见面，注视我的眼睛并提出一些问题。 例如，许多赞美文件中关于其组织中的出色发展的作家都是卑鄙的谎言，但实际上，他们的bash脚本和棍子被粘在电气胶带上。 看着眼睛非常有用。 我真的不仅希望获得一些一般性的建议，而且要问自己的棘手问题-并获得答案。


小观点并不能以任何方式解决许多观点的问题和各种问题。 在书籍的帮助下，您无法深入挖掘并心连心交谈。 没有人可以肯定地决定诸如实用术语之类的问题；空缺的构成取决于情况。 为了获得相关和有用的知识，您可以并且应该同时使用所有资源。


去年，我们意识到一切都令人困惑，是时候该组织一次关于DevOps的大型会议了，只有他一次。 它被称为DevOops ，发生在秋天的圣彼得堡。 下次它将在今年10月14日举行。


大型会议正是解决大多数上述问题的方法。 例如，如果您误解了约翰·威利斯的书中的内容，您不仅可以阅读他的报告并更详细地了解该主题，还可以在讨论区与他会面并直接提问。


仅关于DevOps


首先，其特点是该会议仅涉及DevOps。 原则上，在俄罗斯大多数主要的IT会议上，现在都有几个令人关注的话题。 如果您立即参加一系列会议，您将获得良好的基础。 但是他们将不得不倾听大量Java，.NET，JavaScript程序员等已经变得痛苦的一切，而且通常无济于事。 但是，这一切都太长了，而且太昂贵了。 DevOops会议仅关注DevOps，从而解决了许多烦人的组织问题。


他们将讨论容器及其编排，虚拟化和云，监视和审核，CI和CD，以及有关“ DevOps”一词时通常想到的所有内容。


演讲者


但是最重​​要的是发言人。 在会议宣布之时，已经有来自Google和Microsoft等公司的九个人准备分享他们的经验。 最后，该计划将在三个方面提供约17个报告。 也许会有更多的跟踪和报告。 我们仔细研究了您对先前DevOops的反馈，并尝试邀请您最想要的人。 让我们看看谁已经和我们在一起了。


约翰·威利斯


不可能用言语表达他来找我们有多酷。 John是DevOps的几位父亲之一，在过去的20年中出版了10本书，其中包括著名的DevOps手册和Beyond the Fenix Project ，这是一个运行了35年的大师，只是一个活着的传奇。

他的报告称为“ 使用微服务和云的现代安全性” 。 很难高估安全性在微服务应用程序中的重要性，这使Seth的报告特别重要。 该报告将描述基于微服务的现代系统中安全性的基本原理和最佳实践，还将以Vault为例进行实时演示。

Alena是Rancher Labs的首席软件工程师（是的，这些人都是Rancher的创始人，他的口号听起来像“ Kubernetes Everywhere”）和Apache Software Foundation的项目管理委员会。 以前，我曾在CloudStack项目中为虚拟机构建基础结构服务，现在，正如您可能猜到的，我为重点是Kubernetes的容器构建了服务。 这个人不仅了解Kubernetes的所有知识，而且还可以谈论他，在会议上占据重要位置。
