(Non) proyek komersial: Redis mengubah lisensi tetapi tetap open source

Pada paruh kedua Agustus, sebuah posting muncul di situs web Redis Labs, para pengembang gudang data jurnal jaringan eponymous, yang mengatakan tentang perubahan dalam kebijakan perizinan proyek. Di internet, sebuah pendapat segera muncul bahwa mereka berencana menjadikan Redis produk berpemilik.

Alarm ternyata salah. Redis akan tetap menjadi proyek open source dan akan terus didistribusikan di bawah lisensi BSD . Namun, perusahaan pengembang masih mengubah lisensi untuk sejumlah modul dari GNU AGPL menjadi Commons Clause .

Kami memahami apa artinya ini bagi Redis dan berbicara tentang masalah komersialisasi proyek sumber terbuka secara umum.


/ PxHere / PD

Apa yang terjadi


Setelah pengumuman di situs web Redis tentang perubahan lisensi, komunitas mulai aktif mendiskusikan situasi . Dan di beberapa titik ada kesalahpahaman - beberapa orang memutuskan bahwa Redis mengubah lisensi. Untuk mengklarifikasi sejumlah poin, co-founder CTO dan Redis Labs Yiftach Shoolman menulis sebuah posting resmi atas nama perusahaan tentang apa yang terjadi.

Dikatakan bahwa Redis tetap di bawah lisensi asli - BSD. Pesanan lisensi diubah hanya untuk sejumlah modul Redis Labs, misalnya, RediSearch, Redis Graph, ReJSON, ReBloom dan Redis-ML (ada sekitar selusin dari mereka). Mereka menawarkan fitur-fitur canggih untuk pengguna korporat. Lisensi mereka diubah dari GNU AGPL ke Apache 2.0 Commons Clause.

Menurut ketentuan lisensi baru, siapa pun dapat menggunakan modul ini, mengamati kondisi dasar lisensi perangkat lunak bebas Apache License 2.0. Namun, sekarang dilarang untuk menjual modul - ini mengatur Klausul Commons .

Dengan kata lain, jika suatu aplikasi menggunakan fungsionalitas modul atau dibangun di atasnya, maka tidak ada batasan pada penjualan layanan semacam itu. Tapi sekarang Anda tidak bisa menjual modul aslinya.

Mengapa mengubah lisensi?


Keputusan ini merupakan kompromi dalam memerangi penjualan kembali modul Redis terbuka. Menurut Shulman, sejumlah perusahaan pihak ketiga menggunakan solusi mereka secara gratis dan menjualnya kembali kepada pelanggan mereka, tetapi sama sekali tidak berpartisipasi dalam pengembangan komunitas dan produk.

Kisah serupa terjadi dengan banyak proyek terbuka , misalnya, Hadoop dan Spark - atas dasar mereka, banyak perusahaan membangun bisnis mereka, hanya membuat sedikit modifikasi di dalamnya.

Menurut Redis, mengubah lisensi akan memungkinkan penulis untuk menjamin bahwa pengembang atau sponsor mereka akan menerima uang dari penjualan proyek.

Masalah komersialisasi produk terbuka


Jelas, uang untuk pengembangan proyek sumber terbuka tidak diambil dari udara. Beberapa pengembang menerima subsidi pemerintah , misalnya, NASA membiayai proyek code.nasa.gov . Perusahaan komersial mendukung seseorang. Misalnya, Walmart menghabiskan jumlah yang signifikan untuk membiayai kerangka kerja Hapi.js. Pengecer mendukung pengembang Hapi.js, karena menggunakan solusinya untuk membuat aplikasi sendiri.

Ada juga yang memutuskan untuk menerima uang untuk mengembangkan perangkat lunak open source melalui komersialisasi sebagian produk, seperti Redis Labs. Dan mereka bukan satu-satunya yang mengambil jalan ini - Berkeley DB , Asterisk , Qt dan banyak lainnya yang mengerjakan model ini. Namun, pendapat masyarakat tentang terjemahan proyek-proyek sumber terbuka berdasarkan pembayaran sangat bervariasi.

Pendapat menentang

Keputusan Redis untuk mengubah lisensi untuk beberapa modul yang sebagian besar ditemui oleh masyarakat adalah negatif. Jika Anda melihat utas Berita Hacker dan komentar oleh pengembang di blog tematik, ada sejumlah alasan umum untuk ketidakpuasan.

Alasan pertama adalah bahwa segala bentuk komersialisasi bertentangan dengan prinsip - prinsip open source. Panelis mencatat bahwa produk yang menggunakan lisensi seperti Commons Clause tidak diperbolehkan menjadi open source.

Beberapa anggota komunitas memutuskan bahwa Redis Labs memperdagangkan nilai sumber terbuka untuk mendapatkan uang. Simon Phipps, presiden dari Open Source Initiative (OSI), bahkan menggambarkan Redis Labs sebagai "melepaskan kebebasan perangkat lunak."

Alasan kedua - pengalihan proyek open source atas dasar bayaran mengancam untuk menghancurkan komunitas pengguna, yang telah dibentuk selama bertahun-tahun. Kisah serupa terjadi dengan FoundationDB , ketika pada 2015 itu dibeli oleh Apple. Organisasi membatasi unduhan multi-model DBMS dan menutup semua repositori di GitHub. Akibatnya, komunitas yang berkumpul di sekitar proyek dalam lima tahun mulai hancur.

Ukuran komunitas adalah salah satu aspek kunci dari pemilihan produk bagi banyak perusahaan. Misalnya, baru-baru ini di 1cloud kami memilih sistem logging baru untuk kami sendiri. Pilihan ada pada Redis dan Logstash, karena mereka memiliki komunitas yang luas - ini berarti dokumentasi terperinci, FAQ, dan utas langsung di StackOverflow.

Pendapat untuk

Mitra Andreessen Horowitz Peter Levine menyatakan bahwa setiap proyek open source membutuhkan model bisnis. Jelas bahwa tanpa dana, dia akan "membungkuk". Pengembang mungkin tidak punya cukup uang untuk dukungan dan penskalaan. Lagi pula, berpartisipasi dalam proyek untuk mengembangkan perangkat lunak open source, pada kenyataannya, adalah pekerjaan kedua.

Misalnya, salah satu penghuni Habr menulis sendiri perpustakaan untuk mengatur server soket web dan "mengunggah" ke GitHub . Segera setelah itu, pengguna lain menjadi tertarik pada proyek dan meminta penulis untuk mengimplementasikan fungsionalitas tambahan. Namun, pengembang sendiri sudah cukup dengan apa yang sudah ditulis, dan dia tidak punya keinginan untuk mengembangkan proyek lebih lanjut.

Mungkin mengkomersialkan produk akan membantu pengembangannya. Misalnya, cerita seperti itu terjadi dengan nginx . Penulis terlibat dalam pembuatan server web ini di waktu luangnya. Tetapi sekarang, berdasarkan keputusannya, ia menyediakan layanan komersial untuk pelanggan. Dalam hal ini, kode nginx itu sendiri tetap terbuka .

Orang-orang yang menganjurkan komersialisasi parsial produk terbuka juga mencatat bahwa tanpa pendanaan yang tepat, proyek tidak akan dapat menarik pengembang yang baik.

Untuk alasan ini, pada tahun 2009, proyek Chandler ditutup , di mana mereka mengembangkan perangkat lunak untuk mengelola jadwal. Kurangnya dana dan sejumlah kecil kontributor menyebabkan fakta bahwa minat terhadap program memudar secara harfiah dalam setahun.

Dalam hal ini, kernel Linux dapat digunakan sebagai kasus positif. Pekerjaan banyak pengembang pada proyek ini sepenuhnya atau sebagian didanai oleh perusahaan seperti RedHat.

Dengan demikian, pengembang menempatkan waktu dan energi mereka ke dalam proyek, membuatnya lebih baik untuk yang lain dan dengan demikian memperpanjang umurnya. Pada gilirannya, ini memiliki efek positif pada pengembangan seluruh ekosistem sumber terbuka.
