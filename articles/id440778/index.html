Pertemuan OpenStack kedua di Mail.ru Group: 22 Februari



Halo teman-teman. Jumat ini kami mengumpulkan di Mail.ru @OpenStack Meetup kedua kami, di mana:

  • Rekan Hystax akan menunjukkan betapa sulitnya migrasi mesin virtual antara cloud dan hypervisors dengan contoh kasus klien;
  • Mail.Ru Cloud Solutions akan membagikan pengalaman mereka dengan OpenStack dan Linux Storage Stack: apa yang sering dilupakan saat merencanakan penyimpanan?
  • Canonical akan terus berbicara tentang pendekatan mereka terhadap infrastruktur pemodelan. Kali ini Anda akan belajar bagaimana MAAS dan Juju membuatnya lebih mudah untuk mengotomatisasi bekerja dengan server dengan sejumlah besar antarmuka jaringan dan topologi jaringan yang berorientasi L3 dalam otomatisasi.

Pertemuan akan berlangsung pada 22 Februari (Jumat) pukul 19.00 di kantor Mail.ru Group di Moskow (Leningradsky Prospekt, 39, hal. 79).

Diperlukan pendaftaran dan ditutup pada 20 Februari pukul 23:59 (atau lebih awal jika kursi habis).

Migrasi mesin virtual antara cloud dan hypervisor bisa sulit tidak hanya secara teknis, tetapi juga secara organisasi. Terutama ketika kekuatan ketiga mengganggu proses. Salah satu kasus tersebut akan dijelaskan pada contoh migrasi dari VMware ke OpenStack dari salah satu klien Hystax.

Kami akan beralih ke tumpukan blok Linux, perilaku perangkat blok di lingkungan virtual dan di hypervisor, serta nuansa menerapkan dan menggunakan beberapa jenis penyimpanan blok di OpenStack.

Termasuk kami akan mempertimbangkan topik-topik seperti:

  • Blokir operasi di Linux.
  • Antrian dan Penjadwal (Penjadwal I / O).
  • Cache
  • Konsistensi data dan hubungan konsistensi dan kinerja.
  • I / O dalam QEMU-KVM.
  • Kinerja dan integritas data sistem penyimpanan klasik, SoftRAID dan Ceph.

Saya akan berbicara tentang cara menggunakan pemodelan jaringan di MAAS, Juju dan OpenStack Charms untuk menyederhanakan komisioning cloud OpenStack dengan topologi jaringan yang kompleks. Secara khusus, saya akan mengungkapkan rincian penggunaan beberapa domain broadcast untuk memisahkan lalu lintas jaringan dari berbagai komponen infrastruktur, mengkonfigurasi routing untuk topologi L3, dan juga membahas masalah-masalah penemuan layanan.
