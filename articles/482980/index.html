Oracle mismo copió la API de Amazon S3, y eso está perfectamente bien


Los abogados de Oracle comparan la reimplementación de la API de Java en Android con la copia del contenido de Harry Potter, pdf

A principios de este año, la Corte Suprema de los Estados Unidos considerará un importante caso de Oracle contra Google , que determinará el estado legal de la API de acuerdo con la ley de propiedad intelectual. Si el tribunal se pone del lado de Oracle en su demanda multimillonaria, podría sofocar la competencia y consolidar el dominio de los gigantes de la tecnología, posiblemente incluido el propio Google.

Al mismo tiempo, el negocio de Oracle se basó originalmente en la implementación del lenguaje de programación SQL desarrollado por IBM, e incluso ahora la compañía ofrece un servicio en la nube con API de Amazon S3, y esto es completamente normal. La reimplementación de la API ha sido una parte natural del desarrollo de la informática desde el inicio de la industria.

Oracle acusa a Google de copiar ilegalmente la API de Java, incluida una lista de comandos con nombre vinculados a estructuras gramaticales. El sistema operativo Android es específicamente compatible con la API de Java para facilitar a los programadores de Java la transferencia de software y conocimiento a la nueva plataforma. Para hacer esto, Android copió con precisión los comandos correspondientes de la API de Java y las estructuras gramaticales. El argumento de Oracle es que dicha "reimplementación" de la API de Java se puede comparar con la copia del trabajo de un autor, como la novela literaria Harry Potter (este es un ejemplo real citado por los abogados de Oracle ), y Google viola los derechos de autor de Oracle sobre los nombres de los comandos y estructuras de Java API

Pero la API de Java no es la única API, y Android no es la única reimplementación. Las API son omnipresentes en la industria de TI actual, y la reimplementación es fundamental para mantener la competencia a fin de evitar el monopolio de las grandes empresas, dijo Charles Duan, Director de Política de Tecnología e Innovación del Instituto R Street.

Duan da un ejemplo de la popular plataforma de almacenamiento Amazon S3. Para permitir la escritura y extracción de archivos desde S3, Amazon ha desarrollado una API completa y detallada para interactuar con el servicio. Por ejemplo, para obtener una lista de archivos guardados ( ListObjects ), enviamos un comando GET con el host y los parámetros de tipo encoding-type , continuation-token y x-amz-date . Para trabajar con Amazon S3, el software debe usar estos y muchos otros nombres de parámetros específicos exactamente. 

GET /?Delimiter=Delimiter&EncodingType=EncodingType&Marker=Marker&MaxKeys=MaxKeys&Prefix=Prefix HTTP/1.1 Host: Bucket.s3.amazonaws.com x-amz-request-payer: RequestPayer

Amazon es un claro líder en el mercado de servicios en la nube, y sus competidores ofrecen la reimplementación de la API S3, mientras que tienen que imitar nombres de equipos, etiquetas de parámetros, prefijos x-amz , estructura gramatical y organización general de la API S3. En otras palabras, todo lo que, según Oracle, está protegido por derechos de autor.

Entre las compañías que ofrecen una copia de la API de Amazon S3, está el propio Oracle . Por compatibilidad, la API de compatibilidad de Amazon S3 copia numerosos elementos de la API de Amazon, hasta etiquetas x-amz.



Oracle asegura que la legalidad de sus acciones se basa en la licencia de código abierto Apache 2.0, que permite la copia y modificación gratuitas del código. Por ejemplo, Amazon SDK para Java también viene con una licencia Apache 2.0.

Pero la pregunta es si la ley de propiedad intelectual es generalmente aplicable a objetos como la API. Eso es lo que debe establecer la Corte Suprema.

¿Quién inventó la API?


El término y el concepto de una "biblioteca de subrutinas" apareció por primera vez en el libro de Hermann Goldstein y John von Neumann "Problemas de planificación y codificación para una herramienta de computación electrónica" - Parte II, Volumen III (Instituto de Investigación Avanzada de la Universidad de Princeton, 1948), copia de archivo .org . Esta es la primera descripción de una metodología de programación para computadoras con programas de almacenamiento en memoria (anteriormente no existía). Se extendió ampliamente a las universidades, que en ese momento intentaron crear sus propias computadoras. Y lo más importante, el libro contiene una idea clave: la mayoría de los programas utilizarán operaciones comunes, y las bibliotecas con rutinas reducirán la cantidad de nuevos códigos y errores . Esta idea fue finalizada por Maurice Wilkes y puesta en práctica en la máquina EDSAC, por la cual recibió el Premio Turing de 1967.


La biblioteca de rutina EDSAC está a la izquierda.

El siguiente paso fue crear funciones de orden superior e interfaces de software completas, que Maurice Wilkes y David Wheeler hicieron en el libro "Preparación de programas para una computadora digital electrónica" (1951).

El término Interfaz de programa de aplicación (API) apareció en algún lugar a finales de los años 60.

El autor de la presentación " Una breve historia subjetiva de la API" Joshua Block ofrece varios ejemplos de interfaces de programas, conjuntos de instrucciones y bibliotecas de subprogramas: cómo se crearon y utilizaron más tarde. La idea es que reutilizar es el significado de la API. Es por eso que fueron creados en primer lugar. Copiar y reutilizar API (bibliotecas, conjuntos de instrucciones) no solo es correcto, sino que dicha metodología de programación se recomienda directamente en los cánones de la informática. Incluso antes de copiar las interfaces de programación S3, el propio Oracle lo hizo muchas veces. Además, el negocio de Oracle se basó originalmente en la implementación del lenguaje de programación SQL desarrollado por IBM. El primer producto estrella de Oracle fue el DBMS, en gran parte copiado de IBM System R. En este caso, estamos hablando de la reimplementación de SQL como una "API estándar" para el DBMS.

La superposición de los derechos de propiedad intelectual en las API puede crear un campo de minas legal que todos sufrirán. Las API también implementan otros servicios en la nube . Muchos estándares técnicos, como los protocolos de Internet y Wi-Fi, incluyen API. Las interfaces del programa se implementan necesariamente de alguna forma en cada computadora y servidor en Internet. Oracle Copyright Theory puede convertir casi todo lo que haces con tu computadora en un acto ilegal.

Para evitar estas consecuencias de largo alcance, Oracle y el Tribunal de Apelación, que confirmaron sus argumentos, trataron de limitar la infracción de derechos de autor a solo unas pocas reimplementaciones de API que eran "incompatibles" con el original. Pero las reimplementaciones parciales también son comunes . Incluso en su copia de la API S3, Oracle ha notado numerosas "diferencias" e incompatibilidades con las API originales de Amazon.

El principal peligro de la demanda de Oracle es que podría evitar que las pequeñas empresas de tecnología creen versiones de sistemas compatibles con plataformas dominantes como S3. Sin dicha compatibilidad, los programadores quedarán prácticamente bloqueados en las ofertas de esta empresa.

Los representantes de la industria y los desarrolladores solo pueden esperar que la mente triunfe y los jueces conozcan los conceptos básicos de la programación .
