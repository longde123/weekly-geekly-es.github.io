NB-IoT: bagaimana cara kerjanya? Bagian 3: SCEF - jendela akses tunggal ke layanan operator

Dalam artikel " NB-IoT: Bagaimana Cara Kerjanya?" Bagian 2 â€, berbicara tentang arsitektur inti paket dari jaringan NB-IoT, kami menyebutkan penampilan simpul SCEF baru. Kami menjelaskan di bagian ketiga, apa itu dan mengapa itu dibutuhkan?



Saat membuat layanan M2M, pengembang aplikasi dihadapkan dengan pertanyaan-pertanyaan berikut:

  • Cara mengidentifikasi perangkat
  • Otentikasi dan algoritma otentikasi mana yang digunakan;
  • protokol transport mana yang digunakan untuk berinteraksi dengan perangkat;
  • Bagaimana cara mengirimkan data ke perangkat secara terjamin
  • bagaimana mengatur dan menetapkan aturan untuk bertukar data dengan mereka;
  • cara memantau dan menerima informasi tentang status mereka secara online;
  • cara mengirim data secara bersamaan ke sekelompok perangkat mereka;
  • Cara mengirim data secara bersamaan dari satu perangkat ke beberapa klien;
  • cara mendapatkan akses terpadu ke layanan operator tambahan untuk mengelola perangkat Anda.

Untuk menyelesaikannya, kita harus menciptakan solusi teknis yang â€œberatâ€, yang mengarah pada peningkatan biaya tenaga kerja dan layanan waktu ke pasar. Di sinilah simpul SCEF baru datang untuk menyelamatkan.

Menurut definisi 3GPP, SCEF (fungsi paparan kemampuan layanan) adalah komponen yang sama sekali baru dari arsitektur 3GPP, fungsinya adalah untuk secara aman mengekspos layanan dan kemampuan yang disediakan oleh antarmuka jaringan 3GPP melalui API.

Dengan kata sederhana, SCEF adalah perantara antara jaringan dan server aplikasi (AS), jendela akses tunggal ke layanan operator untuk mengelola perangkat M2M di jaringan NB-IoT melalui API standar intuitif.

SCEF menyembunyikan kompleksitas jaringan operator, memungkinkan pengembang aplikasi untuk abstrak dari mekanisme spesifik dan kompleks untuk berinteraksi dengan perangkat.

Dengan mengubah protokol jaringan menjadi akrab bagi pengembang aplikasi, SCEF API memfasilitasi penciptaan layanan baru dan mengurangi waktu ke pasar. Selain itu, simpul baru mencakup fungsi identifikasi / otentikasi perangkat seluler, menentukan aturan untuk bertukar data antara perangkat dan AS, menghilangkan kebutuhan untuk penerapan fungsi-fungsi ini dari pengembang aplikasi di sisi mereka, mentransfer fungsi-fungsi ini ke bahu operator.

SCEF menutup antarmuka yang diperlukan untuk otentikasi dan otorisasi server aplikasi, menjaga mobilitas UE, transfer data dan pemicu perangkat, akses ke layanan tambahan dan kemampuan jaringan operator.

Menuju ke AS ada satu antarmuka T8 tunggal, sebuah API (HTTP / JSON), 3GPP standar. Semua antarmuka, kecuali T8, beroperasi berdasarkan protokol DIAMETER (Gbr. 1).



T6a adalah antarmuka antara SCEF dan MME. Digunakan untuk prosedur manajemen Mobilitas / Sesi, transmisi data non-IP, penyediaan acara pemantauan dan menerima laporan tentang mereka.

S6t adalah antarmuka antara SCEF dan HSS. Hal ini diperlukan untuk otentikasi pelanggan, otorisasi server aplikasi, memperoleh sekelompok ID eksternal dan IMSI / MSISDN, menyediakan acara pemantauan dan menerima laporan tentang mereka.

S6m / T4 - antarmuka dari SCEF ke HSS dan SMS-C (node â€‹â€‹MTC-IWF didefinisikan dalam 3GPP, yang digunakan untuk memicu perangkat dan mengirim SMS di jaringan NB-IoT. Namun, dalam semua implementasi, fungsi node ini terintegrasi ke dalam SCEF, jadi untuk penyederhanaan skema, kami tidak akan mempertimbangkannya secara terpisah). Digunakan untuk memperoleh informasi perutean untuk mengirim SMS dan berinteraksi dengan pusat SMS.

T8 - SCEF API untuk interaksi dengan server aplikasi. Perintah kontrol dan traffic ditransmisikan melalui antarmuka ini.

* Pada kenyataannya, ada lebih banyak antarmuka, hanya yang paling dasar yang terdaftar di sini. Daftar lengkap diberikan dalam 3GPP 23.682 (4.3.2 Daftar Poin Referensi).

Di bawah ini adalah fitur dan layanan utama SCEF:

  • mengikat pengenal kartu SIM (IMSI) ke ID eksternal;
  • pengiriman lalu lintas non-IP (Pengiriman Data Non-IP, NIDD);
  • operasi grup menggunakan ID grup eksternal;
  • dukungan untuk mode transfer data dengan konfirmasi;
  • buffering data MO (Mobile Originated) dan MT (Mobile Terminated);
  • otentikasi dan otorisasi perangkat dan server aplikasi;
  • penggunaan simultan data satu UE oleh beberapa AS;
  • dukungan fungsi khusus untuk memonitor status UE (MONTE - Acara Pemantauan);
  • memicu perangkat;
  • Menyediakan roaming data non-IP.

Prinsip dasar interaksi antara AS dan SCEF dibangun di atas apa yang disebut langganan. Jika Anda perlu mengakses layanan SCEF untuk UE tertentu, server aplikasi harus membuat langganan dengan mengirimkan perintah ke API spesifik dari layanan yang diminta dan sebagai tanggapan untuk menerima pengidentifikasi unik. Setelah itu, semua tindakan dan komunikasi lebih lanjut dengan UE dalam kerangka layanan ini akan dilakukan menggunakan pengenal ini.

ID eksternal: pengidentifikasi perangkat universal

Salah satu perubahan paling penting dalam skema interaksi antara AS dan perangkat saat bekerja melalui SCEF adalah munculnya pengidentifikasi universal. Sekarang, alih-alih nomor telepon (MSISDN) atau alamat IP, seperti di jaringan klasik 2G / 3G / LTE, pengidentifikasi perangkat untuk server aplikasi menjadi "ID eksternal". Ini didefinisikan oleh standar dalam format yang biasa bagi pengembang aplikasi "<Local Identifier> @ <Domain Identifier>".

Pengembang tidak perlu lagi mengimplementasikan algoritma otentikasi perangkat, jaringan sepenuhnya mengambil fungsi ini atas dirinya sendiri. ID eksternal dilampirkan ke IMSI, dan pengembang dapat yakin bahwa mengacu pada ID eksternal tertentu, ia berinteraksi dengan kartu SIM tertentu. Saat menggunakan chip SIM, situasi unik umumnya diperoleh ketika ID eksternal secara unik mengidentifikasi perangkat tertentu!

Selain itu, beberapa ID eksternal dapat dilampirkan ke satu IMSI - situasi yang lebih menarik diperoleh ketika ID eksternal secara unik mengidentifikasi aplikasi tertentu yang bertanggung jawab untuk layanan tertentu pada perangkat tertentu.

Pengidentifikasi grup juga akan muncul - ID grup eksternal, yang mencakup serangkaian ID eksternal yang terpisah. Sekarang, dengan satu permintaan ke SCEF, AS dapat memulai operasi grup - mengirim data atau perintah kontrol ke beberapa perangkat yang digabungkan menjadi satu grup logis.

Karena kenyataan bahwa untuk pengembang AS, transisi ke pengidentifikasi perangkat baru tidak dapat instan, SCEF meninggalkan kemungkinan komunikasi AS dengan UE melalui nomor standar - MSISDN.

Pengiriman Data Non-IP (NIDD)

Dalam NB-IoT, sebagai bagian dari mengoptimalkan mekanisme untuk mentransfer sejumlah kecil data, di samping jenis PDN yang ada, seperti IPv4, IPv6 dan IPv4v6, jenis lain muncul - non-IP. Dalam hal ini, perangkat (UE) tidak diberi alamat IP, dan data dikirimkan tanpa menggunakan protokol IP. Lalu lintas untuk koneksi seperti itu dapat dialihkan dengan dua cara: klasik - MME -> SGW -> PGW dan kemudian melalui terowongan PtP ke AS (Gambar 2) atau menggunakan SCEF (Gambar 3).



Metode klasik tidak memiliki keunggulan khusus atas lalu lintas IP, kecuali untuk mengurangi ukuran paket yang ditransmisikan karena tidak adanya header IP. Menggunakan SCEF membuka sejumlah peluang baru dan sangat menyederhanakan prosedur untuk berinteraksi dengan perangkat.

Saat mengirimkan data melalui SCEF, ada dua keuntungan yang sangat penting dibandingkan lalu lintas IP klasik:

Pengiriman lalu lintas MT ke perangkat dengan ID eksternal

Untuk mengirim pesan ke perangkat IP klasik, AS harus tahu alamat IP-nya. Di sini muncul masalah: karena perangkat biasanya mendaftar dengan alamat IP "abu-abu", itu berkomunikasi dengan server aplikasi, yang terletak di Internet, melalui simpul NAT, di mana alamat abu-abu diterjemahkan menjadi putih. Banyak alamat IP abu-abu dan putih berlangsung dalam waktu terbatas, tergantung pada pengaturan NAT. Rata-rata, untuk TCP atau UDP - tidak lebih dari lima menit. Yaitu, jika dalam 5 menit tidak ada pertukaran data dengan perangkat ini, bundel akan terpecah dan perangkat tidak lagi dapat diakses di alamat putih tempat sesi dengan AS dimulai. Ada beberapa solusi:

1. Gunakan detak jantung. Setelah koneksi dibuat, perangkat harus bertukar dengan paket AS setiap beberapa menit, sehingga mencegah terjemahan NAT dari penutupan. Tetapi tidak ada keraguan tentang efisiensi energi.

2. Setiap kali, jika perlu, periksa paket untuk perangkat di AS - kirim pesan ke uplink.

3. Buat APN pribadi (VRF), di mana server aplikasi dan perangkat akan berada di subnet yang sama, dan tetapkan alamat IP statis ke perangkat. Ini akan berhasil, tetapi ini hampir mustahil ketika menyangkut armada ribuan, puluhan ribu perangkat.

4. Akhirnya, opsi yang paling cocok: gunakan IPv6, tidak perlu NAT, karena alamat IPv6 langsung dapat diakses dari Internet. Namun, bahkan dalam kasus ini, saat mendaftar ulang perangkat, ia akan menerima alamat IPv6 baru dan tidak akan lagi tersedia pada yang sebelumnya.

Oleh karena itu, perlu untuk mengirim paket inisialisasi tertentu dengan pengenal perangkat ke server untuk menginformasikan alamat IP perangkat baru. Kemudian tunggu paket konfirmasi dari AS, yang juga memengaruhi efisiensi energi.

Metode-metode ini bekerja dengan baik untuk perangkat 2G / 3G / LTE, di mana perangkat tidak memiliki persyaratan yang ketat untuk otonomi dan, akibatnya, tidak ada batasan waktu untuk udara dan lalu lintas. Untuk NB-IoT, metode ini tidak cocok karena konsumsi energinya yang tinggi.

SCEF memecahkan masalah ini: karena satu-satunya pengidentifikasi perangkat untuk AS adalah ID eksternal, cukup bagi AS untuk mengirim paket data ke SCEF untuk ID eksternal tertentu, SCEF akan mengurus sisanya. Jika perangkat berada dalam mode hemat daya PSM atau eDRX, data akan disangga dan dikirim ketika perangkat tersedia. Jika perangkat tersedia untuk lalu lintas, data akan segera dikirim. Hal yang sama berlaku untuk tim manajemen.

Kapan saja, AS dapat menarik pesan yang disangga ke UE atau menggantinya dengan yang baru.

Mekanisme buffering juga dapat digunakan saat mentransfer data MO dari UE ke sisi AS. Jika SCEF tidak dapat mengirimkan data ke AS segera, misalnya, jika pekerjaan layanan sedang berlangsung di server AS, paket-paket ini akan disangga dan dijamin akan dikirimkan segera setelah AS tersedia.

Seperti disebutkan di atas, akses ke layanan tertentu dan UE untuk AS (dan NIDD adalah layanan) diatur oleh peraturan dan kebijakan di sisi SCEF, yang memungkinkan untuk merealisasikan kemampuan unik untuk secara bersamaan menggunakan data satu UE oleh beberapa AS. Yaitu jika beberapa AS telah berlangganan satu UE, maka setelah menerima data dari UE, SCEF akan mengirimkannya ke semua AS yang ditandatangani. Ini sangat cocok untuk kasus-kasus di mana pencipta armada perangkat khusus meraba-raba data antara beberapa klien. Misalnya, dengan membuat jaringan stasiun cuaca yang beroperasi di NB-IoT, Anda dapat menjual data dari mereka ke banyak layanan secara bersamaan.

Mekanisme Pengiriman Pesan Dijamin

Layanan Data yang Andal - mekanisme untuk pengiriman pesan MO dan MT yang terjamin tanpa menggunakan algoritma khusus pada tingkat protokol, seperti, misalnya, jabat tangan dalam TCP. Ini bekerja dengan memasukkan bendera khusus di bagian layanan dari pesan selama pertukaran antara UE dan SCEF. Apakah akan mengaktifkan mekanisme ini atau tidak ketika transmisi lalu lintas ditentukan oleh AS.

Jika mekanismenya diaktifkan, UE, jika perlu, pengiriman lalu lintas MO yang dijamin menyertakan bendera khusus di bagian layanan paket. Setelah menerima paket seperti itu, SCEF membalas ke UE dengan konfirmasi. Jika UE belum menerima paket konfirmasi, paket tersebut akan diteruskan ke SCEF. Hal yang sama terjadi pada lalu lintas MT.

Pemantauan Perangkat (pemantauan peristiwa- MONTE)

Seperti disebutkan di atas, fungsional SCEF, antara lain, termasuk fungsi untuk memantau keadaan UE, yang disebut pemantauan perangkat. Dan jika pengidentifikasi baru dan mekanisme transfer data adalah optimasi (walaupun sangat serius) dari prosedur yang ada, maka MONTE adalah fungsi yang sama sekali baru yang tidak tersedia di jaringan 2G / 3G / LTE. MONTE memungkinkan AS untuk memonitor parameter perangkat seperti status koneksi, ketersediaan komunikasi, lokasi, status roaming, dll. Kami akan memberi tahu Anda lebih banyak tentang masing-masing nanti.

Jika perlu untuk mengaktifkan acara pemantauan untuk perangkat atau sekelompok perangkat, AS berlangganan layanan yang sesuai dengan mengirimkan kepada SCEF perintah MONTE API yang sesuai, yang mencakup parameter seperti ID eksternal atau ID grup eksternal, pengidentifikasi AS, jenis pemantauan, jumlah laporan, yang AS ingin terima. Jika AS berwenang untuk mengeksekusi permintaan, SCEF, tergantung pada jenisnya, akan memicu peristiwa pada HSS atau MME (Gbr. 4). Ketika suatu peristiwa terjadi, MME atau HSS menghasilkan laporan ke sisi SCEF, yang mengirimkannya ke AS.

Semua acara kecuali "Jumlah UE yang hadir di wilayah geografis" disediakan melalui HSS. Dua acara "Ubah Asosiasi IMSI-IMEI" dan "Status Roaming" dilacak langsung di HSS, sisanya dari HSS akan ditetapkan pada MME.
Acara dapat dilakukan satu kali atau berkala, dan ditentukan berdasarkan jenisnya.



Pelaporan peristiwa (pelaporan) dikirim oleh node pelacakan peristiwa langsung ke SCEF (Gbr. 5).



Poin penting: peristiwa pemantauan dapat diterapkan pada perangkat non-IP yang terhubung melalui SCEF dan perangkat IP yang mentransmisikan data dengan cara klasik melalui MME-SGW-PGW.

Mari kita lihat lebih dekat pada masing-masing acara pemantauan:

Hilangnya konektivitas - menginformasikan kepada AS bahwa UE tidak lagi tersedia untuk lalu lintas data atau pensinyalan. Peristiwa terjadi ketika "timer jangkauan seluler" untuk UE berakhir pada MME. Dalam permintaan untuk jenis pemantauan ini, AS dapat menunjukkan nilai "Waktu Deteksi Maksimum" - jika UE tidak menunjukkan aktivitas apa pun selama waktu ini, AS akan diberitahu bahwa UE tidak tersedia, menunjukkan alasannya. Peristiwa ini juga terjadi jika UE dihapus secara paksa oleh jaringan karena alasan apa pun.

* Agar jaringan mengetahui bahwa perangkat masih tersedia, itu secara berkala memulai prosedur pembaruan - Pembaruan Area Pelacakan (TAU). Frekuensi prosedur ini ditetapkan oleh jaringan menggunakan timer T3412 atau (T3412_extended dalam kasus PSM), nilai yang ditransmisikan ke perangkat selama prosedur Lampirkan atau TAU berikutnya. Penghitung waktu jangkauan seluler biasanya beberapa menit lebih lama dari T3412. Jika UE tidak membuat TAU â€‹â€‹sebelum Timer jangkauan seluler berakhir, jaringan menganggapnya tidak lagi tersedia.

UE reachability - Ditampilkan ketika UE tersedia untuk lalu lintas DL atau SMS. Ini terjadi ketika UE tersedia untuk paging (untuk UE dalam mode eDRX) atau ketika UE beralih ke mode ECM-CONNECTED (untuk UE dalam mode PSM atau eDRX), mis. membuat TAU â€‹â€‹atau mengirim paket uplink.

Pelaporan lokasi - Jenis kegiatan pemantauan ini memungkinkan AS untuk meminta data lokasi UE. Baik lokasi saat ini (Lokasi Terakhir) atau yang terakhir diketahui (Lokasi Terakhir Diketahui, ditentukan oleh ID sel dari mana perangkat membuat TAU â€‹â€‹atau lalu lintas yang ditransmisikan untuk yang terakhir kali) dapat diminta, yang penting untuk perangkat dalam mode hemat daya PSM atau eDRX. Untuk â€œLokasi Saat Ini,â€ AS dapat meminta pengulangan yang berulang, dengan MME yang memberi tahu AS setiap kali lokasi perangkat diubah.

Perubahan Asosiasi IMSI-IMEI - Ketika acara ini diaktifkan, SCEF mulai melacak perubahan dalam koneksi antara IMSI (pengidentifikasi kartu SIM) dan IMEI (pengidentifikasi perangkat). Ketika suatu peristiwa terjadi - menginformasikan AS. Ini dapat digunakan untuk secara otomatis menetapkan kembali ID eksternal ke perangkat selama pekerjaan penggantian yang direncanakan atau berfungsi sebagai pengidentifikasi untuk pencurian perangkat.

Status Roaming - jenis pemantauan ini digunakan oleh AS untuk menentukan apakah UE ada di jaringan rumah atau di jaringan mitra roaming. Secara opsional, PLMN (Public Land Mobile Network) dari operator tempat perangkat terdaftar dapat ditransfer.

Kegagalan komunikasi - Jenis pemantauan ini menginformasikan kepada AS tentang kegagalan komunikasi dengan perangkat, berdasarkan alasan koneksi (kode penyebab pelepasan) yang diterima dari jaringan akses radio (protokol S1-AP). Acara ini dapat membantu menentukan alasan kegagalan komunikasi - karena masalah jaringan, misalnya, ketika eNodeb (sumber daya Radio tidak tersedia) kelebihan beban atau karena perangkat itu sendiri (Koneksi Radio Dengan UE Hilang) gagal.

Ketersediaan setelah Kegagalan DDN - acara ini menginformasikan kepada AS bahwa perangkat menjadi tersedia setelah kegagalan komunikasi. Ini dapat digunakan ketika diperlukan untuk mentransfer data ke perangkat, tetapi upaya sebelumnya tidak berhasil, karena UE tidak menanggapi pemberitahuan dari jaringan (paging), dan data tidak terkirim. Jika jenis pemantauan ini diminta untuk UE, maka segera setelah perangkat melakukan komunikasi masuk, membuat TAU â€‹â€‹atau mengirim data ke uplink, AS akan diberi tahu bahwa perangkat tersebut telah tersedia. Karena prosedur DDN (pemberitahuan data downlink) berfungsi antara MME dan S / P-GW, jenis pemantauan ini hanya tersedia untuk perangkat IP.

Status Konektivitas PDN - menginformasikan kepada AS ketika mengubah status perangkat (status konektivitas PDN) - menghubungkan (mengaktifkan PDN) atau memutuskan koneksi (menghapus PDN). Ini dapat digunakan oleh AS untuk memulai komunikasi dengan UE, atau sebaliknya, untuk memahami bahwa komunikasi tidak lagi memungkinkan. Jenis pemantauan ini tersedia untuk perangkat IP dan non-IP.

Jumlah UE yang ada di wilayah geografis - jenis pemantauan ini digunakan oleh AS untuk menentukan jumlah UE di wilayah geografis tertentu.

Pemicu perangkat

Dalam jaringan 2G / 3G, prosedur pendaftaran dalam jaringan adalah dua langkah: pertama, perangkat terdaftar di SGSN (prosedur melampirkan), kemudian, jika perlu, data diaktifkan oleh konteks PDP - koneksi ke gateway paket (GGSN). Di jaringan 3G, kedua prosedur ini berjalan secara berurutan, yaitu perangkat tidak menunggu saat ketika diperlukan untuk mentransfer data, tetapi mengaktifkan PDP segera setelah prosedur pemasangan selesai. Dalam LTE, kedua prosedur ini digabungkan menjadi satu, yaitu, ketika melampirkan, perangkat segera meminta aktivasi koneksi PDN (analog dari PDP dalam 2G / 3G) melalui eNodeB ke MME-SGW-PGW.

NB-IoT mendefinisikan metode koneksi seperti "pasang tanpa PDN", yaitu UE membuat sambungan tanpa membuat koneksi PDN. Dalam hal ini, tidak tersedia untuk transmisi lalu lintas, dan hanya dapat menerima atau mengirim SMS. Untuk mengirim perintah untuk mengaktifkan PDN dan menyambungkan ke AS ke perangkat semacam itu, fungsionalitas "Pemicu perangkat" dikembangkan.

Setelah menerima perintah untuk menghubungkan UE dari AS, SCEF melalui pusat SMS akan memulai pengiriman SMS kontrol ke perangkat. Setelah menerima SMS, perangkat mengaktifkan PDN dan menghubungkan ke AS untuk menerima instruksi selanjutnya atau mentransfer data.
