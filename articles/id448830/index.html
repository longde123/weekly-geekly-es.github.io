Di mana untuk memulai pengembangan arsitektur?

Catatan: Jika Anda berpikir bahwa setidaknya setengah anjing dimakan selama konstruksi arsitektur, maka artikel ini bukan untuk Anda.

Model adalah representasi abstrak dari realitas dalam beberapa bentuk.

Kami berasumsi bahwa arsitek telah selesai mengumpulkan persyaratan untuk sistem masa depan dan menganalisisnya.

Pengembangan arsitektur perlu dimulai hanya dengan konsep dan adopsi konsep dasar bekerja dengan informasi (data): transmisi, penyimpanan dan pemrosesan. Selain itu, bentuk input / output informasi, skema pemrosesan, struktur abstrak array dan elemen data itu sendiri juga merupakan informasi (seperti seluruh aplikasi) dan mematuhi konsep dasar yang sama.



Konsep dasar mengarah pada ide pertama dan penciptaan model pemrosesan data pertama.



Model bekerja dengan data dapat disebut secara berbeda karena perbedaan dalam konsep bersarang. Salah satu namanya adalah aliran data, di mana elemen-elemennya biasanya merupakan modul yang diperbesar di masa depan yang mengimplementasikan tugas manipulasi data tertentu.

Basis arsitektur (model utama) akan menjadi model proses bisnis pergerakan data utama (untuk apa aplikasi akan ditujukan). Tahap ini sangat berbahaya: banyak ide orisinil dapat membanjiri arsitek. Aliran gagasan ini harus diatasi. Menekan keinginan untuk bermimpi atau menggambar banyak skema yang tidak perlu. Sekarang Anda hanya perlu membuat satu skema untuk mendapatkan data master, penyimpanan, dan pemrosesan secara berurutan. Tidak lebih. Tidak diperlukan dekomposisi di sini. Ini adalah skema yang mendasari, adalah representasi utama arsitektur dan harus selalu terlihat. Sebagai proses bisnis dapat menjadi peta teknologi.

Model utama kedua adalah dengan penambahan proses dan data bisnis sekunder. Sebagai contoh, model pengguna, kontrol akses (ACL), model logging, model pemantauan, model acara, dll ditambahkan. Berdasarkan model kedua, sudah dimungkinkan untuk mengembangkan skema database pertama dan / atau data warehouse, dan aplikasi prototipe. Pada tahap ini, pemahaman mungkin muncul tentang apakah membuat aplikasi monolitik, komponen, untuk menerapkan pendekatan layanan-mikro atau lainnya. Faktanya menarik: antara monolith dan masing-masing microservice (dalam arsitektur microservice) tidak ada perbedaan dari sudut pandang konsep dasar. Mulai saat ini, perlu untuk memulai pengembangan dokumen justifikasi arsitektur. Dokumen alasan arsitektur adalah perekam alasan tentang alasan keputusan yang dibuat oleh arsitektur dan penjelasan.

Membuat prototipe aplikasi pertama berdasarkan model kedua dapat menunjukkan manfaat langsung dari aplikasi. Prototipe tentu akan menjadi tata letak aplikasi formal. Misalnya, alih-alih formulir terperinci, Anda bisa menggunakan tombol bersyarat yang membuat elemen jadi berdasarkan generator data acak.

Setelah membuat model utama kedua dan prototipe pertama, Anda dapat memulai dekomposisi, pembesaran, pengembangan representasi dan model arsitektur lainnya. Tetapi model utama pertama dan model utama kedua akan menjadi dasar. Dalam proses pengembangan lebih lanjut, model pertama dan kedua dapat ditingkatkan, disempurnakan dan disempurnakan.
