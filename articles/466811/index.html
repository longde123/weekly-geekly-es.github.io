Microsoft .NET frameworks


El .NET Framework original de Microsoft apareció a finales de los 90 y solo se ha vuelto más eficiente desde entonces. Recientemente, la compañía cambió su rumbo al anunciar el lanzamiento de .NET Core framework, que se distingue por su código multiplataforma y código abierto. Además, al desarrollar la especificación .NET Standard, Microsoft ha proporcionado interoperabilidad en entornos de tiempo de ejecución.


La especificación .NET Standard proporciona código compartido. Define el conjunto de API que deberían estar disponibles en todas las implementaciones de .NET para cumplir con el estándar. Esto garantiza la uniformidad de todos los tiempos de ejecución .NET y evita la fragmentación futura.

Una pequeña digresión: si quieres aprender más sobre la historia de .NET, te recomiendo escuchar a Anders Halesberg o ver un discurso de Richard Campbell : este es un narrador realmente genial.



(También está disponible un podcast . Además, Campbell está escribiendo un libro sobre el mismo tema . Si desea obtener más información sobre la historia de todo el ecosistema .NET, y no solo los tiempos de ejecución, consulte la página Leyendas de .NET ).


Otros tiempos de ejecución de Microsoft


Además del núcleo .NET Framework para uso generalizado, Microsoft ha lanzado otras versiones del entorno para fines especiales.


.NET Compact Framework


Los marcos Compact (.NET CF) y Micro (.NET MF) fueron intentos de crear versiones de tiempo de ejecución más cortas para dispositivos con recursos limitados. Por ejemplo, .NET CF:


... diseñado para dispositivos móviles o integrados con una cantidad limitada de recursos, como PDA, teléfonos móviles, controladores industriales, decodificadores, etc. El marco .NET Compact utiliza algunas de las bibliotecas de la misma clase que la versión completa de .NET, así como varias bibliotecas creadas específicamente para dispositivos móviles, como los controles .NET Compact. Sin embargo, las bibliotecas no son una copia exacta de las utilizadas en .NET Framework. Se reducen para ocupar menos espacio.

.NET Micro Framework


El marco .NET MF es aún más pequeño y:


... es para dispositivos con una cantidad limitada de recursos que tienen al menos 256 KB de memoria y 64 KB de RAM. Incluye una versión reducida de .NET CLR y admite el desarrollo de C # y Visual Basic .NET, así como la depuración (en emulador o hardware) con Microsoft Visual Studio. NETMF implementa un conjunto de bibliotecas base .NET (aproximadamente 70 clases y 420 métodos). Se han agregado bibliotecas especiales para aplicaciones integradas. Este es un software gratuito de código abierto lanzado bajo la Licencia Apache 2.0.

Para aquellos que quieran probar el framework en acción, Scott Hanselman ha escrito una excelente guía para .NET Micro Framework - Hardware para personas de software .


Silverlight


A pesar de que la plataforma está en modo de soporte (o incluso muerta / moviéndose hacia la puesta del sol, dependiendo de su punto de vista), es interesante volver al anuncio original y ver para qué era Silverlight:


Silverlight es un módulo de extensión multiplataforma, navegador cruzado .NET diseñado para crear aplicaciones de Internet enriquecidas para navegadores. Las versiones preliminares lanzadas esta semana son compatibles con Firefox, IE y Safari en Mac y Windows.

En 2007, Silverlight 1.0 implementó las siguientes características (la plataforma incluso funcionó en Linux):


  • soporte para códecs integrados para reproducir archivos de video VC-1 y WMV, así como archivos de audio en formato MP3 y WMA en un navegador ...;
  • Silverlight admite la capacidad de descargar y reproducir gradualmente contenido multimedia desde cualquier servidor web ...;
  • Silverlight también admite opcionalmente la transmisión multimedia integrada ...;
  • Con Silverlight puede crear interfaces de usuario y animaciones multifuncionales, combinar gráficos vectoriales y HTML para crear contenido atractivo ...;
  • Silverlight facilita la creación de rico contenido de video interactivo ...

Además, como se sugiere en los comentarios, Silverlight estaba en el Symbian S60


Mono / Xamarin Runtimes


Mono apareció como resultado de los intentos de Miguel de Icaz y otros desarrolladores para hacer que .NET funcione en Linux (desde la historia temprana de Mono ):


No me importa quién fue el primero, porque creo que Mono es un medio para un fin: una tecnología que ayuda a Linux a establecerse en las computadoras de escritorio.

La misma publicación describe cómo comenzó todo:


En cuanto a Mono, los eventos ocurrieron aproximadamente de la siguiente manera.
Tan pronto como salió la documentación de .NET en diciembre de 2000, me interesé en esta tecnología y comencé a estudiarla, como todos los demás, desde el intérprete de bytecode. Sin embargo, pronto encontré un problema: la falta de especificaciones para los metadatos .
Los últimos cambios en las primeras fuentes de máquinas virtuales salieron el 22 de enero de 2001. Alrededor de este tiempo, comencé a escribir mensajes usando la lista de correo de .NET y solicitando la información que faltaba sobre el formato del archivo de metadatos.
...
Casi al mismo tiempo, Sam Ruby estaba tratando de convencer a los miembros del comité de ECMA para que publicaran una descripción del formato de archivo binario, que no formaba parte de sus planes. No sé cómo se desarrolló todo, pero en abril de 2001, ECMA publicó una descripción del formato de archivo.

Con el tiempo, Mono ( ahora Xamarin ) se ha extendido a otras plataformas. El marco se ejecuta en Android e iOS / Mac. En febrero de 2016, Microsoft adquirió Xamarin. Vale la pena recordar la colaboración de larga data entre Mono / Xamarin y Unity , dirigida a proporcionar soporte para C # en Unity . Al mismo tiempo, Unity ahora es miembro de la Fundación .NET.


Tiempos de ejecución de compilación AOT


Tomé tales entornos a una categoría separada, porque tradicionalmente .NET usaba la compilación JIT , pero con el tiempo, aparecieron más implementaciones con AOT.
Por lo que puedo decir, el primero fue el marco Mono, en el que apareció la compilación AOT en 2006. Además, Microsoft lanzó recientemente el .NET Native, y actualmente están trabajando en CoreRT , un entorno de tiempo de ejecución .NET optimizado para la compilación AOT.


Proyectos comunitarios


Sin embargo, no todos los tiempos de ejecución de .NET fueron desarrollados por Microsoft o por compañías que compró posteriormente. Hay proyectos implementados por las comunidades.


  • El más antiguo de estos es DotGNU Portable.NET . Su desarrollo comenzó simultáneamente con Mono con el objetivo de "crear un conjunto de herramientas gratuitas para compilar y ejecutar aplicaciones en el marco de la CLI ...".
  • En segundo lugar, nombraré DotNetAnywhere , desarrollado por una sola persona, Chris Bacon. DotNetAnywhere es conocido por servir como el tiempo de ejecución inicial para el proyecto Blazor. ¡También es una gran oportunidad para echar un vistazo a lo que constituye un tiempo de ejecución compatible con .NET sin tener que atravesar los millones de líneas de código que forman CoreCLR !
  • Luego viene CosmosOS ( un proyecto de GitHub ), que no es solo un tiempo de ejecución .NET, sino un sistema operativo administrado. Si desea obtener más información, le aconsejo que lea esta sección de preguntas frecuentes o busque "debajo del capó" . Otro intento similar es SharpOS .
  • Finalmente, recientemente me encontré con CrossNet . Este tiempo de ejecución "analiza los ensamblados .NET y genera código C ++ no administrado que puede ser compilado por cualquier compilador de C ++". Para obtener más información, eche un vistazo a la documentación y al código generado de muestra.

Proyectos de investigación


Ahora pasemos a los tiempos de ejecución .NET muy poco conocidos. Estos son proyectos de investigación de Microsoft lanzados para ver cuánto se puede ampliar la funcionalidad del entorno de ejecución administrado y qué tareas se pueden resolver con su ayuda. Algunos hallazgos de investigación han aparecido en implementaciones comerciales del entorno .NET, por ejemplo, Span <T> surgió del proyecto Midori .


Infraestructura de lenguaje común de origen compartido (SSCLI) (o rotor)


Una implementación de la CLI (.NET framework) de Microsoft, en la que el código fuente está disponible bajo la licencia de Fuente compartida. Aunque SSCLI no es adecuado para uso comercial debido a los detalles específicos de la licencia, con su ayuda los programadores pueden analizar las características de implementación de muchas bibliotecas .NET y crear versiones modificadas de la CLI. Microsoft proporciona la CLI de fuente compartida como una CLI de referencia educativa.

Curiosamente, con el lanzamiento de Rotor, la compañía pudo lanzar el proyecto Gyro , que da una idea de cómo aparecieron los genéricos en el tiempo de ejecución de .NET .


Midori


Midori es el nombre en clave de un sistema operativo de código administrado desarrollado por Microsoft en conjunto con Microsoft Research. Se informó que podría convertirse en una implementación comercial del sistema operativo Singularity, un proyecto de investigación lanzado en 2003 para crear un sistema operativo altamente confiable en el que el núcleo, los controladores de dispositivos y las aplicaciones consisten en código administrado. Fue diseñado para computación paralela y podría ejecutar un programa distribuido en varios nodos simultáneamente. También implementó un modelo de seguridad basado en la ejecución de aplicaciones en un entorno aislado. Microsoft ha sugerido varias formas posibles de migrar de Windows a Midori. El trabajo en el sistema operativo se suspendió en 2015, aunque muchas de las ideas implementadas en él cayeron en otros proyectos de Microsoft.

Aparentemente, la mayoría de las ideas del proyecto Midori volvieron al .NET Framework. Puedes leer más sobre esto con Joe Duffy en una serie de artículos sobre Midori :

Singularity (sistema operativo) (también Singularity RDK)


Singularity es un sistema operativo experimental desarrollado por Microsoft Research entre 2003 y 2010. Se suponía que era un sistema operativo altamente confiable en el que el núcleo, los controladores de dispositivos y las aplicaciones consisten en código administrado. La seguridad interna utiliza seguridad de tipo en lugar de protección de memoria de hardware.

Net60


En los comentarios, bmforce sugirió que había otra plataforma, Net60. No hay mucha información al respecto, pero logró encontrar una mención en el foro + artículo sobre CodeGuru :


¡El .NET Compact Framework para Symbian S60 3rd finalmente ha llegado!

Luz de la luna


No se menciona Moonlight, que se basa en la versión Mono - Opensource de Silverlight:


Moonlight es una implementación de código abierto de Silverlight, creada principalmente para Linux y otros sistemas operativos Unix / X11. La última versión de Moonlight (Moonlight 4 Preview 1) proporciona soporte para el conjunto básico de funciones de Silverlight 3, además de compatibilidad con Silverlight 4.

Blazor


Tampoco se menciona el nuevo proyecto Blazor , que tiene todas las posibilidades de éxito: .NET para Web Assembly. Se utiliza, por ejemplo, en el proyecto Try .NET , un juego de herramientas para crear ejemplos de código .NET que se pueden ejecutar en un navegador.
