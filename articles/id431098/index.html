Bahkan kebakaran bukanlah halangan, atau Zimbra Speed ​​Recovery setelah bencana

Dalam artikel sebelumnya, pembaca meminta kami untuk memberi tahu Anda lebih banyak tentang bagaimana Zimbra dipulihkan setelah kerusakan perangkat keras server atau bahkan kerusakan fisiknya. Oleh karena itu, sekarang kita akan berbicara tentang bagaimana proses ini terjadi dan berbagi sejumlah trik, terima kasih yang seluruh proses akan memakan waktu beberapa menit.



Jadi, sesuatu yang sangat buruk terjadi pada server Anda yang menjalankan Zimbra Collaboration Suite. Tidak begitu penting apakah itu kebakaran atau banjir. Adalah penting bahwa pekerjaan di perusahaan telah meningkat dan dalam beberapa menit karena downtime ini perusahaan akan mulai kehilangan uang. Ini berarti saatnya untuk segera mulai mengembalikan fungsionalitas Zimbra Collaboration Suite.

Pemulihan dalam keadaan darurat terdiri dari tiga tahap: pertama Anda perlu menemukan server fisik atau virtual yang sesuai di mana Anda akan menggunakan Zimbra, maka Anda perlu menginstal sistem operasi dan menginstal Zimbra Collaboration Suite itu sendiri, dan kemudian mencadangkan ZCS yang diinstal.

Manajer TI yang berpengalaman akan segera melihat bahwa dua tahap pertama hanya tampak panjang, tetapi sebenarnya mereka dapat dengan mudah dioptimalkan jika Anda memiliki server yang siap untuk commissioning dengan Linux, Zimbra dan Zextras Suite yang terpasang di papan. Dalam hal ini, dua langkah pertama hanya membutuhkan beberapa menit, yang diperlukan untuk menghidupkan server dan mengubah catatan DNS. Dalam hal apa pun, Anda dapat menemukan instruksi terperinci tentang menginstal Zimbra Collaboration Suite di Ubuntu di salah satu artikel kami sebelumnya.

Mengoptimalkan tahap ketiga akan membantu musim dingin Cadangan Zextras, yang merupakan bagian dari Zextras Suite. Ini memungkinkan Anda untuk dengan cepat mengembalikan data dari cadangan. Jadi, misalnya, jika pada cadangan server lama yang dibuat oleh pemindai waktu-nyata disimpan pada disk atau cloud yang terpisah, maka Anda bisa memasangnya di server cadangan dan mengembalikan semua data darinya. Ingat bahwa pemindai waktu-nyata adalah salah satu fitur utama Zextras Backup dan karena integrasinya yang mendalam dengan Zimbra, ia dapat melacak perubahan dalam log server dan langsung mencadangkan semua file yang diubah, membuat cadangan tambahan, dan menggunakan rsync paling umum, Anda dapat menyinkronkan data cadangan dengan penyimpanan yang terletak di tempat yang aman. Berkat mekanisme penyalinan ini, Anda kemungkinan besar tidak akan kehilangan data sama sekali karena keadaan darurat pada server dengan Zimbra.



Perhatikan bahwa saat membuat cadangan, Cadangan Zextras juga menyimpan pengaturan global Zimbra, serta pengaturan server. Namun, selama pemulihan otomatis dari cadangan, opsi ini tidak diterapkan. Dengan kata lain, jika Anda tiba-tiba perlu membuat klon sempurna dari utara lama atau menerapkan pengaturan yang sama pada server baru seperti pada server lama, Anda harus menggunakan antarmuka perintah Zextras dan perintah magic getServerConfig . Sebagai contoh, perintah terakhir tanggal backup standar getserverconfig zxsuite menampilkan pengaturan global dan pengaturan server yang terakhir disimpan, dan manajer TI hanya perlu mengembalikannya sepenuhnya, atau hanya menerapkan yang diperlukan saja.

Jika anggaran TI perusahaan tidak memungkinkan Anda untuk memiliki server cadangan untuk Zimbra siap, maka Anda selalu dapat mengambil keuntungan dari manfaat virtualisasi, serta layanan dari berbagai layanan hosting yang siap untuk memberikan bantuan setiap saat. Anda hanya perlu memiliki wadah yang siap untuk dimuat ke mesin virtual di mana Anda sudah menginstal Linux, serta Zimbra Collaboration Suite dengan versi yang sama seperti pada "server pertempuran" Anda, yang gagal. Ini, tentu saja, akan memakan waktu lebih lama daripada hanya menyalakan server cadangan, tetapi juga akan jauh lebih murah. Dalam hal ini, semuanya tergantung pada berapa lama downtime server dengan Zimbra dapat diterima untuk organisasi Anda. Tahap pemulihan ketiga akan sepenuhnya mengulangi algoritma yang dijelaskan sebelumnya.

Ide yang bagus adalah melakukan semacam latihan. Karena tidak perlu menghentikan server "pertempuran" untuk menggunakan Zimbra pada mesin cadangan, dan terlebih lagi untuk tidak menunggu keadaan darurat, Anda dapat mencoba melakukannya di salah satu hari kerja. Latihan semacam itu akan memungkinkan Anda untuk mengevaluasi waktu yang diperlukan untuk menghilangkan konsekuensinya, dan jika karena alasan tertentu hasilnya tidak sesuai dengan Anda atau kepemimpinan Anda, Anda dapat berpikir tentang mengembangkan algoritma yang lebih optimal jika terjadi insiden seperti itu.

Dengan demikian, tunduk pada sejumlah kondisi dan tindakan pencegahan, bahkan kehancuran fisik server dari Zimbr tidak akan menyebabkan hilangnya data penting, korespondensi dan daftar kontak perusahaan, serta untuk menghindari downtime yang lama.
