Sailfish OS مؤقت تشغيل على فترات طويلة

مقدمة


في كثير من الأحيان ، عند تنفيذ أي منطق في التطبيق ، هناك حاجة لتشغيل وظيفة معينة بعد فترة زمنية معينة. إن أوضح مثال على هذه الحاجة هو تطبيق مؤقت. على سبيل المثال ، cooktimer أو Saildoro .

كما ذكر في مقال سابق ، لإضافة مؤقت إلى تطبيق على نظام تشغيل Sailfish ، يمكنك استخدام عنصر Timer القياسي أو نظيره C ++ ، QTimer . ومع ذلك ، بشكل افتراضي ، يتم تعليق تشغيل هذه المؤقتات لفترات طويلة بسبب حقيقة أن الجهاز قد ينام. واجه مطورو التطبيقين المذكورين أعلاه هذه المشكلة للتو.

تقدم هذه المقالة طريقة قياسية ، ولكن للأسف غير موثقة للتعامل مع هذا السلوك لنظام Sailfish OS.

نقطة البداية


كنقطة انطلاق ، سننظر في تطبيق "تجريدي" لنظام Sailfish OS ، والذي يتطلب تشغيل بعض الوظائف بعد فترة زمنية طويلة. في نفس الوقت ، لا يتم وصف عملية المؤقت في كود QML ، ولكن في فئة C ++:

العنوان 
 class TimerWrapper : public QObject { Q_OBJECT public: //    explicit TimerWrapper(QObject *parent = 0); ~TimerWrapper(); Q_INVOKABLE void start(int interval); //     Q_INVOKABLE void stop(); //     signals: void pomodoroFinished(int start, int end); //    void activeChanged(); //     private: QTimer *_timer; //   int _startTime; //    };

يجب تسجيل كائن من هذه الفئة في QML:

main.cpp

مع هذا النهج ، كما هو مذكور في المقدمة ، قد يتوقف المؤقت مؤقتًا لفترات طويلة من الزمن.

الحل


تم اقتراح الخيار الأول لمنع الموقت من النوم في القائمة البريدية للمطورين وتأصل في تطبيق cooktimer. يقترح هنا إعداد مؤقت إضافي بشكل صريح ، والذي يستدعي مرة واحدة في الدقيقة طريقة D-Bus req_display_cancel_blanking_pause لمنع الجهاز من النوم. من الواضح أن مثل هذا التنفيذ ليس مثالياً ومرهقاً. أولاً ، عند استخدام هذا النهج ، تعمل بطارية الجهاز بشكل أسرع. ثانيًا ، يظهر رمز ثانوي في المشروع يمكن تجنبه.

ويمكنك تجنب استخدام رمز ثانوي لأن Sailfish OS يوفر بالفعل حلين ممكنين للمشكلة: عناصر ScreenBlank و KeepAlive .

ينطوي استخدام النهج الأول على شاشة نشطة باستمرار. هذا نهج عملي ولكنه مباشر ، يستهلك طاقة بطارية الجهاز بنشاط. وبالتالي ، يمكن استخدامه ، ولكن في مجموعة محدودة من المواقف. 

 import QtQuick 2.0 //       QML import Sailfish.Silica 1.0 //     Sailfish OS UI import Sailfish.Media 1.0 //      ScreenBlank ApplicationWindow //     { initialPage: Component { FirstPage { } } //     cover: Qt.resolvedUrl("cover/CoverPage.qml") //    ScreenBlank { //       id: screenBlank //    suspend: true //    } }

في المقابل ، يعد استخدام عنصر KeepAlive نهجًا أكثر ديمقراطية. إلى حد أقل ، يستهلك طاقة البطارية ، لأنه لا يبقي شاشة الجهاز قيد التشغيل باستمرار ، وفي الوقت نفسه ، إما أنه لا يسمح للجهاز بالنوم العميق ، أو يوقظه في وقت معين ، لذلك سيستمر المؤقت في العمل على فترات طويلة. 

 import QtQuick 2.0 //       QML import Sailfish.Silica 1.0 //     Sailfish OS UI import org.nemomobile.keepalive 1.1 //      KeepAlive ApplicationWindow //     { initialPage: Component { FirstPage { } } //     cover: Qt.resolvedUrl("cover/CoverPage.qml") //    KeepAlive { //       id: keepAlive //    enabled: true //       } }

من الجدير بالذكر أنه من حيث المبدأ ، فإن تشغيل جميع الطرق الثلاثة المذكورة هو نداء منتظم لأساليب نظام D-Bus ، والتي تمت مناقشتها في إحدى المقالات السابقة .

الخلاصة


في هذه المذكرة القصيرة ، تم وصف ثلاث طرق ممكنة لمنع الجهاز من النوم بعمق. يمكننا أن نستنتج أنه بالنسبة لمهام الخلفية (على سبيل المثال ، جهاز ضبط الوقت) ، فمن الأفضل استخدام عنصر KeepAlive ، وإذا كانت هناك حاجة لعرض المعلومات باستمرار للمستخدم ، فإن ScreenBlank .
