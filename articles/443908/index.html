Controlamos el dispositivo Android



Este es el tercer intento de hacer que un hogar inteligente se haga amigo de Android, permíteme recordarte que el primer intento de controlar dispositivos Android a través de HTTP fue usar la aplicación Paw Server. Esta aplicación permitió usar el lenguaje BeanShell para incrustar su código en una página xhtml e interactuar con él, recibir datos o administrarlo. Para integrarse con el servidor doméstico inteligente ( ioBroker ), se escribió un controlador, pero para su primera configuración, fue necesario cargar manualmente los scripts. Además, a través del controlador, se actualizaron los scripts, lo que permitió agregar nuevas funciones y corregir errores, pero también impuso una serie de restricciones en los intentos de alejarse de los métodos establecidos en Paw Server para implementar nuevas funciones.

El segundo intento fue construir su aplicación basada en la fuente del servidor Paw. El objetivo principal era simplificar el proceso de configuración para el usuario, así como agregar nuevas características que no podían implementarse en la versión anterior.

En la misma versión, se decidió abandonar por completo el servidor Paw y volver a escribir la aplicación y el controlador para ioBroker. Agregue nuevos métodos de conexión, además del HTTP existente, también MQTT. Agregue más configuraciones para elegir eventos, tanto desde el propio sistema como desde los sensores integrados. Por supuesto, lo primero que la aplicación estará optimizada para ioBroker, pero sin mucha dificultad puede integrarse en otros sistemas.

Google play

La aplicación te permite:

  1. Obtenga el estado de la configuración del sistema (brillo de la luz de fondo, estado de la pantalla, volumen, batería, etc.)
  2. Recibir datos de llamadas entrantes, reconocimiento de voz.
  3. Reciba datos de sensores incorporados.
  4. Obtenga coordenadas de ubicación.
  5. Obtenga una lista de aplicaciones instaladas y ejecútelas.
  6. Administre la configuración del sistema (brillo de la luz de fondo, nivel de volumen, etc.)
  7. Hacer llamadas.
  8. Crear notificaciones y cuadros de diálogo.
  9. Enviar texto a síntesis de voz.
  10. Interactúa con Tasker.
  11. Enviar archivos a dispositivos (solo HTTP).

Apariencia




La apariencia de la aplicación no es final y puede variar. Me gustaría cambiar o agregar mucho, pero eso es todo con el tiempo.

Con la navegación a través de la aplicación, creo, no debería haber dificultades. En el primer lanzamiento, la aplicación informa brevemente sobre los cambios actuales en la nueva versión y sugiere utilizar el "asistente". Al ir a la configuración de la aplicación, puede cambiar los parámetros básicos de operación, seleccionar el tipo de conexión, seleccionar los eventos que se transmitirán al servidor y también permitir o denegar el acceso a algunos datos (directorio telefónico, mensajes, lista de llamadas y fotos).

En la pantalla principal puede ver los "mosaicos", mientras que esta es una versión de prueba, pero con el tiempo planeo ampliar sus capacidades. De los "mosaicos" disponibles, en este momento, hay: botón, atenuador, tiempo, lista, color, información. La tarea principal de los mosaicos es enviar o recibir datos (comandos) del servidor o administrar otros dispositivos. Si bien no hay una imagen general de cómo debería funcionar todo, ahora no describiré todos los matices.

Conexiones


Ahora sobre conexiones y comandos de control, la aplicación tiene dos opciones de conexión a través de HTTP y MQTT. Cada método tiene sus ventajas y desventajas, el método de conexión que elija.

HTTP


Este método implica conectarse mediante Wi-Fi a una red local. La aplicación "eleva" su servidor web (la dirección IP y el puerto se pueden ver en la notificación al conectarse) y le da acceso para administrarlo. Esto se puede hacer directamente (a través de un navegador) o en integración con el servidor UD, a través de solicitudes POST o GET.





Las respuestas de las solicitudes se devolverán en formato JSON, el nombre del dispositivo, la dirección IP y el estado del comando se transmiten en el cuerpo de la respuesta. Algunas solicitudes llaman a una "devolución de llamada" adicional, por ejemplo, cuando se envía texto a síntesis de voz, la aplicación enviará una solicitud al servidor sobre el comienzo de hablar el texto y su finalización. Del mismo modo, la aplicación transmite datos sobre eventos y lecturas de sensores incorporados al servidor. Por lo tanto, para un trabajo completo, se requiere que el servidor UD pueda procesarlos.

MQTT


El protocolo MQTT es bastante popular y es compatible con varios sistemas DD, esto facilita la integración de la aplicación en ellos. Al elegir este método de conexión, puede utilizar corredores MQTT locales y externos.

Cuando se conecta al agente MQTT, se crea la rama principal / PAW / , seguida del nombre de los dispositivos (cada dispositivo debe tener el suyo), que a su vez se dividen en dos ramas del tema / info / y / comm / , puede adivinar por el nombre que La rama de información (información) publica toda la información entrante del dispositivo, y la rama de comunicación (comando) contiene temas para administrarla. Esto se hace para mayor claridad, a fin de comprender mejor qué tema es responsable de qué.



También en la rama principal hay / all_devices / en esta rama hay temas para los que se suscriben todos los dispositivos, lo que le permite administrar todos los dispositivos a la vez.

Para la universalidad, en esos temas cuyos valores pueden ser verdaderos (verdaderos) o falsos (falsos), pueden tomar diferentes valores, es decir, 1, encendido, automático, verdadero es el valor verdadero y 0, apagado, falso, manual es falso . Otra característica de la aplicación es que para verificar la ejecución del comando, en caso de ejecución exitosa, se publica un valor vacío en el mismo tema. Y si el valor después de la publicación no ha desaparecido, esto indica que se produjo un error al ejecutar el comando o que el valor no coincide con el correcto para este tema. Por ejemplo, al cambiar el nivel de volumen, si el valor no corresponde al número o supera el máximo, para un tipo determinado de nivel de volumen, devolverá un error.

También en esta versión, se amplió el conjunto de comandos para notificaciones y "diálogos", que le permiten mostrar información más detallada, así como interactuar con el usuario si el dispositivo se utiliza como informador. Al construirlos, se requiere una gran cantidad de parámetros, por lo tanto, es necesario publicar el valor en el formato JSON en el tema apropiado.

Para notificaciones tema / comunicación / notificación / crear (a continuación se muestra un ejemplo del valor) 

{ "noti":"Any text", "title":"Title 2", "info":"Any text", "vibrate":true, "sound":true, "light":true, "id":2 }

Para los "cuadros de diálogo" tema / comunicación / notificación / alerta . La respuesta de los "cuadros de diálogo" viene en formato JSON y se publica en el tema / información / alerta / respuesta 

 { "alert":"Turn the lights off?", "title":"Light", "negative":"No", "positive":"Yes", "neutral":"Neutral", "sound":true, "id":2 }

Integración de IoBroker




Al incorporar la capacidad de trabajar a través del protocolo MQTT en las aplicaciones, solo quería simplificar la integración con el sistema y deshacerme de escribir un controlador separado. Pero algunas funciones no se pueden implementar a través del protocolo MQTT, y por esta razón no se puede prescindir de un controlador.

La estructura de los objetos del controlador es similar a la estructura de MQTT, y también se divide en dos ramas / info / y / comm / , tiene comandos similares para administrar y la misma reacción a datos incorrectos. No describiré la configuración y el funcionamiento del controlador aquí; toda la información actual se actualizará en GitHub .

Notas


En cuanto a la administración de la configuración del sistema (control del brillo de la luz de fondo, despertarse del modo de suspensión, etc.), los diferentes dispositivos reaccionarán de manera diferente o no a los comandos. Debido a la gran variedad de dispositivos, versiones de SDK, firmware, es difícil establecer un modelo de comportamiento por comando. Aquí debe seleccionar la acción de acuerdo con su dispositivo, por ejemplo, para que la mayoría de los dispositivos apaguen la pantalla (la envíen a dormir) es suficiente para cambiar el tiempo de espera de la luz de fondo, pero en algunos dispositivos esto no funcionará. La misma situación con otras configuraciones del sistema, para la mayoría de los dispositivos, los cambios ocurrirán inmediatamente, pero para otros es necesario enviar el dispositivo a la suspensión y luego despertarlo para que los cambios surtan efecto. Los problemas más pequeños de este tipo surgen con el SDK 19 (Android 4.4), pero eso no es exacto.) Además, no olvide que hay compatibilidad con Tasker, y si no tiene una función, puede agregarla e interactuar a través de la aplicación.
