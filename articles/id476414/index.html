Identifikasi aktivitas disk Linux

TL; DR : artikel ini berbicara tentang cara yang mudah, cepat dan andal untuk mengidentifikasi program berbasis Linux yang menulis data ke disk, yang membantu mengidentifikasi beban besar atau tidak normal pada subsistem disk, dan juga memungkinkan Anda memperkirakan overhead sistem file. Ini terutama berlaku untuk SSD pada PC, EMMC, dan memori Flash pada komputer papan tunggal.
Selama penulisan artikel, ditemukan bahwa menulis beberapa kilobyte data ke sistem file BTRFS mengarah pada penulisan 3 megabyte data nyata ke disk.

Pendahuluan

‚ÄúOh, omong kosong, sel-sel memori pada SSD modern akan keluar dari operasi setelah puluhan tahun penggunaan normal, jangan khawatir tentang hal ini, dan terlebih lagi mentransfer swap, mesin virtual dan folder profil browser ke HDD‚Äù - jawaban khas untuk pertanyaan tentang keandalan SSD dengan jaminan ‚âà150 TBW . Jika Anda memperkirakan berapa banyak perangkat lunak biasa dapat menulis data, tampaknya 10-20 GB per hari sudah merupakan angka besar, biarlah ada maksimum 40 GB, lebih banyak lagi. Dengan angka seperti itu, jawabannya cukup masuk akal - butuh 10 tahun untuk mencapai nilai yang dijamin untuk jumlah sel yang ditimpa, dengan 40 GB data yang direkam setiap hari.
Namun, selama 6 tahun saya telah menggunakan SSD ketiga: yang pertama memiliki pengontrol rusak, dan yang kedua mulai memindahkan data antar sel beberapa kali sehari, yang mengakibatkan keterlambatan 30 detik dalam pemeliharaan rekaman.

Setelah 7 bulan menggunakan SSD baru, saya memutuskan untuk memeriksa jumlah data yang direkam, karena disk itu sendiri melaporkan melalui SMART.
19,7 TB.
Hanya dalam 7 bulan, saya menggunakan 13% dari jumlah yang dijamin dari data yang direkam, meskipun fakta bahwa itu dikonfigurasi sesuai dengan rekomendasi untuk menyelaraskan partisi dan mengatur FS, saya hampir tidak menggunakan swap, disk mesin virtual berada di HDD!
Ini adalah angka tinggi yang tidak normal, pada kecepatan seperti itu garansi TBW akan terlampaui sebelum masa garansi drive 5 tahun tercapai. Dan komputer saya tidak dapat menulis 93 gigabytes per hari! Anda perlu memeriksa berapa banyak data yang ditulis ke disk dalam 10 menit ...

Menentukan overhead sistem file

Perbedaan besar dalam btrace iotop dan btrace mendorong gagasan untuk menguji sistem file dengan secara manual menulis data ke file dan melacak btrace pembacaan.

Jika Anda benar-benar mengecualikan penulisan ke disk dengan memuat ke mode darurat systemd dan secara manual menulis beberapa byte data ke file yang ada, btrace pada SSD dengan btrfs melaporkan 3 megabyte data nyata yang ditulis. Sistem file yang baru dibuat dari flash drive 8 GB menulis setidaknya 264 KiB saat merekam satu byte.
Sebagai perbandingan, menulis sepasang byte ke file di ext4 diakhiri dengan menulis 24 kilobyte data ke disk.

Pada 2017, Jayashree Mohan, Rohan Kadekodi, dan Vijay Chidambaram melakukan studi tentang amplifikasi tulis dari sistem file yang berbeda , hasil mereka untuk btrfs dan ext4 ketika menulis 4 KB mirip dengan milikku.



Kesimpulan dan kesimpulan

Dengan manipulasi yang dijelaskan itu ditemukan:
  1. Sering merekam status pekerjaan untuk printer dengan daemon cetak CUPS di / var / cache / gelas setiap menit. Masalahnya diperbaiki dengan membersihkan / var / spool / gelas (meskipun tidak ada pekerjaan cetak);
  2. Fakta membaca basis data setiap 30 detik oleh Extension Group Speed ‚Äã‚ÄãDial untuk Firefox;
  3. Pencatatan berkala oleh berbagai layanan pelacakan kinerja di Fedora, yang mengarah pada perekaman beberapa megabita data di btrf: pmcd.service, pmie.service, pmlogger.service;
  4. Amplifikasi besar saat merekam sejumlah kecil data saat menggunakan btrfs.

Kesimpulan: Anda tidak boleh menggunakan btrfs jika program sering menulis sejumlah kecil data (beberapa kilobyte), jika tidak maka akan berubah menjadi megabita data yang direkam. Terutama berlaku untuk komputer papan tunggal dengan OS di MicroSD.
