Cohete

Clase de cohete base 
using DG.Tweening; using GlobalEventAggregator; using UnityEngine; namespace PlayerRocket { public class Rocket : PlayerRocketBase { [SerializeField] private float pathorrectionTime = 10; private Vector3 movingUp = new Vector3(0, 1, 0); protected override void StartEventReact(ButtonStartPressed buttonStartPressed) { transform.SetParent(null); rocketState = RocketState.MOVE; transform.DORotate(Vector3.zero, pathorrectionTime); } protected override void Start() { base.Start(); EventAggregator.Invoke(new RegisterUser { playerHelper = this }); if (rocketState == RocketState.WAITFORSTART) return; RocketBehaviour(); } private void FixedUpdate() { RocketBehaviour(); } private void RocketBehaviour() { switch (rocketState) { case RocketState.WAITFORSTART: if (inputController.OnTouch && !inputController.OnDrag) rocketHolder.RotateHolder(inputController.worldMousePos); break; case RocketState.MOVE: rigidbody.AddRelativeForce(Vector3.up*(config.Speed*Time.deltaTime)); forceModel.AddModificator(); break; case RocketState.STOP: Debug.Log(" "); rigidbody.velocity = Vector3.zero; rigidbody.drag = 50; rocketState = RocketState.COMPLETESTOP; break; case RocketState.COMPLETESTOP: break; default: rocketState = RocketState.COMPLETESTOP; break; } } } }


¿Qué necesitamos para que despegue un cohete? En el espacio de juego, necesitamos un planeta condicional con el que comenzamos, un botón de inicio y un cohete. ¿Qué debe hacer un cohete?

  1. Espera el comienzo
  2. Volar
  3. Ser afectado por modificadores
  4. Para parar

Es decir, tenemos un comportamiento / estado diferente del cohete, dependiendo del estado actual, el cohete debería proporcionar un comportamiento diferente. En programación, nos enfrentamos constantemente a una situación en la que un objeto puede tener muchos comportamientos radicalmente diferentes.

Para el comportamiento complejo de objetos, es mejor usar patrones de comportamiento, por ejemplo, un patrón de estado. Para los simples, los programadores novatos a menudo usan mucho si es que lo hacen. Recomiendo usar switch y enum. En primer lugar, esta es una división más clara de la lógica en etapas específicas, gracias a esto sabremos exactamente en qué estado nos encontramos ahora, y lo que está sucediendo, hay menos oportunidades para convertir el código en un fideo de docenas de excepciones.

Cómo funciona

Primero comenzamos enumeración con los estados que necesitamos: 

  public enum RocketState { WAITFORSTART = 0, MOVE = 1, STOP = 2, COMPLETESTOP = 3, }

En la clase padre tenemos un campo: 
 protected RocketState rocketState;

Por defecto, se le asigna el primer valor. Enum establece los valores predeterminados, pero para los datos que los diseñadores de juegos pueden cambiar desde arriba o configurar, configuro manualmente los valores, ¿para qué? Para poder agregar otro valor al inam en cualquier lugar y no violar los datos almacenados. También te aconsejo que estudies enum.

Siguiente:

Definimos el comportamiento en sí en una actualización, dependiendo del valor del campo rocketState 

  private void FixedUpdate() { RocketBehaviour(); } private void RocketBehaviour() { switch (rocketState) { case RocketState.WAITFORSTART: if (inputController.OnTouch && !inputController.OnDrag) rocketHolder.RotateHolder(inputController.worldMousePos); break; case RocketState.MOVE: rigidbody.AddRelativeForce(Vector3.up*(config.Speed*Time.deltaTime)); forceModel.AddModificator(); break; case RocketState.STOP: Debug.Log(" "); rigidbody.velocity = Vector3.zero; rigidbody.drag = 50; rocketState = RocketState.COMPLETESTOP; break; case RocketState.COMPLETESTOP: break; default: rocketState = RocketState.COMPLETESTOP; break; } }

Descifraré lo que está sucediendo:

  1. Cuando esperamos, simplemente giramos el cohete hacia el cursor del mouse, estableciendo así la trayectoria inicial
  2. El segundo estado: volamos, aceleramos el cohete en la dirección correcta y actualizamos el modelo modificador para la aparición de objetos que afectan la trayectoria
  3. El tercer estado es cuando el equipo llega a nosotros para detenerse, aquí resolvemos todo para que el cohete se detenga y se traduzca en el estado: nos detuvimos por completo.
  4. El último estado es que no estamos haciendo nada.

La conveniencia del patrón actual: todo es muy fácil de expandir y ajustar, pero hay uno excepto el eslabón débil: es cuando podemos tener un estado que combina varios otros estados. Aquí, ya sea una bandera inam, con una complicación de procesamiento, o ya cambiar a patrones más "pesados".

Descubrimos el cohete. El siguiente paso es un objeto simple pero divertido: el botón de inicio.

Botón de inicio


Se requiere la siguiente funcionalidad de ella: hizo clic, notificó que hicieron clic en ella.

Clase de botón de inicio 
 using UnityEngine; using UnityEngine.EventSystems; public class StartButton : MonoBehaviour, IPointerDownHandler { private bool isTriggered; private void ButtonStartPressed() { if (isTriggered) return; isTriggered = true; GlobalEventAggregator.EventAggregator.Invoke(new ButtonStartPressed()); Debug.Log(""); } public void OnPointerDown(PointerEventData eventData) { ButtonStartPressed(); } } public struct ButtonStartPressed { }


Según el diseño del juego, este es un objeto 3D en el escenario, se supone que el botón está integrado en el diseño del planeta inicial. Bueno, está bien, hay un matiz: ¿cómo rastrear un clic en un objeto en una escena?

Si buscamos en Google, encontraremos un montón de métodos OnMouse, entre los cuales habrá un clic. Parecería una opción fácil, pero es muy malo, comenzando con el hecho de que a menudo funciona de manera torcida (hay muchos matices para el seguimiento de clics), "querido", y termina con el hecho de que no da esa cantidad de bollos que están en UnityEngine.EventSystems.

Al final, recomiendo usar UnityEngine.EventSystems y las interfaces IPointerDownHandler, IPointerClickHandler. En sus métodos, nos damos cuenta de la reacción al presionar, pero hay varios matices.

  1. Un EventSystem debe estar presente en la escena, este es un objeto / clase / componente de la unidad, generalmente creado cuando creamos el lienzo para la interfaz, pero también puede crearlo usted mismo.
  2. Física RayCaster debe estar presente en la cámara (esto es para 3D, para gráficos 2D hay un racaster separado)
  3. Debe haber un colisionador en la instalación

 public void OnPointerDown(PointerEventData eventData) { ButtonStartPressed(); } private void ButtonStartPressed() { if (isTriggered) return; isTriggered = true; GlobalEventAggregator.EventAggregator.Invoke(new ButtonStartPressed()); Debug.Log(""); }

¿Qué está pasando aquí?

Tenemos un campo booleano en el que rastreamos si se presiona el botón o no (esto es protección contra la presión repetida para que no tengamos un script de inicio cada vez).

A continuación, llamamos al evento: se presiona el botón, al que se suscribe la clase de cohete, y se pone el cohete en un estado de movimiento.

Saltando un poco hacia adelante, ¿por qué está aquí y allá para los eventos? Esta es una programación orientada a eventos. En primer lugar, un modelo de evento es más barato que el procesamiento continuo de datos para descubrir sus cambios. En segundo lugar, esta es la conexión más débil, no necesitamos saber en el cohete que hay un botón, que alguien lo presionó, y así sucesivamente, solo sabemos que hay un evento para comenzar, lo recibimos y estamos actuando. Además, este evento es interesante no solo para el cohete, por ejemplo, un panel con modificadores está firmado para el mismo evento, está oculto al comienzo del cohete. Además, este evento puede ser de interés para el controlador de entrada, y la entrada del usuario no puede procesarse o procesarse de manera diferente después del lanzamiento del cohete.

¿Por qué a muchos programadores no les gusta el paradigma del evento? Debido a que una gran cantidad de eventos y suscripciones a estos eventos convierten fácilmente el código en fideos, en los que no está del todo claro dónde comenzar y si terminará en algún lugar, sin mencionar el hecho de que también necesita monitorear su cancelación / suscripción y mantener vivos todos los objetos.

Y es por eso que para la implementación de eventos utilizo mi agregador de eventos, que de hecho no transmite eventos, sino contenedores de datos a través de eventos, y las clases se suscriben a los datos que les interesan. Además, el agregador mismo monitorea objetos vivos y arroja objetos muertos de los suscriptores. Gracias a la transferencia del contenedor, la inyección también es posible; puede pasar un enlace a la clase de interés para nosotros. Por el contenedor, puede rastrear fácilmente quién procesa y envía estos datos. Para la creación de prototipos es una gran cosa.

Rotación de cohetes para determinar la ruta de inicio



Según el diseño del juego, el cohete debería poder girar alrededor del planeta para determinar la trayectoria inicial, pero no más de un ángulo. La rotación se lleva a cabo con el tacto: el cohete simplemente sigue al dedo y siempre se dirige al lugar donde tocamos la pantalla. Por cierto, es solo el prototipo que hizo posible determinar que este es un punto débil y que hay muchos episodios asociados con la administración que bordean esta funcionalidad.

Pero en orden:

  1. Necesitamos que el cohete gire en relación con el planeta en la dirección de la carretilla
  2. Necesitamos sujetar el ángulo de rotación

En cuanto a la rotación relativa al planeta: puede girar astutamente alrededor del eje y calcular el eje de rotación, o simplemente puede crear un objeto con un maniquí centrado dentro del planeta, mover el cohete allí y rotar silenciosamente el maniquí alrededor del eje Z, el maniquí tendrá una clase que determinará el comportamiento del objeto. El cohete girará con él. El objeto que llamé RocketHolder. Lo descubrimos.

Ahora sobre las restricciones para girar y girar en la dirección de la carretilla:

clase RocketHolder 
 using UnityEngine; public class RocketHolder : MonoBehaviour { [SerializeField] private float clampAngle = 45; private void Awake() { GlobalEventAggregator.EventAggregator.AddListener(this, (InjectEvent<RocketHolder> obj) => obj.inject(this)); } private float ClampAngle(float angle, float from, float to) { if (angle < 0f) angle = 360 + angle; if (angle > 180f) return Mathf.Max(angle, 360 + from); return Mathf.Min(angle, to); } private Vector3 ClampRotationVectorZ (Vector3 rotation ) { return new Vector3(rotation.x, rotation.y, ClampAngle(rotation.z, -clampAngle, clampAngle)); } public void RotateHolder(Vector3 targetPosition) { var diff = targetPosition - transform.position; diff.Normalize(); float rot_z = Mathf.Atan2(diff.y, diff.x) * Mathf.Rad2Deg; transform.rotation = Quaternion.Euler(0f, 0f, rot_z - 90); transform.eulerAngles = ClampRotationVectorZ(transform.rotation.eulerAngles); } }


A pesar de que el juego es en teoría 3D, pero toda la lógica y el juego son en realidad 2D. Y solo necesitamos apretar el cohete alrededor del eje Z en la dirección del lugar de presión. Al final del método, fijamos el grado de rotación por el valor especificado en el inspector. Controlador de entrada


Una de las clases más importantes, es él quien recopila y procesa el comportamiento del usuario. Al presionar teclas de acceso rápido, botones de gamepad, teclados, etc. Tengo una entrada bastante simple en el prototipo, de hecho, solo necesita saber 3 cosas:

  1. ¿Hay un clic y sus coordenadas?
  2. ¿Hay un deslizamiento vertical y cuánto deslizar?
  3. ¿Opero con interfaz / modificadores?

Todo está en la frente y sin problemas, lo interesante puede ser la implementación primitiva de propiedad reactiva: cuando recién comenzaba a programar, siempre fue interesante cómo descubrir que los datos han cambiado, sin una ventilación constante de los datos. Bueno, ya está.

Se ve así:

clase ReactiveValue 
 public class ReactiveValue<T> where T: struct { private T currentState; public Action<T> OnChange; public T CurrentValue { get => currentState; set { if (value.Equals(currentState)) return; else { currentState = value; OnChange?.Invoke(currentState); } } } }


Nos suscribimos a OnChange y nos contraemos si solo ha cambiado el valor.

Con respecto a la creación de prototipos y la arquitectura: los consejos son los mismos, propiedades y métodos públicos, todos los datos solo deben cambiarse localmente. Cualquier procesamiento y cálculo: sumar de acuerdo con métodos separados. Como resultado, siempre puede cambiar la implementación / los cálculos, y esto no afectará a los usuarios externos de la clase.
