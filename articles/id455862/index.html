API DATA untuk Amazon Aurora Serverless


Bagaimana cara menghubungkan ke database tradisional bekerja? Anda membuka koneksi, menggunakannya untuk memproses satu atau lebih query SQL atau pernyataan lain, dan kemudian menutup koneksi. Anda mungkin menggunakan perpustakaan klien khusus untuk sistem operasi, bahasa pemrograman, dan basis data Anda. Pada titik tertentu, Anda menyadari bahwa membuat koneksi memerlukan banyak waktu dan mengambil memori pada mesin basis data.

API DATA menangani pengelolaan dan penskalaan koneksi basis data jangka panjang dan mengembalikan data dalam bentuk JSON untuk analisis yang mudah. Semua traffic adalah crypto dan terjadi melalui HTTPS.

Jadi apa itu DATA API?

ExecuteStatement - menjalankan satu query SQL.
BatchExecuteStatement - mengeksekusi satu query SQL untuk sebuah array data.
BeginTransaction - mulai transaksi dan kembalikan pengenal transaksi.
CommitTransaction - selesaikan transaksi dan komit operasi SQL yang dilakukan di dalamnya.
RollbackTransaction - gulung balik transaksi.

Satu-satunya batasan adalah bahwa setiap fungsi harus selesai dalam satu menit dan dapat mengembalikan hingga 1 megabyte data.

Amazon Kinesis Data Analytics sekarang memungkinkan Anda untuk menetapkan tag sumber daya AWS untuk aplikasi real-time Anda


Dengan meningkatnya aliran data, kebutuhan untuk memproses dan menganalisis data ini dengan cara khusus meningkat. Amazon Kinesis Data Analytics adalah salah satu layanan ini yang memberi Anda kesempatan, dengan menulis SQL atau kode Java (menggunakan Apache Flink), untuk mulai mengumpulkan data dari berbagai sumber, serta memproses dan menganalisisnya secara real time. Sekarang, setelah menerima gambaran data Anda yang lebih lengkap dan multidimensi, Anda dapat dengan mudah mengekstraksi informasi berguna yang baru, serta dengan cepat menanggapi kebutuhan bisnis dan pelanggan.

Dengan semakin banyaknya aplikasi semacam itu, kebutuhan untuk mengelolanya secara kompeten semakin meningkat. Amazon baru-baru ini mengumumkan bahwa sekarang mungkin untuk menambahkan tag sumber daya ke aplikasi Anda di Amazon Kinesis Data Analytics, seperti sumber daya AWS lainnya. Tag adalah pasangan nilai kunci, di mana nilai adalah bagian opsional. Menggunakan tag adalah cara sederhana namun efektif untuk mengelola sumber daya AWS dan mengatur data. Contoh kemungkinan tag:

Lingkungan: Pementasan
Aplikasi: Nama aplikasi
Proyek: Nama proyek

Dengan tag, Anda dapat menambahkan lebih banyak konteks ke aplikasi. Dalam kasus aplikasi Amazon Kinesis Data Analytics, tag dapat digunakan:
Untuk menentukan tagihan untuk aplikasi Analisis Data Amazon Kinesis tertentu.
Untuk mengontrol akses ke sumber daya aplikasi.
Untuk tujuan yang ditentukan oleh pengguna sendiri - Anda dapat menentukan fungsionalitas aplikasi berdasarkan keberadaan tag pengguna.

Amazon S3 Path Deprecation Plan - Sisa Kisah


Di salah satu layanan S3 paling populer dari Amazon Web Services (AWS), cara untuk mendapatkan file dengan menentukan path-style akan segera menjadi usang, misalnya, s3.amazonaws.com/usmanovbf/docs/Bulat_Usmanov_CV.pdf atau dengan wilayah - s3-us-east-2.amazonaws.com/usmanovbf/docs/Bulat_Usmanov_CV.pdf , di mana usmanovbf adalah bucket, dan /docs/Bulat_Usmanov_CV.pdf adalah kunci untuk file pada bucket ini.

Preferensi akan diberikan kepada metode lain - menggunakan host virtual (gaya virtual-host), misalnya, usmanovbf.s3.amazonaws.com/docs/Bulat_Usmanov_CV.pdf atau juga dengan wilayah - usmanovbf.s3-us-east-2.amazonaws. com / docs / Bulat_Usmanov_CV.pdf

Transisi ini diharapkan akan dilakukan pada 30 September 2020. Karena keterbukaan para insinyur AWS kepada komunitas, semua tautan gaya jalur yang dibuat hingga dan termasuk tanggal ini tidak akan dihapus. Artinya, mereka akan tetap beroperasi penuh, tepat setelah tanggal ini Anda hanya dapat menggunakan metode kedua - gaya virtual-host.

Perubahan ini ditentukan oleh dua alasan utama:
1. Model yang lebih terpusat, yang mengakses file menggunakan path-style, membatasi penskalaan yang efektif, karena semua lalu lintas sampai pada set kecil titik input: sub-domain dari tingkat ketiga s3, s3-us-east-2, s3-us-east -1 dan seterusnya. Untuk alasan yang sama, menjadi tidak nyaman untuk bekerja dengan resolusi nama DNS, keamanan, dan refleksi dari serangan DDOS.
Gaya virtual-host memungkinkan Anda untuk lebih fleksibel mengontrol aliran data karena mengikat ke subdomain-ember yang terpisah dan ke suatu wilayah, misalnya, usmanovbf dan s3-us-east-2 seperti pada contoh di atas.

2. Insinyur AWS tidak tinggal diam, dan untuk memperkenalkan fungsionalitas baru dan mengabaikan yang lama, mereka harus beralih ke gaya yang dihosting secara virtual. Sebagai contoh, direncanakan untuk meninggalkan metode enkripsi yang lama.
